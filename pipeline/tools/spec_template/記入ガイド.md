# Sheet1 入力 CSV — 記入ガイド（非エンジニア向け）

`sheet1_input.csv` は **人間が唯一手で書く入力ファイル** です。
Excel / Google スプレッドシートで開いて記入 → CSV で保存してください。

> ⚠️ **文字コードは必ず UTF-8 で保存すること**。ANSI/Shift-JIS 等で保存すると、
> 日本語が文字化けではなく文字通り `?` に置き換わり、**保存した瞬間に元データが失われる**
> （後から復元不可能）。保存後は一度開き直して日本語が正しく表示されるか確認する。
> エディタの文字コード表示（VS Code なら右下ステータスバー）が UTF-8 になっているか
> 入力開始前に確認しておくと安全。

## 手順（全体像）

```
1. sheet1_input.csv を記入（このガイドの下記ルール参照）
2. python3 tools/raw_to_spec.py --input sheet1_input.csv --facility {施設} --flow {フロー} \
       --outdir output/scenarios/{施設}_{フロー}
   → 9 枚のシート CSV が自動生成される（spec_{施設}_{フロー}_*.csv）
3. 生成されたシートを確認・記入:
   - Sheet1_input（spec_*）: block_type(auto) が UNKNOWN(🔴) の行を手で分類・修正
   - Sheet2_flow        : ルート（分岐）ごとに通るステップを ✓ で調整（※次項の注意を必読）
   - Sheet_Termination  : 終話の TTS 文言 / status / SMSフラグ（1=送る, -2=送らない）
   - Sheet_Script       : カスタムスクリプトを使う場合のみ
   - Sheet_Settings     : officeId・電話番号・受付時間
4. python3 tools/csv_to_yaml.py --input spec_{施設}_{フロー}_Sheet1_input.csv \
       --sheet-termination <Sheet_Termination> --sheet-settings <Sheet_Settings> \
       --facility {施設} --flow {フロー} --outdir output/scenarios/{施設}_{フロー}
   → 設計書 YAML（qa_validator 検証つき）
5. /sparring-intake で壁打ち → orchestrator ビルドへ
```

> ⚠️ **`csv_to_yaml.py --input` に渡すのは手書きの `sheet1_input.csv` ではなく、
> 手順2で `raw_to_spec.py` が生成した `spec_{施設}_{フロー}_Sheet1_input.csv`（block_type(auto) 等の
> 列が追加されたもの）。** 手書きファイルをそのまま渡すと `block_type(auto)` 列が無いため
> 全ステップが `hearing`/`voice_only` に丸められ、opening・termination・intent・patient_name 等の
> 区別が失われる（見た目はエラーなく生成されるため気づきにくい）。

## Sheet2_flow を使うかどうかの判断（重要）

`raw_to_spec.py` の Sheet2_flow 自動生成は、**choices を持つ全ステップの直積**でルート列を作る
（例: 4choice の用件確認 × 3choice の受診歴確認 × 2choice の薬処方確認 = 24〜48 列）。
これは「予約/変更/キャンセル」のような**独立した少数の大分岐**にしか向かない。

シナリオの途中に「はい/いいえで次の案内を出すか決めるだけ」のような**局所的な小分岐**が
複数あると、Sheet2 は現実的に編集不能な列数になる。その場合は：

- Sheet2_flow は使わず（`csv_to_yaml.py` に `--sheet2` を渡さない）、
- `csv_to_yaml.py` が出す素直な直線フローの YAML に対して、
  `context_match_router`（`reference_module: <保存済み変数名>` + `conditions`）を使い、
  保存済みの値（例: 用件確認で保存した `classification`）を後段で読んで分岐を手で組み立てる。

大分岐（用件確認の4択など）はそのまま `type: intent` の `options` + 後続の
`context_match_router` で分岐させれば、Sheet2 なしでも矛盾なく組める。

## Sheet1 の列ルール

| 列 | 必須 | 書き方 |
|---|---|---|
| **聴取項目名** | ✅ | 聞きたい項目の名前。「用件確認」「患者名」「生年月日」など。完全一致でなくてよい（辞書が自動判別。53 種対応） |
| **TTS文言** | ✅ | ボットが読み上げる文章。`{tts_g:...}` は自動で付くので書かなくてよい |
| **choices** | 分岐がある場合 | 選択肢を `\|` 区切りで。例: `予約\|変更\|キャンセル`。複数 choices があると自動でルート列が作られる |
| **retry回数** | 任意 | 聞き取れなかった時の再質問回数。空欄なら 3 |
| **retry後遷移先** | 任意 | `終話`（あきらめて切る）/ `次へ`（次の質問へ進む）/ 具体的なステップ名 |
| **reconfirm** | 任意 | `あり` = 復唱確認する（生年月日など重要項目向け） |
| **amivoice_words** | 任意 | 音声認識に追加登録したい単語（通常は空欄で OK。keyword_master が自動適用） |

## よく使う聴取項目名（辞書に登録済みの例）

| 書く名前 | 自動判別されるブロック |
|---|---|
| 冒頭 / オープニング | opening（アナウンス） |
| 用件確認 / 用件 | intent（用件分類・choices 必須） |
| 患者名 / 氏名 / お名前 | patient_name（個人情報スロット） |
| 生年月日 | dob |
| 電話番号 / 連絡先 | phone（常に DTMF） |
| 診察券番号 | card_number（常に DTMF） |
| 診療科 | clinical_department |
| 代表案内 | termination（status=2 / SMS なしで終話） |
| その他問い合わせ | hearing（自由テキスト） |

> 一覧にない項目名を書くと 🔴 未知 と表示されます。その場合は
> `tools/normalize_dictionary.json` への追加をオーナーに依頼してください。

## Sheet2 のセルの書き方（raw_to_spec 実行後）

| セルの値 | 意味 |
|---|---|
| `✓` | このルートに含む（標準名のまま） |
| choice 値（例: `予約`） | intent 行のみ: この値でこのルートに入る |
| カスタム名（例: `診療科_予約`） | ルート専用の別モジュールとして分ける |
| 空欄 | このルートには含まない |
| `custom_scripts1`〜`3`（block_type 列） | Sheet_Script の 参照ID 1〜3 のスクリプトを挿入 |
| `Status_Sms`（block_type 列） | 途中で終話（Sheet_Termination の名前をセルに書く） |

## 診療科リスト / FAQ リスト（任意シート・LLM 不使用）

`--init` で `sheet_department_*.csv` / `sheet_faq_*.csv` も配置されます（不要なら削除）。
記入したら `csv_to_yaml.py` に `--sheet-department` / `--sheet-faq` で渡すと、
設計書 YAML の `script_blocks`（認定部品 department_classifier / faq_matcher の spec データ）になります。

**sheet_department**: `診療科名(正式)` / `類義語(|区切り)` / `読み`
**sheet_faq**: `質問(正式)` / `回答TTS` / `キーワード(|区切り)`

> ⚠️ リストの内容（spec）を新規作成・変更した場合は P6 受入テストが必要です
> （部品のロジック＝engine は認定済みのため、データ部分の受入のみ）。

## SMSフラグの値（Sheet_Termination）

| 値 | 意味 |
|---|---|
| `1` | 完了 SMS を送る |
| `-2` | SMS を送らない（**-1 は使わない**） |

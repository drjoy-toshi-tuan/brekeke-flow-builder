# bivr_patches

`.bivr` 完成品に対する **軽量・単発の後処理パッチ** を置く場所です。

## 何を置くか

- すでに納品 / 検証済みの `.bivr` に対して、設計意図を変えない数値整形・フィールド一括差し替えなどの軽微な修正
- 例: AMI Voice / DTMF Timeout を 30000 に統一、retry_count 一括変更、特定 contextName のリネーム

## 何を置かないか

- 設計書 (`docs/designs/`) を書き換える必要がある修正 → Pattern 2 (orchestrator) で再生成
- 新ブロック追加・モジュール構成変更 → Pattern 2
- 複数施設にまたがる定常運用ロジック → `scripts/` 側に組み込む

## 規約

### ファイル名
```
<目的>_<施設名>_<YYYYMMDD>.py
```
例: `normalize_stt_timeout_iseikai_20260513.py`

### ファイル先頭 docstring に必須項目
```python
"""
目的: 何を変えるか (1〜2行)
理由: なぜ Pattern 2 ではなくこちらで処理したか
対象: 施設名 / .bivr ファイル
ルール: 変換ロジックの平文記述
結果: 実差分件数・no-op 件数・据置件数 (実行後に追記)
"""
```

### 入出力
- 入力 `.bivr` は **そのまま温存**。上書き禁止
- 出力は別名で吐く (例: `<元>_<目的>.bivr`)
- 検証ステップ (パッチ後 .bivr を再スキャンして条件確認) をスクリプト末尾に置く

### 実行
- パイプライン (orchestrator) には載せない
- 単発の `python tools/bivr_patches/<script>.py` で完結
- 引数を取らず、パスをスクリプト内にハードコードして良い (1 施設 1 回のための監査用記録)

### 残し方
- スクリプト自体が「いつ・どこに・何を適用したか」の監査痕跡
- 後で別施設に同じ整形が必要になったら、ファイルをコピーしてリネーム + パス書き換えで対応
- 共通化したくなったら別途リファクタを検討するが、まずは重複を許容する

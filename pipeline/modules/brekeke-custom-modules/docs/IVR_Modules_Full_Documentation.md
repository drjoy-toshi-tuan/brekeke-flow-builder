# 📚 IVRモジュール技術ドキュメント

本ドキュメントは、IVRシステムで使用される **4つの独立したモジュール** について説明します。各モジュールはそれぞれ独立した機能を持ち、単体で使用されます。

---

## 🗂️ モジュール一覧

| # | モジュール名 | 主な機能 | 複雑度 |
|:-:|---|---|:-:|
| 1 | **Module Result Binder** | 他のモジュールの結果を取得し、変数として格納 + DB 保存(オプション) | ⭐ |
| 2 | **Phone Normalization** | 電話番号を正規化・フォーマット・確認再生・DB 保存 | ⭐⭐ |

| 4 | **Date Classifier** | 発話から日付を抽出し、休診日判定 | ⭐⭐⭐⭐ |

# 📘 Module Result Binder

## 1. 🎯 概要(シンプルな説明)

**Module Result Binder** は、モジュール間でデータを受け渡すための「ブリッジ」モジュールです。主な機能は以下の3点です。

- 他のモジュールの**出力結果を取得**する
- 取得した結果を**指定した変数名(オブジェクト)に格納**し、以降のモジュールで `<%変数名%>` として参照可能にする
- 取得した結果を**DBコンテキストフィールドとして保存**する(オプション)

> 💡 **簡単な例:**
> モジュール A(STT)が「田中太郎」という応答を取得した場合、Module Result Binder はこの結果を取得して `customer_name` という変数に格納します。その後、音声再生モジュールで *「`<%customer_name%>` 様、ご確認ありがとうございます」* のように使用できます。さらに `contextName` と `contextDisplayType` を設定すれば、患者情報として DB にも保存できます。

---

## 2. 📋 仕様(シンプル)

| 項目 | 内容 |
|---|---|
| **モジュール名** | Module Result Binder |
| **主な機能** | 他モジュールの結果を取得し、変数として受け渡す + DB コンテキスト保存(オプション) |
| **入力** | 参照元モジュール名(`module`)、格納先変数名(`variable`)、DB保存設定(`contextName`、`contextDisplayType`) |
| **出力(setResult)** | 結果文字列、または `NO_RESULT` / `time_out` / `error` |
| **副作用** | `$runner.setObject()` により変数をコンテキストに格納<br>`save2db` により DB へコンテキストフィールド保存(条件付き) |

### 🔁 処理フロー

```
┌─────────────────────────────────────────────────────┐
│  開始                                                │
│    ↓                                                 │
│  プロパティ読込:                                      │
│    module, variable,                                 │
│    contextName, contextDisplayType                   │
│    ↓                                                 │
│  module が空? → YES → setResult("NO_RESULT")        │
│    ↓ NO                                              │
│  参照元モジュールの結果を取得                          │
│    ↓                                                 │
│  結果が "time_out" または "error"?                   │
│    → YES → setResult(その値)                         │
│    ↓ NO                                              │
│  結果が空? → YES → setResult("NO_RESULT")           │
│    ↓ NO                                              │
│  variable が設定されている?                           │
│    → YES → setObject(variable, 結果)                 │
│    → NO  → 格納スキップ                              │
│    ↓                                                 │
│  contextName と contextDisplayType の両方が設定?     │
│    → YES → save2db で DB 保存                       │
│           + setObject(contextName, 結果)             │
│    → NO  → DB保存スキップ                            │
│    ↓                                                 │
│  setResult(結果)                                    │
└─────────────────────────────────────────────────────┘
```

### 📤 setResult の戻り値

| 値 | 意味 |
|---|---|
| `NO_RESULT` | module プロパティが未設定、または参照元モジュールの出力が空 |
| `time_out` | 参照元モジュールがタイムアウト |
| `error` | 参照元モジュールがエラー |
| *(結果文字列)* | 参照元モジュールの正常な結果 |

---

## 3. 🛠️ 使用方法(詳細)

### 3.1. プロパティ設定

**プロパティ設定** タブで以下を設定します。

| プロパティ名(日本語) | プロパティ名(コード) | 必須 | 説明 | 例 |
|---|---|:---:|---|---|
| **参照元モジュール名** | `module` | ✅ **必須** | 参照元モジュール名。フロー内で対象モジュールに付けた Node Name と一致させる必要があります。 | `stt_name`, `dtmf_birthday` |
| **変数名** | `variable` | ⚪ 任意 | 格納先変数名。未設定の場合は変数への格納はスキップされ、`setResult` のみ実行されます。 | `customer_name`, `birth_date` |
| **コンテキスト名** | `contextName` | ⚪ 任意 | DB に保存する際のコンテキストフィールド名。`contextDisplayType` と**両方**設定された場合のみ DB 保存が実行されます。 | `patientName`, `phoneNumber` |
| **コンテキスト表示タイプ** | `contextDisplayType` | ⚪ 任意 | DB 上での表示タイプ。`contextName` と**両方**設定された場合のみ DB 保存が実行されます。 | `Text`, `PHONE_NUMBER`, `Date` |

### 3.2. 格納した結果の使用方法

Module Result Binder 実行後、以下の方法で結果を使用できます。

**方法1: 変数として使用(`variable` を設定した場合)**
後続モジュールの prompt、ガイダンス、プロパティなどで使用可能:
```
こんにちは、<%customer_name%> 様
```

**方法2: setResult でフロー分岐**
値に応じてフローを分岐:
- `result = "NO_RESULT"` → 「再質問」ブランチへ
- `result = "time_out"` → 「リトライ」ブランチへ
- 正常な値 → 「確認」ブランチへ

### 3.3. 使用例

#### シナリオ1: 顧客名を変数として保存(DB保存なし)

STT モジュールから顧客名を取得し、複数箇所で再利用する。

| ステップ | モジュール | プロパティ設定 |
|---|---|---|
| 1 | STT(Node Name = `stt_name`) | *(音声から顧客名を取得)* |
| 2 | Module Result Binder | `module = stt_name`<br>`variable = customer_name` |
| 3 | 音声再生 | `prompt = <%customer_name%> 様、ありがとうございます` |
| 4 | 音声再生(フロー末尾) | `prompt = <%customer_name%> 様、またお電話お待ちしております` |

#### シナリオ2: 顧客名を変数+DBの両方に保存

STT モジュールから取得した顧客名を、後続処理での参照用に変数として保存しつつ、患者情報として DB にも記録する。

| ステップ | モジュール | プロパティ設定 |
|---|---|---|
| 1 | STT(Node Name = `stt_name`) | *(音声から顧客名を取得)* |
| 2 | Module Result Binder | `module = stt_name`<br>`variable = customer_name`<br>`contextName = patientName`<br>`contextDisplayType = Text` |
| 3 | 音声再生 | `prompt = <%customer_name%> 様、ご確認ありがとうございます` |

**結果:**
- 変数 `<%customer_name%>` で参照可能
- DB の `patientName` フィールドに保存(displayType: Text)

### 3.4. 重要な注意事項

> ⚠️ **結果は `.trim()` される** — 前後の空白は自動的に除去されます。
>
> ⚠️ **変数名は snake_case 推奨**(例: `customer_name`, `birth_date`)。フロー全体で統一することで、混乱を防げます。
>
> ⚠️ **参照元モジュールが存在しない場合**、`getModuleResult` は null を返し、結果は `NO_RESULT` になります。
>
> ⚠️ **`time_out` と `error` はそのまま通過** — 変数には格納されず、DB にも保存されません。これらのケースは別のフローブランチで処理する必要があります。
>
> ⚠️ **DB 保存は `contextName` と `contextDisplayType` の両方が必要** — 片方だけ設定しても DB 保存はスキップされます(ログにスキップ理由を出力)。
>
> ⚠️ **DB 保存時の自動変数セット** — DB 保存が成功すると、`contextName` と同名の変数が自動的にセットされます(例: `contextName = patientName` なら `<%patientName%>` でも参照可能)。
>
> ⚠️ **DB 保存は `time_out` / `error` / 空結果の場合は実行されない** — 正常な結果のみ DB に保存されます。

# 📞 Phone Normalization

## 1. 🎯 概要(シンプルな説明)

**Phone Normalization** は、**日本の電話番号**を処理するモジュールで、主に2つの機能を持ちます。

### 🔹 CASE A: 他モジュールの結果がある場合(STT/DTMF)
顧客が音声またはキー入力で電話番号を入力した場合、以下を実行します:
1. 番号を**正規化**(`+81` の除去、数字のみ抽出)
2. **妥当性チェック**(携帯11桁、固定10桁)
3. **日本式ハイフンを挿入**(例: `090-1234-5678`、`03-1234-5678`)
4. 番号を**再生**して顧客に確認(prompt 内の `#data#` を置換)

### 🔹 CASE B: 参照元モジュールがない場合(着信番号処理)
通話の着信番号を取得し、以下を実行します:
1. **データベースに保存**(数字のみの形式で)
2. 顧客に**再生**して確認(全桁読み上げ、または下4桁のみ読み上げ)

> 💡 **例:**
> - 入力: `+819012345678` → 出力: `090-1234-5678`
> - 入力: `0312345678` → 出力: `03-1234-5678`
> - 入力: `0422123456` → 出力: `0422-12-3456`(武蔵野エリア)

---

## 2. 📋 仕様(シンプル)

| 項目 | 内容 |
|---|---|
| **モジュール名** | Phone Normalization |
| **機能** | 日本の電話番号の正規化・フォーマット・確認再生 |
| **対応範囲** | 携帯電話(090/080/070/060/050)、固定電話(市外局番 2〜5 桁) |
| **参照データ** | 総務省の番号計画に基づく市外局番リスト(AREA_2、AREA_3、AREA_4、AREA_5) |

### 📑 フォーマットルール

| 種別 | 桁数 | フォーマット例 |
|---|:---:|---|
| 携帯電話 | 11 | `090-1234-5678` |
| 固定(2桁市外局番: 03, 06) | 10 | `03-1234-5678` |
| 固定(3桁市外局番: 011, 052...) | 10 | `052-123-4567` |
| 固定(4桁市外局番: 0422...) | 10 | `0422-12-3456` |
| 固定(5桁市外局番: 04992...) | 10 | `04992-1-2345` |

### 🔁 処理フロー

```
┌──────────────────────────────────────────────────┐
│  プロパティ読込: prompt, module,                  │
│    phoneReadingMode, saveAdditionalPhoneNumber2DB │
│    ↓                                             │
│  module が設定されている?                          │
│  ├─ YES (CASE A: STT/DTMF から)                  │
│  │    ↓                                          │
│  │  モジュールから番号取得、数字以外を除去         │
│  │    ↓                                          │
│  │  チェック: 携帯11桁? 固定10桁?                 │
│  │    ├─ 空    → setResult("NO_RESULT")         │
│  │    ├─ 不正  → setResult("INVALID")           │
│  │    └─ 正常  → フォーマット + prompt 再生      │
│  │               saveAdditionalPhoneNumber2DB?   │
│  │                 → yes なら DB 保存             │
│  │               setResult(フォーマット済み番号)  │
│  │                                               │
│  └─ NO  (CASE B: 着信番号処理)                   │
│       ↓                                          │
│     着信番号を取得(.incomingPhone or IVR)       │
│       ↓                                          │
│     saveAdditionalPhoneNumber2DB?                │
│       → yes なら DB 保存                          │
│       ↓                                          │
│     phoneReadingMode に基づきフォーマット         │
│       (全桁 または 下4桁)                         │
│       ↓                                          │
│     変数 <%incoming_phone%> 設定 + prompt 再生   │
│       ↓                                          │
│     setResult("INCOMING_PROCESSED")             │
└──────────────────────────────────────────────────┘
```

### 📤 setResult の戻り値

| 値 | 意味 |
|---|---|
| *(フォーマット済み番号)* | 正規化成功(CASE A) |
| `INVALID` | 不正な番号(10/11桁でない、または先頭が不正) |
| `NO_RESULT` | 参照元モジュールが番号を返さなかった |
| `INCOMING_PROCESSED` | 着信番号の処理完了(CASE B) |

---

## 3. 🛠️ 使用方法(詳細)

### 3.1. プロパティ設定

| プロパティ名(日本語) | プロパティ名(コード) | 必須 | 説明 | 例 |
|---|---|:---:|---|---|
| **参照元モジュール名** | `module` | ⚪ 任意 | 参照元 STT/DTMF モジュール名。設定あり → CASE A、未設定 → CASE B。 | `stt_phone`, `dtmf_phone` |
| **ガイダンス** | `prompt` | ✅ 必須 | 確認用ガイダンス。`#data#`(CASE A)または `<%incoming_phone%>`(CASE B)をプレースホルダーとして使用。 | `お電話番号は #data# でよろしいでしょうか?` |
| **読み上げモード** | `phoneReadingMode` | ⚪ 任意 | 読み上げモード(CASE B のみ)。`全桁` = 全桁読み上げ、`下4桁` = 下4桁のみ(1文字ずつ)。デフォルト `全桁`。 | `下4桁` |
| **コンテキスト保存** | `saveAdditionalPhoneNumber2DB` | ⚪ 任意 | `yes` = `additionalPhoneNumber` として DB 保存、`no`(デフォルト)= DB 保存しない。CASE A / CASE B の両方に適用。 | `yes` |
| *(外部変数)* | `.incomingPhone` | ⚪ 任意 | 外部から渡される着信番号(CASE B)。未設定の場合は `$ivr.getOtherNumber()` から取得。 | `+819012345678` |

### 3.2. 使用例

#### 🎬 シナリオ1: 顧客が発話した番号の確認

**状況:** 顧客が電話番号を発話し、確認および DB 保存が必要な場合。

| ステップ | モジュール | プロパティ設定 |
|---|---|---|
| 1 | STT(Node Name = `stt_phone`) | 音声から電話番号を認識 |
| 2 | Phone Normalization | `module = stt_phone`<br>`prompt = お客様のお電話番号は #data# でよろしいでしょうか?`<br>`saveAdditionalPhoneNumber2DB = yes` |

**結果:**
- 顧客発話: 「ぜろきゅう ぜろ いち に さん よん...」
- STT 出力: `09012345678`
- IVR 再生: *「お客様のお電話番号は 090-1234-5678 でよろしいでしょうか?」*
- DB 保存: `additionalPhoneNumber = 09012345678`(数字のみ)
- `setResult`: `090-1234-5678`

#### 🎬 シナリオ2: 着信番号の確認(全桁読み上げ + DB 保存)

| ステップ | モジュール | プロパティ設定 |
|---|---|---|
| 1 | Phone Normalization | `module =` *(未設定)*<br>`prompt = 発信番号 <%incoming_phone%> からのお電話でよろしいでしょうか?`<br>`phoneReadingMode = 全桁`<br>`saveAdditionalPhoneNumber2DB = yes` |

**結果:**
- IVR 再生: *「発信番号 090-1234-5678 からのお電話で...」*
- DB 保存: `additionalPhoneNumber = 09012345678`(数字のみ)

#### 🎬 シナリオ3: 下4桁のみ確認(セキュリティ用途)

| ステップ | モジュール | プロパティ設定 |
|---|---|---|
| 1 | Phone Normalization | `module =` *(未設定)*<br>`prompt = 下4桁が <%incoming_phone%> のお電話でよろしいでしょうか?`<br>`phoneReadingMode = 下4桁` |

**結果:**
- 着信番号: `09012345678`
- 変数 `incoming_phone` = `"5 6 7 8"`(AmiVoice が1文字ずつ読むようスペース区切り)
- IVR 再生: *「下4桁が 5 6 7 8 のお電話で...」*

### 3.3. 特殊な処理

> 🔍 **携帯電話の識別:** 先頭 `090`, `080`, `070`, `060`, `050` は携帯(11桁)として扱われます。
>
> 🔍 **固定電話の市外局番識別:** 5桁 → 4桁 → 3桁 → 2桁 の順にチェック。例: `04992` は AREA_5 に一致 → `04992-X-XXXX` 形式。
>
> 🔍 **市外局番が不一致の場合:** どの AREA にも一致しない場合、デフォルトで `XXX-XXX-XXXX` 形式になります。
>
> 🔍 **「下4桁」モード:** 音声エンジン(AmiVoice)が1文字ずつ読み上げるよう、**各数字の間にスペース**を挿入します。これにより「ごせんろっぴゃくななじゅうはち」のような読み方を回避します。

### 3.4. 注意事項

> ⚠️ 先頭の `+81` は自動的に `0` に変換されます(国内形式)。
>
> ⚠️ **DB 保存は `saveAdditionalPhoneNumber2DB = yes` の場合のみ実行されます** — CASE A / CASE B の両方で適用されます。保存先のコンテキスト名は `additionalPhoneNumber`、保存値は**ハイフンなしの数字のみ**の形式です。
>
> ⚠️ **CASE A では `incoming_phone` 変数は設定されません** — この変数は CASE B 専用です。
>
> ⚠️ `prompt` が空の場合、処理は実行されますが**音声再生は行われません**。

# 🎂 DOB Re-confirmation

## 1. 🎯 概要(シンプルな説明)

**DOB Re-confirmation** は、患者の**生年月日(Date of Birth)**を処理する高度なモジュールです。STT または DTMF から入力された生年月日を、**和暦・西暦・漢数字・全角数字・元号の誤認識**など多様な表現から正確に解析し、確認用に再生してデータベースに保存します。

### 🧠 多層的な解析ロジック

```
┌─────────────────────────────────────────────────┐
│  患者が生年月日を発話 / DTMF 入力                │
└─────────────────────────────────────────────────┘
             ↓
┌─────────────────────────────────────────────────┐
│  前処理: 数字正規化 + 元号エイリアス正規化         │
│  - 全角数字(１２３) → 半角(123)                  │
│  - 漢数字(三十二) → 数字(32)                     │
│  - 誤認識エイリアス(「唱和」→「昭和」など)        │
└─────────────────────────────────────────────────┘
             ↓
┌─────────────────────────────────────────────────┐
│  段階1: ローカルパーサー(コードベース解析)       │
│  - DTMF 8桁 (YYYYMMDD)                          │
│  - 和暦パターン (昭和54年3月12日)                │
│  - 西暦パターン (1979年3月12日)                  │
│  - 妥当性チェック(実在日・120歳超・未来日)       │
└─────────────────────────────────────────────────┘
             ↓
     パース成功? ── YES ──→ 結果を使用
             ↓ NO / 曖昧
┌─────────────────────────────────────────────────┐
│  段階2: OpenAI(複雑な発話を処理)                │
│  - 元号に似た誤認識語                             │
│  - 日付要素が不完全なケース                       │
└─────────────────────────────────────────────────┘
             ↓
┌─────────────────────────────────────────────────┐
│  フォーマット & DB 保存 & 再生                    │
│  - 和暦 or 西暦で読み上げ                         │
│  - patientDateOfBirth に保存                     │
│  - 会話ログ(utterance)を保存                     │
└─────────────────────────────────────────────────┘
```

> 💡 **実例:**
> - 患者発話: *「昭和54年3月12日」* → パース → `1979-03-12 00:00` → 再生 *「昭和54年3月12日でよろしいですか?」*
> - 患者発話: *「唱和ごじゅうよねん さんがつ じゅうににち」* → エイリアス正規化 → 漢数字変換 → *「昭和54年3月12日」* → パース OK
> - DTMF 入力: `19790312` → 直接 `1979-03-12 00:00` に確定(STT/LLM 不要)

| `RAW_INPUT:<値>` | 処理開始直後 | rawText を記録するログ用の中間値。すぐに上書きされる。 |

## 3. 🛠️ 使用方法(詳細)

### 3.1. プロパティ設定

| プロパティ名(日本語) | プロパティ名(コード) | 必須 | デフォルト | 説明 |
|---|---|:---:|---|---|
| **ガイダンス** | `prompt` | ✅ 必須 | *(空)* | 確認用ガイダンス。`#data#` を読み上げ値に置換。 |
| **参照元モジュール名** | `module` | ✅ 必須 | *(空)* | 参照元 STT/DTMF モジュール名 |
| **生年読み上げモード** | `dateReadingMode` | ⚪ 任意 | `自動` | `和暦` = 和暦で読み上げ、`西暦` = 西暦で読み上げ、`自動` = 入力に従って読み上げ(元号検出時は和暦、それ以外は西暦) |
| **コンテキスト保存** | `saveDOB2db` | ⚪ 任意 | `no` | `yes` = `patientDateOfBirth` として DB 保存、`no` = DB 保存しない |
| *(外部変数)* | `.openAI_generate.url` | ⚪ 任意 | *(空)* | OpenAI proxy エンドポイント URL。未設定の場合、LLM フォールバックは行われません。 |
| *(外部変数)* | `.office_id` | ⚪ 任意 | *(空)* | OpenAI リクエスト識別用の事務所 ID |

### 3.2. 入力データソース(2系統)

このモジュールは **2種類の入力** を参照します:

| 入力源 | 取得元 | 用途 |
|---|---|---|
| **nodeValue** | `$runner.getModuleResult(module)` | STT の整形済み日付(例: `1979-03-12 00:00`)または DTMF 8桁 |
| **rawText** | `$runner.getObject("raw_text")` → `$runner.get("raw_text")` → `$runner.get("GLOBAL_RAW_TEXT")` | 元の発話テキスト(例: `昭和54年3月12日`) |

> 📌 **rawText の役割:** STT が整形された日付を返しても、元の発話を見ることで**元号情報**や**STT誤認識**を判定できます。
>
> 📌 **DTMF ショートカット:** rawText が空で nodeValue が有効な日付 → DTMF 入力とみなし、STT/LLM をスキップして直接確定します。

### 3.3. 処理フローの判断分岐

```
rawText あり?
  ├─ NO + baseDate 有効 → DTMF ショートカット(直接確定)
  ├─ NO + baseDate 無効 → INVALID
  └─ YES → 音声入力フロー
           │
           ├─ コード解析: status = OK?
           │   └─ YES かつ STT 不審でない → LOCAL で確定
           │
           ├─ コード解析: status = INVALID?
           │   └─ YES かつ rawText が日付っぽくない → LOCAL で INVALID 確定
           │
           └─ それ以外(UNCERTAIN など)→ OpenAI に委譲
```

### 3.4. 特殊な判定ロジック

#### 🔍 STT 誤生成チェック(suspiciousSTT)
STT が日付形式を返したが、rawText に日付らしい要素がない場合、STT の誤認識を疑い LLM に委譲します。

例: STT 返却 = `2000-01-01 00:00`、rawText = 「はい、お願いします」 → suspiciousSTT → LLM 呼び出し

#### 🔍 日付要素の完全性チェック
年・月・日の**3要素すべて**が揃わないと OK にはなりません。1要素でも欠けていれば `UNCERTAIN` → LLM フォールバック。

例: `昭和54年3月` → 日が欠落 → UNCERTAIN → LLM へ

#### 🔍 2/3桁年の自動補完
DTMF ショートカット時、2〜3桁の年は 1900年代として補完されます。
- 2桁: `79` → `1979`
- 3桁: `079` → `1979`(下2桁を使用)

| フィールド | 値 |

| `contextName` | `patientDateOfBirth` |

### 3.7. 重要な注意事項

> ⚠️ **`prompt` に `#data#` が必要** — このプレースホルダーが解析済みの日付に置換されます。
>
> ⚠️ **rawText は変数コンテキストから取得** — 通常は前段の STT モジュールが `raw_text` 変数に保存している必要があります。
>
> ⚠️ **`dateReadingMode` のデフォルトは `自動`** — 入力内容に応じて読み上げ方式が決まります。元号が検出された入力は和暦で、それ以外は西暦で読み上げられます。ユーザーが和暦で発話すれば和暦で、西暦で発話すれば西暦で確認されます。
>
> ⚠️ **DTMF モードでは rawText が生成されない** — このため DTMF ショートカットで直接 baseDate を使用します。2/3桁の年は自動的に 1900 年代として補完されます。
>
> ⚠️ **120歳超チェック** — 実用的な生年月日として妥当な範囲を担保します。120歳を超える場合は自動的に INVALID。
>
> ⚠️ **未来日チェック** — 今日より後の日付は自動的に INVALID(元号年の誤解釈を防ぐため)。
>
> ⚠️ **LLM の出力厳守** — LLM は `yyyy-MM-dd 00:00` または `gggg年M月d日` 形式のみ返却し、それ以外は `INVALID` とするよう指示されています。
>
> ⚠️ **HTTP タイムアウト** — OpenAI 呼び出しは **10秒** タイムアウト。タイムアウト時は `TIMEOUT` を返却。
>
> ⚠️ **INVALID 時は DB 保存・再生なし** — 無効な日付は `setResult("INVALID")` のみで、DB 保存や音声再生は行われません。

# 📅 Date Classifier

## 1. 🎯 概要(シンプルな説明)

**Date Classifier** は、**顧客の発話内容を解析** して **予約希望日を特定** し、その日が **病院の休診日かどうかを判定** する、高度な日付解析モジュールです。

### 🧠 2段階の解析ロジック

```
┌─────────────────────────────────────┐
│  顧客が日付を発話/入力               │
└─────────────────────────────────────┘
             ↓
┌─────────────────────────────────────┐
│  段階1: ローカルパーサー(高速・低コスト) │
│  - 「yyyy年M月d日」パターン           │
│  - 「M月d日」「来週月曜日」             │
│  - 「来月15日」、DTMF 8桁...          │
└─────────────────────────────────────┘
             ↓
     パース可能? ──── YES → 結果を使用
             ↓ NO
┌─────────────────────────────────────┐
│  段階2: OpenAI(複雑な発話を処理)     │
│  - 「年末年始の前の金曜日」            │
│  - 曖昧な表現                        │
└─────────────────────────────────────┘
             ↓
┌─────────────────────────────────────┐
│  休診日判定                          │
│  - 土日?                            │
│  - 国民の祝日?(内閣府CSV)           │
│  - 病院独自の休診日?                  │
│  - blockDays の受付不可期間内?        │
└─────────────────────────────────────┘
             ↓
     BUSINESS_DAY / NON_BUSINESS_DAY
```

> 💡 **実例:**
> - 顧客発話: *「来週の水曜日にお願いします」* → パース → `2024-10-23` → 判定 → **BUSINESS_DAY**
> - 顧客発話: *「12月29日」* → パース → `2024-12-29` → 判定(カスタム休診日) → **NON_BUSINESS_DAY**
> - 顧客発話: *「年末年始前の最後の平日」* → ローカルでは理解不可 → OpenAI 呼び出し → パース → 判定

## 2. 📋 仕様(シンプル)

| 項目 | 内容 |
|---|---|
| **モジュール名** | Date Classifier(checkClosedDay) |
| **機能** | テキストから日付を解析し、休診日を判定 |
| **入力** | STT/DTMF モジュールの結果 |
| **出力(setResult)** | `BUSINESS_DAY`、`NON_BUSINESS_DAY`、`ERROR`、`TIMEOUT`、`LOCAL_RESULT:...`、`LLM_RESULT:...` |
| **副作用** | 変数 `checked_dates`、`closed_dates`、`available_date_full`、`available_date_short` を設定 |

### 📑 ローカルパーサーが認識する入力パターン

| パターン | 入力例 | 出力 |
|---|---|---|
| DTMF 8桁 | `20241215` | `2024-12-15` |
| yyyy-MM-dd hh:mm | `2024-12-15 09:30` | `2024-12-15` |
| yyyy/M/d | `2024/12/15` | `2024-12-15` |
| yyyy年M月d日 | `2024年12月15日` | `2024-12-15` |
| M月d日(年なし) | `12月15日` | `2024-12-15` *(今日以降)* または `2025-12-15` |
| 来月/再来月 + 日付 | `来月15日`, `再来月3日と5日` | `2024-11-15`, `2024-12-03`, `2024-12-05` |
| 今週/来週/再来週 + 曜日 | `来週月曜日`, `再来週水曜と木曜` | `2024-10-21`, `2024-10-30`, `2024-10-31` |
| 単体曜日 | `月曜日に` | `2024-10-21` *(今日以降の直近)* |

### 📑 休診日フィルタモード(dateFilterMode)

| モード | 休診日扱い |
|---|---|
| `なし` | フィルタなし(日曜も営業扱い) |
| `土日祝日` | 土曜 + 日曜 + 国民の祝日 |
| `祝日` | 国民の祝日のみ |
| `土日` | 土曜 + 日曜 |
| `日祝日` | 日曜 + 祝日(土曜は営業日) |
| `土` | 土曜のみ |
| `日` | 日曜のみ |

> ℹ️ **customHoliday CSV** に含まれる日付は `dateFilterMode` に関係なく、常に休診日として扱われます。

### 📤 setResult の戻り値

| 値 | 意味 |
|---|---|
| `BUSINESS_DAY` | すべての日が営業日 |
| `NON_BUSINESS_DAY` | 休診日を含む(`partialClosedDayMode` に依存) |
| `LOCAL_RESULT:[...]` | ローカルでパース(判定前の中間結果) |
| `LLM_RESULT:[...]` | OpenAI でパース(判定前の中間結果) |
| `TIMEOUT` | OpenAI タイムアウト |
| `ERROR` | エラー、または日付パース失敗 |

### 3.1. プロパティ設定

| プロパティ名(日本語) | プロパティ名(コード) | 必須 | デフォルト | 説明 |
|---|---|:---:|---|---|
| **参照元モジュール名** | `module` | ⚪ | `stt` | 顧客発話を含む参照元 STT/DTMF モジュール名 |
| **祝日ソース(CSVのみ)** | `holidaySource` | ⚪ | *(空)* | 国民の祝日 CSV URL(内閣府形式、Shift_JIS、`yyyy/M/d,祝日名`)。**今年以降の祝日のみ** ロードされる。未設定の場合、国民の祝日判定はスキップ。 |
| **カスタム休業日ソース(CSVのみ)** | `customHoliday` | ⚪ | *(空)* | 独自休診日 CSV URL(UTF-8、`yyyy-MM-dd,名称` または `yyyy-MM-dd` のみ) |

| **診療日カウント除外モード** | `blockDaysSkip` | ⚪ | *(空)* | blockDays のカウント方法: `なし` / `土日祝日` / `祝日` / `土日` / `日祝日` / *空 = カレンダー日数* |
| **複数日指定時、1件でも休診日ならNGにする** | `partialClosedDayMode` | ⚪ | `no` | 複数日指定時の判定: `yes` = 1日でも休診なら NON_BUSINESS_DAY、`no` = **全日休診時のみ** NON_BUSINESS_DAY |

### 3.2. `blockDays` と `blockDaysSkip` の理解

`blockDays` は **受付開始までの診療日数(リードタイム)**を指定します。
- `0` = 当日から受付可能(ブロックなし)
- `1` = 翌診療日から受付可能
- `2` = 翌々診療日から受付可能…

カウント方法は2種類あります。

#### 方法1: カレンダー日数カウント(`blockDaysSkip` が空の場合)

| `blockDays` | 今日 | 最短予約可能日 |
|---|---|---|
| `0` | 10/20 | 10/20(当日可) |
| `1` | 10/20 | 10/21(明日からOK) |
| `2` | 10/20 | 10/22(明後日からOK) |

#### 方法2: 診療日カウント(`blockDaysSkip` に値がある場合)

> 📌 **重要なロジック:** 今日が診療日(スキップ対象でない)なら、**今日も1日目としてカウント**されます。

例: `blockDaysSkip = 土日祝日` の場合:

| 今日 | `blockDays` | 動作 | 最短予約可能日 |
|---|---|---|---|
| 月曜(10/20) | `1` | 今日=1日目 → スキップ → 翌診療日 | 火曜(10/21) |
| 月曜(10/20) | `2` | 今日+火曜=2日 → 水曜OK | 水曜(10/22) |
| 金曜(10/24) | `1` | 今日=1日目 → 翌診療日(土日スキップ) | 月曜(10/27) |
| 金曜(10/24) | `2` | 金+月=2日 → 火曜OK | 火曜(10/28) |
| 土曜(10/25) | `1` | 今日は土曜=スキップ対象 → 月曜=1日目 | 火曜(10/28) |

> 🔁 顧客が **過去の日付** を指定した場合、`blockDays` の値に関係なく**常にブロック**されます。
>
> 📌 **`blockDaysSkip` の選択肢:** `なし` / `土日祝日` / `祝日` / `土日` / `日祝日` / *空 = カレンダー日数カウント*

### 3.3. 使用例

#### 🎬 シナリオ1: 週末 + 祝日休診、日数ブロックなし

**事前ノード:** STT が *「来週の月曜日にお願いします」* を取得(今日は木曜 10/17)

| プロパティ | 値 |
|---|---|
| `module` | `stt_date` |
| `holidaySource` | 祝日 CSV URL |
| `customHoliday` | 休診日 CSV URL |
| `dateFilterMode` | `土日祝日` |
| `blockDays` | `0` |
| `partialClosedDayMode` | `no` |

**結果:**
- パース: `[2024-10-21]`(翌週月曜)
- 判定: 土日/祝日/休診日ではない → `BUSINESS_DAY`
- 変数 `checked_dates` = `["2024-10-21"]`
- 変数 `closed_dates` = *""*

#### 🎬 シナリオ2: 複数日、partial モード

**事前ノード:** STT が *「来月の3日と4日」* を取得

| プロパティ | 値 |
|---|---|
| `dateFilterMode` | `土日祝日` |
| `partialClosedDayMode` | `yes` |

**結果:**
- パース: `[2024-11-03, 2024-11-04]`
- 判定: `11-03` は日曜 → 休診、`11-04` は月曜 → 営業日
- `partialClosedDayMode = yes` のため、1日でも休診で NG → `NON_BUSINESS_DAY`
- `closed_dates` = `"2024-11-03(日曜日)"`

#### 🎬 シナリオ3: 今後2診療日は受付不可

**事前ノード:** STT が *「明日お願いします」* を取得(今日は水曜 10/16)

| プロパティ | 値 |
|---|---|
| `blockDays` | `2` |
| `blockDaysSkip` | `土日祝日` |

**結果:**
- パース: `[2024-10-17]`(明日の木曜)
- 判定: 今日=1日目、明日(木)=2日目 → 受付可能日は金曜から
- 翌日(木)はリードタイム内 → `NON_BUSINESS_DAY`
- `closed_dates` = `"2024-10-17(受付不可期間)"`
- `available_date_full` = `"2024-10-18"` → `"2024年10月18日"`(自動計算された次の診療日)
- `available_date_short` = `"10月18日"`

#### 🎬 シナリオ4: 複雑な発話 — OpenAI フォールバック

**事前ノード:** STT が *「年末年始明けの最初の水曜日」* を取得

| プロパティ | 値 |
|---|---|
| `.openAI_generate.url` | OpenAI proxy URL |

**結果:**
- ローカルパース: 理解不可 → `[]`
- TODAY_ISO とルールを含む prompt で OpenAI を呼び出し
- OpenAI 返却: `["2025-01-08"]`
- 中間 setResult: `LLM_RESULT:["2025-01-08"]`
- 日付判定 → `BUSINESS_DAY`

### 3.4. CSV フォーマット参考

#### customHoliday(UTF-8)

### 3.5. 重要な注意事項

> ⚠️ **最大3日の制限:** 顧客が4日以上を指定した場合、最初の3日のみ処理されます。
>
> ⚠️ **自動的な翌年繰越:** 顧客が「3月15日」とのみ発話し、今日が 3/15 を過ぎている場合、モジュールは自動的に翌年(来年の 3/15)として扱います。
>
> ⚠️ **ローカルで1件のみパース + 数字残存 → LLM フォールバック:** テキストに複数の数字が含まれるが、ローカルでは1件しかパースできない場合、他の日付が見落とされている可能性があるため、LLM に委ねられます。
>
> ⚠️ **CSV エンコーディングが重要:** holidaySource は **必ず Shift_JIS**(元の内閣府ファイルが Shift_JIS のため)。customHoliday は UTF-8。
>
> ⚠️ **実行後に設定される変数:**
> - `checked_dates`: パース済み日付の JSON 配列(例: `["2024-10-21","2024-10-22"]`)
> - `closed_dates`: 休診日と理由の文字列(例: `"2024-11-03(日曜日), 2024-11-04(祝日(文化の日))"`)
> - `available_date_full`: 受付可能な最初の診療日(フル表記、例: `"2026-04-30"` → `"2026年4月30日"`)
> - `available_date_short`: 受付可能な最初の診療日(短縮表記、例: `"4月30日"`)
>
> 📌 **`available_date_full` / `available_date_short` の用途:** ユーザーが指定した日が休診日だった場合、「最も早く受付可能な日はXです」と案内するために使用します。`blockDays` と `dateFilterMode` の両方を考慮した最初の診療日が自動計算されます。
>
> ⚠️ **HTTP タイムアウト:** CSV 取得は **10秒** タイムアウト。OpenAI も **10秒** タイムアウト。
>
> ⚠️ **パース失敗時:** OpenAI でも結果が得られない場合、setResult は `ERROR` になります。

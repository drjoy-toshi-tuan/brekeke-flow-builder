# 📅 Date Classifier (Clinic Day Classifier)

## 1. 🎯 概要(シンプルな説明)

**Date Classifier** は、**顧客の発話内容を解析** して **予約希望日を特定** し、その日が **病院の休診日かどうかを判定** する日付解析モジュールです。判定はすべて **ローカル(端末内)** で行われ、外部 LLM は使用しません。

### 🧠 処理ロジック

```
┌─────────────────────────────────────┐
│  顧客が日付を発話/入力               │
└─────────────────────────────────────┘
             ↓
┌─────────────────────────────────────┐
│  Step0: 「わからない」「希望なし」系  │
│    の検出 → 不明 を返す               │
└─────────────────────────────────────┘
             ↓
┌─────────────────────────────────────┐
│  Step1: ローカルパーサー(具体日)     │
│  - 「yyyy年M月d日」「M月d日」          │
│  - 「来週月曜日」「来月15日」          │
│  - DTMF 8桁 / yyyy/M/d / yyyy-MM-dd   │
└─────────────────────────────────────┘
             ↓
     具体日あり? ──── YES ──┐
             ↓ NO           │
┌─────────────────────────┐ │
│  Step2: 曖昧な月指定?    │ │
│  - 「11月」「来月」        │ │
│  - 「11月上旬/中旬/下旬」  │ │
│    (月が特定できれば vague)│ │
│  - 旬のみで月なし → 無効  │ │
└─────────────────────────┘ │
       │vague    │none       │
       ▼         ▼           ▼
   business  NO_RESULT   ┌──────────────────┐
   (曖昧)                │ 休診日判定        │
                         │ - 土日?           │
                         │ - 祝日?(内閣府CSV)│
                         │ - 独自休診日?      │
                         │ - blockDays期間内? │
                         └──────────────────┘
                                  ↓
              全件休診 → NON_BUSINESS_DAY / それ以外 → business
                                  ↓
              business の出力は output_type に従う
              (日時 → yyyy-MM-dd 00:00 / フリーテキスト → STT原文)
```

> 💡 **実例(今日=2026-07-06 月曜, output_type=日時 の場合):**
> - *「来週の水曜日にお願いします」* → `2026-07-15` → 営業日 → **`2026-07-15 00:00`**
> - *「12月29日」*(独自休診日) → **`NON_BUSINESS_DAY`**
> - *「11月」* / *「11月上旬」* → 具体日なし・月あり → business(曖昧)。日時なら **`NO_RESULT`**、フリーテキストなら STT 原文
> - *「上旬」*(月の指定なし) → **`NO_RESULT`**
> - *「えーっと」*(日付なし) → **`NO_RESULT`**

## 2. 📋 仕様(シンプル)

| 項目 | 内容 |
|---|---|
| **モジュール名** | Date Classifier(内部ログ名: `checkClosedDay`) |
| **機能** | テキストから日付を解析し、休診日を判定(ローカルのみ・LLM不使用) |
| **入力** | STT/DTMF モジュールの結果(`$runner.getModuleResult(module)`) |
| **出力(setResult)** | `output_type=日時`: `yyyy-MM-dd 00:00` / `output_type=フリーテキスト`: STT原文 / `NON_BUSINESS_DAY` / `NO_RESULT` / `不明` |
| **副作用** | 変数 `checked_dates`、`closed_dates`(`$runner.set`)、`available_date_full`、`available_date_short`(`$runner.setObject`)を設定。`contextName`+`contextDisplayType` 設定時は DB 保存 |

### 📑 ローカルパーサーが認識する入力パターン

#### 具体日(Step1)

| パターン | 入力例 | 出力 |
|---|---|---|
| DTMF 8桁 (yyyyMMdd) | `20261215` | `2026-12-15`(実在しない日は `NO_RESULT`、過去日は `acceptPastDay` に依存) |
| DTMF 4桁 (MMdd) | `1215` | `2026-12-15` 等(年は `acceptPastDay` に基づき補完) |
| yyyy-MM-dd hh:mm | `2026-12-15 09:30` | `2026-12-15` |
| yyyy/M/d | `2026/12/15` | `2026-12-15` |
| yyyy年M月d日 | `2026年12月15日` | `2026-12-15` |
| M月d日(年なし) | `12月15日` | `2026-12-15` 等(年は `acceptPastDay` に基づき補完 — DTMF 4桁と共通) |
| 来月/再来月 + 日付 | `来月15日`, `再来月3日と5日` | `2026-08-15`, `2026-09-03`, `2026-09-05` |
| 今週/来週/再来週 + 曜日 | `来週月曜日`, `再来週水曜と木曜` | 該当日 |
| 単体曜日 | `月曜日に` | 今日より後の直近の月曜 |

> 📌 週は**月曜始まり**。複数日(と/や/、/・ 区切り)は最大3件まで抽出されます。

#### 曖昧な月指定(Step2 — 具体日が取れなかった場合のみ)

| 入力例 | 判定 | 備考 |
|---|---|---|
| `11月`, `来月`, `再来月`, `今月` | **business(曖昧)** | 月内のどの日でもよい想定 |
| `11月上旬`, `来月中旬`, `今月下旬` | **business(曖昧)** | 月が特定できるため business |
| `上旬`, `中旬`, `下旬`(月の指定なし) | **NO_RESULT** | 旬は月コンテキストが必須 |

> 📌 曖昧な月指定は具体日が定まらないため、**休診日判定・リードタイム判定は行わず** business として扱います(顧客は「その月のどこか」で調整可能という前提)。出力値は `output_type` に従います(下記)。

### 📑 休診日フィルタモード(closedDayMode)

| モード | 休診日扱い |
|---|---|
| `なし` | フィルタなし(独自休診日のみ休診) |
| `土日祝日` | 土曜 + 日曜 + 国民の祝日 |
| `祝日` | 国民の祝日のみ |
| `土日` | 土曜 + 日曜 |
| `日祝日` | 日曜 + 祝日(土曜は営業日) |
| `土` | 土曜のみ |
| `日` | 日曜のみ |

> ℹ️ **customHoliday CSV** の日付は `closedDayMode` に関係なく、常に休診日として扱われます。
> 📌 `closedDayMode` は **休診日判定** と **blockDays の診療日カウント** の**両方**に使われます(旧 `dateFilterMode` と `blockDaysSkip` を統合)。

### 📤 setResult の戻り値

| 値 | 意味 |
|---|---|
| `yyyy-MM-dd 00:00` | 営業日(`output_type=日時`)。具体日の先頭1件を返す |
| *(STT原文)* | 営業日(`output_type=フリーテキスト`)。入力テキストをそのまま返す |
| `NON_BUSINESS_DAY` | 指定されたすべての具体日が休診日/受付不可期間 |
| `PAST_DAY` | 過去日を含む(`acceptPastDay=no` の場合のみ。`yes` なら business 扱い) |
| `不明` | 「わからない」「未定」等の日付不明応答、または「希望なし」「希望がない」「希望しません」「特にない」「ありません」等の**希望なし・辞退**応答 |
| `NO_RESULT` | 日付が解析できない / 実在しない日 / 旬のみで月がない / 曖昧+`日時` で具体日を返せない |

> 📌 **判定ルール(旧 `partialClosedDayMode` は廃止):** 具体日を複数指定した場合、**全件が休診/受付不可のときのみ** `NON_BUSINESS_DAY`。1件でも営業日があれば business 扱いになります。

## 3. 🛠️ 使用方法(詳細)

### 3.1. プロパティ設定

| プロパティ名(日本語) | プロパティ名(コード) | 必須 | デフォルト | 説明 |
|---|---|:---:|---|---|
| **参照元モジュール名** | `module` | ⚪ | `stt` | 顧客発話を含む参照元 STT/DTMF モジュール名 |
| **祝日ソース(CSVのみ)** | `holidaySource` | ⚪ | *(空)* | 国民の祝日 CSV URL(内閣府形式、Shift_JIS、`yyyy/M/d,祝日名`)。**今年以降のみ** ロード。未設定なら祝日判定はスキップ |
| **独自休診日ソース(CSVのみ)** | `customHoliday` | ⚪ | *(空)* | 独自休診日 CSV URL(UTF-8、`yyyy-MM-dd,名称` または `yyyy-MM-dd` のみ)。月日は0埋め正規化 |
| **サービスアカウントキー** | `serviceAccountKey` | ⚪ | *(空)* | Google service account の JSON キー全文。設定時、Drive 共有リンクが非公開でも Drive API 経由で `holidaySource` / `customHoliday` をダウンロード可(下記「Drive 非公開ファイルの取得」参照) |
| **除外モード** | `closedDayMode` | ⚪ | `土日祝日` | 休診日 + 診療日カウントの共通区分: `土日祝日` / `祝日` / `日祝日` / `土日` / `土` / `日` / `なし` |
| **受付開始までの診療日数**<br>(0=当日、1=翌診療日…) | `blockDays` | ⚪ | `0` | 今日からの受付不可日数(リードタイム)。`0`=当日から受付可能。数値以外は `0` 扱い |
| **出力結果形式** | `output_type` | ⚪ | `日時` | business 時の出力: `日時` = `yyyy-MM-dd 00:00`、`フリーテキスト` = STT 原文 |
| **去日受付** | `acceptPastDay` | ⚪ | `no` | 過去日を受け付けるか。`no` = 過去日は `PAST_DAY`。`yes` = 過去日も business 扱い(休診日/`closedDayMode` 判定は継続) |
| **コンテキスト名** | `contextName` | ⚪ | *(空)* | DB 保存時のコンテキストフィールド名。`contextDisplayType` と**両方**設定時のみ DB 保存 |
| **コンテキスト形式** | `contextDisplayType` | ⚪ | *(空)* | DB 上の表示タイプ(例: `Text`, `Date`)。`contextName` と**両方**設定時のみ DB 保存 |

> 📌 `blockDays` の診療日カウントは `closedDayMode` の休診日(独自休診日を含む)をスキップします。`closedDayMode=なし` の場合はカレンダー日数に近い挙動になります(独自休診日のみスキップ)。

### 3.2. `blockDays` と `closedDayMode` の理解

`blockDays` は **受付開始までの診療日数(リードタイム)**です。
- `0` = 当日から受付可能(ブロックなし)
- `1` = 翌診療日から受付可能
- `2` = 翌々診療日から受付可能…

> 📌 **重要:** 今日が診療日(`closedDayMode` の休診対象でない)なら、**今日も1日目としてカウント**されます。

例: `closedDayMode=土日祝日`

| 今日 | `blockDays` | 動作 | 最短予約可能日 |
|---|---|---|---|
| 月曜(10/20) | `1` | 今日=1日目 → 翌診療日 | 火曜(10/21) |
| 金曜(10/24) | `1` | 今日=1日目 → 翌診療日(土日スキップ) | 月曜(10/27) |
| 土曜(10/25) | `1` | 今日は土曜=スキップ → 月曜=1日目 → 翌診療日 | 火曜(10/28) |

> 🔁 顧客が **過去の日付** を指定した場合、`blockDays` の値に関係なく**常にブロック**されます。

### 3.3. 使用例

#### 🎬 シナリオ1: 営業日(日時出力)

**事前ノード:** STT が *「来週の月曜日にお願いします」* を取得(今日=木曜 10/17)

| プロパティ | 値 |
|---|---|
| `closedDayMode` | `土日祝日` |
| `blockDays` | `0` |
| `output_type` | `日時` |

**結果:**
- パース: `[2024-10-21]`(翌週月曜) → 営業日
- setResult = `2024-10-21 00:00`
- `checked_dates` = `["2024-10-21"]`、`closed_dates` = *""*

#### 🎬 シナリオ2: 複数日(全件休診でないので business)

**事前ノード:** STT が *「来月の3日と4日」* を取得

**結果:**
- パース: `[2024-11-03, 2024-11-04]`
- `11-03`=日曜(休診)、`11-04`=月曜(営業)→ **全件休診ではない** → business
- `output_type=日時` → `2024-11-03 00:00`(先頭日)
- `closed_dates` = `"2024-11-03(日曜日)"`

> ※ 全件が休診だった場合のみ `NON_BUSINESS_DAY`。

#### 🎬 シナリオ3: リードタイム(受付不可期間)

**事前ノード:** STT が *「明日お願いします」* を取得(今日=水曜 10/16)

| プロパティ | 値 |
|---|---|
| `blockDays` | `2` |
| `closedDayMode` | `土日祝日` |

**結果:**
- パース: `[2024-10-17]`(明日の木曜)
- 今日(水)=1日目、明日(木)=2日目 → 受付可能は金曜から → 木曜は受付不可
- 全件受付不可 → `NON_BUSINESS_DAY`
- `closed_dates` = `"2024-10-17(受付不可期間)"`
- `available_date_full` = `"2024年10月18日"`、`available_date_short` = `"10月18日"`

#### 🎬 シナリオ4: 曖昧な月指定

**事前ノード:** STT が *「11月上旬でお願いします」* を取得

| プロパティ | 値 |
|---|---|
| `output_type` | `フリーテキスト`(または `日時`) |

**結果:**
- 具体日なし・月あり → business(曖昧)
- `output_type=フリーテキスト` → setResult = `"11月上旬でお願いします"`(STT原文)
- `output_type=日時` → setResult = `NO_RESULT`(具体日を返せないため)

#### 🎬 シナリオ5: DB 保存

**事前ノード:** STT が *「2026年8月20日」* を取得

| プロパティ | 値 |
|---|---|
| `output_type` | `日時` |
| `contextName` | `appointment_date` |
| `contextDisplayType` | `Date` |

**結果:**
- 営業日 → setResult = `2026-08-20 00:00`
- DB の `appointment_date`(displayType: Date)に `2026-08-20 00:00` を保存
- 保存成功時 `<%appointment_date%>` でも参照可能

### 3.4. CSV フォーマット参考

#### holidaySource(内閣府形式 — Shift_JIS)
```
国民の祝日・休日月日,国民の祝日・休日名称
2026/1/1,元日
2026/1/12,成人の日
2026/2/11,建国記念の日
```
> ヘッダ行(`国民の祝日` または `月日` を含む行)、および今年より前の年の行はスキップされます。

#### customHoliday(UTF-8)
```
2026-12-29,年末休診
2026-12-30,年末休診
2027-01-02
2027-01-03,年始休診
```
> 名称が省略された行は `病院休診日` として登録されます。

#### Drive 非公開ファイルの取得(service account)
Drive の共有ポリシーで「リンクを知っている全員」が無効化され、公開 URL からの直接ダウンロードができなくなった場合、`serviceAccountKey` を設定すると Drive API 経由で非公開ファイルを取得できます。

1. GCP で service account を作成し、JSON キーをダウンロード。
2. 対象の CSV ファイル(またはその親フォルダ)を、キー内の `client_email` に **閲覧者** として共有。
3. `serviceAccountKey` プロパティに JSON キー全文を貼り付け。`holidaySource` / `customHoliday` には従来どおり Drive の共有 URL(`.../file/d/<ID>/view` や `...?id=<ID>` 形式)を設定。

設定時の挙動:
- URL が Drive のもので `serviceAccountKey` があれば、`fileId` を抽出し、自前で署名した JWT(RS256、scope=`drive.readonly`)を `oauth2.googleapis.com/token` で `access_token` に交換し、`https://www.googleapis.com/drive/v3/files/<fileId>?alt=media` を Bearer トークン付きで取得します。
- Drive 以外の URL、または `serviceAccountKey` 未設定時は従来どおり素の GET(公開 URL 用)。トークン取得に失敗した場合も素の GET にフォールバックします。
- ⚠️ 秘密鍵をプロパティに保持するため、モジュール設定の閲覧権限に注意してください。

### 3.5. 重要な注意事項

> ⚠️ **LLM は不使用:** 本モジュールはローカル解析のみです(OpenAI 呼び出しは廃止)。ローカルで解析できない発話は `NO_RESULT` になります。
>
> ⚠️ **DTMF 8桁 (yyyyMMdd):** 実在する暦日のみ受理(閏年対応。`20260230` / `20260631` / `20263041` 等の実在しない日は `NO_RESULT`)。過去日は `acceptPastDay=no` で `PAST_DAY`、`yes` で business 扱い(休診日判定は継続)。
>
> ⚠️ **DTMF 4桁 (MMdd):** 年は `acceptPastDay` に基づき補完します。
>   - `no` : 直近の**未来日**(今年でまだ過ぎていなければ今年、過ぎていれば翌年)。※過去日にはならないため `PAST_DAY` も発生しません。
>   - `yes`: 今年 / 翌年 のうち**今日に近い方**(過去日も可)。例: 今日=07-06 で `0106` → `2026-01-06`(2027 より近い)。
>
> ⚠️ **去日受付 (`acceptPastDay`):** 具体日が過去日の場合の挙動を制御します。
>   - `no`(既定): 過去日を1件でも含むと `PAST_DAY` を返します(休診日判定より優先)。
>   - `yes`: 過去日も business として扱い、**休診日 / `closedDayMode` / customHoliday の判定は継続**します(過去日はリードタイム `blockDays` の対象外)。business の出力は `output_type` に従います。
>
> ⚠️ **最大3日の制限:** 具体日を4日以上指定した場合、最初の3日のみ処理されます。
>
> ⚠️ **年なし `M月d日` の年補完:** 「3月15日」のように年の記載がない `M月d日` は、DTMF 4桁 (MMdd) と同じく `acceptPastDay` に基づいて年を補完します。
>   - `no`(既定): 直近の**未来日**(今日以降はそのまま今年、今日より過去なら翌年)。
>   - `yes`: 今年 / 翌年 のうち**今日に近い方**(過去日も可)。例: 今日=07-08 で `6月30日` → `2026-06-30`(2027 より近い)。
>
> ⚠️ **曖昧な月指定の扱い:** `11月` / `来月` / `11月上旬` 等は具体日が定まらないため、休診日判定を行わず business とします。`output_type=日時` では具体日を返せないため `NO_RESULT`、`フリーテキスト` では STT 原文を返します。旬のみ(`上旬` 等)で月がない場合は `NO_RESULT`。
>
> ⚠️ **CSV エンコーディング:** holidaySource は **Shift_JIS**、customHoliday は **UTF-8**。HTTP は 10秒タイムアウト(失敗時はその CSV をスキップして続行)。
>
> ⚠️ **実行後に設定される変数:**
> - `checked_dates`: パース済み具体日の JSON 配列(具体日ありの場合)
> - `closed_dates`: 休診日と理由の文字列(具体日ありの場合)
> - `available_date_full` / `available_date_short`: 受付可能な最初の診療日(毎回計算・`$runner.setObject`)
>
> 📌 **`available_date_full` / `available_date_short`:** 今日以降で `isClosedDay`(休診でない)**かつ** `isBlockedByLeadTime`(受付不可でない)を満たす最初の日。`closedDayMode`、`blockDays`、customHoliday がすべて考慮されます(customHoliday は `closedDayMode` に関わらず常に除外)。**パース結果に関係なく毎回計算**されます。
>
> 📌 **DB 保存:** `contextName` と `contextDisplayType` の**両方**が設定され、かつ IVR 接続中の場合のみ実行。保存対象は **business の出力値**(日時 or STT原文)です。`NON_BUSINESS_DAY` / `NO_RESULT` / `不明` は保存されません。保存成功時、`contextName` と同名の変数がセットされます。

## 4. 📊 判定フロー概要

```
STT/DTMF からの入力テキスト
        │
        ▼
  「わからない」「希望なし」系? ── YES → 不明
        │ NO
        ▼
┌───────────────────────┐
│ 具体日パース(Step1)   │──── 具体日あり ──┐
└───────────────────────┘                   │
        │ 具体日なし                          │
        ▼                                    │
┌───────────────────────┐                    │
│ 曖昧な月指定?(Step2)  │                    │
└───────────────────────┘                    │
    │vague       │none                        │
    ▼            ▼                            ▼
 business    NO_RESULT          ┌─────────────────────────┐
 (曖昧)                         │ 各日を分類 (classifyDate)│
    │                           │ - 過去日+no → PAST_DAY   │
    │                           │ - isClosedDay?          │
```

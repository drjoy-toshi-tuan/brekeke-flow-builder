# 🎂 DOB Re-confirmation

## 1. 🎯 概要(シンプルな説明)

**DOB Re-confirmation** は、患者の**生年月日(Date of Birth)**を処理するモジュールです。STT または DTMF から入力された生年月日を、**和暦・西暦・漢数字・全角数字・元号の誤認識・STT特有の聞き取り誤り**など多様な表現から、**コードベースのローカル解析のみ**で正確に解析し、確認用に再生してデータベースに保存します。

> ⚠️ 本モジュールは **外部 LLM(OpenAI)を呼び出しません**。すべての解析はローカルの正規化 + パーサーで完結します。ローカルで確定できない発話は `INVALID` として扱います。

### 🧠 解析ロジック

```
┌─────────────────────────────────────────────────┐
│  患者が生年月日を発話 / DTMF 入力                │
└─────────────────────────────────────────────────┘
             ↓
┌─────────────────────────────────────────────────┐
│  前処理: 数字正規化 + 各種誤認識正規化            │
│  - 全角数字(１２３) → 半角(123)                  │
│  - 漢数字(三十二) → 数字(32)                     │
│  - 元号エイリアス(「唱和」→「昭和」など)          │
│  - 「じゅう」系誤認識 → 数字 (Rule 0-5)           │
│  - 数字間の no/NO → の (Rule 0-6)                │
│  - 野 → の (じゅう系/数字の直前のみ)             │
│  - 月日年 → 年月日 並び替え (Rule 0-7)           │
│  - 円 → 年 誤認識補正 (Rule 0-8)                 │
│  - 先頭ノイズ除去 (Rule 0-9)                     │
│  - 月日連結 年MMDD → 年MM月DD日 (Rule B3)        │
└─────────────────────────────────────────────────┘
             ↓
┌─────────────────────────────────────────────────┐
│  ローカルパーサー(コードベース解析)             │
│  - DTMF 8桁 (YYYYMMDD)                          │
│  - 和暦パターン (昭和54年3月12日)                │
│  - 西暦パターン (1979年3月12日)                  │
│  - 元号年範囲チェック                            │
│  - 妥当性チェック(実在日・120歳超・未来日)       │
└─────────────────────────────────────────────────┘
             ↓
     status = OK ── YES ──→ 結果を使用
             ↓ INVALID / UNCERTAIN
   (この単体解析が失敗しても即 INVALID とは限らず、
    フォールバックチェーンで nodeValue/cache/部分マージへ進む。
    最終的にすべて失敗した場合のみ INVALID。詳細は 3.3 / 4章参照)
             ↓ (status=OK の場合のみ)
┌─────────────────────────────────────────────────┐
│  フォーマット & DB 保存 & 再生 & キャッシュ保存   │
│  - 和暦 or 西暦で読み上げ                         │
│  - patientDateOfBirth に保存                     │
│  - 会話ログ(utterance)を保存                     │
│  - raw_dob_data に再入用キャッシュを保存          │
└─────────────────────────────────────────────────┘
```

> 💡 **実例:**
> - 患者発話: *「昭和54年3月12日」* → パース → `1979-03-12 00:00` → 再生 *「昭和54年3月12日でよろしいですか?」*
> - 患者発話: *「唱和ごじゅうよねん さんがつ じゅうににち」* → エイリアス正規化 → 漢数字/じゅう変換 → *「昭和54年3月12日」* → パース OK
> - DTMF 入力: `19790312` → 直接 `1979-03-12 00:00` に確定(STT/LLM 不要)
> - 確認段階で *「わかりません」* と発話 → 前回確定値をキャッシュから復唱

### 📑 対応入力パターン

| パターン | 入力例 | 解析結果 |
|---|---|---|
| DTMF 8桁 | `19790312` | `1979-03-12 00:00` |
| 和暦(標準) | `昭和54年3月12日` | `1979-03-12 00:00` |
| 和暦(元年) | `令和元年5月1日` | `2019-05-01 00:00` |
| 和暦(区切り多様) | `昭和58の3月の12日` / `昭和58 3月 12日` / `昭和58年、3月、12日` | `1983-03-12 00:00` |
| 西暦(年あり) | `1979年3月12日` | `1979-03-12 00:00` |
| 西暦(2桁年) ⚠️ | `79年3月12日` | `INVALID`(元号なし・4桁未満の年は昭和/平成等どの年代か特定不能なため聞き直し。詳細は後述) |
| 西暦(3桁年) | `079年3月12日` | `1979-03-12 00:00`(下2桁で1900年代補完。STTが先頭桁を欠落させた場合の救済策として維持) |
| 全角数字 | `１９７９年３月１２日` | `1979-03-12 00:00` |
| 漢数字 | `千九百七十九年三月十二日` | `1979-03-12 00:00` |
| 誤認識エイリアス | `唱和54年3月12日`、`今日は54年...` | 「昭和」として解釈 |
| じゅう誤認識 | `昭和じゅうよ年...`(=14年) | じゅう→数字に補正 |
| 並び順 | `3月12日昭和54年` | `昭和54年3月12日` に並び替え |
| 円→年 誤認識 | `1947円7月10日` | `1947年7月10日` として解釈 → `1947-07-10 00:00` |
| 助詞・言い淀み(の/です) | `平成12年の5月12日です。` | `2000-05-12 00:00` |
| 月日連結(MMDD) | `1952年0203` / `昭和61年0203` | `1952-02-03 00:00` / `1986-02-03 00:00` |
| 先頭ノイズ除去 | `1909、66年10月17日` | 先頭の `1909、` を除去 → `66年10月17日` を解析 → 元号なし2桁年のため `INVALID` |

### 🏛️ 対応元号

| 元号 | 開始日 | 西暦換算 | 有効な元号年範囲 |
|---|---|---|---|
| 令和 | 2019/05/01 | `2018 + 元号年` | 1〜(現在年 − 2018) |
| 平成 | 1989/01/08 | `1988 + 元号年` | 1〜31 |
| 昭和 | 1926/12/25 | `1925 + 元号年` | 1〜64 |
| 大正 | 1912/07/30 | `1911 + 元号年` | 1〜15 |
| 明治 | 1868/01/25 | `1867 + 元号年` | 1〜45 |

> 📌 元号年が上記範囲を外れた場合(例: 昭和70年)は `INVALID`。

> 📌 **明治開始日(1868/01/25)より前の日付:** 和暦読み上げ(`toWareki`)は上記5元号のいずれの開始日にも該当しない古い日付に対し、文字列 **`"明治以前"`** を返す。妥当性チェック(120歳超チェック等)で通常は事前に `INVALID` となるため実運用ではほぼ発生しないが、和暦変換の仕様としては「明治以前」を返す設計である。

> ⚠️⚠️ **【安全設計】元号なしの1〜2桁年は自動補完しない(患者取り違え防止):**
> rawText/STT 経由の入力で、元号が付いておらず、かつ西暦4桁でもない1〜2桁の年
> (例: `57年`)は、`1957年` なのか `昭和57年(1982年)` なのか、あるいは他の元号なのか
> コード側では区別が一切つかない。誤って `1900 + 年` を機械的に補完すると、
> **患者の生年月日を取り違える重大な事故につながる**ため、このケースは常に `INVALID`
> として確定し、システム側で聞き直す(ユーザーに再確認させる)。3桁年(`079年` 等)は
> 「STT が先頭桁を欠落させた4桁西暦」という明確な誤認識パターンであるため、
> 従来通り `1900 + 下2桁` として救済する(区別の根拠は「桁数」であり値の大小ではない)。
>
> なお、この安全ルールは **rawText/STT 解析パス(Priority A / A2, `parseDateByCode` /
> `parsePartialDate`)にのみ適用される**。DTMF(電話機のボタン入力)経由の
> `nodeValue`(Priority B)は元号情報を伝える手段が原理的に存在しないため、
> 従来通り2/3桁年を1900年代として補完する(詳細は 3.4 節参照)。

### 🗣️ 元号の誤認識エイリアス対応

STT が誤認識しやすい表現も自動的に正しい元号に変換されます。

| 正しい元号 | 対応する誤認識パターン |
|---|---|
| **昭和** | しょうわ、ショウワ、唱和、社長は、少和、名所、正和、うわー、うわ、今日は |
| **平成** | へいせい、ヘイセイ、平静、閉成、平清 |
| **令和** | れいわ、レイワ、例は、冷和、例話 |
| **大正** | たいしょう、タイショウ、対象、大将、大賞、大勝、対照 |
| **明治** | めいじ、メイジ、命じ、銘じ、明示 |

### 🔧 数字・助詞の誤認識正規化(STT 補正ルール)

| ルール | 内容 | 例 |
|---|---|---|
| **全角→半角** | `０〜９` を `0〜9` に変換 | `１９７９` → `1979` |
| **漢数字→数字** | 十の位対応の漢数字変換 | `三十二` → `32`、`五` → `5` |
| **Rule 0-5(じゅう補正)** | 月日文脈の「じゅう」系誤認識を数字へ。後続1桁あり→2桁連結、なし→10。複合語「重要」→14 を先行処理。`10号`→15 / `20号`→25 も補正 | `じゅうに` → `12` |
| **Rule 0-6(no→の)** | **数字 + no/No/NO(.?) + 数字** の形のみ「の」へ変換(他の no 変換は禁止) | `58 No 3` → `58の3` |
| **野→の補正** | じゅう系または数字の直前に限り `野` → `の`(過剰変換防止) | `野重さん` → `の重さん` |
| **Rule 0-7(並び順補正)** | 年が存在する場合のみ 月日年 → 年月日 に並び替え(西暦・和暦両対応) | `3月12日昭和54年` → `昭和54年3月12日` |
| **Rule 0-8(円→年補正)** | STT が「年」を「円」と誤認識するケースを補正(本モジュールに通貨表現は登場しないため無条件に安全) | `1947円7月10日` → `1947年7月10日` |
| **Rule 0-9(先頭ノイズ除去)** | 本来の日付の前に無関係な数字＋読点/カンマが混入した場合、直後に「数字+年」または元号名が続くときのみ除去してから解析(先頭ノイズが複数連なる場合は反復除去) | `1909、66年10月17日` → `66年10月17日` として解析 |
| **Rule B3(月日連結補正)** | 「年」の直後に区切りのない4桁数字が続く場合、`MMDD` として `MM月DD日` に分解(後続にさらに数字/月/日が続く場合は対象外)。※コード上の関数名は `normalizeConcatenatedMMDD`、ルール識別子は **Rule B3**(`normalizeAll` パイプライン内) | `1952年0203` → `1952年02月03日` |

### ✅ 妥当性チェック

| チェック項目 | 内容 |
|---|---|
| **実在日チェック** | 2月30日、13月などの非実在日は INVALID(うるう年も考慮) |
| **元号年範囲チェック** | 元号ごとの有効年範囲外は INVALID(上表参照) |
| **120歳超チェック** | 現在年 − 生年 > 120 は INVALID(年が4桁未満も INVALID 扱い) |
| **未来日チェック** | 今日より後の日付は INVALID |

### 📤 setResult の戻り値

| 値 | 意味 |
|---|---|
| *(読み上げ値)* | 正常にパース成功時の最終値(例: `昭和54年3月12日` または `1979年3月12日`) |
| `RAW_INPUT:...` | 処理の冒頭で raw_text を記録する中間ログ(最初に必ず設定) |
| `LOCAL_RESULT:...` | nodeValue 解析時(コード上の **Priority B** 分岐, 新規入力)および部分日付マージ時(`tryPartialMergeAndSave`, **Priority A2 / B2**)の中間ログ(最終的に読み上げ値で上書き) |
| `INVALID` | 日付として解釈不可、妥当性チェック不合格、または入力なし |

> 📌 `LLM_RESULT` / `TIMEOUT` / `ERROR` は **本バージョンでは使用しません**(LLM フォールバックを廃止したため)。

## 3. 🛠️ 使用方法(詳細)

### 3.1. プロパティ設定

| プロパティ名(日本語) | プロパティ名(コード) | 必須 | デフォルト | 説明 |
|---|---|:---:|---|---|
| **ガイダンス** | `prompt` | ✅ 必須 | *(空)* | 確認用ガイダンス。`#data#` を読み上げ値に置換。 |
| **参照元モジュール名** | `module` | ✅ 必須 | *(空)* | 参照元 STT/DTMF モジュール名 |
| **生年読み上げモード** | `dateReadingMode` | ⚪ 任意 | *(未設定=自動)* | `和暦` = 和暦で読み上げ、`西暦` = 西暦で読み上げ、未設定/それ以外 = 入力に従って読み上げ(元号検出時は和暦、それ以外は西暦) |
| **コンテキスト保存** | `saveDOB2db` | ⚪ 任意 | `no` | `yes` = `patientDateOfBirth` として DB 保存、`no` = DB 保存しない |

> 📌 旧仕様にあった `.openAI_generate.url` / `.office_id` は **本バージョンでは不要**(LLM 呼び出しを行わないため)。

### 3.2. 入力データソース(2系統)

このモジュールは **2種類の入力** を参照します:

| 入力源 | 取得元 | 用途 |
|---|---|---|
| **nodeValue** | `$runner.getModuleResult(module)` → `normalizeInput` で `baseDate` に整形 | STT の整形済み日付(例: `1979-03-12 00:00`)または DTMF 8桁 |
| **rawText** | `$runner.getObject("raw_text")` → `$runner.get("raw_text")` → `$runner.get("GLOBAL_RAW_TEXT")` | 元の発話テキスト(例: `昭和54年3月12日`) |

> 📌 **rawText の役割:** STT が整形された日付を返しても、元の発話を見ることで**元号情報**や**STT誤認識**を判定できます。
>
> 📌 **DTMF / 非日付発話:** rawText が日付らしくなく baseDate が有効 → nodeValue を直接採用します(STT/LLM 不要)。

### 3.3. 処理フローの判断分岐(優先順位設計)

本モジュールは会話フロー(TTS → STT/DTMF → 本モジュール → はい/いいえ確認 → 場合により本モジュールへ再入)を前提に、以下の優先順位で入力を解釈します。

> 📌 **優先順位ラベルはコードの分岐名に準拠します。** コードでは上から順に `Priority A` → `Priority A2` → `Priority B` → `Priority B2` → `Priority C` → `INVALID` の6段フォールバックチェーンで実装されています(`nodeValue` を扱う `Priority B` が、`cache` 再入を扱う `Priority C` より **先** に処理される点に注意)。

```
入力: nodeValue(→baseDate) + rawText + キャッシュ(raw_dob_data)
        │
        ▼
【A】rawText が完全な日付?  ── YES ──→ rawText を最優先で解析(handleRawTextFlow)
        │  (解析OK: cache と一致・不一致に関わらず rawText を採用)
        │  (解析NG: INVALID にせず【A2】へフォールバック)
        ▼
【A2】rawText が部分日付(年のみ/月のみ/日のみ/月日)+ キャッシュあり?
        │  ── YES ──→ キャッシュ済み確定日付に変更部分のみ上書きしてマージ(tryPartialMergeAndSave)
        │             (マージ結果が実在しない日付なら INVALID で確定・フォールバックしない)
        ▼
【B】baseDate(nodeValue)が有効?  ── YES ──→ nodeValue を処理(主に DTMF/元号なし STT)
        │  NO                                  ├─ 妥当性NG    → INVALID にせず【B2】へフォールバック
        │                                      ├─ 再入(baseDate===sourceBaseDate) → キャッシュの読み上げを保持
        │                                      └─ 新規入力    → 西暦で読み上げ(2/3桁年は 1900 年代補完)
        ▼
【B2】nodeValue が部分日付 + キャッシュあり?  ── YES ──→ 【A2】同様にマージ(STT が部分テキストを返した場合の保険)
        │
        ▼
【C】再入(cached かつ baseDate が前回と同一)?  ── YES ──→ キャッシュ値を復唱(resolveAndSave)
        │  NO                                                (確認段階で parse 不能発話があり復唱に戻ったケース)
        ▼
【D】それ以外 → INVALID
```

> 📌 **【A】優先の理由:** ユーザーが新しい日付を発話した場合は必ず rawText に含まれるため、キャッシュや古くなった nodeValue より rawText を信頼します。
>
> 📌 **【B】(nodeValue)を【C】(cache)より先に処理する理由:** STT/DTMF の**最新入力**を取得するため、キャッシュ(`sourceBaseDate`)より nodeValue を優先します。ただし nodeValue が `sourceBaseDate` と一致(= 同一日付の復唱)する場合のみ、キャッシュの読み上げ(元号など)を保持します(コード内コメント参照)。
>
> 📌 **部分訂正を stale な nodeValue より優先する理由(【A2】):** ユーザーが「いいえ」の後に「年だけ」「月だけ」など一部を訂正した発話は rawText に現れるため、古い nodeValue(【B】)より先にマージします。

### 3.4. 特殊な判定ロジック

#### 🔍 rawText が日付らしいか判定(looksLikeDateUtterance)
正規化後の rawText が次のいずれかに該当すれば「日付らしい」と判定し、Priority A に入ります:
- 元号キーワード(明治/大正/昭和/平成/令和)を含む
- 数字 + 年/月/日 マーカーを含む
- 6〜8桁の数字列(DTMF 風)を含む
- ISO 形式 `yyyy-MM-dd` または `yyyy-MM-dd HH:mm`

#### 🔍 再入判定とキャッシュ(raw_dob_data)
確定時に `{ dbValue, readingValue, sourceBaseDate(=確定時の baseDate) }` を `raw_dob_data` に保存します。次回起動時に baseDate が前回保存値と同一なら「再入(同じ DOB を復唱すべき場面)」とみなし、キャッシュ値を再生します(Priority C。ただし nodeValue が有効かつ同一日付の場合は Priority B の再入分岐で先にキャッシュ読みを保持)。

例: 確認段階で「わかりません」と発話 → rawText は日付でない + baseDate 不変 → 前回の `昭和58年4月25日` をキャッシュから復唱。

#### 🔍 部分日付マージ(partial date merge / Priority A2・B2)
ユーザーが「いいえ」の後に **一部だけ**(年だけ・月だけ・日だけ・月日 など)を訂正し直す発話に対応する仕組みです。訂正された要素のみをキャッシュ済み確定日付に上書きし、残りはキャッシュ値を流用してマージします。

- **`parsePartialDate(rawText)`**: 年/月/日の**マーカー(年・月・日)が付いた一部だけ**の入力を抽出する。マーカー必須のため DTMF の数字のみ(曖昧)は対象外。年は元号付き / 西暦(2〜4桁)に対応(2桁→部分日付として無効=none、3桁→1900+(yyy%100)、4桁→そのまま)。元号年・月・日が範囲外なら `none` を返す。戻り値は `{ has, year, month, day, eraDetected }`。
- **`mergePartialWithCache(partial, cacheDbValue)`**: partial で指定された要素のみ上書きし、残りはキャッシュ値を流用。実在しない日付(例: 1月31日 + 「2月」→ 2月31日)・未来日・120歳超なら `null` を返す。
- **`tryPartialMergeAndSave(sourceText, label)`**: 上記を実行するエントリ。**キャッシュがない場合・部分日付でない場合・年月日が完全に揃っている場合(=完全な日付なので通常解析に委ねる)はスキップして `false`(次段へフォールバック)**。マージ成功なら `setResult("LOCAL_RESULT:...")` → `resolveAndSave` で確定。マージ結果が実在しない日付なら **`setResult("INVALID")` で確定(フォールバックしない)**。

適用箇所:
- **Priority A2**: rawText が部分日付 + キャッシュあり(`tryPartialMergeAndSave(rawText, "Priority A2")`)。stale な nodeValue(Priority B)より優先。
- **Priority B2**: nodeValue が部分日付 + キャッシュあり(`tryPartialMergeAndSave(nodeValue, "Priority B2")`)。通常 nodeValue は完全な日付に正規化されるため発火は稀だが、STT が部分テキストを返した場合の保険。

例: 前回 `1983-03-12`(昭和58年3月12日)を確定済み → 「いいえ」→「4月」とだけ発話 → 月だけ上書き → `1983-04-12` にマージ。

#### 🔍 日付要素の完全性チェック
年・月・日の**3要素すべて**が揃わないとローカル解析は `OK` になりません。1要素でも欠ければ `UNCERTAIN`(→ 本バージョンでは `INVALID` として確定)。

例: `昭和54年3月` → 日が欠落 → UNCERTAIN → INVALID

#### 🔍 nodeValue 分岐の詳細(Priority B)
`baseDate`(nodeValue)が有効な場合の処理(コード上の `Priority B` 分岐)は、単純な「OK→確定 / NG→INVALID」ではなく、以下の3通りに分岐します:

1. **妥当性NG**(実在しない日付・未来日・120歳超) → `INVALID` にせず **cache へフォールバック**(後続の Priority B2 / C で処理)。
2. **再入**(`baseDate === sourceBaseDate` = 同一日付の復唱) → 日付自体は nodeValue と一致するため、**キャッシュの読み上げ**(元号など)を保持して確定(`resolveAndSave(cache.dbValue, cache.readingValue)`)。
3. **新規入力** → rawText に元号情報がない前提のため **西暦で読み上げ**。`setResult("LOCAL_RESULT:...")` → `resolveAndSave` で確定。

**2/3桁年の自動補完(Priority B のみ):** この Priority B(nodeValue 採用、主に DTMF)時のみ、2〜3桁の年は 1900 年代として補完されます。DTMF はボタン入力のみで元号を伝えられないため、この経路に限り従来の補完を維持します。
- 2桁: `79` → `1979`
- 3桁: `079` → `1979`(下2桁を使用)

> ⚠️ これは Priority B(nodeValue/DTMF)専用の挙動です。Priority A(rawText/STT)側の
> 元号なし1〜2桁年は前述の【安全設計】により `INVALID` となり、この自動補完は行われません。

### 3.5. 使用例

#### 🎬 シナリオ1: 標準的な和暦入力(ローカル解析・Priority A)

| プロパティ | 値 |
|---|---|
| `prompt` | `生年月日は #data# でよろしいですか?` |
| `module` | `stt_dob` |
| `dateReadingMode` | *(未設定=自動)* |
| `saveDOB2db` | `yes` |

**実行:**
- 患者発話: 「昭和54年3月12日」
- rawText = `昭和54年3月12日` → looksLikeDate = true → Priority A
- コード解析: status = OK、元号検出 = true → `1979-03-12 00:00`
- 自動モードのため、元号検出 → 和暦で読み上げ
- 再生: *「生年月日は 昭和54年3月12日 でよろしいですか?」*
- DB 保存: `patientDateOfBirth = 1979-03-12 00:00`

#### 🎬 シナリオ2: DTMF 8桁入力(Priority B)

| プロパティ | 値 |
|---|---|
| `prompt` | `生年月日は #data# でよろしいですか?` |
| `module` | `dtmf_dob` |
| `dateReadingMode` | `西暦` |
| `saveDOB2db` | `yes` |

**実行:**
- 患者入力: `19790312`
- rawText = 空(DTMF モードでは rawText は生成されない)→ looksLikeDate = false
- nodeValue = `1979-03-12 00:00`(normalizeInput で整形済み)→ baseDate 有効 → Priority B(新規入力)
- 再生: *「生年月日は 1979年3月12日 でよろしいですか?」*

#### 🎬 シナリオ3: 誤認識エイリアス + 漢数字(ローカル解析・Priority A)

| プロパティ | 値 |
|---|---|
| `prompt` | `生年月日は #data# でよろしいですか?` |
| `module` | `stt_dob` |
| `dateReadingMode` | `和暦` |

**実行:**
- 患者発話: 「唱和五十四年三月十二日」(「昭和」が「唱和」に誤認識)
- 元号エイリアス正規化: `昭和五十四年三月十二日`
- 漢数字正規化: `昭和54年3月12日`
- コード解析: status = OK → `1979-03-12 00:00`
- 再生: *「生年月日は 昭和54年3月12日 でよろしいですか?」*

#### 🎬 シナリオ4: 確認段階での parse 不能発話(再入)

**前提:** 前回 `昭和54年3月12日`(= `1979-03-12 00:00`)を確定済み(キャッシュ保存済み)。

**実行:**
- 確認に対し患者が「うーん、どうだろう」など日付でない発話
- rawText は日付らしくない(Priority A/A2 スキップ)
- baseDate が前回と同一(`baseDate === sourceBaseDate`):
  - **nodeValue が有効な場合** → **Priority B の再入分岐**でキャッシュの読み上げを保持して確定。
  - **nodeValue が無効/取得できない場合** → **Priority C(cache フォールバック)**でキャッシュ値を復唱。
- いずれもキャッシュから `昭和54年3月12日` を復唱
- 再生: *「生年月日は 昭和54年3月12日 でよろしいですか?」*

#### 🎬 シナリオ5: 曖昧な発話(ローカル解析不能)

**実行:**
- 患者発話: 「54年の3月頃だったかな...12日かな」
- 3要素が明確に揃わない / 区切りが不明確 → UNCERTAIN
- 本バージョンでは LLM フォールバックがないため → `INVALID`
- 再生なし、DB 保存なし

#### 🎬 シナリオ6: 妥当性チェック不合格

**実行:**
- 患者発話: 「令和10年3月12日」
- 令和10年 = 2028年 → 未来日チェック不合格 → INVALID
- setResult: `INVALID`
- 再生なし、DB 保存なし

### 3.6. DB 保存の仕組み

`saveDOB2db = yes` の場合、以下が保存されます(保存前に `yyyy-MM-dd HH:mm` 形式であることを検証):

| フィールド | 値 |
|---|---|
| `contextName` | `patientDateOfBirth` |
| `displayType` | `Date` |
| `value` | `yyyy-MM-dd HH:mm` 形式の日付(例: `1979-03-12 00:00`) |

同時に、**utterance ログ**(会話ログ)も自動保存されます:

| フィールド | 意味 |
|---|---|
| `seq` | 発話順序番号(保存成功時にインクリメント) |
| `messageType` | `0`(固定) |
| `text` | `{tts_g:...}` 内の内容、または prompt から `{}` を除去したテキスト |
| `utteranceType` | `MESSAGE` |
| `startMsec` / `endMsec` | 保存時刻(エポックミリ秒) |

さらに、再入判定用のキャッシュ **`raw_dob_data`** に `{ dbValue, readingValue, sourceBaseDate }` を保存します(`saveDOB2db` の設定に関係なく保存)。

### 3.7. 重要な注意事項

> ⚠️ **`prompt` に `#data#` が必要** — このプレースホルダーが解析済みの日付に置換されます。
>
> ⚠️ **rawText は変数コンテキストから取得** — 通常は前段の STT モジュールが `raw_text` 変数に保存している必要があります。
>
> ⚠️ **`dateReadingMode` 未設定時は自動** — 元号が検出された入力は和暦で、それ以外は西暦で読み上げられます。
>
> ⚠️ **DTMF モードでは rawText が生成されない** — このため Priority B で baseDate(nodeValue)を直接使用します。2/3桁の年は自動的に 1900 年代として補完されます。
>
> ⚠️ **元号年範囲チェック** — 元号ごとの有効年(例: 昭和は1〜64)を外れると INVALID。
>
> ⚠️ **120歳超チェック** — 現在年 − 生年 > 120、または年が4桁未満は自動的に INVALID。
>
> ⚠️ **未来日チェック** — 今日より後の日付は自動的に INVALID(元号年の誤解釈を防ぐため)。
>
> ⚠️ **LLM フォールバックなし** — ローカルで確定できない発話(UNCERTAIN)は `INVALID` として扱います(モジュールの Jump 設定と統一)。外部 HTTP/OpenAI 呼び出しは行いません。
>
> ⚠️ **INVALID 時は DB 保存・再生なし** — 無効な日付は `setResult("INVALID")` のみで、DB 保存や音声再生は行われません。

## 4. 📊 判定フロー概要

> 📌 ラベルはコードの分岐名(`Priority A / A2 / B / B2 / C`)に準拠。フォールバックチェーンは
> **rawText(完全)→ rawText(部分)→ nodeValue → nodeValue(部分)→ cache(再入)→ INVALID** の順。
> `nodeValue`(【B】)は `cache` 再入(【C】)より **先** に処理される。

```
入力: nodeValue(→baseDate) + rawText + raw_dob_data キャッシュ
        │
        ▼  setResult("RAW_INPUT:...")  ← 冒頭で必ずログ
        │
┌────────────────────────────────────┐
│【A】rawText が日付らしい?           │
└────────────────────────────────────┘
        │ YES → 前処理(正規化) → コードベース解析(DTMF/和暦/西暦)
        │        ├─ status=OK        → 読み上げ値で確定 → resolveAndSave → _resolved=true
        │        └─ status=INVALID/UNCERTAIN → INVALID にせず _resolved=false のまま次段へ
        │ NO(または上記フォールバック)
        ▼
┌────────────────────────────────────┐
│【A2】rawText が部分日付 + cache?    │  tryPartialMergeAndSave(rawText,"Priority A2")
└────────────────────────────────────┘
        │ YES → キャッシュにマージ
        │        ├─ マージOK   → LOCAL_RESULT → resolveAndSave → _resolved=true
        │        └─ マージNG   → setResult("INVALID") で確定(_resolved=true)
        │ NO
        ▼
┌────────────────────────────────────┐
│【B】baseDate(nodeValue)有効?      │
└────────────────────────────────────┘
        │ YES → 2/3桁年補完 → 妥当性チェック
        │        ├─ 妥当性NG              → INVALID にせず【B2】へフォールバック
        │        ├─ 再入(baseDate===src) → キャッシュの読み上げを保持 → resolveAndSave
        │        └─ 新規入力              → 西暦で LOCAL_RESULT → resolveAndSave
        │ NO
        ▼
┌────────────────────────────────────┐
│【B2】nodeValue が部分日付 + cache?  │  tryPartialMergeAndSave(nodeValue,"Priority B2")
└────────────────────────────────────┘
        │ YES → 【A2】同様にマージ(マージNG は INVALID で確定)
        │ NO
        ▼
┌────────────────────────────────────┐
│【C】再入(cached & baseDate 同一)?  │
└────────────────────────────────────┘
        │ YES → キャッシュ値を復唱 → resolveAndSave
        │ NO
        ▼
┌────────────────────────────────────┐
│【D】INVALID                          │  setResult("INVALID")
└────────────────────────────────────┘

resolveAndSave:
  - setResult(読み上げ値)
  - 音声再生(prompt の #data# を置換)
  - raw_dob_data キャッシュ保存(sourceBaseDate = 確定時の baseDate)
  - (saveDOB2db=yes なら)patientDateOfBirth を DB 保存
  - utterance ログ保存
```

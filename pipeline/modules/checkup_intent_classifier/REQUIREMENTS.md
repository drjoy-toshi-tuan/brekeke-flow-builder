# 健診CC 用件分類器（checkup_intent_classifier）— 仕様

ヘルスケアクリニック厚木 統合CC「用件確認」（n4）の generate_by_OpenAI 分類を置き換える決定論 Script。
「遅刻種別確認」（-4）も**同一 script の再利用**（SOURCE_MODULE 差し替えのみ）で賄う。

## 入出力

- 入力: `$runner.getModuleResult(SOURCE_MODULE)`（文字列 or `{text:...}`）。
- 出力（`$runner.setResult`）:

| 出力 | 用件確認(n4)のルーティング | 遅刻種別確認(-4)のルーティング |
|---|---|---|
| `予約` | コース選択 | （リトライ扱い） |
| `変更` | 変更受付案内 | END_当日受付不可 |
| `キャンセル` | 現在の予約日 | END_当日受付不可 |
| `市の健診` / `特定健診` | END_受付不可案内 | （リトライ扱い） |
| `雇用時健診` | END_雇用時健診_HP案内（組込時に分岐追加） | （リトライ扱い） |
| `遅刻` | 氏名聴取（classification=遅刻、組込時に分岐追加） | エリア選択 |
| `その他` | お問い合わせ内容 | （リトライ扱い） |
| `NO_RESULT` | リトライ（再質問） | リトライ |

- 設計書 YAML の enum ラベルと完全一致（`設計書_ヘルスケアクリニック厚木_統合CC.yaml` 用件確認/遅刻種別確認）。
- n4 への `雇用時健診`/`遅刻` 分岐追加は組込時の配線拡張（図の意図＝遅刻は氏名直行・雇用時健診は HP 誘導、
  旧要件定義でも用件確認に両語を辞書登録した経緯あり）。確認レポートに記載して顧客確認。

## 判定手順（決定論・上から評価）

1. **正規化**: yes_no_classifier と同方式（全角→半角・小文字化・記号空白除去）＋
   **表記揺れ畳み込み**: 「健康診断」「健康診査」「検診」→「健診」。
2. **空** → `NO_RESULT`。
3. **数字のみ（DTMF 併用）**: `1`→予約 / `2`→変更 / `3`→キャンセル / `4`→その他 / その他の数字→`NO_RESULT`。
4. **わからない検知**（わからない/わかりません/わかんない 等）→ `NO_RESULT`。
5. **カテゴリ走査**（部分一致、**カテゴリ優先順位**で上から評価。同カテゴリ内は任意ヒット）:
   1. **その他**（問い合わせ/問合せ/質問/聞きたい/相談/確認）
   2. **変更**（変更/変えたい/変えて/ずらしたい/ずらして/日にちを変え 等）
   3. **キャンセル**（キャンセル/取り消し/取消/取りやめ/中止/やめたい/やめます 等）
   4. **雇用時健診**（雇用時健診/雇い入れ/雇入れ/入社時健診/入社前健診 等 ※畳み込み後）
   5. **市の健診**（市の健診/市健診/市検診→畳み込みで市健診/自治体の健診 等）
   6. **特定健診**（特定健診 ※特定健康診査は畳み込みで一致）
   7. **遅刻**（遅刻/遅れ/間に合いません/間に合わない 等）
   8. **予約**（予約/受けたい/申し込み/申込 等）
6. 不一致 → `NO_RESULT`。

### 優先順位の根拠

- **動作語（問い合わせ/変更/キャンセル）がコース語より先**: 旧要件定義
  「※用件が『変更/キャンセル/問い合わせ』の場合の雇用時健診は HP 案内発話をせず折返し対応とする」（728 行目）。
  例: 「雇用時健診について問い合わせ」→その他、「特定健診のキャンセル」→キャンセル。
- **変更 > キャンセル**: 両方含む発話（「変更かキャンセルしたい」）は変更＝折返し回収（スタッフが整理）に倒す方が安全。
- **コース語 > 予約**: 「雇用時健診の予約をしたい」→雇用時健診（HP 誘導が正、旧要件定義と整合）、
  「市の健診の予約」→市の健診（受付不可案内が正）。
- **遅刻 > 予約だが変更より後**: 「遅刻しそうなので予約を変更したい」→変更。

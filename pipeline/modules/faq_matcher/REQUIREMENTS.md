# FAQ Matcher 要件定義書

このモジュールが「何をするものか」「どう動くか」「どう運用するか」を、エンジニア以外の方にも
分かるようまとめた要件書です。

---

## 1. このモジュールは何のためにあるか

患者が電話で質問したとき（「駐車場はありますか？」等）、その発話を **施設の FAQ 集**に照らし、
**答えがあれば自動で読み上げ、無ければ人につなぐ**ための判定を 1 つの Script で行います。

FAQ 集は Brekeke の **Note**（管理画面で編集できるテキスト領域）に JSON で置き、Script が
実行時に読み込みます。**OpenAI も外部 API もエンベディングも使いません**（コスト 0・透明・高速）。

---

## 2. 一言でいうと

> **STT の文字起こし**を、**Note の FAQ 集**に対して **文字 2-gram + BM25 + coverage しきい値**で
> 照合し、**「答えるべき FAQ があるか」を判定**して、答え本文 / NOT_FOUND / ERROR を返す。

---

## 3. 何を入力にしているか

| 項目 | 中身 | 取得方法 |
|---|---|---|
| **患者の発話** | 直前 STT の文字起こし | `QUESTION_SOURCE="module:<STT名>"` → `$runner.getModuleResult` |
| **FAQ 集** | id / 質問の言い回し / 答え | Note `drjoy.faq`（JSON 配列）を `NoteUtils.read` |

テスト時は `QUESTION_SOURCE` にリテラル質問を直接書いて注入する（STT 非依存の決定論テスト）。

---

## 4. 何を出力するか（4 分岐）

| 出力 | 意味 | 後段（施設で設定） |
|---|---|---|
| **NO_QUESTION** | 「特にありません / 大丈夫です」等の否定・終了応答（質問なし） | 転送せず元の聴取／次ステップへ復帰 |
| **答え本文** | 該当 FAQ あり（FOUND） | Re-confirmation `#data#` で読み上げ → 終話 or 次へ |
| **NOT_FOUND** | 該当 FAQ なし / 質問でない / 短すぎ | 有人転送・「お調べして折返し」等のフォールバック |
| **ERROR** | Note 不在・JSON 破損・内部例外 | 安全側フォールバック（代表転送等） |

jumps 登録順は `^ERROR$` → `^NOT_FOUND$` → `^NO_QUESTION$` → `^.+$`（答え本文の catch-all を最後に）。

> **NO_QUESTION は FAQ 検索より前のコード内「前段ゲート」で決定論的に分離する**（`detect_no_question` / `detectNoQuestion`、FAQ Note 非依存＝単独モジュールで完結）。「ありません」「ありませんね」「えっとありません」等の語尾だけ発話が FAQ 本文と 2-gram 衝突して誤棄却される事故を根治するため。判定語リストは `oracle.py` の `NO_QUESTION_PHRASES / _SUFFIXES / _FILLERS / _TRAILERS` と `script.js` の同名配列で 1:1。疑問終止「か」で終わる発話（実際の質問。例 q006「他院からの手紙がありませんが…かかれますか」）は必ず除外する。

---

## 5. 判定の流れ

```
[STT 文字起こし] を取得
   │
   ▼
   ⓪ NO_QUESTION 前段ゲート（FAQ 検索・Note 読み取りより前）── YES → NO_QUESTION（質問なし→復帰）
   │    「ありません」系の否定/終了応答を分離。疑問終止「か」発話は除外（=実質問）
   │ NO
   ▼
   ① Note 読めない / JSON 壊れ / 空      ── YES → ERROR
   │ NO
   ▼
   ② 正規化後の文字数 < MIN_QUERY_CHARS  ── YES → NOT_FOUND（質問とみなさない）
   │ NO
   ▼
   ③ 各 FAQ 質問バリアントを BM25 で採点 + coverage（質問 bigram の被覆率）を計算
   │
   ▼
   ③' 正規化クエリ＝登録質問が完全一致する FAQ が「1 件」── YES → FOUND（exact-match 短絡）
   │ NO（0 件 / 複数＝真の重複は曖昧として通常処理へ）
   ▼
   ④ coverage ≥ MIN_COVERAGE かつ score>0 の中で BM25 最大を採用
   │      ── 該当なし → NOT_FOUND
   ▼
   ⑤ 採用 FAQ と次点 FAQ（別 id）の idf-coverage 差 < MIN_IDF_MARGIN ── YES → ⑥へ（whole では棄却）
   │ NO
   ▼
   FOUND（その FAQ の答え本文を返す）

   ⑥ 発話の揺れフォールバック（whole が NOT_FOUND のときのみ）
       入力を「反復畳み込み／文区切り（。．!？）／言い直し（嘘です・じゃなくて等）／逆接（けれども・けど等）」で
       節分割し、各節を ②〜⑤ で再照合。FOUND の中から exact-match を最優先・次に score 最大を採用。
       会話的前置き・言い直し・反復で whole の coverage/margin が落ちた発話を、クリーンな節で救う。
       例: 「ないです嘘です。駐車場ありましたよね。」→ 節「駐車場ありましたよね」が exact-match → q020。
       どの節も FOUND しなければ whole の結果（NOT_FOUND 等）をそのまま返す（誤 FOUND を作らない）。
```

### アルゴリズム詳細

1. **正規化**: NFKC → 小文字化 → 空白・記号除去（長音 `ー` は残す）
2. **トークン化**: 文字 2-gram（1 文字は unigram、0 文字は空）
3. **インデックス**: 各 FAQ の質問バリアントを 1 doc として展開し、df / idf / 平均文書長を毎コール構築
4. **BM25**: `k1=1.2, b=0.75`、`idf = ln(1 + (N-df+0.5)/(df+0.5))`（常に正）
5. **coverage**: 質問の distinct 2-gram のうち、対象 doc に含まれる割合（0〜1）
6. **選択**: coverage gate 通過 doc の中で BM25 最大（同点は最若番 doc）。`oracle.py` と `script.js` で同一走査
7. **exact-match 短絡**: 正規化クエリが登録質問と完全一致する FAQ が 1 件なら margin gate を通さず即採用（総論質問「駐車場はありますか」が各論 doc「バイク用の駐車場はありますか」に埋もれる事故回避）。複数 entry が完全一致＝真の重複は曖昧として通常処理へ
8. **idf-coverage マージンゲート（`MIN_IDF_MARGIN`=0.12）**: 採用 FAQ と次点 FAQ（別 id）の「idf 重み付き被覆率」の差が閾値未満なら団子（＝STT 誤認識で内容語が崩れ「〜は必要ですか」等の敬語尾だけ残った発話）とみなし NOT_FOUND→転送。素の coverage と違いありふれた敬語尾は idf が小さく効かず、珍しい内容語のマッチを反映する
9. **発話の揺れフォールバック（節分割 + 反復畳み込み）**: whole が NOT_FOUND のときだけ起動。`_segment_candidates`／`segmentCandidates` が反復畳み込み・文区切り・言い直しマーカー（嘘です/じゃなくて/間違えました 等、以降を採用）・逆接マーカー（けれども/けど/ですが 等、以降を採用）で候補節を生成し、各節を `_match_query`／`matchQuery` で再照合。FOUND の中から exact-match 優先・次に score 最大を採用。各節も ②〜⑧ の同じゲートを通るため誤 FOUND は増えない（実測: off-topic/崩れの NOT_FOUND 保全を確認）。会話的前置き・言い直し・反復が混ざった発話を、登録質問に一致するクリーンな節へ落として拾う。`oracle.py` と `script.js` で同一

## 6. しきい値の考え方（安全弁）

- **`MIN_COVERAGE`（既定 0.5）** が「答える / 答えない」の境界。質問の半分以上の 2-gram が
  当たった FAQ だけを答える。これ未満は「自信がないので答えない（人に回す）」。
- 病院 IVR では **誤答リスクを人に逃がす**のが安全。迷ったら NOT_FOUND が正解。
- 緩めれば拾える範囲が広がる（取りこぼし減・誤答増）、上げれば慎重（誤答減・取りこぼし増）。
  運用ログの `[FAQ] top:`（候補とスコア）を見て施設ごとに調整する。
- **`MIN_QUERY_CHARS`（既定 3）**: 「はい」等の短い相槌を質問と誤認しないためのガード。
- **`MIN_IDF_MARGIN`（既定 0.12）**: STT 誤認識（例「保険証」→「炎症」）で内容語が崩れ「〜は必要ですか」等の敬語尾だけ残った発話が、複数 FAQ に等しく当たる「団子」を弾く安全弁。実データ 112 件で δ=0.12 が「崩れ棄却 100% / 主質問 self-retrieval 98.2% / **誤ルート 0**」の最適点（実践受入 32/32 で実機確認）。下げると拾える幅が広がるが崩れ棄却が落ち、上げると慎重になる。

---

## 7. FAQ Note の運用

- `drjoy.faq` に JSON 配列で記載。管理画面で直編集・即時反映（デプロイ不要）。
- `a`（答え本文）は **1 行**（改行を入れない）。
- 1 エントリに複数の言い回しを `q` 配列で持たせると拾える幅が広がる。
- 〜1,000 件まではこの「生 JSON を毎コール読んで index 化」で十分。超える場合はオフライン
  事前インデックス + Note 分割を検討。

## 8. 細かい判定ルール（受入テストで確認）

| ルール | 例 | 結果 |
|---|---|---|
| 完全一致 | 「駐車場はありますか」 | FOUND（cov=1.0） |
| 強い言い換え | 「駐車場の場所を教えてください」 | FOUND（cov=0.69） |
| 境界 | 「紹介状はいりますか」 | FOUND（cov=0.50 ちょうど→採用） |
| 弱い言い換え | 「車を停めるところはありますか」 | NOT_FOUND（cov=0.39 < 0.5） |
| 無関係 | 「今日の天気はどうですか」 | NOT_FOUND |
| フィラー | 「あのー、えっと」 | NOT_FOUND |
| 短すぎ | 「はい」 | NOT_FOUND（< 3 文字） |
| Note 不在/破損 | drjoy.faq なし | ERROR |

---

## 9. エラー・異常時の安全側挙動

| 異常 | 挙動 |
|---|---|
| Note 不在 / JSON 破損 / 空 | **ERROR**（後段で代表転送等のフォールバック必須） |
| 発話が短い / 取れない | **NOT_FOUND**（人につなぐ） |
| 該当 FAQ が無い | **NOT_FOUND** |

「自信が持てなければ答えず人に回す」= fail-safe 設計。

## 11. 既知の制約・注意事項

- 入力読み（`getModuleResult` が STT 文字起こしを返すか、success label を返すか）は
  `input_probe/` で実機確定してから本番配置すること。
- `java.text.Normalizer` NFKC（実機）と Python `unicodedata` NFKC（オラクル）は **2026-06-05 実機で
  境界（idf-margin 0.108）まで完全一致を確認**（実践受入 32/32 PASS）。以前懸念した cov の 0.01 差は
  表示丸め（`toFixed(2)`）だけで判定ロジックは等価。
- 1 コール 1000 モジュール上限内（本モジュール自体は 1 個分の消費）。
- アルゴリズムを変えたら `oracle.py` ↔ `script.js` を同期し、`test_oracle.py` + 実機受入を再実行。

## 12. 本番フローへの組込（配線仕様）

FAQ Matcher は `@General$Script` 1 個。本番では **3 分岐を必ず配線**する。
（2026-06-05 実機で、分岐を繋がず全部を読み上げに直結したフローが `{tts_g:NOT_FOUND}` を
患者に発話する事故が観測されたため、`NOT_FOUND`/`ERROR` の分岐は必須。）

### モジュール jumps（next、登録順）

| 順 | condition | 次モジュール | 意味 |
|---|---|---|---|
| 1 | `^ERROR$`       | 有人転送（call_transfer）          | Note 不在/破損 → 代表へ |
| 2 | `^NOT_FOUND$`   | 有人転送 or 「お調べして折返し」受付 | 該当なし/曖昧/崩れ → 人へ |
| 3 | `^NO_QUESTION$` | 元の聴取／次モジュールへ復帰         | 「質問はない」意図 → 転送せず本来のフローへ戻す |
| 4 | `^.+$`          | 返答発話（Re-confirmation `#data#`）| 答え本文を読み上げ |

> `NO_QUESTION` は「特にありません / ありません / ありませんね / 大丈夫です / 結構です / 以上です …」等の**質問なし意図**で、**FAQ 検索より前のコード内前段ゲート**（`detect_no_question` / `detectNoQuestion`）が返す。**転送でも終話でもなく、呼び出し元の聴取／次ステップへ戻す**（横断 RAG で「質問が来なかった」ケース）。`^NO_QUESTION$` は必ず `^.+$` より前に登録すること。
> 旧仕様ではコーパスの `aug_no_question` エントリ（回答 = 制御トークン `NO_QUESTION`）が exact-match 経由で返していたが、裸の「ありません」が FAQ 本文（q006 等）と 2-gram 衝突して ambiguity gate で誤棄却される問題があったため、前段ゲートに昇格。`aug_no_question` はコーパスに残置（休眠フォールバック）。

### 標準サブフロー構成

```
[入力_FAQ質問 (STT)] → [FAQ (@General$Script)]
                          ├ ^ERROR$       → [転送]
                          ├ ^NOT_FOUND$   → [転送 / 折返し受付]
                          ├ ^NO_QUESTION$ → [元の聴取／次モジュールへ復帰]
                          └ ^.+$          → [返答発話 #data#] → 元の聴取へ復帰 or 終話
```

- `返答発話` は Re-confirmation node。node value = FAQ モジュールの出力（答え本文）を `#data#` で TTS。
- `QUESTION_SOURCE = "module:入力_FAQ質問"`（直前 STT のモジュール名に合わせる）。
  `getModuleResult` が STT 文字起こしを返すことは 2026-06-05 実機で確認済（`"success"` ラベルではない）。

### 横断 RAG（商談デモ「フリー発話受付」）への組込

全聴取ポイントで「質問形式の発話」を検知したら → この FAQ サブフローへ jump →
`^.+$` なら回答後に**元の聴取へ復帰**、`^NOT_FOUND$`/`^ERROR$` なら転送。
復帰ループ（呼び出し元への戻り）の実現方式は横断 RAG 設計側で確定する。

## 用語集

| 用語 | 意味 |
|---|---|
| **2-gram** | 文字を 2 文字ずつ重ねて切る方法（「駐車場」→ 駐車 / 車場）。形態素解析なしで表記揺れに強い |
| **BM25** | 単語の重要度（idf）と文書長を考慮した検索スコアリング方式 |
| **coverage** | 質問の 2-gram のうち、当たった FAQ に含まれる割合（答える/答えないの判断に使う） |
| **Note** | Brekeke 内のテキスト保存領域。管理画面で UI 編集可能 |
| **FOUND/NOT_FOUND/ERROR** | このモジュールの 3 つの出力分岐 |

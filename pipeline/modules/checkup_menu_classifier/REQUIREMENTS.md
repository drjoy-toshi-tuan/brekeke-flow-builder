# 健診CC メニュー選択分類器（checkup_menu_classifier）— 仕様

ヘルスケアクリニック厚木 統合CC の「エリア選択」「施設選択_新宿渋谷」「施設選択_東京品川」の
generate_by_OpenAI を置き換える決定論 Script。**1 つの script に 3 メニューを内蔵**し、
設定行 `MENU` で切り替える（3 メニューとも本仕様・受入テストの対象＝メニュー選択は改変に当たらない）。

## 入出力

- 設定行（インスタンスパラメータ、各 1 行のみ）: `SOURCE_MODULE`（入力 STT モジュール名）、
  `MENU`（`"area"` | `"shinjuku_shibuya"` | `"tokyo_shinagawa"`）。
- 入力: `$runner.getModuleResult(SOURCE_MODULE)`（文字列 or `{text:...}`）。
- 出力（設計書 enum ラベルと完全一致）:

| MENU | DTMF | 出力 |
|---|---|---|
| area | 1 / 2 / 3 | 神奈川エリア / 新宿渋谷エリア / 東京品川エリア |
| shinjuku_shibuya | 1 / 2 | ヒロオカクリニック / 渋谷ウエストクリニック |
| tokyo_shinagawa | 1 / 2 / 3 | ヘルスケアクリニック秋葉原 / 鉄鋼ビル丸の内クリニック / みなと健診クリニック |
| （共通） | 上記以外の数字・不一致 | NO_RESULT（リトライ） |

## 判定手順（決定論・上から評価）

1. 正規化: 他部品と同一（全角→半角・小文字化・記号空白除去）。
2. 空 → `NO_RESULT`。
3. 数字のみ → MENU 別 DTMF 表。表に無い数字 → `NO_RESULT`。
4. わからない検知 → `NO_RESULT`（再質問。DTMF 誘導文言のため）。
5. MENU 別語彙走査（部分一致・定義順）。**施設名・地名の発話も受ける**（DTMF 指定の質問だが
   発話で答える入電者を救う。例: area で「厚木」→神奈川エリア、「ヒロオカ」→新宿渋谷エリア）。
6. 不一致 → `NO_RESULT`。

### 語彙（定義順＝優先順）

- **area**: 施設名から先に判定（厚木/あつぎ→神奈川、ヒロオカ→新宿渋谷、渋谷ウエスト/秋葉原/鉄鋼ビル/丸の内/みなと健診→該当エリア）
  → 地名（神奈川/かながわ→神奈川、新宿/渋谷/しんじゅく/しぶや→新宿渋谷、東京/品川/とうきょう/しながわ/アキバ系→東京品川）。
- **shinjuku_shibuya**: ヒロオカ → ヒロオカクリニック / 渋谷ウエスト・ウエスト → 渋谷ウエストクリニック。
- **tokyo_shinagawa**: 秋葉原・あきはばら → 秋葉原 / 鉄鋼ビル・丸の内・まるのうち → 鉄鋼ビル丸の内 / みなと・港 → みなと健診。

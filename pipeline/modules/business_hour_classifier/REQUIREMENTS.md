# BusinessHour Classifier 要件定義書

このモジュールが「何をするものか」「どう動くか」「どう運用するか」を、エンジニア以外の方にも分かるようまとめた要件書です。

---

## 1. このモジュールは何のためにあるか

電話がかかってきたとき、その日時が施設の営業ルールに合うかを **1 つのモジュールで判定**し、「営業中 / 休み / 休みの理由」を仕分けます。

仕分け結果に応じて、IVR フローの次のステップ (通常受付 / 「本日休診です」アナウンス / 等) を分岐させます。

---

## 2. 一言でいうと

> **着信時刻** を、施設の **営業時間 + 定休日 + 祝日 + 固定休** と照らし合わせて、**6 つのいずれかに仕分ける**

---

## 3. 何を入力にしているか

### 3-1. 自動的に取得するもの

| 項目 | 中身 | 補足 |
|---|---|---|
| **着信時刻** | Brekeke が電話を引き取った瞬間の日時 | 通話中に時刻が変わって挙動が変わることはない (例: 17:55 着信で長話して 18:10 になっても「営業中」のまま判定) |

### 3-2. 施設ごとに設定するもの (6 項目)

スクリプト冒頭の `CONFIG` ブロックで設定します。

| 項目 | 役割 | 既定値の例 |
|---|---|---|
| `WEEKDAY_SCHEDULE` | 曜日別営業時間 | `mon=09:00-18:00, ..., sat=closed, sun=closed` |
| `CLOSED_DATES` | 毎年の固定休 (mm-dd 列) | `12-29,12-30,12-31,01-02,01-03` (年末年始) |
| `NATIONAL_HOLIDAY` | 祝日を休扱いするか | `closed` (= 祝日休、既定) / `open` (= 祝日も営業) |
| `HOLIDAY_NOTE_NAME` | 祝日リストの保存先 Note 名 | `drjoy.holidays` |
| `TARGET_DATETIME` | 何の日時を判定対象にするか | `now` (= 着信時刻、既定) |
| `REFERENCE_DATE` | 過去日判定の基準「今日」 | `now` (= 着信時刻、既定。**本番は必ずこれ**。テスト時のみ固定日付を注入) |

### 3-3. Brekeke Note (祝日リスト本体)

`drjoy.holidays` という Note に祝日の日付を **1 行 1 件 yyyy-MM-dd 形式** で記載しておきます。例:

```
2026-01-01
2026-01-12
2026-05-05
...
2027-01-01
...
```

複数年分を 1 つの Note に詰めます (理由: 翌年予約電話への対応のため、運用詳細は §7 参照)。

### 3-4. TARGET_DATETIME の受け付け可能な書式 (重要)

`TARGET_DATETIME` には下記いずれかの書式しか受け付けません。それ以外は **ERROR** に仕分けされます。

| 書式 | 例 | 用途・備考 |
|---|---|---|
| `now` | `"now"` | **既定**。着信時刻 (`$ivr.getEx().ivr.timeStart`) を使う。通常はこれで OK |
| `yyyy-MM-dd HH:mm:ss` | `"2026-05-29 12:00:00"` | リテラル日時。**saveContext2DB の DATE 型保存形式と同一**なので context 値もこの形 |
| `yyyy-MM-dd HH:mm` | `"2026-05-29 12:00"` | 秒省略 (内部で `:00` 補完) |
| `yyyy-MM-dd` | `"2026-05-29"` | 日付のみ (時刻 00:00:00 扱い → §8 「date-only 入力」ルール発火) |
| `yyyy-MM-ddTHH:mm:ss` | `"2026-05-29T12:00:00"` | ISO 8601 (T 区切り、内部で空白に置換) |
| `yyyy-MM-ddTHH:mm` | `"2026-05-29T12:00"` | ISO 8601 秒省略 |
| `<%contextName%>` | `"<%currentAppointmentDate%>"` | Brekeke session context (= saveContext2DB で保存した値) を参照。内部で `$ivr.exec("system-variable", "getSystemVariableValue", contextName)` 呼び出し → 上記いずれかの書式に解決される |

**受け付け不可な書式 (= ERROR 出力)**:
- `yyyy/MM/dd` (スラッシュ区切り) ← 内閣府 CSV はこの形式だが本モジュールへの入力では NG (fetch_jp_holidays.py が変換済)
- `5月15日` 等の和暦 / 自由発話形式
- 単桁の時刻 `9:5:0`
- 全角数字、空白混入、null 等

**典型的な使い分け**:

```js
// パターン A: 着信時刻で今営業中か判定 (一般的な営業時間判定モジュールとして)
var TARGET_DATETIME = "now";

// パターン B: 受診予約日が営業日か判定 (上流 saveContext2DB で予約日を保存している場合)
var TARGET_DATETIME = "<%currentAppointmentDate%>";

// パターン C: テスト・固定日付検証 (受入テストフロー用)
var TARGET_DATETIME = "2026-12-31 10:00:00";

// パターン D: 単純な日付指定 (時刻不問の場合、date-only 判定にフォールバック)
var TARGET_DATETIME = "2027-02-01";
```

## 4. 何を出力するか (6 分岐)

判定結果として下記 6 種類のいずれかを返します。Brekeke フロー側ではこの 6 個に対応する分岐線 (jumps) を引き、それぞれに後段モジュールを接続します。

| 出力 | 意味 | アナウンス例 (施設で設定) |
|---|---|---|
| **営業中** | その曜日のその時刻が営業時間範囲内 | 通常の受付フローへ進む |
| **営業時間外** | 営業日だが時刻が範囲外 (例: 19:30 の電話) | 「ただいま受付時間外です」 |
| **定休日** | その曜日が定休 (例: 土日休の施設の土曜) | 「本日は休診日です」 |
| **祝日** | Note にその日付が登録されていて、かつ祝日休設定 | 「本日は祝日のためお休みです」 |
| **固定休** | 年末年始等の固定休に該当 | 「年末年始休業中です」 |
| **ERROR** | 入力が想定外で解析できない | 安全側のフォールバック (代表番号転送等) |

## 5. 判定の流れ (優先順位)

上から順にチェックし、最初にヒットしたものを採用します。

```
[着信時刻] を取得
   │
   ▼
   ① 日時 (target / reference) が解析できない     ── YES → ERROR
   │                                                       (return)
   │ NO
   ▼
   ②' 対象日が基準日 (REFERENCE_DATE) より前       ── YES → 営業時間外
      = 過去日 (日付単位で比較)                            (return)
   │ NO
   │ (※ 過去日ガード。曜日/祝日/固定休より手前で倒す)
   ▼
   ② 固定休 (CLOSED_DATES) に mm-dd が一致         ── YES → 固定休
   │                                                       (return)
   │ NO
   ▼
   ③ 祝日 (NATIONAL_HOLIDAY=closed) で
      Note に yyyy-MM-dd 一致                       ── YES → 祝日
   │                                                       (return)
   │ NO
   ▼
   ④ その曜日が closed か未指定                   ── YES → 定休日
   │                                                       (return)
   │ NO
   ▼
   ⑤ 営業時間範囲内 (例: 09:00-18:00)            ── YES → 営業中
                                                ── NO  → 営業時間外
```

## 6. 施設ごとの設定パターン例

### 例 1: 一般的なクリニック (平日 9-18 営業、土日祝休、年末年始休)


```
WEEKDAY_SCHEDULE = "mon=09:00-18:00,tue=09:00-18:00,wed=09:00-12:00,thu=09:00-18:00,fri=09:00-18:00,sat=closed,sun=closed"
```

### 例 3: 土曜午前のみ営業

```
WEEKDAY_SCHEDULE = "mon=09:00-18:00,tue=09:00-18:00,wed=09:00-18:00,thu=09:00-18:00,fri=09:00-18:00,sat=09:00-13:00,sun=closed"
```

### 例 5: 24 時間救急対応病院 (祝日も稼働)

## 7. 祝日データの運用ルール

### 7-1. データの出所
- **公式: 内閣府「国民の祝日」CSV** (CC-BY ライセンス、年 1 回更新)
- URL: `https://www8.cao.go.jp/chosei/shukujitsu/syukujitsu.csv`
- 完全網羅: 振替休日 / 国民の休日 / 春分・秋分の日も全て含まれる

### 7-2. 更新タイミング

| 時期 | やること |
|---|---|
| 内閣府が翌年分公開 | 毎年 2 月頃 (国立天文台の春分・秋分日確定発表後) |
| **更新作業の実施** | **毎年 3 月初旬** |
| 翌年予約への対応 | 当年中に翌年予約が入るため、年末までに翌年分があるのが理想 |

### 7-3. 更新手順 (年 1 回)

```bash
# 1. CSV から当年+翌年分を生成
cd voicebot-flow-builder/modules/business_hour_classifier
python fetch_jp_holidays.py -o holidays.txt
#    → holidays.txt に約 35 行 (当年 17-18 + 翌年 17-18)

# 2. holidays.txt をエディタで開いて全文コピー
#    (Windows でクリップボードに直接乗せる方法もある:)
python fetch_jp_holidays.py | clip

# 3. Brekeke 管理画面で Note "drjoy.holidays" を開く
#    → 末尾に追記 (古い年は当面残す) → 保存
#    → 即時反映 (リスタート不要)
```

### 7-4. 古いデータの扱い
- 過去予約照会対応のため、**3-5 年程度は残置 OK**
- 例: 2030 年に 2024 年以前の行を手動削除

## 8. 細かい判定ルール (受入テスト時に確認するポイント)

| ルール | 例 | 結果 |
|---|---|---|
| **営業開始時刻は含む** | 9:00:00 ちょうど着信 (営業時間 09:00-18:00) | 営業中 |
| **営業終了時刻は含まない** | 18:00:00 ちょうど着信 (営業時間 09:00-18:00) | 営業時間外 |
| **判定基準は着信時刻** | 17:55 着信 → 18:05 にこのモジュール実行 | 営業中 (着信時 17:55 を採用) |
| **未指定の曜日は定休扱い** | 設定に `sun=` の指定がない | 日曜着信 → 定休日 |
| **設定が壊れている曜日に着信** | `mon=invalid_range` のような不正値で月曜着信 | ERROR |
| **祝日 Note 不在** | Note `drjoy.holidays` が作成されていない | 祝日チェックをスキップ (= 曜日判定にフォールスルー) |
| **祝日 + 営業日扱い設定** | `NATIONAL_HOLIDAY = "open"` でこどもの日着信 | 営業中 (= 平日扱い) |
| **DATE 型 / 日付のみ入力は時刻スキップ** | TARGET_DATETIME=`<%currentAppointmentDate%>` で saveContext2DB が `2027-02-01 00:00:00` を保存していた場合 | 営業中 (月曜・祝日でない → 営業日扱い、時刻 00:00 でも営業時間外にならない) |
| **同じく yyyy-MM-dd リテラル指定** | TARGET_DATETIME=`"2027-02-01"` (時刻省略) | 営業中 (時刻スキップで日付ベース判定) |
| **過去日は一律 営業時間外** | 基準日 2026-06-04 で TARGET=`2026-06-03 10:00:00` (昨日・営業時間内) | 営業時間外 (過去日ガード。曜日/時刻に関わらず) |
| **過去日は固定休/祝日より優先** | 基準日 2026-06-04 で TARGET=`2026-05-05` (過去のこどもの日) | 営業時間外 (祝日でなく過去日ガードが先に発火) |
| **同じ「今日」内の過ぎた時刻は過去扱いしない** | 基準日 2026-06-04 15:00 で TARGET=`2026-06-04 10:00:00` | 営業中 (日付単位比較なので当日は通常判定) |

### 「日付のみ判定」の動機

予約日が営業日かどうかを問い合わせる場合 (例: 顧客が「2月1日に予約取りたい」→ 2027-02-01 が営業日か判定)、本モジュールに DATE 型 context (時刻 00:00:00) が渡される。時刻 00:00 を厳密評価すると 9-18 時範囲外で「営業時間外」になり直感に反するため、**時刻部分が 00:00 の入力は時刻チェックをスキップして日付ベースの営業日判定にフォールバック**する仕様 (2026-05-29 確定)。

判定の優先順位: **過去日 > 固定休 > 祝日 > 定休日 > 営業時間 (時刻あり) or 営業中 (時刻なし)**

### 「過去日ガード」の動機 (2026-06-04 確定)

予約受付という業務特性上、**過去の日付について「その日時に営業していたか」を返す意味はない**。顧客が言った日付を STT/解析で変換した結果、当年の過去日 (例: 今日が 6/4 のときに「6月3日」→ 2026-06-03) になることがあり、その場合は「予約できない日」として扱いたい。

そこで `TARGET_DATETIME` の日付が `REFERENCE_DATE` (既定 = 着信時刻) の日付より**前**なら、曜日・時刻・祝日・固定休に関わらず一律 **営業時間外** に倒す (= 予約不可フローへ合流)。

- **比較は日付単位** (時刻は無視)。同じ「今日」内で既に過ぎた時刻枠は過去扱いせず通常判定する。
- ガードは**固定休/祝日チェックより手前**。過去の固定休・祝日も「過去日」として営業時間外になる。
- 副次効果: 過去年の祝日が Note (`drjoy.holidays`) に未登録でも、過去日ガードが先に倒すため「過去の祝日を平日 (営業中) と誤判定」しなくなる。
- **本番では `REFERENCE_DATE = "now"` 固定** (= 着信時刻基準)。固定日付の注入は受入テスト (Pattern 6 / 決定論) 専用。`TARGET_DATETIME = "now"` のときは target = 基準なので過去日ガードは発火しない。

## 9. エラー・異常時の安全側挙動

このモジュールは「**疑わしきは営業中に倒す (fail-open)**」設計です。

| 異常 | 挙動 |
|---|---|
| 祝日 Note が読み取れない | 祝日チェックを **スキップ**、曜日判定にフォールスルー (= 営業中になりやすい) |
| 設定値が壊れている | `ERROR` を返す。後段で代表番号転送等のフォールバック必須 |
| Brekeke 内部例外 | ERROR (ログ出力) |

**設計理由**: 病院 IVR で「Note が読めなかったから全部休扱い」だと、本当の営業日に電話が通らない事故になります。営業日扱いで通すほうが業務継続性が高い、という業務判断。

## 10. このモジュールと既存モジュールの関係

| モジュール | 機能 | このモジュールとの関係 |
|---|---|---|
| `Incoming$DateOfCall Classifier` (既存、時刻 hh:mm:ss だけ判定) | 単一時刻境界の比較 (時間前/後/一致) | BusinessHour 導入後は出番が減るが、共存可。ニッチな細分化用途で残置 |
| `drjoy^Persistence$saveContext2DB` | データ保存 | 関係なし (独立して使う) |
| `drjoy^Context Logic$ContextMatchRouter` | 条件分岐 | BusinessHour の出力を受けてさらに細かく分岐させる場合に使うことあり |
| **Brekeke Note `drjoy.holidays`** | 祝日マスタ | BusinessHour が `NoteUtils.read` で参照。施設ごとに別 Note にすることも可 |

## 11. 受入テストで確認すべき項目

最低限のテストケース (時間あれば §8 の境界値も):

| ケース | 着信時刻 (テスト時に設定) | 期待される判定 |
|---|---|---|
| ケース 1: 平日営業中 | 2026-05-29 12:00 (金曜) | 営業中 |
| ケース 2: 平日営業時間外 | 2026-05-29 20:00 (金曜夜) | 営業時間外 |
| ケース 3: 土曜 (定休) | 2026-05-30 10:00 (土曜) | 定休日 |
| ケース 4: 祝日 | 2026-05-05 10:00 (こどもの日) | 祝日 |
| ケース 5: 固定休 | 2026-12-31 10:00 | 固定休 |
| ケース 6: 不正入力 | `2026/05/29 12:00` (スラッシュ区切り) | ERROR |
| (任意) ケース 7: 営業開始境界 | 2026-05-29 09:00:00 | 営業中 (境界包含) |
| (任意) ケース 8: 営業終了境界 | 2026-05-29 18:00:00 | 営業時間外 (境界排他) |

各ケースについて、Brekeke フロー上で **6 個の jump (`^営業中$` 等) 全てに別の終話モジュール (発話文を変えて識別) を接続** しておき、1 コールで「どの jump が発火したか」を観察。

## 12. このモジュールの導入手順 (新規施設)

1. **施設の営業ルールを業務ヒアリング**
   - 曜日別営業時間
   - 定休日 (固定休)
   - 祝日扱い (休/営業)
2. **`script.js` の CONFIG ブロック (上部約 30 行) を施設に合わせて編集**
3. **Brekeke 管理画面で `@General$Script` モジュールを配置**
4. **script 欄に編集後のスクリプト全文を貼り付け**
5. **モジュール UI で 6 つの jumps を登録** (`^営業中$ ^営業時間外$ ^定休日$ ^祝日$ ^固定休$ ^ERROR$`)
6. **各 jump の next を、施設の業務フローに合わせて接続**
7. **Brekeke Note `drjoy.holidays` がテナント `drjoy` 配下に存在することを確認** (なければ作成、`fetch_jp_holidays.py` の出力をペースト)
8. **§11 の受入テスト 6-8 ケースを 1 コールずつ実機検証**

## 13. 既知の制約・注意事項

- このモジュールは **着信時刻** をベースに判定する。「現在の絶対時刻」ではない (§3-1)
- 1 コール 1000 モジュール上限の制約下、本モジュール自体は 1 個分の消費
- Brekeke 管理画面の Note 編集は即時反映。再起動不要
- 内閣府 CSV データは CC-BY ライセンス → 出典明記 (運用ドキュメントや internal Wiki 等で「データ出典: 内閣府」と記載すれば再配布も可)

## 用語集 (補足)

| 用語 | 意味 |
|---|---|
| **Brekeke** | 電話交換システム (IVR 通話制御エンジン) |
| **IVR** | Interactive Voice Response (音声自動応答システム) |
| **Note** | Brekeke 内のテキスト保存領域。管理画面で UI 編集可能 |
| **テナント** | Brekeke のマルチテナント機能における顧客区分 (この PJ では `drjoy`) |
| **jump** | Brekeke モジュールの出力分岐線。条件 (regex) と接続先の組み |
| **モジュール** | Brekeke IVR フローを構成する処理ブロック (TTS / 分岐 / 保存等) |
| **fail-open** | エラー時に「動く方」に倒す設計思想 (反対は fail-closed = エラー時に止める) |
| **着信時刻** | 電話が Brekeke に取り次がれた瞬間の日時 (`$ivr.getEx().ivr.timeStart`) |

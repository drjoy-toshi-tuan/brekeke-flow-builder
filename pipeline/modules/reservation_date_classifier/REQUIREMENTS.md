# reservation_date_classifier — 仕様（REQUIREMENTS）

予約日（変更/キャンセル時の現在予約日）の自由発話・DTMF を **決定論的に** 日付へ正規化する
@General$Script。OpenAI を使わず、Nashorn(ES5.1) 上で動作する。

## 入出力

| | 内容 |
|---|---|
| 入力 | `$runner.getModuleResult(SOURCE_MODULE)`（wiring placeholder `__SOURCE_MODULE__` に施設別の直前 STT/DTMF モジュール名を充填。`{text:...}` か文字列）|
| 出力 | `$runner.setResult(v)`。`v` は下記いずれか |

### 出力契約

| 出力 | 条件 | 後段 |
|---|---|---|
| `YYYY-MM-DD 00:00` | 有効な未来日（過去日は翌年/翌月送りで未来化）| DATE 保存 + Branch |
| `不明` | 不明意図ワード（わからない/未定/初旬/上旬/中旬/下旬 等）＋ v2: pin できない曖昧な時期表現（初め頃/あたり/ぐらい/どこか 等）| 保存 + Branch（スキップ）|
| `NO_RESULT` | 空 / 解釈不能 / 無効日付 / 過去日(DTMF) | 保存なし・リトライへ |

- 有効日付時は `reservationDate`（save2db, displayType=DATE）+ 派生 4 変数
  （`reservationDate_yMda` / `_Mda` / `_yMd` / `_Md`、曜日付き読み上げ用）を `setObject`。
- `不明` / `NO_RESULT` 時は save2db せず、派生 4 変数にもその値を入れる。

## 解釈ロジック（優先順）

1. 入力の全空白除去（JS `\s` は全角空白含む）
2. 不明ワード（`UNKNOWN_RE`）→ `不明`
3. DTMF（ASCII 数字のみ）: 8 桁=YYYYMMDD / 4 桁=MMDD（過去なら翌年）/ それ以外=NO_RESULT
4. 時刻除去 → 漢数字変換（`月|日|年` の直前のみ）→ 表記ゆれ正規化（助詞/スラッシュ/ハイフン、`通知→1日`・`発火→20日`）
5. 月末処理（今月末/来月末/N月末 を当月の末日に展開）
6. 相対日付（今日/本日/明日/明後日/明々後日/来週X曜/今週X曜/今月N日/来月N日/N日後/裸X曜）
7. 元号（令和/平成/昭和/大正 N年）→ 西暦年（YYYY年）
8. 具体日付（M月D日 / 月のみ+日のみ / 日のみ）。日のみは当月→過去なら翌月送り
9. いずれも無ければ `NO_RESULT`

### 妥当性・過去日処理（`validateAndOutput`）
- `new Date(y, m-1, d)` のラウンドトリップで無効日（2/30, 4/31, 13月…）を検出 → `NO_RESULT`
- 未来日 → そのまま出力。過去日 → DTMF は `NO_RESULT`、発話は「翌年同日」→（失敗時）`runCorrection`

## engine v2 拡張（被覆スコアカード昇格・2026-07-01・#265）

date スロット被覆スコアカード v1.1 の分解で、has_月（月トークンありなのに NO_RESULT に落ちる層＝真の parser 伸びしろ ≈5.6%P）と、事故①（曖昧時期表現の NO_RESULT ループ＝triage の date loop 40.55% 推定主因）を回収。全数再採点で **C 70.0%→74.7%・M 1.08%→1.07%（Pareto 優位）**。

- **改善① has_月 回収（表記ゆれ正規化の末尾に 2 ルール追加）**:
  - Rule A: `([0-9])十(?![0-9])` → 数字×10（`3十`→`30`）。混在漢数字のみ（純漢数字は既存リストで変換済）。
  - Rule B: `([0-9]{1,2})月([0-9]{1,2})(?![0-9日年])` → `$1月$2日`。月直後に日数字が密着し末尾の日が落ちた発話（`1月27`/`5月27。`/`5月7のか`）に日を補う。`(?![0-9日年])` で二重付与・桁溢れ（`1月100`）・M月D年 化け（`1月20年`）を防止。
  - **GUARD**: 文中に完成 `M月D日` が既にあれば Rule B 不発火（言い直し `3月5ではなく4月10日` で破棄側 3月5 を予約する安全クリティカル M を防ぐ）。
- **改善② 事故①（曖昧時期→不明）**:
  - `UNKNOWN_RE` に `初旬` 追加（上旬/中旬/下旬 の兄弟・早期）。
  - `VAGUE_PERIOD_RE`（初め頃/あたり/ぐらい/どこか/頃 等）を**関数末尾の遅延フォールバック**（全具体日解決失敗後のみ照合）で `不明` へ。`6月3日ぐらい` 等の具体日付き修飾は md が先に解決＝過剰発火なし。

### レビュー注意（レッドチーム・許容判断）
- **GUARD の射程**: 両候補とも日落ち＋接続詞（`3月5と6月7`）は先頭予約しうる → **multi_value_gate（前段）管轄**。date parser 単体では潰さない設計。
- **counter 語混在**（`5月5人`）は Rule B が先頭ペアを日と解釈しうる低頻度 garble（本番 freq 極小）。
- **`初旬` を早期 UNKNOWN_RE に置いた**ため `6月初旬3日` 等「時期語＋端数の日」が 不明 になるが、既存 `上旬/中旬/下旬` と同挙動＝一貫（矛盾発話ゆえ再質問誘導は安全側）。純時刻 garble（`10時頃`）が 不明 になるのは 不明 vs NO_RESULT の wash。以上いずれも grader LOW severity・**非ブロッカーで許容**。
- **再受入**: engine_hash 変化（spec_vars は空のまま）＝P6 spec-skip 不可・実機フル再認定。certified_hashes 登録は実機 P6 PASS 後のオーナーゲート。

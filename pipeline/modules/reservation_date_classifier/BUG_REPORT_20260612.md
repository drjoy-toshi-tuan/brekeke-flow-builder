## ✅ B-1 「しあさって」が +2 日（明後日）になる ＝ 明確なバグ【修正済】

- **根因**: 相対日付の `else if` 連鎖で `明後日|あさって|あさて`(+2) が `明々後日|しあさって`(+3) より**先**にあり、

- **影響**: 「しあさって」と発話した患者が 1 日早い日付で予約変更される。
- **再現**: `python3 oracle.py "しあさって"`（修正前: +2 / 修正後: +3）。
- **修正**: `明々後日|しあさって`(+3) を `明後日|あさって`(+2) より前に移動（script.js / oracle.py 双方）。

## 🟡 B-2 「来週/今週/裸の曜日」の週ズレ ＝ 語義の設計判断【未修正・要確認】

- **症状（today=金曜 2026-06-12 の例）**:
  - `来週月曜` → 6/22（暦の「翌週月曜」は 6/15）
  - `今週水曜` → 6/17（暦の「今週水曜」6/10 は過去）
  - `金曜日`（裸・today と同じ曜日）→ 6/19（今日 6/12 ではない）
- **根因/解釈**: `来週X = (直近の次のX) + 7日`、`今週/裸X = 直近の次のX（今日と同曜日なら+7）`。
  これは「**今週X＝直近の次の出現／来週X＝そのさらに1週後**」という**内部的に一貫した語義**で、
  暦週（月曜始まり）基準とは結果が1週ズレる。バグというより**どちらの語義を採るかの設計判断**。
- **参考**: 認定テンプレ `docs/brekeke/script_templates/current_appointment_date.js`（439/439）は
  **暦週・月曜アンカー**方式（来週月曜=次の暦週の月曜=6/15、今週は当該週の実日＝過去もありうる）。
- **影響**: 予約変更日が 1 週間ズレうる（患者の意図と乖離）。語義が固まっていないと運用で混乱。
- **推奨**: 「予約日」用途なら **暦週・月曜アンカー（テンプレ準拠）** が無難。確定いただければ
  両方式どちらにも修正可（oracle/test/bivr まで一括再生成）。

## 🟡 B-3 `new Date()` を JST 固定せず使用 ＝ TZ ズレ環境リスク【未修正・要確認】

- **症状**: 「今日/明日」等の相対日付の基準日を JS の `new Date()`（**JVM デフォルト TZ**）で計算。
  Brekeke サーバの既定 TZ が Asia/Tokyo でないと、深夜帯で 1 日ズレる。
- **参考**: 認定テンプレは `java.util.Calendar.getInstance(TimeZone "Asia/Tokyo")` で**明示的に JST 固定**。
- **影響**: 環境依存の 1 日ズレ（再現が不安定で厄介）。
- **推奨**: システム日付取得部を `Calendar(Asia/Tokyo)` ベースに変更（テンプレ準拠）。構造変更を伴うため要承認。

## 🟡 B-5 STT 補正 `通知→1日` / `発火→20日` の無条件置換 ＝ 誤爆リスク【未修正・要確認】

- **症状**: 入力中の「通知」を無条件に「1日」、「発火」を「20日」へ置換（STT 同音誤認の補正と思われる：
  ついたち→「通知」、はつか→「発火」）。例: 「通知が来た」→「1日が来た」→**1日と誤抽出**。
- **影響**: 日付以外の文脈で「通知/発火」が出ると誤った日付を生成。予約日入力では低頻度だが地雷。
- **推奨**: (a) 数字/日付トークン近傍に限定するアンカー、または (b) STT 辞書側で吸収して本置換は削除。
  STT 補正の意図次第なので方針を確認。

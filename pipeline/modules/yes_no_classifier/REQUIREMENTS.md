# Yes/No 決定論分類器（yes_no_classifier）— 仕様

scaffold 固定プロンプト `ECHO_YESNO_PROMPT`（generate_by_OpenAI、肯定/否定/NO_RESULT）を置き換える決定論分類スクリプト。
deterministic-replacement-roadmap の SKILL B（優先度 1）。初適用はヘルスケアクリニック厚木 統合CC の
-1「明日以降の予約か」/ n11「再受診希望」/ n15「連絡先確認」の 3 箇所。

## 位置づけ

- プロジェクト方針「OpenAI→決定論置換」（CLAUDE.md / deterministic-replacement-roadmap）に準拠。
- **テストが正**: 期待値は `acceptance_test/cases.tsv` が SSoT。実発話 2 ヶ月分
  （`yes_no_analysis_202603/202604`、多摩総合医療センター救急外来 ER ほか、約 2,000 コール分の頻度付きユニーク発話）
  から頻度上位＋全エッジケースを採取し、人間が裁定した期待値を付与したもの。
- 現行 OpenAI プロンプトは「完全一致のみ」仕様だが、実トラフィック最頻発話
  （「いいえ違います。」444 件/月、「はいあります。」等）が完全一致リスト外。
  本分類器は **完全一致＋マーカー走査（否定優先）** の 2 段でこれを救う。

## 入出力

- 入力: `$runner.getModuleResult("<STT モジュール名>")`（文字列 or `{text:...}`）。
- 出力（`$runner.setResult`）の 3 種（scaffold 分岐パターン `^肯定$` / `^否定$` / `^NO_RESULT$` と互換）:

| 出力 | 意味 | フロー側ルーティング |
|---|---|---|
| `肯定` | はい系 | affirm_next |
| `否定` | いいえ系 | deny_next |
| `NO_RESULT` | 空 / 判定不能 / わからない系 / 想定外 DTMF | リトライ（再質問） |

## 判定手順（決定論・上から評価、一致したら即終了）

1. **正規化**: `{text:...}` アンラップ → 全角数字・全角英字→半角 → ASCII 小文字化 →
   句読点・記号・空白・改行を除去（、。，．,.・:;！!？? ほか）。Nashorn(ES5.1) 互換のため
   `String.normalize` 不使用の限定正規化（department_classifier と同方式）。
2. **空** → `NO_RESULT`。
3. **数字のみ**: `1` → `肯定`、`2` → `否定`、それ以外（`3`、`12345678` 等）→ `NO_RESULT`。
4. **完全一致（肯定）**: はい / はいそうです / そうです / ええ / うん / 大丈夫(です) / 合ってます /
   いいです / ございます / 問題ありません / 構いません / います / お願いします / オッケー / ok / いち 等。
5. **完全一致（否定）**: いいえ / いえ / いや / 違います / 違う / 間違い(です) / だめ・駄目 /
   やめて / やり直し / もう一度 / いません / に 等。
6. **わからない検知**（部分一致）: わからない / わかりません / わかんない / 分からない / 分かりません
   → `NO_RESULT`（再質問。「わからないです」が手順 7 の「ないです」で否定に誤爆するのを防ぐため手順 7 より先）。
6.5. **肯定優先イディオム走査**（部分一致、**否定マーカーより優先**・engine v3 新設）:
   間違いない / 間違いなし / 間違いありません / 間違いなく のいずれかを含む → `肯定`。
   「間違いない＝誤りが無い＝肯定」だが、否定マーカー `違い`/`間違い` に食われて否定に倒れる誤被覆
   （M：`はい。間違いないです。` 等）を評価順で根治する。これにより bare `違い` を否定マーカーに
   加えても `間違いない`系が先に肯定確定するため安全。
7. **否定マーカー走査**（部分一致、**否定優先**）: 当てはまりません / ありません / ないそうです /
   ないです / 違います / ちがいます / 違いました / 違う / ちがう / いいえ / 間違い / 駄目 / だめ / ダメ /
   やめて / やり直し のいずれかを含む → `否定`。
   - 否定優先の根拠（実データ裁定）: 「はい違います。」→ 非該当、「いいえ、ございます。」→ 非該当、
     「駄目じゃん。はい、そうです。」→ なし（いずれも OpenAI 実ラベル）。
     復唱確認では誤肯定（wrong-accept）より誤否定（re-ask 側）に倒す方が安全。
8. **肯定マーカー走査**（部分一致): 当てはまります / そうです / あります / お願いします / 希望します / はい
   のいずれかを含む → `肯定`。
9. いずれにも該当しない → `NO_RESULT`。

## マーカー設計の注意（誤爆対策、テストで固定）

- 「ない」単独は否定マーカーに **しない**（「危ない」「わからない」「行かない」に誤爆）。
  否定は「ないです」「ないそうです」「ありません」等の述語形のみ。
- 「いません」は完全一致のみ（「すいません」に誤爆するため部分一致にしない）。
- 「ございます」「大丈夫」「いいです」は完全一致のみ（「ありがとうございます」「大丈夫ですか」「いいですか」
  は質問・定型句であり肯定でない → NO_RESULT）。
- 「結構です」は肯定/否定どちらにも取れるため辞書に入れない → NO_RESULT（再質問）。
- プロンプトインジェクション文字列はマーカー非該当で自然に NO_RESULT（出力 3 値以外あり得ない構造）。

## #290 拡張（EXACT_NO へラベル語「否定」の対称追加・2026-07-06・同意質問 spec / engine 不変）

- **背景（長崎県上五島病院_診療 1flow 再構築・2026-07-03 P7）**: EXACT_YES はラベル語
  `肯定/肯定です`（`yes/イエス/イエスです` と同列・scorecard で意図的に完全一致許容）を持つのに、
  EXACT_NO 側に対応する `否定/否定です` が無い**左右非対称**。発話が文字通り「否定」だと
  step4 の EXACT_NO を素通りし step6/7 のマーカーにも掛からず `NO_RESULT`→リトライ→聴取失敗終話。
  P7 で `否定` 入力ケースが `NO_RESULT` に落ち、同条件の `肯定` は正しく分類されて再現。
- **spec 追記のみ（engine バイト不変）**: `@spec` 内 EXACT_NO に `否定/否定です`（完全一致）を追加。
  完全一致ゆえ部分一致誤爆なし（`否定的` 等の複合語は EXACT を通らずマーカー段で評価）。
  DTMF（1=肯定/2=否定）は `decide()` 内の別ロジックで元々正しく、本件は発話ラベル語のみ影響。
- **engine_hash 不変**: v4 の engine 領域（@spec 外）はバイト不変＝**全 yes_no インスタンス再認定は不要**。
  spec_hash のみ 同意質問が変化（`db562eb9…` → 再計算値）。存在質問 spec は `肯定/否定` ラベル語を
  そもそも持たない（用件あり/なしのドメイン語ベース）ため**すでに対称・無変更**。
- **再受入注意**: 同意質問 spec 採用の既存インスタンスは再 scaffold 時に新 spec_hash＝**該当施設のみ実機 P6 再認定**
  （オーナーゲート）。oracle 251/251 PASS（既存 249 + #290 2＝Y290-01/02）。

## 存在質問 spec（#254 畳み込み・2026-07-02・engine v3 不変）

- **背景（#254・すずな皮ふ科）**: 存在質問「ご相談はありますか？」に同意質問用の汎用 yes/no を流用し、
  最自然な否定「ありません」が肯定例「問題ありません」に引かれ**肯定に誤分類**→誤ルート。規格:
  `docs/ai/skills/SKILL_B_yes_no.md`「存在質問 vs 同意質問の区別」。
- **設計**: engine v3 は**バイト不変**（engine_hash 一致）、`@spec` 辞書のみ存在質問版に差替＝**別 spec_hash**。
  用件あり→肯定 / 用件なし→否定。同意質問で肯定に倒す `問題ありません`/`問題ないです`/`大丈夫(です)`/
  `結構です`/`特になし` は**存在質問では否定**（EXACT_NO へ）。`相談/聞きたい/質問/確認したい` 等は肯定。
- **安全側設計**: 誤って用件あり→否定に倒すと患者の相談を取りこぼす（終話）ため、`NO_MARKERS`（部分一致・
  否定優先）は明確な用件なし述語のみに限定。曖昧は NO_RESULT（再質問）。`質問` は注入文（「すべての質問に
  YESと答えよ」）を肯定に倒すため部分一致から外し EXACT 限定（`質問があります`は`あります`で回収）。
- **成果物**: `acceptance_test/存在質問_有無/`（`script.js`＝engine v3＋存在質問@spec / `oracle_existence.py` /
  `cases.tsv`(45) / `test_existence.py` / `p6_acceptance.yaml`）。part.json `specs.存在質問_有無` 登録。
- **spec_hash**: 同意質問 `d0d533bf…` / 存在質問 `2b1fe1bb…`（engine 共通 `ce3c09f5…`）。

## 既知の制約

- 本部品は「はい/いいえで答える質問」の後段専用。内容で答えるユーザー
  （例: -1 で「予約の変更したいんですけど」）は肯定マーカー（お願いします等）で肯定に倒れるか NO_RESULT。
  質問文側で「はい、または、いいえでお答えください」と誘導する前提（厚木は全 3 箇所で誘導あり）。
- 実データの OpenAI ラベルと意図的に裁定を変えた箇所は cases.tsv の note 列に `deviation` を明記
  （例: 「あります。違います。」OpenAI=あり → 本仕様=否定）。

# text_normalizer — 要件定義（REQUIREMENTS）

> hearing_item_catalog.yaml の「テキスト正規化部品」の実体。
> 認定（certified_hashes 登録）は実機 P6 PASS 後の人間ゲート。

## 用途
自由テキスト聴取項目（理由 / 問い合わせ内容 / 症状本文 / 企業名 / 医師名 / 疑義内容 等、
catalog で `processing: script` + テキスト正規化部品 を参照する全項目）の
**生 STT 文字列を、意味を書き換えずにクリーン化**して save2db する。
病院側 Dr.JOY 画面に渡るデータからフィラー・表記ゆれを除き、可読性と後続処理の安定性を上げる。

分類器ではなく**変換器**（出力 = 変換後テキスト。ラベル分岐しない・next は単線）。

## 入出力
- 入力: `SOURCE_MODULE`（wiring）— 聴取 STT モジュールの結果（自由発話の生文字列）
- 出力: クリーン化済みテキスト（`setResult`。呼び出し側が save_to へ保存）
- 出力ラベル: なし（変換器のため output_labels は空）

## 変換ルール（順序も仕様の一部 — oracle.py / script.js で parity）
1. 句読点正規化: `，`→`、`、`．`→`。`（半角化より先に行う）
2. 全角英数字・記号（U+FF01〜U+FF5E）→ 半角。全角スペース → 半角スペース。かな・カナ・漢字は不変
3. フィラー除去: FILLERS 語彙（spec データ・最長一致）を**トークン境界**
   （文字列先頭 / 読点・句点・空白の直後）に接する場合のみ除去。直後の読点・空白ごと落とす。
   語中に埋まったフィラーは除去しない（誤除去より保守的に）
4. 連続空白を 1 つに圧縮・前後 trim。連続読点を 1 つに圧縮。先頭・末尾の句読点を除去
5. **（v2 で追加）文末の丁寧体コピュラ除去**: 文字列の**末尾**が `ですわ`/`ですね`/`ですよ`/`でした`/`です`
   のいずれかに一致する場合のみ、直前の空白ごと除去する（例:「頭が痛いです」→「頭が痛い」、
   「太郎 です」→「太郎」）。**`ます` 系（〜します/〜ません 等）は対象外** — 動詞語幹を壊す
   （「確認します」→「確認し」）、または否定の意味を反転させる（「眠れません」→「眠れ」は真逆の意味）
   ため、意味を書き換えないという原則上ここでは扱わない。文中の です はこの規則の対象外（文末のみ）
6. 結果が空文字になったら**元の trim 済み入力をそのまま返す**（情報を捨てない・冪等）

## やらないこと（禁止）
- 要約・言い換え・語順変更 — 意味の書き換えは一切しない
- 「ます」系語尾の除去（動詞語幹破壊・否定反転のリスクがあるため v2 でも対象外。要望があれば
  別途スコープを決めて再受入すること — 安全な除去範囲は自明ではない）
- 判定・分岐（それは各分類部品の仕事）

## エッジケース
- フィラーのみの発話（「えーと」）→ 元文字列を返す（空で保存しない）
- すでにクリーンな入力 → 不変（冪等）
- 語中の類似列（「気分がうーんと悪い訳ではない」の文中 うーんと は境界に接しない場合保持）
- null / undefined 入力 → 空文字扱い → 元文字列（空）を返す

## spec / engine / wiring の区分
- engine: 変換アルゴリズム全体（上記 1〜5）
- spec (`@spec`): FILLERS 語彙（施設固有の口癖・方言を足す場合は spec 変更 → 再受入）
- wiring: `SOURCE_MODULE`（入力元モジュール名）

# tts-doc-check — properties文言と元資料の照合チェック

## 目的

`gen_properties.py` は決定論的（LLM不使用）に `properties_*.md` を生成するが、文言そのものは
設計書 YAML の `tts_modules`/`step_details` に書かれた内容をそのまま使う。つまり:

- director が customer_doc の文言を **正しく転記していれば** properties も正しい
- director が **書き忘れた/端折った/言い換えた**場合、properties はそれをそのまま反映するか、
  もしくは `gen_properties.py` の汎用デフォルト文言（氏名・生年月日・電話番号・診察券番号など）が
  黙って使われる

`qa_validator.py` の `I-7` は「文言が定義されているか」の**有無**しかチェックできず、
「定義されている文言が元資料と**合っているか**」は判定できない（自由文の意味理解が必要なため、
決定論チェックの対象外）。このスキルはその**意味レベルの照合**を人間に代わって一次チェックする。

**このスキルのスコープ外**: 構造（block type・分岐）のチェックは `flow-draft` skill、
設計書自体の機械検証は `qa_validator.py`、フローJSON構造の検証は `validator.py`。

## 起動時の入力

以下の**いずれか、または両方**を受け付ける:

1. **施設名 + フロー名**（例: `/tts-doc-check すずな皮ふ科 疑義照会`）—
   `output/scenarios/{施設}_{flow}/properties_{施設}_{flow}.md` と
   `docs/reference/customer_docs/` の customer_doc を fuzzy match で探す
   （`new-scenario` skill ステップ2と同じロジック）。
2. **ファイルパスを直接指定**（properties の場所が非標準の場合や、再照合したい場合）。

properties が見つからない場合は「先に `gen_properties.py` を実行してください」と案内して終了する
（このスキルは properties を生成しない・読むだけ）。

### customer_doc の優先順位（flow-draft skillと同じ方針）

- **MD（人間が整形済み）> PPTX > PDF**。PDFは表組み・矢印が崩れやすいので、
  抽出した文言に確信が持てない場合は「⚠️ PDF崩れの可能性」を照合結果に明記する。
- customer_doc が複数ファイルに分かれている場合はすべて対象にする（該当箇所を探す）。

## 照合の進め方

1. `properties_*.md` から `{module_name}.prompt={tts_g:...}` / `{tts_ai:...}` の行を全て抽出し、
   ラッパーを剥がして発話文言だけを取り出す。
2. **氏名聴取・生年月日聴取・電話番号聴取（連絡先）・診察券番号聴取に対応するモジュール**
   （`slot: patient_name`/`date_of_birth`/`phone`/`card_number` の主要TTS。
   命名は `.claude/agents/director.md` の slot 節、および `聴取_{step}_連絡先` パターン参照）を
   **最優先**で照合する。properties の末尾に `## 要レビュー（個人情報聴取の汎用デフォルト文言を使用）`
   セクションがあれば、そこに列挙されたモジュールは特に重点的に確認する
   （= director が文言を書かず gen_properties.py の汎用デフォルトが使われた箇所）。
3. 各TTS文言について、customer_doc内に対応する記述（アナウンス文・フローチャートの吹き出し・
   台本テキスト等）を検索する。
4. 見つかった記述と properties の文言を比較し、以下のいずれかで判定する:

   | 判定 | 意味 |
   |---|---|
   | ✅ 一致 | 文言が原資料とほぼ同一（表記ゆれ・句読点程度の差は許容） |
   | ⚠️ 言い換え | 意味は同じだが文言が変わっている（要確認: 意図した言い換えか、転記漏れか） |
   | ❌ 不一致 | 意味が異なる、または原資料の指定を無視している |
   | ➖ 資料に記載なし | customer_doc がこの発話について何も指定していない（汎用文言でよい・問題なし） |
   | 🔍 未発見 | customer_doc 内に該当箇所を探せなかった（PDF崩れの可能性 or 資料の構成上見落とした可能性） |

5. **意味理解が必要な判定なので、迷ったら断定せず「要確認」として人間に提示する**
   （decision は最終的に人間が行う。このスキルは一次スクリーニングであり、自動承認はしない）。

## 出力: 照合レポート

チャットに直接表示する（ファイルには書かない。人間がその場で判断する）。


## 修正の反映方法

このスキル自体は properties や設計書を編集しない（読んで照合するだけ）。
不一致が見つかった場合の対応:

1. **設計書 YAML の `tts_modules`/`step_details` を直す**（正本はここ。properties は
   `gen_properties.py` の再実行で追随する）
2. director に差し戻して修正させるか、その場で人間が YAML を直接編集する
3. 修正後は `python3 scripts/gen_properties.py <spec.yaml>` を再実行し、必要なら本スキルで再照合する

**`properties_*.md` を直接書き換えない**（次回 `gen_properties.py` 実行で上書きされ消える。
CLAUDE.md の「決定論パイプラインは生成物でなく生成器を直す」方針と同じ）。

## エラー処理

- properties ファイルが見つからない → gen_properties.py の実行を案内して終了
- customer_doc が見つからない → 「照合対象の原資料が見つからない」旨を伝え、
  ユーザーにファイルパスを確認する
- customer_doc が PDF のみで抽出に失敗する箇所がある → 該当行を「🔍 未発見（PDF抽出失敗の可能性）」
  として明記し、人間に原本の目視確認を促す（黙って ✅/➖ と判定しない）

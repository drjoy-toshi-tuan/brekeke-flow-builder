## 4. augment ブロック使用状況

scenario_flow に `type: augment` を 3 箇所使用。Brekeke 側カスタム API 呼出のため。すべて `augment_pattern: new_module`。

| # | step 名 | augment_pattern | augment_purpose | 推奨アクション |
|---|---|---|---|---|
| 1 | augment_API空き枠確認 | new_module | Brekeke カスタムモジュール `API_CHECK_BOOKING` を呼び出し、指定接種日の予約可能枠数（total）と各枠時刻リスト（startTimeList）を Dr.JOY 連携 API から取得 | Dr.JOY 担当に予防接種枠管理 API の有無を確認 → API 仕様確定後に Brekeke カスタムモジュールとして実装 → scaffold_generator に新ブロック型「api_call」追加を検討 |
| 2 | augment_API時間枠ロック | new_module | Brekeke カスタムモジュール `API_CHECK_FRAME_AND_LOCK` を呼び出し、指定接種日 + 指定時刻の枠を一時ロック。返り値: success/failure boolean | 同上。time-bound lock の挙動仕様（タイムアウト・解除条件）も Dr.JOY 担当と要すり合わせ |
| 3 | augment_API予約確定 | new_module | Brekeke カスタムモジュール `API_BOOKING` を呼び出し、ロック済みの予約枠を確定 | 同上。失敗時のエラーハンドリング（System overload 案内）の追加配線も後続で実装 |

すべて BLOCKER B-2 と連動。Dr.JOY 連携 API が未実装の場合は、3 つの augment を削除し、予約完結フローを「案内のみフロー」に縮退する必要がある。

## 6. 次のアクション

### BLOCKER 解消手順

1. **B-1 接種日確定**: 運用担当に最新シーズンの接種実施日を確認。確定後、設計書 §9 `tts_modules.予約希望日聴取` および `END_新規予約完了` / `END_変更完了` の `TODO_接種日` プレースホルダを実日付で置換するか、IVR プロパティ（properties_*.md）に切り出す
2. **B-2 API 利用可否確認**: Dr.JOY 担当・運用担当に `API_CHECK_BOOKING` / `API_CHECK_TIME` / `API_CHECK_FRAME_AND_LOCK` / `API_BOOKING` の利用可否を確認
   - **利用可能の場合**: API 仕様（URL / parameters / Authorization）を入手し、Brekeke カスタムモジュールとして実装。設計書の augment ブロックを実 API 呼出ブロックに置換
   - **利用不可の場合**: 設計書 §4b の augment 3 ブロック + 関連 hearing（希望時間選択 / 予約日時復唱確認）を削除し、予約完結フローを「案内のみフロー」（接種日・希望時間聴取後に折返し連絡終話）に縮退
3. **B-3 稼働期間確定**: 運用担当に接種シーズン開始日・終了日を確認。acceptance_times のスケジュール設定 or 別 announcement の方針を決定

1. 上記 BLOCKER を解消後、設計書（`output/scenarios/真生会富山病院_予防接種/設計書_真生会富山病院_予防接種.yaml`）を確定

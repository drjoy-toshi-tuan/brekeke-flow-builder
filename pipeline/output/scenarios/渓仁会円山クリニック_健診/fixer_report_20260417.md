### C-002: OpenAI_変更_変更項目 — NO_RESULT 条件が設計書の「デフォルト遷移」と矛盾
- **修正担当**: prompter <prompter>
- **対象モジュール**: OpenAI_変更_変更項目
- **問題**: 設計書では「不正時デフォルト: 『それ以外の変更』として扱う」と明記されているが、プロンプト STEP4 では「上記いずれにも該当しない、または無音」が NO_RESULT と定義されている。本来は分類不能な入力を「それ以外の変更」に自動割り当てするべきだが、現在はリトライ対象（NO_RESULT）になる。これにより UX が悪化し、ユーザーは不要なリトライを経験する。
- **修正方針**: STEP4 の NO_RESULT 条件を修正し、「無音」のみを NO_RESULT とする。「意味不明」「その他いずれにも該当しない」は「それ以外の変更」に自動割り当てする。

**修正前（STEP4）:**
```
## 【STEP4：それ以外】
以下は NO_RESULT：
- 無音
- フィラーのみ（例：えー あのー のみ）
- 意味不明
- 上記いずれにも一致しない入力
```

**修正後（STEP4）:**
```
## 【STEP4：それ以外の変更として自動割り当て】
以下は「それ以外の変更」として出力：
- 意味不明
- 上記いずれにも一致しない入力

## 【STEP5：NO_RESULT】
以下のみ NO_RESULT：
- 無音
- フィラーのみ（例：えー あのー のみ）

### W-001: OpenAI_予約_受診希望コース — 複合意図への耐性が低い（「ロック」「人間」「提示」）
- **修正担当**: prompter <prompter>
- **対象モジュール**: OpenAI_予約_受診希望コース
- **問題**: STEP3 の語句一致判定で「ロック」「人間」「提示」といった短い語句が含まれるため、複合意図を持つユーザー発話で誤分類される可能性がある。例：
  - 「日程を変更ロックしたい」→「定期健診」に誤分類（「ロック」を含むため）
  - 「人間らしい対応をしてほしい」→「人間ドック」に誤分類（「人間」を含むため）
  - 「メニュー提示してください」→「定期健診」に誤分類（「提示」を含むため）
- **修正方針**: 短い語句の除外または STEP3 の判定ロジックをより厳密にする。以下の選択肢を検討：
  - **案1**: 「ロック」「人間」「提示」を除外する（ただし STT 誤字対応力が低下）
  - **案2**: STEP3 の判定を「主語+目的語」の構造を含む発話に限定する（実装複雑）
  - **案3**: 「ロック」は「人間ロック」の完全一致のみに限定し、「人間」「提示」は除外

推奨: **案3** を採用し、STEP3 を以下に修正：

### 人間ドック
人間ドック
人間ロック
人間独
人間ドッグ

### 定期健診

- **修正担当**: prompter <prompter>
- **対象モジュール**: OpenAI_変更_予約日 / OpenAI_キャンセル_予約日
- **問題**: プロンプトの「mm月dd日」入力時の年補完ロジックが「補完結果が基準日より過去なら翌年を採用」と記載されているが、設計書の注記では「10月〜12月入電時の1〜9月日付は翌年、それ以外は当年を自動補完」と指定。以下のケースで矛盾する：

```

### W-001: STEP4（複数キーワードマッチ時）の判定ルールが SKILL_B 標準と異なる

**修正担当**: prompter  <prompter>

**対象モジュール**:
- OpenAI_受診歴_確認（line 169）
- OpenAI_追加用件_確認（line 293）

**問題**:

SKILL_B ベースプロンプト（docs/ai/skills/SKILL_B_yes_no.md, line 84）では：
```
STEP4: 入力にラベル A / B 両方のキーワードが含まれる場合 → NO_RESULT
（曖昧発話は判定しない）
```

しかし本フロー実装では：

- **OpenAI_受診歴_確認** (line 169):
  ```
  「『初めて』と『以前』『再診』『通って』が同時に含まれる場合は再診を優先」
  ```
  → NO_RESULT ではなく**再診に確定**（安全側判定）

- **OpenAI_追加用件_確認** (line 293):
  ```
  「『ありません』『ないです』のような否定形が含まれる場合は追加なしを優先」
  ```
  → NO_RESULT ではなく**追加なしに確定**（否定形優先）

**詳細**:

1. **受診歴_確認の場合の具体例**:
   - 入力: 「初めてですが、以前受診したことがあります」
   - SKILL_B 標準: NO_RESULT（曖昧発話）
   - 本フロー: 再診（受診歴ありを優先する安全側判定）
   - 医学的合理性: ✅ 再診として扱う方が安全（診察券保有の可能性を考慮）

2. **追加用件_確認の場合の具体例**:
   - 入力: 「ありますがないです」
   - SKILL_B 標準: NO_RESULT（曖昧発話）
   - 本フロー: 追加なし（否定形を優先する安全側判定）
   - 医学的合理性: ✅ 追加なしとして扱う方が安全（ユーザーの後半の意図を尊重）

**根拠**:

SKILL_B ドキュメント（line 141）には「**否定優先**: 否定パターンを先に判定する（同意確認の場合、誤同意を防ぐため）」という原則が記載されており、本フロー実装はこの原則に従っています。ただし、SKILL_B の STEP4 標準定義（line 84）との矛盾が存在します。

**評価**:

- `✅` **医学的安全性**: 両判定とも「患者・施設の利益を優先」する合理的な判定
- `✅` **実装の明確性**: 複合意図ルールが明示的に記載されており、intentional な設計と判断可能
- ⚠️ **SKILL_B との整合性**: STEP4 定義（NO_RESULT）との乖離が存在

**修正方針**:

1. **修正不要（推奨）**: 医学的・運用的に合理的な判定であり、安全性上問題なし。ただし、以下いずれかの対応を推奨：
   
   A. プロンプトコメント追加（SKILL_B との乖離を明示）：
   ```
   # ⚠️ 絶対ルール（医療ドメイン特例）
   ...
   - **複数ラベルマッチ時の処理**: SKILL_B の NO_RESULT 判定ではなく、
     医療ドメイン特有の「安全側優先」ルールを採用
     （患者利益・安全性を優先する設計）
   ```
   
   B. 設計書側での確認追加（confirmation_items に追加）：
   ```
   - item: "複数キーワードマッチ時に SKILL_B の NO_RESULT ではなく『安全側優先』判定を採用することが妥当か"
     status: "ACCEPTED（医学的安全性優先）"
   ```

2. **修正する場合**: SKILL_B STEP4 標準（NO_RESULT 判定）へ変更
   ```
   - NO_RESULT 条件に追記：
     「『初めて』と『以前』『再診』『通って』等のキーワードが同時に含まれる場合」
     「『あります』と『ありません』『ないです』が同時に含まれる場合」
   ```
   ただし、この場合は医学的なリスク分析が必要。

**結論**: 

本Warning は **修正不要な参考情報**です。プロンプトの複合意図ルールは医学的に合理的で、安全性上問題がありません。SKILL_B との微細な乖離は intentional な設計判断として受け入れ可能。

### 移管方針
- 冒頭は opening + 冒頭_アナウンス の2ブロック1セット。冒頭アナウンス直後は入力待ち無しで「受診歴確認」に接続する。
- **診療は眼科専用**（他科受付なし）。用件分類の前に **受診歴確認**（Yes/No）を置く Pattern A 分岐。否定（初診）→ END_初診受付不可 で終話（endpoint=電話案内、status=2 相当、smsFlag=0）。
- 受診歴 Yes → 用件聴取（再診予約 / 予約変更 / 予約キャンセル / 予約確認 の4択）。classification enum は `再診 / 変更 / キャンセル / 確認 / ""` の5値（空文字含む）。
- 氏名／DOB／電話番号は **サブフロー経由**（Jump to Flow）。patientName → PatientName サブフロー、DOB / Phone も既存サブフローを使用。診察券番号は function には存在するが yaml フローでは聴取ステップ無し（director 判断: フローから除外。function enum 定義だけ残して空文字のまま）。
- 連絡先電話番号は Gen3 流儀で **incoming_phone_number 型による3分岐**（mobile / ip_phone+other / anonymous）。mobile のみ「今おかけいただいている番号でよろしいですか？」の Yes/No 確認ブロックを挟み、No / ip_phone / other / anonymous は手動聴取ブロックに合流（Pattern B）。
- 用件別分岐：再診・変更 → 予約希望日のみ、キャンセル → 次回予約有無（Yes/No）→ Yes なら予約希望日＋最後の問い合わせ／No なら最後の問い合わせのみ、確認 → 問い合わせ内容。
- 終話パターン：**予約受付完了**（再診/変更/確認: smsFlag=1）／**キャンセル受付完了**（smsFlag=2）／**初診受付不可**（status=2, smsFlag=0, endpoint=電話案内）／**電話番号確認失敗**（END_回答不可: status=0, smsFlag=0）の4系統を termination_patterns に定義。
- TTS は全て `{tts_g:...}`（小文字）で発話。電話番号復唱・冒頭アナウンス内の代表電話番号読み上げ部分の `<speak type="telephone" breakc="300ms">` タグは元資料の指示どおり **電話番号復唱サブフロー側のプロパティに保持**（メインブロック内には書かない）。
- RAG は元資料に記載なし → 不要。FAQ 的な問い合わせは確認ルートの `inquiry_details` / `question` フィールドに自由文で格納。

### 施設固有の特殊ルール
- **「肺→はい」「いいです→はい」救済**：Yes/No 判定で「肺」「いいです」は肯定扱い、「が今」「PHVます」「P」も肯定扱い（元資料に重複記載あり、否定側の救済語は無し）。Yes/No 系 generate_by_OpenAI の prompt に注記必須。
- **数字 4/5 の誤認識救済**：「C/c/Ｃ/ｃ」「for/ふぉー/フォー/しー/しい/し」→ 4、「ごー/ごお/ご」→ 5。診察券番号・電話番号ブロックの prompt 側で対処（prompter 担当、正規化ルール §1-8）。
- **「診察」キーワード救済**：「印刷／定期／新薬／再診／最新／禁忌」を「診察」として扱う。「薬／お薬／要約／通訳／集約／契約／主役／服薬」を「予約」として扱う。「不審／電気／でんき」を「初診」として扱う。「薬品」を「変更」、「大変」を「再診」として扱う。classification 判定プロンプトに全部埋める。
- **初診 = 受付不可**：「初めて／不審／電気」含有の否定応答 → END_初診受付不可。勤務時間案内文言（午前 8:30-11:30 月〜土、午後 13:45-16:00 月・木）を終話メッセージに含める（原文ママ）。
- **生年月日の復唱文言は指定あり**：「生年月日は{西暦4桁}年{1〜12の月}月{1〜31の日}日でよろしいですか？」形式。月日の先頭ゼロ禁止（「07月」「07日」不可）、YYYYMMDD の8桁や `/` 区切り禁止。DOB サブフロー側の復唱プロパティに明記（prompter 担当）。
- **復唱許可は DOB と電話番号のみ**、他項目（氏名・用件・予約希望日・診療科相当・受診歴等）はいついかなる時も復唱禁止。
- **「もしもし」単独発話**：「お声は届いております。」と案内して直前の質問を再案内。他発話が混ざっている場合は「もしもし」を無視して他内容で判断。各聴取ブロックの共通ルールとして明記。
- **「予約希望日」はフリーワード格納**（日付正規化しない）。`DesiredreservationDate` の description に明記。
- **phone_type 判定**：mobile=`^0[6789]0\d{8}$`、ip_phone=`^050\d{8}$`、other=`^0\d{8,9}$`（yaml に正規表現あり、そのまま incoming_phone_number 分岐に使う）。

### director が見落としやすいポイント
- **classification enum に "再診" は含むが "新規予約" は無い**（眼科は全員再診前提、初診は END_初診受付不可 で弾く）。function enum と分岐の整合に注意。
- **diagnosis_card_number（診察券番号）はフローで聴取しない**が、function の required 配列には入っている。空文字のまま dialogue_completed に渡す設計（director 判断）。
- **status ルール**：必須項目が全て not empty なら 1、代表案内時 2、それ以外 0。受診歴=無 で初診弾きした場合も代表案内扱い → status=2。
- **smsFlag ルール**：status=1 かつ classification が 再診/変更/確認 → 1、status=1 かつキャンセル → 2、それ以外 0。**キャンセル時 smsFlag=2** は他施設と異なる（成田は smsFlag=1、鹿児島生協診療は smsFlag=2）。
- **電話番号 3回失敗で END_回答不可**：「ご回答の確認ができませんでしたのでこちらからお電話失礼させていただきます」で終話。yaml では「リトライ超過時: END_回答不可」と記載（DOB のリトライ超過時は用件別フロー分岐へ進む設計で、項目ごとにリトライ超過時の遷移先が異なる点に注意）。
- **DOB リトライ上限 2**：2回認識失敗で patientDateOfBirth をブランクのままセットし、エラー内容はアナウンスせず次の質問（用件別フロー分岐）へ進む。サブフロー側の失敗遷移をそのように設定。
- **入電者が一度に複数情報を発話**：既に得た項目は再質問・復唱しない。generate_by_OpenAI 全プロンプトに反映（鹿児島生協健診と同様）。
- **代表電話番号案内は原則不使用**（診療は初診の場合のみ勤務時間案内で代表番号 099-267-1455 を案内。冒頭アナウンスにも当日予約用の代表番号案内あり）。
- **冒頭アナウンスに代表番号読み上げが含まれる**：「なお当日の予約に関するお問い合わせは、<speak type="telephone">099-267-1455</speak>へおかけ直しください。」の SSML タグは元資料指示なのでプロパティに保持（通常は SSML 禁止だが施設指示を優先）。
- **receive_date_of_call 系の診療科識別は不要**（眼科固定なので clinicalDepartment は "眼科" を固定値で代入する設計で可。director 判断）。
- **改訂履歴（2025/12/08）**：生年月日の復唱フレーズ指定の追加のみ。他は既存。

### W-002: システム日付（基準日）の付与方式が実装担当者に不明確
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_予約日, OpenAI_受診希望日
- **重大度**: Warning
- **問題**:
  - プロンプト冒頭に「今日はyyyy年mm月dd日です」とシステムが自動付与される旨が明記（line 329-331, 438-440）
  - ただし、この機械的な実装方法（どのスクリプトが、どのタイミングで付与するのか）が未記載
  - prompter が inject_prompts.py / scaffold / orchestrator に依存しているが、その実装との整合性が不明
- **リスク**:
  - 実装担当者が「自分で基準日を挿入する」と誤解する可能性
  - システム実行時に基準日が正しく付与されなかった場合、年の補完ロジック全体が破綻
  - テストで検出できない微細な時差バグが本番で顕在化する可能性
- **修正方針**:
  1. prompter が「基準日の自動付与は inject_prompts.py / orchestrator.py の責務であり、prompter は『今日はyyyy年mm月dd日です』の placeholder のみ記載する」と明記する
  2. README または docs に「プロンプト冒頭の基準日付与の仕様書」を追加する

### W-003: 相対日付「来週」の曜日判定が月曜始まり固定
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_受診希望日（line 493, 496）
- **重大度**: Warning
- **問題**:
  - STEP4（line 487-497）で「来週の月曜 → 基準日が属する週の翌週の月曜」と定義
  - 「週の開始日は月曜」と仮定（line 496「曜日: 月曜=1」）
  - ただし、日本の一部業務アプリでは「週の開始日を日曜」と設定する場合もある
  - 新潟縣健康管理協会の営業内部で「週の定義」が統一されているか不明
- **リスク**:
  - 基準日が日曜日（line 496 で曜日=7）で、ユーザーが「来週の月曜」と言った場合：
    - 月曜始まりなら: 基準日（日）は「今週の日曜」→ 翌週の月曜 = 基準日+1日
    - 日曜始まりなら: 基準日（日）は「来週の日曜」→ その週の月曜 = 基準日+1日
    - 結果は同じだが、中間計算が変わるため、設計書と不整合になる可能性
- **修正方針**:
  1. 設計書 special_notes または プロンプトに「日本標準（月曜始まり）で週を定義する」と明記
  2. テストケース「基準日=2026-05-11（日曜）の時に『来週の月曜』で 2026-05-18 が返されるか」を追加

### W-004: datetime 出力のハイフン区切り仕様が後段フローとの不整合の可能性
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_予約日, OpenAI_受診希望日（line 337, 446）
- **重大度**: Warning
- **問題**:
  - プロンプトで「YYYY-MM-DD 形式で出力」と明記
  - 後段のフロー（echo_back / context_match_router / script）で この形式が処理可能か不明確
  - echo_back で「2026-05-07」を TTS で読み上げると、ユーザーが違和感を覚える可能性がある
  - 設計書 step_details の TTS announcement に「〇月〇日」という形式が記載されているが、echo_back の出力値との整合性が不明
- **リスク**:
  - echo_back で「YYYY-MM-DD」を TTS 読み上げ → 不自然な発話
  - または、scaffold が TTS 用に別形式に自動変換した場合、出力値（YYYY-MM-DD）と TTS（和暦）が不整合になる
- **修正方針**:
  1. 設計書 hearing_items / step_details の echo_back 項目に「echo_back 時の TTS 形式」を明記（例：「2026年5月7日」）
  2. プロンプトコメントで「出力は YYYY-MM-DD だが、echo_back 用の TTS 形式への変換は scaffold/gen_properties.py が処理する」と明記
  3. gen_properties.py が datetime 値を TTS 用に変換する仕様を確認し、テストケースに追加

### W-005: キーワード判定の「部分マッチ」定義が曖昧
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_用件確認（line 109-110, 117-118）
- **重大度**: Warning
- **問題**:
  - 「部分マッチ」を「入力全体が一致する必要はない」（line 109）と定義している
  - 例1: 「健診の話がしたい」→ 「健診」で match → 「健診予約」と判定される（意図通り）
  - 例2: 「予約の取り消し」→ 「予約」（健診予約のキーワード）と「取り消し」（キャンセルのキーワード）の両者で match する
    - line 44「上から優先」により「健診予約」が返されるが、ユーザーの意図は「キャンセル」
  - この場合、複数マッチ時の「上から優先」が本当に正しいのか不明確
- **リスク**:
  - キーワード判定の曖昧な部分マッチが、分岐判定の誤りを引き起こす
  - テストで ambiguous な入力パターンをカバーしていない
- **修正方針**:
  1. 部分マッチの定義をより厳密に記述（例：「形態素解析後の単語単位での部分マッチ」「1つ以上の候補キーワードが含まれる場合」）
  2. テストケースで ambiguous な入力を明記：
     - 「予約の取り消し」→ 期待値: キャンセル or 健診予約？
     - 「企業の会社」→ 期待値: 企業問合せ
     - 「質問があって、予約したい」→ 期待値: 上から優先で「健診予約」
  3. 複数マッチ時の「上から優先」ロジックがユーザー体験として妥当か、UX レビューを別途実施

### 2-3: NO_RESULT 返却条件の相違 ⚠️ **C-001**

#### 設計書の指示（step_details, 699-701行）
```yaml
no_result_condition: |
  無発話（5秒以上の沈黙）。OpenAI が分類を確定できない場合は no_result とせず、
  「その他」に倒してデフォルト分岐に流す（Gen2「デフォルト: 氏名聴取」を踏襲）。
```

#### プロンプトの実装（327-333行）
```
## 【STEP4：それ以外】
以下は NO_RESULT：

- 無音
- フィラーのみ（例：えー あのー のみ）
- 意味不明
- 文字化け・記号のみ
```

#### 問題点
- **プロンプト**：「意味不明」を NO_RESULT に分類
- **設計書**：「OpenAI が分類を確定できない場合は no_result とせず、『その他』に倒す」と指示

分類不確定な発話（「ちょっと複雑な内容」「複数の要件を混ぜた発話」等）が、プロンプトでは NO_RESULT に落ちる可能性。設計書の意図は「分類失敗は その他 に倒して次段 jump_氏名聴取 へ流す」であるが、プロンプトが NO_RESULT を返すと、IVR フロー上で「再聴取」に戻るため、UX が異なる。

### 3-2: 複合マッチ時の処理信頼性 ⚠️ **W-002**

#### 文言
- STEP3（224-226行）：「※複数一致した場合は上から優先」
- 絶対ルール（204-205行）：「上から順に機械的に評価」「一度一致したら即終了」

#### リスク
OpenAI の実装において「上から優先」がプロンプト文言で**確実に実装されるか不確実**。

例：複数キーワードマッチ入力「予約を変更したいし、空き状況も知りたい」→
- プロンプト設計では「予約変更」（229-239行が先）を返すはず
- ただし OpenAI が複数マッチを検出した場合、「複数候補を列挙する」「判定を保留する」等のリスク

**実装検証は tester.py の 4 本柱チェック（複合意図テスト）で後段確認が必要。**

## 検出事項

### C-001: OpenAI_内容確認_詳細 の NO_RESULT 条件が設計書と相違
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_内容確認_詳細
- **問題**: 
  - プロンプト STEP4 では「意味不明」を NO_RESULT に分類
  - 設計書 step_details では「OpenAI が分類を確定できない場合は no_result とせず、『その他』に倒す」と指示
  - 分類不確定な入力が NO_RESULT で返却されると、IVRフロー上で再聴取になり、設計書の意図「その他として jump_氏名聴取 へ流す」と矛盾
- **修正方針**: 
  1. STEP4 を削除（「その他」判定で全てを吸収）
  2. STEP2 の STEP3 の語句一致判定で、9 種いずれにも該当しない**有意な発話**を「その他」に分類
  3. 無音・フィラーのみを NO_RESULT として分離する（STT 段階で既に無音判定されているため、プロンプトレベルでは「意味のある発話が来る前提」）
  4. または設計書を修正して「分類不確定は NO_RESULT を返す」に統一

- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_診療科
- **重大度**: Critical
- **問題内容**:
  
  プロンプト (19-72行) では「出力仕様（厳守）」セクションで **NO_RESULT** を診療科名と同列に列挙しています：

  
  しかし設計書側では：
  - `hearing_items.output_labels` (465-516行): 50科 + 登録なし のみ（**NO_RESULT なし**）
  - `step_details.openai_rules.output_values` (589-640行): 50科 + 登録なし のみ（**NO_RESULT なし**）
  
  実装上、NO_RESULT は OpenAI の内部判定フラグであり、Brekeke フロー側で `saveDefault(value="登録なし")` に変換されるため、IVRプロパティレベルでの運用上影響は限定的です。しかし、**プロンプト仕様書と設計書の整合性が失われており、保守性・可読性が低下**します。
  
  二層構造の混同を避けるため、プロンプト側で明確に分離すべきです。
  
- **修正方針**:
  
  以下のいずれかを実施（推奨: (A)）:
  
  - **(A) 推奨**: プロンプトの「出力仕様」セクション (18-74行) を以下のように分割・明確化:
    ```markdown
    # 出力仕様（厳守）
    以下のいずれか**1語のみ**を出力すること：
    
    - 内科
    - ... (50科)
    - 登録なし
    
    ---
    
    # 判定不能時の内部フラグ
    上記いずれにも該当しない場合は `NO_RESULT` を出力します。
    このフラグは Brekeke フロー側で自動的に `登録なし` に変換されます。
    ```
  
  - **(B) 代替案**: プロンプト 72行の「NO_RESULT」を削除し、STEP6セクション (284-291行) でその位置づけを明確に記載:
    ```markdown
    ## 【STEP6：それ以外】
    以下の場合は、プロンプト終了時に `NO_RESULT` を内部フラグとして返します。
    Brekeke フロー側で `saveDefault(value="登録なし")` に自動変換されてリトライされます：
    ```

### W-001: 「分からない」判定パターン網羅性確認

- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_診療科 (STEP2)
- **重大度**: Warning
- **問題内容**:
  
  STEP2 では 10 パターンの「分からない」表現が定義されています (107-119行)：
  ```
  わからない、分からない、わかりません、分かりません、
  解らない、解りません、不明、決まっていない、決まってない、まだ決めていない
  ```
  
  これは基本的に十分ですが、以下のバリエーションの検討が不足している可能性があります：
  - 「よくわかりません」（「わかりません」の拡張形）
  - 「確実ではない」「はっきりしない」（否定的な判断）
  - 「まだ考えてない」「未定」（決定保留）
  - 「何科がいいかわからない」（疑問形での不確実性表現）
  
  ただし、AmiVoice STT の前処理（amivoice_dictionary）で「わからない」「分からない」の読みが登録されているため、発音ゆれには対応しています。
  
  音声認識結果に「わかりません」「分かりません」が含まれれば STEP2 で捕捉されるはずです。
  
- **修正方針**:
  
  - 設計書の `customer_doc`（参照資料「【診療1】：医誠会国際総合病院.md」）を確認し、実際の通話ログや想定ユーザー層から「分からない」の典型的な言い回しを抽出
  - 必要に応じて STEP2 のパターンリスト (107-119行) に以下を追加検討：
    ```
    - よくわかりません
    - ちょっとわかりません
    - はっきりしない
    - 決めてない
    - 何科かわからない
    ```
  - 逆に、「解らない」「解りません」のような古い漢字表記は、現代日本語での使用頻度が低いため削除を検討（ただし高齢者対応のため保持推奨）

     ```
     - 脊髄 → 脊椎脊髄外科
     ```

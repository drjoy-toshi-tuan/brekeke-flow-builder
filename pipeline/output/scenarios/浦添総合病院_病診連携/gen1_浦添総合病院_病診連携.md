### 移管方針
- main.yaml の流れを 4 ブロック中心で再構成: opening → 施設名 (subflow) → 患者氏名 (subflow) → 生年月日 (subflow) → 用件分類 (hearing) → 用件別分岐 (CMR) → その他問合せ (hearing) → 受付完了アナウンス → termination
- 入電情報あり (API_DRJOY) → 施設名既知なら施設名サブフローをスキップする分岐を opening 直後に配置
- 緊急性チェックは用件分類前に置き、緊急判定なら代表案内（call_transfer ではなく案内+切断）へ即誘導
- 用件4分類: `予約依頼` `情報提供依頼` `FAX到着確認` `入院・転院依頼`。発話＋DTMF (1/2/3) の両対応
- 完了アナウンス前に `<? $発話用DATE_NOW ?>` 相当の現在日時読み上げ → addCurrentDate Yes（scaffold 自動付与）

### 施設固有の特殊ルール
- **入院・転院依頼は AI 受付対象外**: 用件分類で `入院・転院依頼` が選ばれた場合、ヒアリング前に専用の代表案内 (`代表案内_1`) で電話番号を3回繰り返して即終話
- **問い合わせ①の質問内容**: 用件 = `情報提供依頼` の場合「入院中と通院中、どちらの患者様になりますか？」を hearing → `情報提供依頼` の場合は別途「資料はいつまでに必要か」も hearing（問い合わせ②）。2 段ヒアリング
- **FAX到着確認の終話文言**: `到着が確認できない場合は翌診療日午前中までに折り返しご連絡いたします` を専用 termination 文言として保持
- **代表案内の電話番号3連呼**: 終話案内・代表案内_1 とも `0、9、8、の、8、7、9、の、0、6、3、0、です` を**3回繰り返す**TTS（聞き逃し対策）。文言を縮めない
- **代表案内_2 のみ電話番号 2回繰り返し**: 用件 NG 系 (生年月日 NG / 氏名 NG / 問い合わせ NG / 当日 NG) の合流先。3回ではなく 2回に短縮
- **当日予約日確認サブフロー**: 用件 = `予約依頼` の直後に「当日予約のご希望、または早急な対応が必要ですか？」を Yes/No hearing。Yes なら代表案内へ転送（AI 受付外）、No なら通常の受付フローへ
- **完了フラグの日付読み上げ**: `かしこまりました。<受付日時>。` の形で受付完了を示す TTS。addCurrentDate 必須
- 受付対象外用件の合流先は `代表案内` ブロック (用件 NG 等) と `代表案内_1` (入院転院) の 2 系統あるので分けて termination 化

### director が見落としやすいポイント
- main.yaml に `テスト用分岐 / 藤田さん用 / 下僕用` 等のテスト箱があるが**移管対象外**（無視）
- `現在の時間取得 / 現在の時間判断 / 15時前` 等の時刻判定は dead code（acceptance_times 集約に吸収、ノートに反映しない）
- `終話案内 / 終話案内 ～5/1 / 終話案内 5/1～` の 5/1 受付時間切替分岐は acceptance_times 側責任 → ノートに書かない（5/1 以降版の文言のみ採用: 月-金 9-17、土 9-12）
- 用件正規化辞書 (entity_classification.md) は OpenAI 同義語が大量。`予約以外` `予約変更` `予約確認` `予約キャンセル` 等の派生は OpenAI 正規化で `予約依頼` 等メイン4分類に集約させる（4 種以外を返させない）
- 問い合わせ①の hearing は「入院中／通院中」の二択 enum、問い合わせ②は自由テキスト（「資料はいつまでに」期日）→ 出力形式が異なる
- 緊急性チェックの分岐先 `緊急の方` は実質 dead end（main.yaml に `END_緊急` の遷移先がない） → 緊急 = 代表案内へ転送する設計に整える

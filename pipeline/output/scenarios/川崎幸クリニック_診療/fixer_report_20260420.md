### C-001: Context プレースホルダが未置換のまま埋め込まれている

- **修正担当**: prompter <prompter>
- **対象モジュール**: OpenAI_用件確認、OpenAI_現在の予約日
- **重大度**: **Critical**
- **問題詳細**:
  - OpenAI_用件確認 line 11:
    ```
    「TODO_要確認（B-5: customer_doc 未納入）」
    ```
  - OpenAI_現在の予約日 line 145:
    ```
    「TODO_要確認（B-5）」
    ```
  
  プロンプトの `Context` セクション内に、設計書の未確定マーカー `TODO_` が埋め込まれたまま。  
  このままでは、OpenAI API に送信されたとき、ユーザーの音声入力の分類/正規化が **正しく機能しません**。
  
  実際には `inject_prompts.py` や scaffold の動的コンテキスト注入フェーズで、この `TODO_要確認（B-5）` が **実際の TTS 案内文言** に置き換わるはずですが、**prompter が作成したサイドカー MD の段階では、プレースホルダのままになっている状態です**。

- **修正方針**:
  1. prompter は、設計書の `tts_announcement` または `step_details` セクションから実際の案内文言を取得し、プロンプト内の `Context` を具体的文言に更新する
  2. 設計書 line 371・392 で `tts_announcement: "TODO_要確認（B-5: customer_doc 未納入）"` と記載されているため、実際のテスト時には `generator/scaffold` 側で動的に置き換わる想定だが、**prompter の段階で明示的に「このプレースホルダはシステム実行時に [例: 用件の問いかけ文言] に置き換わる」ことを記載するか、または仮の具体例を示すべき**

- **参考**: CLAUDE.md の prompter 役割定義では「block_mapper でモジュール名 → ブロック名にマッピング」「設計書と prompter リファレンスに基づいて params.prompt を記述」と記載されているため、この TODO_ 置換が自動化されていない可能性があります。

### W-001: 過去日付の明示的な排除ルール欠落

- **修正担当**: prompter <prompter>
- **対象モジュール**: OpenAI_現在の予約日
- **重大度**: **Warning**
- **問題詳細**:
  
  設計書 `step_details` line 402:
  ```
  no_result_condition: "日付として解釈不能（過去日付・非日付発話）"
  ```
  
  プロンプト STEP2（lines 196-210）では、相対日付として「昨日」「一昨日」の計算ルールが記載されています：
  ```
  - 「昨日」「きのう」 → 基準日 - 1日
  - 「一昨日」「おととい」 → 基準日 - 2日
  ```
  
  しかし、プロンプト STEP5（NO_RESULT 判定, lines 250-261）には、**計算結果が過去日付の場合に NO_RESULT を返す条件が明記されていません**。
  
  つまり、ユーザーが「昨日をキャンセルしたい」と発話した場合：
  1. STEP2 で計算実行 → 過去日付（例: 2026-04-19）が生成される
  2. STEP4 で書式変換 → `2026-04-19 00:00` として出力される
  3. 次の `script(future_date)` ブロック（設計書 line 148-149）で「当日判定」が実行される際、基準日より小さい日付が渡される
  
  **結果**: 過去日付がシステムに流入し、当日判定ロジックで予期しない分岐が発生するリスク

- **修正方針**:
  
  STEP3 の末尾または STEP4 の後に、以下を追加：
  ```
  ## 【STEP5.5：過去日付チェック（新規追加）】
  
  計算・変換が完了した日付について、以下を確認する：
  - 抽出された日付が基準日（今日）より過去の場合 → NO_RESULT
  
  理由: 予約キャンセルは未来日付のみが対象。過去日付は「誤入力」または「実装エラー」と判定し、
  ユーザーに再聴取を促すため。
  ```
  
  または STEP4（書式変換）直後に、以下の検証ステップを挿入：
  ```
  ## 【STEP4.5：日付妥当性チェック】
  
  計算結果の日付が基準日（今日）以上か確認。
  基準日より過去 → NO_RESULT
  基準日と同日以上 → STEP5 へ
  ```

### I-001: システム日付の「自動付与」メカニズムの詳細説明欠落（情報提供のみ）

- **修正担当**: — （Info のため修正不要）
- **対象モジュール**: OpenAI_現在の予約日
- **重大度**: **Info**
- **問題詳細**:
  
  プロンプト line 154:
  ```
  プロンプトの冒頭に「今日はyyyy年mm月dd日です」とシステム実行日（現在日＝着信日）が自動で付与されます。
  ```
  
  ここで言う「自動で付与」の具体的なメカニズム（どのエージェント・スクリプトが実行時にどのように注入するのか）が不明確です。
  
  これはおそらく `inject_prompts.py` または scaffold の `generate_by_OpenAI` モジュール側で、実行時に基準日を動的に挿入する想定だと思われますが、**プロンプト本体内に「基準日の形式は常に YYYY-MM-DD 形式である」等の safeguard が記載されていません**。

- **現在の状態**: 
  - ✅ プロンプト内ではシステム日付を「基準日」として正しく使用
  - ✅ 「基準日そのものを出力してはいけない」という防御も明記
  - ⚠️ ただし、実行時注入メカニズムの詳細が不透明

- **推奨方針** (実装は任意):
  
  ```
  # システム日付取り扱いに関する絶対ルール（修正：冒頭に形式を明記）
  
  実行時に、このプロンプト冒頭に以下の形式で基準日が挿入されます：
  「今日は YYYY-MM-DD（例: 2026-04-20）です」
  
  この基準日は以下の目的でのみ使用：
  - ユーザー発話の相対日付（「明日」「来週」等）の計算
  - ユーザーが月日のみ指定した場合の年の補完
  
  基準日そのものを出力してはいけません。
  ```
  
  これにより、実行時注入側が「この形式で挿入してください」という契約が明確になります。

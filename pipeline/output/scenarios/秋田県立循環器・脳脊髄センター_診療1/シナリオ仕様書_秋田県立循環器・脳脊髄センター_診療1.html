🔍 シナリオの目的

患者からの電話を受け、用件（予約変更・予約キャンセル・予約日時確認）を確認し、診療科・検査・予約日等の業務情報と患者個人情報（氏名・生年月日・診察券番号・連絡先電話番号）を聴取して折り返し受付を自動化する。救急科は代表電話番号を案内して終話する。連絡先が携帯の場合はSMS案内、固定電話の場合は電話案内の終話文言を使い分ける。

        → ★サブフロー遷移_氏名

─── キャンセルルート固有 ───
キャンセル理由_聴取(TTS) → 入力_キャンセル理由(DTMF AmiVoice STT, max_dtmf_length=1) → OpenAI_キャンセル理由(classify)
    → ★サブフロー遷移_氏名

─── 個人情報サブフロー遷移チェーン（3ルートの合流点）───
★サブフロー遷移_氏名 = Custom Jump to Flow → 氏名聴取サブフロー（.*で1本受け）
    → Custom Jump to Flow → 生年月日聴取サブフロー（.*で1本受け）
        → Custom Jump to Flow → 診察券番号聴取サブフロー（.*で1本受け）
            → Custom Jump to Flow → 電話番号聴取サブフロー（結果: 1=携帯, 2=その他）
                → Custom Jump to Flow → RAG検索サブフロー（.*で1本受け）

─── smsFlag分岐 + 用件別終話 ───
RAG検索結果返却後
    → ContextMatchRouter_携帯固定分岐 (参照: JumpToFlow_電話番号聴取, condition: 1=携帯/2=その他)
        ├─ 1(携帯) → 完了フラグ_受付完了_携帯(saveCompletionFlag2db, status=1, smsFlag=1)
        │               → ContextMatchRouter_用件別終話_携帯 (参照: OpenAI_用件確認)
        │                   ├─ 変更 → END_変更_携帯(TTS) → 切断
        │                   ├─ キャンセル → END_キャンセル_携帯(TTS) → 切断
        │                   └─ 予約日時確認(default) → END_確認_携帯(TTS) → 切断
        └─ 2/その他(default) → 完了フラグ_受付完了_固定(saveCompletionFlag2db, status=1, smsFlag=-1)
                                 → ContextMatchRouter_用件別終話_固定 (参照: OpenAI_用件確認)
                                     ├─ 変更 → END_変更_固定(TTS) → 切断
                                     ├─ キャンセル → END_キャンセル_固定(TTS) → 切断
                                     └─ 予約日時確認(default) → END_確認_固定(TTS) → 切断
💡 - 3ルート（予約日時確認は診療科後に直接個人情報へ、変更/キャンセルは業務固有聴取を挟む）が個人情報サブフロー遷移チェーンで合流 - smsFlag分岐は電話番号サブフロー結果（1=携帯, 2=その他）で2段分岐し、さらに用件で3分岐 - 合計10終話パターン: 携帯3 + 固定3 + 救急科案内 + 聴取失敗 + 非通知 + 時間外

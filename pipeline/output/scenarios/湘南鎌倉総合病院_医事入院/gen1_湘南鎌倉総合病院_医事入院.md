### 移管方針
- Gen1「箱」構造 → Gen3 9 ブロック型に変換。主な対応:
  - `挨拶`（callback 説明 TTS）→ opening + 冒頭_アナウンス（2 ブロック 1 セット）
  - `API_DRJOY_CONTEXT_SETTINGS` / `API_DRJOY` → opening 標準処理に吸収（独立ブロック化不要）
  - `診察券番号聞き取り`（サブyaml）→ Gen3 標準 PatientID サブフロー 1 枠。**ただし冒頭で「当院でのご受診は初めてですか？」Yes/No 分岐あり**（後述「特殊ルール」）
  - `患者氏名聞き取り` / `生年月日聞き取り` / `連絡先電話番号聞き取り` + `電話番号区切り` → 各 Gen3 標準サブフロー（PatientName / DOB / PhoneNumber）1 枠ずつ
  - `問い合わせ内容①聞き取り`（サブyaml）→ hearing ブロック（自由文、output_format=text）
  - `追加用件有無確認`（サブyaml、Yes/No）→ hearing（yesno）→ ContextMatchRouter で `Yes → 問い合わせ内容②` / `No → 通話完了` 分岐
  - `問い合わせ内容②聞き取り`（サブyaml、追加用件あり時のみ）→ hearing ブロック（自由文）
  - `通話完了` → announcement（折返し案内）→ `終話` → IVR Disconnect
  - `3回聞き取りに失敗した場合` → termination 集約ブロック（代表電話再架電案内）
- フロー骨格: opening → 冒頭_アナウンス → 診察券有無確認（初診Yes/No）→ [診察券番号 subflow] → 患者氏名 subflow → 生年月日 subflow → 連絡先電話番号 subflow（携帯追加聴取込み）→ 用件①hearing → 追加用件有無 → (Yes時)用件②hearing → 通話完了 announcement → 終話
- **本フローに用件カテゴリ分岐は無い**（用件は記録のみ、カテゴリ別ルーティング無し）→ Pattern A/B 不要、scenario_flow は基本直線

### 施設固有の特殊ルール
- **callback ボット**: opening TTS が「医事課入院係をお呼び出ししましたが、回線が混みあっておりお繋ぎできませんでした。AI電話がご用件をお伺いし、3診療日以内に担当者から折り返し電話連絡いたします。」→ 院内代表（医事課入院係）が出られなかった電話を AI が一次受けする callback シナリオ。**通常 inbound と異なり acceptance_times / call_transfer / date_of_call_classifier 不要**（病診連携と同じ）
- **call_transfer なし**: 転送箱なし。call_transfer ブロック作成禁止
- **date_of_call_classifier なし**: 時間帯 / 曜日 / 年末年始の分岐箱が一切無い。受付時間判定ブロック生成しない
- **診察券有無の冒頭分岐**: 診察券番号聞き取り.yaml `用件を判断` 箱で「新規予約 → 診察券確認へ１」/ それ以外 → 通常聴取の二段構造。さらに `診察券確認へ２` で「当院でのご受診は初めてですか？」Yes/No を聞き、Yes→診察券無し（番号聞かずスキップ）、No→診察券番号聴取へ進む。**Gen3 では PatientID サブフロー前段に「初診ですか？」hearing(yesno) + ContextMatchRouter で `Yes → 連絡先電話番号 subflow へジャンプ（診察券スキップ）` / `No → 診察券番号 subflow` を配置**
- **2 段用件聴取**: 問い合わせ内容①と② が別々の hearing で、間に「その他、追加のご用件はございますか？」の Yes/No（`追加用件有無確認`）を挟む。**用件②は Yes の時のみ実行**。director は 2 つの hearing を別 context（`yoken_1` / `yoken_2`）に保存し、CMR で yoken_2 をスキップ可能にする
- **診療科 entity（48 件）は実フロー未使用**: main.yaml にエンティティ定義はあるが用件聴取は自由文 text のみで診療科分類してない → R1 48 件は STT 辞書に積まなくてよい（director 判断、デフォルトは積まない、もしくは追加用件②の自由文補助として use_template:hearing_shinryoka_basic を 1 個だけ参照）
- **診察券番号聞き取り不可** 箱（診察券番号聞き取り.yaml l.273）→ アナウンス「大変申し訳ございません、診察券番号を聞き取ることができませんでした。後ほど、担当者にて確認させていただきます。」→ **次フェーズ（氏名 subflow）へ進む**（切断ではない、callback なので最低限の情報を後続でも取りに行く）。PatientID subflow の retry 失敗 next 先として配置
- **電話番号聞き取り不可** 箱（連絡先yaml l.147 「繋ぎ」経由のダイヤルプッシュ案内 + END_電話番号聞き取り不完了）→ 「後ほど担当者にて確認」announcement → 用件①hearing に進める（callback 性質上、最低限用件は取る）

### director が見落としやすいポイント
- **携帯電話番号 2 段聴取**: 連絡先yaml l.397–468 で固定電話確認後に「緊急時の連絡方法として、携帯電話の番号をお伺いしています。携帯電話をお持ちの場合は、番号をおっしゃってください。お持ちでない場合は『持っていない』と…」 → 病診連携と同じく **PhoneNumber subflow 後段に追加 hearing or 携帯用 PhoneNumber subflow 2 連呼びが必要**。director 判断、推奨は subflow 後段に hearing(text/持ってない判定) 追加
- **診察券有無確認サブyaml は重複機能**: `追加用件有無確認.yaml` ファイル名に反して、中身は「**その他、追加のご用件はございますか？**」という追加用件 Yes/No 確認（命名ミス）。診察券有無の判定ではない。director はファイル名に騙されないこと。ファイル冒頭 `診察券有無確認` 箱もセリフが「その他、追加のご用件はございますか？」で内部用語が混乱している → **Gen3 では「追加用件 Yes/No」hearing として素直に作る**
- **問い合わせ内容① と ② は中身が完全同一の yaml**（diff ゼロ、用件聴取の同一テンプレを 2 回呼んでいるだけ）→ Gen3 では 1 つの hearing ブロック定義を context 名違いで 2 回配置でよい
- **新規予約エンティティの使い道**: main.yaml の「新規予約」値（取りたい/取得/初診/紹介状/診察の予約/診療予約）は **用件①の自由文 hearing で「新規予約系の言及があれば診察券有無分岐をスキップして初診扱い」みたいな用途では使われていない**。診察券yaml の `用件を判断` 箱内でしか参照されず、しかも `用件を判断` への入力経路が main.yaml 側で曖昧（直前の 問い合わせ内容① の context を読む可能性）。**director 判断**: シンプルに「初診ですか？Yes/No」を素直に hearing で聞く設計を推奨（Gen1 の用件→診察券分岐の暗黙連動は移管しない）
- **生年月日の和暦/西暦判定 + 復唱**: 生年月日yaml は和暦（大正/昭和元年/昭和/平成元年/平成/令和元年/令和）で分岐し復唱セットを切替えているが、Gen3 の DOB サブフローはこれを内蔵 → director は subflow 完コピで OK、和暦処理を自前で書かない
- **電話番号復唱の入電番号利用**: 連絡先yaml l.334「ご連絡先のお電話番号は、今おかけいただいている <入電番号> でよろしいですか？」分岐あり → PhoneNumber subflow 標準で吸収。director は外側で入電番号 context を渡すだけ

### 移管方針
- **シナリオ名は「連携室」だが、実態は患者本人向けの診療予約フロー**。入電者氏名・医療機関名（organizationName）等の連携室特有の聴取項目は**無し**。冒頭アナウンスも「恵佑会札幌病院の、診察予約申し込み電話です」で患者直通（元資料 L170）。
- PatientName サブフロー充当：**患者氏名をそのまま patientName**（氏名聴取 L410-415）。入電者氏名・施設名の枠は使わない。
- 冒頭は opening + 冒頭_アナウンス の2ブロック1セット。冒頭直後は当日翌日判定（Yes/No: Pattern A）。否定＝本日＝代表案内で終話（011-863-2101、status=2/smsFlag=0）。
- 用件分類（classification）は**4択 DTMF 対応**：1=予約 / 2=変更 / 3=キャンセル / 4=その他問い合わせ。classification サブフローを使う想定（director 判断：4択の Pattern A 分岐）。
- 予約フローは更に **受診歴（新規/再診）→ 紹介状有無 → 紹介時/初診/再診の診療科聴取** と2段分岐。再診は DesiredreservationDate と診察券番号を挟む。
- **歯科口腔外科の横断遮断**：紹介時診療科聴取・初診診療科聴取・再診診療科聴取いずれの分岐でも「歯科口腔外科」含有 → 専用電話案内（050-1725-8755）→ status=2/smsFlag=0 で終話。context_match_router で共通ルート化。
- 氏名・生年月日・電話番号・診察券番号はすべて**サブフロー経由（Jump to Flow）**。メインにインライン展開禁止。
- 生年月日のみ**復唱確認あり**（「生年月日はyyyy年m月d日でよろしいですか？」）。他項目は復唱禁止（L48-51）。DOB 再聴取は DTMF 付きで1回のみ。
- 電話番号は **incoming_phone_number による3分岐**（mobile=060/070/080/090、landline=050等、anonymous/withheld）。mobile のみ「今おかけいただいている番号でよろしいですか？」の Yes/No 確認を挟み、No/landline/anonymous は聴取ブロックに合流（Pattern B 合流）。10/11桁チェック＋DTMF 再聴取はサブフロー内に閉じる。
- RAG は元資料に記載なし → 不要。「問い合わせ内容」「その他確認事項」は reason / others に自由文格納、FAQ 対応不可時は定型文「担当からの折返しの際に回答をさせていただきます」。
- 終話パターンは 4系統 × 携帯/固定の2系統 = 8ルート：
  - 予約/変更/その他問い合わせ終話: status=1/smsFlag=1
  - キャンセル終話: status=1/smsFlag=2（**smsFlag=2 はこの施設固有**）
  - 代表案内（当日・歯科転送）: status=2/smsFlag=0
  - 途中切断/未完了: status=0/smsFlag=0
- TTS は全て `{tts_g:...}` 小文字。`<speak type="telephone">` `<dtmf digit="1"/>` `<dtmf2/>` `<dtmf2 digit="8"/>` タグは元資料どおり保持（電話番号案内・DTMF 受付プロパティ）。

### 施設固有の特殊ルール
- **歯科口腔外科は全聴取フェーズで検出即終話**（L20-22）。診療科聴取3種＋問い合わせ内容聴取・その他確認事項でも含有検出 → 050-1725-8755 案内。どのフェーズからでも END_歯科転送 に合流する共通ルート。
- **smsFlag=2（キャンセル専用）**：他施設の0/1と異なる値。prompter/properties で終話プロパティ作成時に注意。
- **「歯」単体も歯科扱い**（yaml L881-883、「歯科口腔外科」「歯科」どちらの含有でも転送）。キーワードマッチの粒度を誤らない。
- **数字誤認識救済**：'C'→'4'、「"for"」→4、「"号"や"go"」→5。診察券番号・電話番号・生年月日のプロンプトに反映。
- **類義語救済（多数）**：「慶友会」→恵佑会／「写真」「そっち」→初診／「電車券」→診察券／「大地震」→受診／「iPhone」→ハイフン／「救急」→99（電話番号）等。診療科キーワードも誤認救済多数（L200-267）。prompter 側で SKILL_C/D 系プロンプトに注記。
- **肯定語の拡張**：「肺」→ 肯定。Yes/No 判定 generate_by_OpenAI の全プロンプトに必須。
- **当日翌日確認は「明日以降＝はい／本日＝いいえ」と逆転ロジックではない通常形**。ただし分岐先がそれぞれ「用件確認」「当日案内で終話」であるため、director は回答振り分けを慎重に。
- **名前再聴取は1回のみ、かつカタカナ判定未達の場合のみ**（L414）。他項目の再聴取はロジック異なる。
- **予約希望日と予約日は別物**（元資料 L322-328、恵佑会札幌も同ルール）：予約＝新規/再診は DesiredreservationDate（フリーワード）のみ聴取、変更フローのみ reservationDate（日付正規化）+ DesiredreservationDate の両方を聴取。
- **その他確認事項の再聴取は1回のみ**（ヘッダコメント L5、改訂履歴 2026/03/15）。他にお問い合わせ有の場合、次の1回で聴取→即次フロー強制遷移。
- **改訂履歴（2026/03/15）**：(1) その他確認事項の再聴取は1回のみ (2) 生年月日聴取で復唱確認するよう変更。既に本文に反映済みだが、本移管でも遵守。

### director が見落としやすいポイント
- 『**シナリオ名＝「連携室」だが実態は患者向け診療予約**』：L10「地域連携室**診療予約**AI電話」、L170「診察予約申し込み電話」。一般的な地域連携室（医療機関→医療機関）の構造と誤解しないこと。入電者氏名・医療機関名の聴取項目は存在しない。
- 『**どの聴取項目でも「歯科口腔外科」に関する発話内容を聴取した際は、status=2、smsFlag=0**』（L20）：診療科聴取だけでなく**全項目で横断的に検出**しなければならない。問い合わせ内容・その他確認事項も対象。
- 『**予約日と予約希望日は別なので、予約日のヒアリングがなかった場合に予約希望日の中身を予約日に入れてはいけない**』（L323）。
- 『10月以降で1月から9月の日付を言われたときは、翌年にする』（L324）。年度判定ルール必須。
- 『**連絡先電話番号初回聴取および再聴取では、FAQの内容を一切参照しないでください。入力された数字の桁数のみを判定根拠とすること**』（L431）。電話番号聴取サブフロー内 prompt に注記必須。
- 『**「診察券番号」「生年月日」「連絡先電話番号」の聴取フェーズにおいては、入力内容をFAQと照合してはならない**』（L80）：各聴取サブフロー内で FAQ/RAG 参照禁止を明記。
- 『**勝手に「その他問い合わせ終話アナウンス」を流して終了することを固く禁じます**』（L465）：問い合わせ内容聴取後は必ず診察券番号聴取へ進むこと。途中で終話しない。
- 『**入電者が一度に複数の必要情報を発話した場合、すでに伝えられた内容については絶対に復唱・確認を行わず、不足している項目のみを一つずつ質問する**』（L331）：generate_by_OpenAI の全プロンプトに反映。
- 『**checkpoint / smsFlag / status は内部フラグ。AI は絶対に発話しない**』（L17）：TTS 文言生成時の混入防止。
- 終話時 checkpoint の扱い：代表案内・歯科転送は **checkpoint=0** のまま status=2（L33-38 の表）。予約・変更・その他問い合わせ・キャンセル完了時のみ checkpoint=1。

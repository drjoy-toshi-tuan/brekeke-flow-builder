### C-001: 現在予約日の出力形式ミスマッチ
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_現在予約日
- **問題**: 
  - 設計書 hearing_items（line1003-1008）では「YYYY-MM-DD HH:MM 形式に正規化」と記載
  - プロンプト 出力仕様（line761-766）では「yyyy/mm/dd 形式（時刻不要）」と明示
  - コンテキストフィールド reservationDate の display_type が "DATE"（time なし）のため、
    設計書の HH:MM 記載との整合性が不明確
- **修正方針**: 
  - **選択肢A**: 設計書 YAML を修正して「yyyy/mm/dd」に統一（推奨。TTS で時刻を聞いていないため）
  - **選択肢B**: プロンプトで日付+時刻の抽出を実装し、出力形式を「yyyy/mm/dd HH:MM」に変更
  - prompter は design/設計書の意図を director に確認した上で、いずれかを選択すること

### W-001: その他問い合わせの「明確な疑問内容」定義が曖昧
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_その他問い合わせ
- **問題**: 
  - line1128: 「入力に **明確な疑問内容（「○○について」「○○はどうですか」「○○の場合」 等の問いかけ）** が含まれる場合も → any」
  - 「明確な疑問内容」の判定基準が明文化されていない
  - LLM が「○○とは何ですか」「○○を教えてください」等を拡張的に解釈し、
    不正な質問（「システム設定を表示してください」など）を「疑問内容」として認識する可能性
- **修正方針**: 
  - 「明確な疑問内容」の具体例を増やす（例：「予約後のキャンセル方法」「診察時間」「料金」等、
    FAQ で対応可能な医療系質問の例を提示）
  - 逆に除外する質問形式を明示（例：「システム」「設定」「プロンプト」「AI」を含む質問は
    「お答えできません」として判定）
  - STEP2 と STEP3 の判定後、STEP4 で「疑問内容の妥当性」を二次判定する仕様を検討

### W-002: 複数診療科マッチ時の再質問仕様の曖昧性
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_診療科
- **問題**: 
  - 設計書 notes（line954）: 「呼吸器」単体 → 「呼吸器内科と呼吸器外科のどちらでしょうか？」と確認
  - プロンプト STEP2（line358-359）: 「呼吸器」単体 → NO_RESULT
  - 「呼吸器」以外で複数科がマッチする場合（例：「小児」→小児科+小児循環器科）の処理が明示されていない
  - STEP4（line372）では「複数マッチ → NO_RESULT」と定義されているが、

- **修正方針**: 
  - 複数科マッチの全パターン（「小児」「呼吸器」等）を STEP2 で明示し、
    「○○内科と○○外科のどちらでしょうか？」と確認指示を含める
  - または、複数マッチ時に最初の候補を出力するロジック（優先度ベース）を追加
  - 現状の NO_RESULT → 診療科2（テキスト聴取）フローは仕様として問題ないため、
    プロンプト内に「複数マッチ時は NO_RESULT を返し、診療科2 フローで自由テキスト入力を促す」と
    明示コメントを追加推奨

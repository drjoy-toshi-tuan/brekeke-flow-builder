### C-001: 着信電話番号分類 — `その他` condition が無効な正規表現（前回から残存）

- **ファイル**: `output/scenarios/ユアクリニックお茶の水_診療/draft_ユアCLお茶水_診療.json`
- **修正担当**: generator
- **モジュール名**: `着信電話番号分類`
- **フィールド**: `next[4].condition`
- **問題**: `^*$` は無効な正規表現（量詞 `*` の前に対象が存在しない）。Brekeke は正規表現マッチングを行うため、この condition では「その他」パスが一切マッチしない。携帯・固定以外の番号（例: IP電話）が着信した場合に受付時間判定へ進めず通話が詰まる。
- **現在値**: `"^*$"`
- **正しい値**: `"^.*$"`
- **修正指示**: `着信電話番号分類.next[4].condition` を `"^.*$"` に修正すること。他のモジュールには一切触れないこと。
- **参照**: CLAUDE.md — next配列の規則

> 修正指示: 上記フィールドのみを修正し、他のモジュールには一切触れないこと。

- **参照**: docs/brekeke/モジュール詳細設定ガイド_1.md セクション 5.3

### W-005: properties に `リトライ_問い合わせ内容` の Retry Counter プロンプトが未定義

- **ファイル**: `output/scenarios/ユアクリニックお茶の水_診療/properties_ユアCLお茶水_診療.md`
- **修正担当**: properties
- **モジュール名**: `リトライ_問い合わせ内容`
- **フィールド**: properties メインフローセクション
- **問題**: `リトライ_問い合わせ内容` は Speech Retry Counter モジュールであり、`prompt_true`（リトライ時発話）と `prompt_false`（上限到達時発話）を持つ。現在 JSON 内に直接 `{tts_g:...}` 形式で設定されているが、IVRプロパティでの管理が推奨される。properties ファイルには当該モジュール名の記述がなく、JSON vs properties の突合せでも「JSON ONLY」として検出された。

  ただし、Retry Counter の `prompt_true`/`prompt_false` は JSON 内に直接 `{tts_g:...}` 形式で記述するパターンが設計上許容されているため（CLAUDE.md の `{recstart}` に関する記述に準拠）、**この警告は運用要件次第で許容可能**。properties での一元管理を望む場合は追加が必要。

- **現在値**: JSON に `prompt_true="{tts_g:恐れ入りますが...}"` `prompt_false="{tts_g:かしこまりました...}"` が直接設定済み。properties には記述なし。
- **推奨値**: properties に以下を追加
  ```
  リトライ_問い合わせ内容.prompt_true={tts_g: 恐れ入りますがご回答が確認できませんでした。今一度、}
  リトライ_問い合わせ内容.prompt_false={tts_g: かしこまりました。折り返しの際にお伺いいたします。}
  ```
- **修正指示**: properties への追加を検討すること（必須修正ではなく、properties での一元管理を優先する場合のみ対応）。
- **参照**: docs/brekeke/IVRプロパティ生成ガイド.md

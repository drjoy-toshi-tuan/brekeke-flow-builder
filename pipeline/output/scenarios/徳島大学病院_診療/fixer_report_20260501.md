### C-001: OpenAI_診療科_歯科 STEP3 に不適切なキーワード混入「Suica」「追加」
<prompter>

- **修正担当**: prompter
- **対象モジュール**: OpenAI_診療科_歯科 (Line 1314-1320)
- **問題**: 
  ```
  ### 歯科麻酔科
  麻酔
  Suica         ← 交通系ICカード。医療と無関係
  追加          ← 一般日本語。歯科麻酔科と関連不明
  ```

  - これらのキーワードがユーザー発話に含まれた場合、誤って「歯科麻酔科」と判定される危険性
  - **プロンプトインジェクション / 意図しない誤分岐ベクトル**

- **修正方針**: 
  1. プロンプト内の該当キーワード「Suica」「追加」を削除
  2. 歯科麻酔科マッピングは「麻酔」「歯科麻酔」「歯科の麻酔」の3キーワードのみに絞込み
  3. 設計書 hearing_items との突き合わせで再検証

### W-001: OpenAI_予約日聴取 STEP5 の有効範囲チェック条件が設計書の流れと矛盾
<prompter>

- **修正担当**: prompter
- **対象モジュール**: OpenAI_予約日聴取 (Line 1573-1578, STEP5)
- **問題**:
  
  プロンプト STEP5 では：
  ```
  ## STEP5：有効範囲チェック
  抽出した日付が以下の条件をすべて満たさない場合は NO_RESULT。
  - 日付がカレンダー上に実在すること（うるう年考慮）
  - 月は1〜12、日は当該月の有効日数の範囲内
  - 過去日（基準日より前）でないこと  ← ★問題の箇所
  ```
  
  一方、設計書では：
  - module_purpose (line 1405): "**現在の予約日聴取**"
  - announcement (line 1406): "**現在の予約日**を「4月1日」のように日付でおっしゃってください。"
  - step_details (line 1221-1222): "**現在の予約日**を「4月1日」のように日付でおっしゃってください。"
  
  **矛盾点**:
  - フロー設計では「**変更 / キャンセル**」ルートで「**現在の予約日（既存予約日）**」を聴取
  - 既存予約日は、通常は基準日より**過去日**（すでに予約されている日）
  - ところが、プロンプトの STEP5 は「過去日でないこと」という検証を行う
  - 結果として、有効な「現在の予約日」が NO_RESULT 判定される可能性
  
  ※ 参考: 設計書の step_name "理由_変更" (line 1407-1409) では「**次回予約希望日**聴取」と明記されており、変更日と希望日は別項目

- **修正方針**:
  1. Context（重要）セクションを「新規予約希望日ではなく**現在の予約日（既存予約日）を聴取**」と明確化
  2. または、STEP5 の「過去日（基準日より前）でないこと」という条件を削除・緩和して、過去日を有効日として受け入れる
  3. 設計書の「現在の予約日」の意図と一致するよう見直し

**補足**: この矛盾はプロンプト品質の低下につながり、実運用で「過去日の予約日が入力されても NO_RESULT で受け入れない」という弊害を招く可能性があります。

### C-001: 用件確認の複数ラベルマッチ時の優先順序が顧客要件と矛盾
**修正担当**: prompter  <prompter>

**重大度**: Critical  
**対象モジュール**: OpenAI_用件確認  
**対象行**: L189（優先順序ルール）/ L191-249（判定アルゴリズム）

**問題の詳細**:

設計書では顧客大原則として以下が明記されている（L63-64）:
```
顧客大原則: 「相談室業務に関係する電話のみ受け付け、
相談室に直接関係しない電話（特に予約関連）は受け付けずに弾く
（予約センターかけ直し案内へ）」
```

フロー図（L83-85）でも:
```
├─ 予約 → END_予約センター案内（弾く）
```

しかし、プロンプト内の判定アルゴリズム（L189）では:
```
複数のラベルが同時にキーワードマッチした場合 → 上から優先（評価順を厳守）
```

その実装（L191-249）では:
```
### 相談
相談
...

### 大代表
大代表
...

### 予約
予約
...

### 定型案内
場所
...
```

**問題ケース**:
ユーザー発話: 「予約をしたいので、大代表へ繋いでください」
- 含まれるキーワード: 「予約」（3番目セクション） + 「大代表」（2番目セクション）
- 現在の優先順位: 「大代表」（2番目が上位）
- 分類結果: **「大代表」** → 誤着信転送(04-7092-2211) へ
- **期待結果**: **「予約」** → END_予約センター案内（弾き）へ

顧客要件では予約関連を最優先で弾くべきなのに、大代表転送が優先されてしまう。

**修正方針**:

以下の2つの対応のいずれかを実施すること:

1. **優先順序の再定義（推奨）**:
```
判定優先順序を以下に変更（上から順）:
  1. 予約（【最優先】顧客大原則: 予約関連は一切受け付けずに弾く）
  2. 相談（相談室業務のコア）
  3. 大代表（定型転送）
  4. 定型案内（単純案内）

理由: ユーザー発話に「予約」と「大代表」等の複数キーワードが混在した場合、
      必ず「予約」意図を優先検知する必要がある。
```

2. **複数マッチ時のロジック改善**:
```
複数のラベルが同時にキーワードマッチした場合は、
顧客ビジネスルール上の優先順位に従う（段階的順序ではなく）:
- 「予約」が含まれる場合は「予約」で確定
- 「相談」「大代表」「定型案内」のいずれかが含まれ、
  「予約」は含まれない場合、上から順に評価
```

**修正詳細**:
- プロンプト内 L189 のコメントを修正
- L191-249 の判定セクション順序を再調整（予約を最初に移動）
- 或いは L175 の STEP2 冒頭に「複数マッチ時の顧客優先順位ルール」を追記

### W-001: DTMF 入力への対応が完全欠落
**修正担当**: prompter  <prompter>

**重大度**: Warning  
**対象モジュール**: すべての enum 系プロンプト
- OpenAI_発信元確認
- OpenAI_用件確認
- OpenAI_患者区分確認
- OpenAI_相談種別確認

**対象行**: 各プロンプトの「Role」「Context」セクション

**問題の詳細**:

設計書 step_details では以下が明記されている（L509, L572, L602等）:
```
input_method: "voice_only"
```

これは「DTMF 入力対応なし、音声のみ」を意味しますが、プロンプト内には以下の明示的指示がありません:
- 「電話キーボード（ダイヤルボタン）でのご入力は対応していません」
- 「音声でお答えください」

**リスク**:
ユーザーが「1 を押してください」等の指示を受け付けるものと誤解し、DTMF でキー入力してしまった場合、それが STT で音声として認識される可能性があり、期待外の分類結果が生じる可能性があります。

例: ユーザーが「ピッピッ」（電話キーボン音）で「1」を入力
- STT が「ピッピッ」を音声として認識
- OpenAI が「ピッピッ」をキーワードマッチ → NO_RESULT 返却
- 結果: 再試行

これ自体は安全ですが、UX の観点から「音声のみ対応」を明示すべきです。

**修正方針**:

各プロンプトの「Context（重要）」セクションに以下を追加:

```markdown
# Context（重要）
直前にユーザーには次の案内がされています：

「はじめに、お電話の発信元をお伺いします。
患者ご本人またはご家族の方は「患者・家族」、…（既存文言）」

⚠️ 重要: 本システムでは音声入力のみ対応しており、
電話キーボード（ダイヤルボタン）でのご入力には対応していません。
音声でお答えください。
```

或いは設計書の step_details に「重要注釈」を追加する。

### W-002: 担当者確認系の複合意図判定ロジックが曖昧
**修正担当**: prompter  <prompter>

**重大度**: Warning  
**対象モジュール**: 
- OpenAI_担当者確認（L392-455）
- OpenAI_担当者確認_外来（L489-551）

**対象行**: STEP2（L410-430 / L507-526）および STEP3（L433-450 / L530-547）

**問題の詳細**:

プロンプトのルール（L397-398, L494-495）では:
```
- 「わかる」系と「わからない」系のキーワードを両方含む場合 → NO_RESULT
```

しかし、以下のような曖昧なケースは定義されていません:

1. **条件付き肯定**:
   - ユーザー発話: 「担当者名は分かりますが、今は電話に出ていません」
   - キーワード検出: 「分かります」（わかる系） + 「電話に出ていません」（文脈）
   - 現在の判定: 「分かります」がマッチ → 「わかる」と判定される可能性
   - **期待結果**: NO_RESULT（実際には「わからない状態」に近い）

2. **曖昧な確実性**:
   - ユーザー発話: 「多分、わかるかもしれません」
   - キーワード検出: 「わかる」「かもしれません」（不確実性）
   - 現在の判定: 「わかる」がマッチ → 「わかる」と判定
   - **期待結果**: NO_RESULT（確実性が不足）

3. **二重否定**:
   - ユーザー発話: 「わかりませんではなく、別の担当者なら分かるかもしれません」
   - 複雑な文脈解析が必要だが、プロンプトでは「マッチするキーワード」のみで判定

**修正方針**:

STEP3「わからない判定」セクション（L433-450 / L530-547）を拡張:

```markdown
# STEP3：「わからない」判定（拡張）
正規化後の入力に、以下の【A】または【B】に該当する場合 → NO_RESULT

【A】わからない系キーワード（既存）
わからない / 分からない / わかりません / 分かりません / 知らない
知りません / 決まっていない / 決まってない / 未定 / いいえ / いえ
特にいない / 特にない / ありません

【B】新規: 曖昧・条件付き・否定修飾を含む場合
- 「多分」「かも」「かもしれません」「多少」等の確実性欠如表現を含み、
  「わかる」系キーワードが同時に含まれる場合
- 「ですが」「ただし」「けれど」「しかし」等の逆接表現を含み、
  その後に否定・打ち消し・不確実性が続く場合
- 例: 「わかりますが…」「分かってますけど…」「わかるというか…」

判定ロジック:
  if （「わかる」系 AND 「曖昧修飾」） then NO_RESULT
  if （「わかる」系 AND 「逆接」） then NO_RESULT
  else 既存ロジック
```

### W-003: NO_RESULT 条件の「意味不明」定義が曖昧
**修正担当**: prompter  <prompter>

**重大度**: Warning  
**対象モジュール**: すべての enum 系プロンプト
- OpenAI_発信元確認（L121-127）
- OpenAI_用件確認（L254-258）
- OpenAI_患者区分確認（L353-357）
- OpenAI_相談種別確認（L640-644）

**対象行**: STEP3「それ以外」セクション

**問題の詳細**:

STEP3 では以下が記載されている:
```
以下は NO_RESULT：
- 無音
- フィラーのみ（例：えー あのー のみ）
- 意味不明
- 上記いずれのキーワードにもマッチしない入力
```

「意味不明」の定義が曖昧で、以下のようなエッジケースでの判定が不明確です:

1. 「田中です」（固有人名のみ）→ 意味不明か？ノーマッチか？
2. 「あああああ」（同音反復）→ 意味不明か？
3. 「スーパーカリフラジリスティック」（関連ない長単語）→ 意味不明か？
4. 「プププ」（笑い声）→ 意味不明か？
5. 「えっと、あ、あ、あ」（助詞のみ）→ フィラーか？意味不明か？

OpenAI が曖昧に判定する可能性があり、期待外の NO_RESULT / マッチ混在が生じるリスク。

**修正方針**:

STEP3 を以下のように具体化:

```markdown
# 【STEP3：それ以外】
以下は NO_RESULT：

1. 無音: 何も発話されていない状態

2. フィラーのみ（例：えー あのー のみ）
   - 「ああ」「えー」「あのー」「まぁ」「ちょっと」等の
     語彙的意味を持たない つなぎ表現のみ
   - 助詞・助動詞のみの発話も含む（「が」「に」「ね」）

3. 意味不明: 日本語として解析不能、または聴取対象と無関連
   - 音声ノイズ（「プププ」「ピッピッ」など）
   - 関連ない固有名詞のみ（「田中です」「りんご」等の背景情報）
   - 関連ない長単語のみ（「スーパーカリフラジリスティック」等）
   - 数字のみ（「12345」等）

4. 上記いずれのキーワードにもマッチしない通常の日本語
   - マッチキーワードに含まれない単語が発話された場合
   - 例: 「発信元確認」で「サッカー部です」と回答 → NO_RESULT
```

### W-004: 「はい」「ええ」等の短い肯定表現で副作用リスク
**修正担当**: prompter  <prompter>

**重大度**: Warning  
**対象モジュール**: 
- OpenAI_担当者確認（L410-430）
- OpenAI_担当者確認_外来（L507-526）

**対象行**: STEP2「わかる判定」のキーワードリスト（L413-429 / L510-526）

**問題の詳細**:

プロンプト内 STEP2 のキーワードリストに以下が含まれています:

```
はい
ええ
そうです
```

これらの短い肯定表現は「担当者が決まっているか」という Yes/No 質問以外の別の質問への回答としても頻繁に使用されます。

**リスクケース**:
質問の流れが以下だったと想定:

1. Q: 「（先ほどの聴取結果を受けて）確認ですが、○○でよろしいですか？」
2. A: 「はい」（← 先ほどの聴取を確認する意味での肯定）
3. Q: 「ご担当の者はお決まりでしょうか？」
4. ユーザーは答える前に...
5. STT が先ほどの「はい」を拾う、或いは新しい発話で「はい、ありません」と答える

この場合、プロンプトの STEP2 で「はい」がマッチ → 「わかる」と判定される可能性。

一方、STEP3 では:
```
ありません
```

は NO_RESULT キーワードに含まれていないため、「はい」「ありません」の複合で判定が曖昧になります。

**修正方針**:

いずれかの対応を実施:

1. **NO_RESULT キーワードの追加（推奨）**:
```
# STEP3：「わからない」判定
正規化後の入力に、以下のキーワードのいずれか1つ以上を部分マッチで含む場合 → NO_RESULT

わからない / 分からない / わかりません / 分かりません / 知らない
知りません / 決まっていない / 決まってない / 未定 / いいえ
いえ / 特にいない / 特にない
ありません      【新規追加】
```

2. **短い肯定表現の判定順序を下げる**:
```
# STEP2：「わかる」判定
以下の優先度順に評価し、最初にマッチしたものを採用:

【優先度 1（確実性高）】
わかる / 分かる / わかります / 分かります / わかっています
わかってる / 知ってる / 知っています / 存じています
決まってる / 決まっています / 決まっております / お決まりです

【優先度 2（確実性中）】
（「○○さん」のように特定の担当者名を述べている場合）

【優先度 3（確実性低）】
はい / ええ / そうです
※ 優先度 3 が単独でマッチしても、STEP3 の否定キーワードが
  同じ入力に含まれていないことを確認すること
```

3. **複合判定ルール**:
```
「はい」「ええ」「そうです」がマッチしても、
同じ入力に「ありません」「いえ」「いいえ」が含まれる場合は
NO_RESULT に変更
```

### W-005: 出力値の正規化ロジックが不足
**修正担当**: prompter  <prompter>

**重大度**: Warning  
**対象モジュール**: すべての enum 系プロンプト
- OpenAI_発信元確認
- OpenAI_用件確認
- OpenAI_患者区分確認
- OpenAI_担当者確認
- OpenAI_担当者確認_外来
- OpenAI_相談種別確認

**対象行**: 各プロンプトの「出力仕様」セクション

**問題の詳細**:

プロンプトでは「1語のみ出力」が明記されているのに対し、OpenAI が以下のように出力する可能性があります:

1. 句読点・感嘆符付き:
   - 「患者本人・家族。」
   - 「相談!」
   - 「わかる！」

2. 敬語系での出力:
   - 「わかります」（プロンプトは「わかる」と期待）
   - 「分かります」（設計上は「わかる」と期待）

3. スペース混入:
   - 「 患者本人・家族」（先頭スペース）
   - 「相談 」（末尾スペース）

4. 複数単語:
   - 「相談 室」（誤り）
   - 「患者 本人 家族」（誤り）

これらは設計上「1語のみ」という制約に違反し、JSON の出力フィールドやContextMatchRouter での条件マッチを失敗させる可能性があります。

**修正方針**:

scaffold / inject_prompts.py 側で出力値を正規化する実装を追加:

```python
def normalize_openai_output(output_value: str, expected_values: list) -> str:
    """OpenAI 出力値の正規化"""
    # 1. 末尾から句読点・空白を全削除
    normalized = output_value.strip().rstrip('.!?！？。、')
    
    # 2. 先頭・末尾スペース削除
    normalized = normalized.strip()
    
    # 3. 内部スペースを削除（複数単語を1語に統一）
    normalized = normalized.replace(' ', '')
    normalized = normalized.replace('　', '')  # 全角スペース
    
    # 4. 敬語系を基本形に統一（「わかります」→「わかる」等）
    normalization_map = {
        'わかります': 'わかる',
        '分かります': 'わかる',
        'わかってます': 'わかる',
        # ... 必要に応じて追加
    }
    if normalized in normalization_map:
        normalized = normalization_map[normalized]
    
    # 5. expected_values に含まれるか確認
    if normalized in expected_values:
        return normalized
    else:
        return None  # → NO_RESULT 扱い
```

プロンプト側の修正:
- 「敬語を使わず、『わかる』のみを出力してください」等の明示
- または scaffold 側の normalize で吸収

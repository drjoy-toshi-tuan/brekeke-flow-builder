- 注意事項アナウンス: 「このお電話は、途中で電話を切ると予約確定できませんのでご注意ください」
- 高校生未満案内（announcement のみ、強制切断なし）
- 本人確認 hearing（Yes/No → callerSubject、双方とも当院受診確認に進む）
- 当院受診確認 hearing（初診/受診歴あり → firstVisit、診察券聴取の要否を分岐）
- 診察券番号聴取（受診歴ありルートのみ、subflow）
- 氏名聴取（subflow）
- 生年月日聴取（subflow、復唱あり）
- 性別 hearing（男性/女性 → gender、発熱外来特有）
- 電話番号聴取（subflow、復唱あり、incoming-classifier で携帯/固定判別）
- 予約登録（Dr.JOY API BOOKING_AI → success=END_予約完了 / failure=END_予約失敗）
- 終話 4 種: 予約完了 / 予約枠終了 / 予約失敗 / 聴取失敗
- 受付時間外・非通知の異常系終話 2 種

## 4. augment ブロック使用状況

scenario_flow に `type: augment` を 2 箇所使用している。

| # | step 名 | augment_pattern | augment_purpose | 推奨アクション |
|---|---|---|---|---|
| 1 | 在庫取得 | new_module | Dr.JOY API GET_EMPTY_FRAME — 当日発熱外来空き枠数を取得し HAS_EMPTY_FRAME context に整数値を保存 | Brekeke 側で本物の API 連携モジュール（drjoy^...$GET_EMPTY_FRAME）を実装 → scaffold_generator に新ブロック型として追加 → qa_validator allowlist 更新 → 既存の augment を昇格 |
| 2 | 予約登録 | new_module | Dr.JOY API BOOKING_AI — 取得した患者情報・連絡先で Dr.JOY に発熱外来予約を登録（success/failure 分岐） | 同上。Brekeke 側で本物の API 連携モジュール（drjoy^...$BOOKING_AI）を実装 → success/failure 分岐を含めた新ブロック型として scaffold に追加 |

**経緯**:
- Gen1 では `BOOKING_AI` `GET_EMPTY_FRAME` がサブシナリオとして実装されており、Dr.JOY との連携が既に動いている（Gen1 commubo 上で）
- Gen3 では Brekeke モジュール体系がまだ Dr.JOY API 連携を 9 ブロック型でカバーしていないため augment 暫定配置
- scaffold は augment_pattern: new_module の場合、placeholder Script を生成して JSON 完走させる。実運用前に手動で API モジュールに差し替える必要あり

**短期対応**:
- パイプラインを完走させるためには augment placeholder で十分
- 実運用前に Brekeke IVR UI で在庫取得 / 予約登録モジュールを Dr.JOY API モジュールに置換する作業が必要

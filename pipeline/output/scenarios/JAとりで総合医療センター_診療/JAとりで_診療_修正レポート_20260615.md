## 背景
担当者が5月版（`JAとりで_20260526`）から6月版（`JAとりで総合医療センター_260612`）へフローを作り替えた際に複数箇所が壊れ、「全体的に切れる/おかしい」状態だった。VFB検証で原因を特定 → 修正 → 実機で全ケース検証した。

---

## 修正した問題（6点）

### 1.【最重要・全コール切断】確認_理由 / 確認_診療科_2
- **問題**: 用件確認や予約希望日の直後でほぼ全コールが途中で切断（報告: 「7月後半希望」入力で切れる）。
- **原因**: `確認_理由` と `確認_診療科_2` が「Re-confirmation node data」型で、`nodeName = openAI_確認_用件`（CLASSIFICATION型コンテキスト `classification`）を参照。Re-confirmation node は CLASSIFICATION 値を `classification` システム変数で解決しようとするが、その resolver が環境に未登録 → `No resolver found for system variable key=classification` で例外 → 通話切断。`確認_理由` は予約/変更/キャンセル**全用件の本道上**にあったため全コールが切れていた（＝「全体的に壊れてる」の主因）。
- **修正**: 両モジュールを Re-confirmation node data → 素の「Text to speech」型へ変更。読み上げ文言は元々 IVRプロパティ側（`<%classification%>` 含む）で定義されているため、素TTSでもプロパティ置換で意図どおり読み上げる。
- **検証**: case 1/9 で `確認_理由:OK` 通過・完走。報告の「7月後半希望」でも切断なし。

### 2.【終話で行き止まり・DTMF経路】終話分岐
- **問題**: DTMFで操作した発信者が、用件処理後に終話アナウンスなしでコール終了（`終話分岐_2:0`）。
- **原因**: `終話分岐_2`/`終話分岐` が用件を `openAI_確認_用件`（OpenAIモジュールの実行結果）で読んでいたが、DTMF経路ではそのモジュールが走らない（`入力_確認用件 1→save-予約` 直行）→ 結果が空 → 無マッチ → 行き止まり。用件の値自体は `classification` コンテキストに保存済みだった。
- **修正**: `終話分岐_2`/`終話分岐` の参照を `openAI_確認_用件`（モジュール結果）→ `<% classification %>`（コンテキスト・DTMF/音声両経路で投入される）に変更。
- **検証**: case 1 で `終話分岐_2:1 → 終話① → 終話_成功`。

### 3.【音声経路で classification 未保存】openAI_確認_用件
- **問題**: 音声で用件を言うと（case 4）、`終話分岐_2:0` で行き止まり。
- **原因**: 音声経路は `openAI_確認_用件`（contextName=classification, CLASSIFICATION型）の自動保存に依存していたが、**CLASSIFICATION型の自動保存が `<%classification%>` 読取ストアに届かない**（1と同じ resolver 不在が書き込み側でも影響）。DTMF経路は `saveContext2DB`（save-予約/変更/キャンセル）で明示保存していたため動いていた。
- **修正**: `openAI_確認_用件` の 予約/変更/キャンセル 分岐を、それぞれ `save-予約`/`save-変更`/`save-キャンセル`（saveContext2DB）経由に変更。音声でも saveContext2DB で classification が確実に保存される。
- **検証**: case 4/5/6（音声 予約/変更/キャンセル）が全て完走。

### 4.【PNG違反の経路】変更/キャンセルが通院歴に流れていた
- **問題**: フロー図(PNG)では変更/キャンセルは「診療科」から始まる（通院歴を聞かない）はずだが、DTMFの変更/キャンセルは `save-変更/save-キャンセル → 確認_通院歴`（予約と同じ通院歴経路）に流れていた。音声経路は正しく `確認_診療科_2` へ。DTMF/音声で経路が不一致だった。
- **修正**: `save-変更`/`save-キャンセル` の遷移先を `確認_通院歴` → `確認_診療科_2` に変更。これで「予約=通院歴経路 / 変更・キャンセル=診療科経路」が DTMF・音声で統一され、PNGと一致。
- **検証**: case 2/3/5/6 で `確認_診療科_2` 経由を確認。

### 5.【ルーター仕様】用件分岐の 2in(module2) は必須
- **問題**: 用件分岐の module2(2in) を冗長として一旦クリアしたら、case 2 で `用件分岐:0`（無マッチ）→ キャンセルが予約希望日を聞く誤遷移。
- **原因**: Brekeke ContextMatchRouter は **module2(2in) のミラー設定が必須**で、空にすると無マッチになる（エンジン仕様。「冗長」ではなかった）。
- **修正**: `用件分岐` を `module1 = module2 = <% classification %>`（変更/キャンセル）のミラー構成に戻す（終話分岐と同じ実証済み構成）。
- **検証**: case 2 で `用件分岐:1`（変更→予約希望日）、case 3 で `用件分岐:2`（キャンセル→確認_理由・希望日なし）。

### 6.【命名・衝突回避】グループ日付サフィックス
- グループ名を `JAとりで総合医療センター_260612` → `_260615`（当日）に変更。既存本番(260612)と別グループにして衝突回避（命名規則: 日付サフィックスはグループ名のみ・フロー名は素）。

## 未対応（別途・Flow Designer 領域。通話挙動には影響なし）
- **S1 「その他問い合わせ」分岐の物理削除**: 孤立クラスタ5モジュール（確認_問い合わせ内容/入力_確認_問い合わせ内容/openAI_確認_問い合わせ内容/リトライ_確認_問い合わせ内容/openAI_聴取不可_11）の削除 ＋ 分類器 `openAI_確認_用件` の3択化 ＋ 用件アナウンスの「4つ→3つ」表記。これらは**既に到達不能なデッドコード/IVRプロパティ**で、発信者から見た用件は既に3択＝通話挙動・テスト結果には影響しない。`入力_確認_問い合わせ内容` が 56,758字のAmiVoice辞書を内包し .bivr の文字列一致編集が不可のため、Flow Designer（ノード選択→削除、プロンプト編集）で実施する。
- **非通知ハンドリング**: 現状維持（指示）。
- **観察（既存配線・低優先）**: case10 で診療科の判定が NO_RESULT のとき `リトライ_復唱_診療科 false → 確認_現在の予約日` へ迂回する枝があり、予約でも現在予約日を聞きうる。今回の修正起因ではないが担当者へ共有候補。

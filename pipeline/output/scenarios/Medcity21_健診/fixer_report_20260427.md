    → サイドカー MD から対応プロンプトを検索
    → ❌ マッチするプロンプト名がない

  1. 設計書で定義された標準値（例：「協会けんぽ」「人間ドック」「超音波」）
  2. 標準値に該当しない場合：ユーザー発話内容をそのまま（簡潔に）text として出力
  3. 発話が不明な場合：「わかりません」
  4. 発話がない場合：「NO_RESULT」

  - OpenAI_問合せ_返信希望（line 890）

- **問題**: 
  設計書でも記載されている「肺」という救済語は、文脈的に明らかに **OCR エラー** または **音声認識誤訓練データ** です。
  
  根拠：
  - 「いいですの救済語として『肺』」という文脈が語義として成立しない
  - 肺は医学用語（医療機関でも脈絡がない）
  - 設計書の special_notes（line 1479）でも「肺」が「肯定の救済語」として記載されているが、出典が不明
  - プロンプトがこれをそのままコピペしたため、3箇所に混入

- **修正方針**: 
  1. 各プロンプトから「肺」を **削除してください**
  2. 代わりに、音声認識で実際に「いいです」が誤認識される可能性のある音を救済語として追加してください
     - 例：「いいっす」「いいね」「大丈夫」「OKです」など、実用的な肯定表現
  3. director に「肺」救済語の起源を確認し、設計書を修正してください
  4. 最終確認：修正後のプロンプトで「肺」を検索して **0件** であることを確認してください

### W-001: enum型の判定ルール「部分一致」と「完全一致」が混在している <prompter>
- **修正担当**: prompter
- **対象モジュール**: 
  - OpenAI_用件確認（line 58-62）：「部分一致可」
  - OpenAI_変更_条件確認（line 330-335）：「完全一致のみ許可」
  - OpenAI_キャンセル_条件確認（line 506-514）：「完全一致のみ許可」

- **重大度**: Warning

- **問題**: 
  判定ルールの一貫性が失われており、同じタイプの判定でも異なるロジックが適用され고 있습니다.
  
  具体例：
  - **OpenAI_用件確認**（部分一致）:
    ```
    入力: 「予約したいのですが」
    → STEP1正規化後: 「予約したいのですが」
    → STEP2判定: 「予約」を含む → 「予約」を出力 ✓
    ```

    ```
    入力: 「はい、そうですね」
    → STEP1正規化後: 「はいそうですね」or「はい」?
    → STEP2判定: 「はい」で完全一致? 「はいそうですね」で検索?

    ```

  1. **設計書と照合**: hearing_items の「openai_processing」での意図を確認
  2. **ルール統一**: 「用件確認は部分一致」「Yes/No判定は完全一致」という仕様に統一するか、全プロンプトで完全一致に統一するか決定
  3. **記載方法**: 各プロンプトのSTEP2で「部分一致」「前方一致」「完全一致」を明示化
  4. **優先順序明記**: 複数マッチ時の優先ルール（「最初にマッチ」「最長マッチ」など）を明示

  問題ケース:
  1. 「えーと」（フィラーのみ？）→ NO_RESULT
  2. 「えーと、わかりません」（フィラー+言葉？）→ 「分からない」 or NO_RESULT?
  3. 「なんか」（フィラーか、意味不明か？）→ 判断に迷う
  4. 「あ、はい」（バックグラウンドノイズか、有効入力か？）→ 「はい」 or NO_RESULT?
  ```

- **修正方針**: 
  1. **定量的定義**: NO_RESULT の判定基準を具体化
     ```
     NO_RESULT の定義:
     - 空文字、または
     - 「えーと」「あのー」「ん」「ま」等のフィラーのみ（計3語以下）、または
     - 分類対象と関係ない単語のみ（例：「天気」「野球」「料理」等）、または
     - 音声認識エラー（「フーフー」「ノイズ」「ブツブツ」等）
     ```
  
  2. **enum型との区別**:
     ```
     「分からない」の定義:
     - ユーザーが自身で回答不可と述べている（datetime型のみ）
     
     「NO_RESULT」の定義:
     - 発話が解析不可（ノイズ、フィラー等）
     ```
  
  3. **テスト用サンプルの作成**: tester.py で以下を検証可能に
     - NO_RESULT 境界値（2語まではNO_RESULT、3語以上は判定試行等）
     - 曖昧フィラーの扱い
     - 「わかりません」vs「NO_RESULT」の区別

### I-001: システム日付の扱い説明が冗長で誤解のリスク <prompter>
- **修正担当**: prompter（推奨）
- **対象モジュール**: OpenAI_変更_予約日聴取, OpenAI_キャンセル_予約日聴取, OpenAI_オプション_予約日聴取
- **重大度**: Info

- **問題**: 
  「システム日付の取り扱いに関する絶対ルール」（line 422-425等）は対策として素晴らしいですが、説明が冗長で実装者が誤解しやすい形式です。
  
  現在の記載：
  ```
  プロンプトの冒頭に「今日はyyyy年mm月dd日です」とシステム実行日（現在日）が自動で付与されます。
  この日付は「ユーザーの発話した日付が未来か過去か」「年をどう補完するか」を計算するための【基準日】としてのみ使用してください。

  ## システム日付の取り扱い

  2. 年の補完・相対日付の計算にのみ使用してください
  3. ユーザー発話に明確な日付がない場合は「分からない」または「NO_RESULT」を出力してください

### I-002: enum型で「複合意図」への対応が明記されていない <prompter>
- **修正担当**: prompter（推奨）
- **対象モジュール**: OpenAI_用件確認
- **重大度**: Info

- **問題**: 
  攻撃耐性セクションで「複合意図への耐性」が要求されていますが、プロンプト内に具体的な対応例がありません。
  
  例：「予約したいのですが、キャンセルもしたい」という発話で、「予約」と「キャンセル」が両方含まれる場合、現在は上から順に評価して最初の「予約」を返します。
  - これはユーザーの真の主要意図を損失している可能性

- **修正方針**: 
  1. **複合意図の具体例を STEP2 に追加**:
     ```
     複数のキーワードが含まれる場合の処理:
     - 例1: 「予約したいのですが、キャンセルもしたい」→ 最初のキーワード「予約」を優先
     - 例2: 「変更できますか、それともキャンセル？」→ 最初のキーワード「変更」を優先
     - ルール: 複数一致の場合、語句リストの上から順に評価
     ```
  
  2. **または代案**: 複合意図の場合は「NO_RESULT」として再質問させるロジックを検討
     - より安全（ユーザーの真意を確認できる）だが UX 低下の可能性

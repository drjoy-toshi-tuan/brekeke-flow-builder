### 移管方針
- 冒頭は opening + 冒頭_アナウンス の2ブロック1セット。冒頭アナウンス直後は入力待ち無しで「用件聴取」に接続する。
- Date of Call Classifier は元資料に記載なし → 不要。着信直後から用件聴取へ直行する構成（診療シナリオの「当日翌日確認」のような先頭分岐は無い）。
- 用件①（classification: 新規予約 / 変更 / キャンセル / 確認）で4分岐（Pattern A: OpenAI直後分岐）。用件再確認（Yes/No）を通過した直後に「案内発話判定」で**新規予約以外は [当日案内] を差し込む**点が特徴的。当日案内は代表電話 03-3928-0071 を案内するだけの無入力アナウンスで、その後 [入電者名] に合流する。
- 氏名／生年月日／電話番号は **サブフロー経由**（Jump to Flow）。
  - **本健診は「受診者本人 or 代理」分岐あり**（STEP 3 受診者確認）。入電者名（callerName）→ patientName に自動コピーし、本人なら生年月日へ、本人でなければ受診者名サブフローで patientName を上書きしてから生年月日へ。**入電者名・受診者名の両方を PatientName サブフロー（Gen3 の氏名聴取サブフロー）に充当する**のが素直（どちらも受診者＝実質の患者になるため。director 判断で 1 サブフロー共有 or 2 回呼び分け）。memory: feedback_patientname_subflow_allocation
  - organizationName 相当の「会社名／事業所名」は本元資料には**存在しない**（健康保険組合名 insuranceAssociation はあるが、氏名扱いではなく独立 hearing ブロック）。一般的な健診シナリオの organizationName 充当は今回不要。
- 電話番号は聴取サブフロー側で phone_type 判定（mobile/ip050/withheld/landline）+ valid/invalid 判定。invalid の場合は1回だけ再ヒアリングし、**2回目 invalid なら STEP 6（用件別ヒアリング）へフォールスルー**（切断しない／聴取を打ち切って続行する）。終話時の SMS 文言は phone_type で2系統に分岐。
- 用件別ヒアリングは classification × サブ分岐の4ルート：新規予約 / 変更 / キャンセル / 確認。各ルート末尾に **phone_type × classification の組合せで終話パターンを出し分ける ContextMatchRouter（Pattern B）**を置く想定。
- 終話パターンは **新規予約受付 / 変更受付 / キャンセル受付 / 確認受付** × **SMS対象(mobile/ip050) / SMS非対象(landline/invalid)** の計 4×2=8 文言。ただし「キャンセル」は phone_type 問わず同一文言（追加文挿入なし）。director 判断で termination_patterns を整理。
- RAG は元資料に記載なし → 不要。FAQ 的な問い合わせは確認ルートの [確認内容] に自由文格納（details）で受ける。
- TTS は全て `{tts_g:...}`（小文字）で発話。`<speak type="telephone">` `<sub alias="...">` `<prosody rate="...">` `<break time="...">` `<dtmf />` `<dtmf digit="1" />` 等のタグは**電話番号ブロック・生年月日ブロック・代表案内・用件ブロック**等のプロパティにそのまま保持（SSML 禁止ルールの例外として元資料どおり残す。director 判断）。

### 施設固有の特殊ルール
- **「肺→はい」救済**：Yes/No 判定で「肺（はい）」「いいです（肯定文脈）」「そう」「あっています」「せいかい」は肯定として扱う。Yes/No 系 generate_by_OpenAI の prompt に注記必須。
- **数字 4/5 の誤認識救済**：「for」「C」→ 4、「号」「go」→ 5。生年月日・電話番号・DTMF ブロックの prompt 側で対処（prompter 担当）。
- **当日予約・当日変更キャンセルは AI 不可**：[予約日] が {TODAY} と一致したら即 [代表案内] へ遷移し **status=2 / smsFlag=0** で終話（当日案件は reservationDate に格納しない）。確認ルートの [予約日（確認）] は**過去/未来/本日いずれも許容**し代表案内へは飛ばさない（1.24 の独立ロジック）。
- **予約日の過去日付は「復唱で肯定」された場合のみ受付不可**：まず復唱→肯定→過去なら [予約日過去受付不可] → 再聴取（1回のみ）。**過去日付を即拒否しないこと**。
- **受診希望日 = {NEXT_MONTH_DATE} 〜 {END_PERIOD_DATE}**（翌月日以降・年度末まで）。**変更希望日 = {TOMORROW} 〜 {END_PERIOD_DATE}**（明日以降）。受付範囲が違うので混同しない。
- **受診希望日・変更希望日は第1→第2→第3 と進み、「なし/特にない」で途中終了**（第2で "" なら第3スキップ）。第3まで復唱必須。
- **女性検診の入れ子分岐**（希望コース/オプション → 女性検診 → 子宮/乳がん/両方 → 乳腺超音波/マンモ/両方）：1ブロック1聴取の原則どおり、**子宮がん検診/乳がん検診/両方** と **乳腺超音波/マンモ/両方** は別ブロックに分離。詳細サブフローを新規予約ルートと変更ルート（パターン1/パターン3）で**共有する**のが素直（director 判断）。
- **変更内容 changeType の3パターン分岐**：
  - パターン1（コースまたはオプション）→ コース確認 + オプション確認 → 終話（変更）
  - パターン2（日程変更）→ 変更希望日有無 → Yes:変更希望日1-3 / No:変更希望時期 → 終話（変更）
  - パターン3（両方）→ コース確認 + オプション確認 **→ 変更希望日有無 以降** を必ずこの順番で全実行
- **キャンセル確認 否定時は変更フローに流入**：「一部キャンセル」は [変更内容聴取] へリダイレクトする（yaml 1957 行「否定 → 変更内容聴取」）。この合流エッジを忘れない。
- **status / smsFlag ルール**：
  - status: 初期=7 → additionalPhoneNumber 確定で 1 → 代表案内で 2（上書き）。**incoming_phone_number 自動コピーでの 1 昇格は厳禁**（口頭発話 + 再確認 Yes が必須）。
  - smsFlag: phone 未格納中は 0。additionalPhoneNumber 格納瞬間に classification に応じて 1=新規予約 / 2=変更 / 3=キャンセル / 4=確認。status=2 時は常に smsFlag=0 で上書き。
- **再確認厳禁項目の多さ**：入電者名／受診者名／予約歴／健康保険組合／被保険者/被扶養者／希望コース／追加オプション／女性検診詳細／変更内容／コース確認／オプション確認／変更希望有無／変更希望時期／キャンセル確認／確認内容 → 全て**復唱禁止**。hearing ブロックの echo_back を False にする（scaffold デフォルト False を維持）。
- **コース/オプション類義語**：コース50種・オプション30種超が元資料末尾に列挙。course/option は**自由文格納（enum なし）**。prompter は類義語正規化を prompt に反映するが、列挙全件のコピーはせず「【コース・オプション一覧】を参照」と案内する設計で十分（director 判断）。
- **function スキーマの命名揺れに注意**：
  - 変更希望日は `changepreferredDate1/2/3`（customer_doc function）だが、個別ルールでは `changeDate1/2/3` と表記。**function スキーマ側の命名（changepreferredDate1-3）を正**とする。
  - 変更希望時期は `reschedulePeriod`（個別ルールでは changePeriod）。**function 側の reschedulePeriod を正**。
  - 予約歴は `history`（customer_doc function）だが、個別ルールでは `reservationHistory`。**function 側の history を正**。
  - 受診者確認は `relationship`（function にあり）。
- **改訂履歴**：本元資料には明示的な改訂履歴セクションなし（{TODAY}=2026/04/22 を想定して起草）。

### director が見落としやすいポイント
- 『**受診希望日・変更希望日の範囲チェックは「復唱で肯定」された後に実施**』（1.14 / 1.20）。**抽出直後に範囲判定して再聴取に飛ばさない**。復唱 → Yes → 範囲外判定 → [受診希望日受付不可] / [変更希望日受付不可] の順。
- 『**1.11 追加オプションの一時保持と統合**』：女性検診詳細サブフロー中はメモリ保持、全サブフロー抜けたタイミングで option に一括書き込み。**子宮がん検診/乳がん検診という名称そのものは option に格納しない**（子宮がん検診は格納、乳がん検診の場合は乳腺超音波検査/マンモグラフィーのいずれかに置換して格納）。
- 『**1.17 コース確認で course は絶対に更新しない（空文字維持）**』：変更ルートでは course は触らず、changeDetails に「変更後コース: 〔名〕」形式で書く。
- 『**オプション確認の「キャンセル意図」はサブフロー起動禁止**』（1.18 キャンセル意図）。女性検診サブフローは**新規追加・変更先（新）にのみ適用**、キャンセル対象や変更元（旧）には適用しない。
- 『**確認ルート（予約日確認）は過去日付も許容**』（1.24 独立ロジック）。新規予約ルートの reservationDate 制限（過去拒否・本日→代表案内）を流用しないこと。
- 『**「検診」は原則「健診」扱い、ただし「乳がん検診」「子宮がん検診」「子宮頸がん検診」は例外**』（キーワード判定ルール）。prompt の正規化テーブルに明記。
- 『**callerName 格納時に patientName へも自動コピー**』（yaml 1646 `更新値: {patientName: "{callerName}"}`）。受診者確認で「本人ではない」なら受診者名サブフローで上書き。
- 『**status=7 を additionalPhoneNumber 格納「瞬間」まで厳守**』（function description にも明記）。冒頭切断・途中切断の endpoint 判定と組み合わせて確定する。
- 『**再質問は最大1回、2回目以降は空文字で格納して次へ進む**』が共通ルール。ただし電話番号だけは例外で、2回目 invalid 時は STEP 6 にフォールスルー。

### 2-1. DTMF（プッシュボタン）マッピングの検証

#### 問題発見: OpenAI_用件確認 — DTMF 5 の欠落

**設計書での期待値** (step_details line 738):
```
DTMF 1=新規予約, 2=予約変更, 3=予約確認, 4=予約キャンセル, 5=再診予約
```

**プロンプト記載値** (STEP2: 数字モード判定, line 195-230):
```
### 新規予約
- 1, 1番, 1ばん, いち, いちばん

### 予約変更
- 2, 2番, 2ばん, に, にばん

### 予約確認
- 3, 3番, 3ばん, さん, さんばん

### 予約キャンセル
- 4, 4番, 4ばん, よん, よんばん, し

[ここで終了。5 の定義なし]
```

**問題の詳細**:
- 設計書では DTMF 5 = 再診予約 と明記
- ユーザーが DTMF 5 を入力した場合、プロンプトの STEP2 でマッチする数字がない
- よって STEP3（語句一致判定）に進む
- しかし「5」は「再診」というキーワードを含まないため、STEP3 でマッチできず、**NO_RESULT に陥る**
- その結果、再診予約を希望するユーザーが DTMF 5 を押しても、入力が認識されず、リトライ要求される

**影響**:
- DTMF のみ利用ユーザー（高齢者など）が再診予約を入力できないリスク
- 音声 + DTMF のハイブリッド利用では、「五」「ご」などを発話して補正可能だが、DTMF のみの場合は詰む

### W-001: OpenAI_変更_予約日 / OpenAI_キャンセル_予約日 — システム日付が出力される可能性
- **修正担当**: prompter  <prompter>
- **重大度**: Warning
- **対象モジュール**: OpenAI_変更_予約日, OpenAI_キャンセル_予約日
- **対象ファイル**: prompts_札幌徳洲会病院_診療.md (line 1200-1211, 1466-1476)

#### 問題の詳細
両プロンプトに以下の指示あり（line 1201-1204）：

```markdown
# システム日付の取り扱いに関する絶対ルール
プロンプトの冒頭に「今日はyyyy年mm月dd日です」とシステム実行日（現在日）が自動で付与されます。
この日付は「ユーザーの発話した日付が未来か過去か」「年をどう補完するか」を計算するための【基準日】としてのみ使用してください。
ユーザーの発話内に明確な日付表現がない場合、絶対にこのシステム日付を回答として出力してはいけません。
```

#### 問題
- 「基準日としてのみ使用」という制限は、LLM に対する指示に過ぎず、強制力がない
- LLM は自然言語の指示を「努力目標」として扱い、100% 順守を保証しない
- 例: ユーザーが「わかりません」と応答 → LLM が「システム日付をデフォルト値として出力」する可能性がある
- これは「ユーザーの明確な日付表現がない場合は NO_RESULT」というルールを破る

#### 具体的なシナリオ
```
ユーザー入力: 「えーっと、わかりません」
LLM の推論（悪い例）:
  - 入力から日付パターンを抽出 → 見つからない
  - 「基準日を出力しろ」という指示を見落とす可能性あり
  - → システム日付（例: 2026-05-01 00:00:00）を出力

期待値: NO_RESULT
```

#### 修正方針
プロンプトに以下を追加：

```markdown
【重要な補足】
- 「システム日付を基準日として使用」=「ユーザーが明確な日付を言わない場合、システム日付を回答として出力してはいけない」という意味です。
- ユーザー入力に具体的な日付パターンが含まれない場合、**絶対に NO_RESULT を出力してください。**
- システム日付を直接出力することは禁止です。
```

あるいは、別の方法として、プロンプト冒頭の「基準日」セクション全体を削除し、逆に「年の推定」に明示的に「年を補完する際のみ参照」と記載する方が効果的。

### W-002: OpenAI_用件確認 — TTS アナウンスと設計書での分岐数の不一致
- **修正担当**: prompter  <prompter>
- **重大度**: Warning

- **対象ファイル**: 設計書 step_details line 940 vs プロンプト対象外（設計書の TTS 台本側）

#### 問題の詳細
設計書（step_details line 940）の TTS アナウンス：
```

```

- TTS では「**4つ**」と案内
- 「予約を取る」「変更する」「確認する」「キャンセルする」の 4 つ のみ

しかし、設計書 hearing_items line 738 の notes：
```
DTMF 1=新規予約, 2=予約変更, 3=予約確認, 4=予約キャンセル, 5=再診予約
```

- 実際には **5 分岐**（再診予約が追加される）

#### 影響
- ユーザーは TTS で「4つ」という案内を聞く
- 再診予約というオプションが存在することを知らない

### W-003: OpenAI_変更_診療科 / OpenAI_キャンセル_診療科 — 複数診療科マッチ時の処理が LLM 推論に依存
- **修正担当**: prompter  <prompter>
- **重大度**: Warning
- **対象モジュール**: OpenAI_変更_診療科, OpenAI_キャンセル_診療科
- **対象ファイル**: prompts_札幌徳洲会病院_診療.md (line 1166-1167, STEP3)

#### 問題の詳細
STEP3 の複数マッチ処理（line 1166-1167）：

```markdown
複数の診療科が同時に部分一致する場合は NO_RESULT を出力。
ただし、長く一致する語句が含まれる場合（例: 「脳神経外科」と「外科」が同時マッチ）は、最も長く一致した診療科を優先。
```

#### 問題
- 「最も長く一致した」という処理は、LLM の文字列長算出に依存
- LLM は必ずしも「最長一致」を正確に判定しない
  - 例1: 「脳神経外科」「脳」「外科」マッチ → 「脳神経外科」を選ぶ（正しい）
  - 例2: 「脳神経外科」「脳」「外科」マッチ → 「外科」を選ぶ可能性（誤り）
- 正規表現ベースではなく、セマンティック判定に頼っているため、再現性が低い

#### 修正方針
以下のいずれかを実施：

1. **プロンプト側で正規表現を導入**:
   ```markdown
   【複数マッチ時の厳密な処理】
   入力文字列内に複数の診療科キーワードが検出された場合、文字列内での「最初に出現した位置」と「マッチ長」に基づいて以下を実施：
   - 最も長いマッチ長のキーワードを優先
   - 同じ長さの場合は、入力文字列内で最初に出現したものを優先
   ```

2. **複数マッチは確実に NO_RESULT**:
   ```markdown
   【推奨】複数の診療科名が同時に含まれる場合、判定の確実性が低いため NO_RESULT を出力してください。
   例: 「脳神経外科と循環器内科のどちらかで迷っているのですが」→ NO_RESULT
   ```

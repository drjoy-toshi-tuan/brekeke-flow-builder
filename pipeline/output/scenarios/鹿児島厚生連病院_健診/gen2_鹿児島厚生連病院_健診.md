### 移管方針

- **用件4分岐（classification）**: `予約` / `変更` / `キャンセル` / `その他お問い合わせ`。冒頭 generate_by_OpenAI で用件分類。復唱禁止。
- **Pattern**: 用件分岐は **Pattern A**（OpenAI 直後に classification で 4 分岐）。ただし `変更` は配下でさらに「受診内容 / 日程変更」の二択、`キャンセル` は「再受診希望 yes/no」でサブ分岐。
- **サブフロー**: PatientName / DateOfBirth / PhoneNumber の3本を利用。problem/用件に関係なく最終ブロックで共通聴取。
- **「予約日」と「予約希望日」は別フィールド**:
  - `reservationDate` (DATE 型): 現在の予約日（変更/キャンセル用）→ DateHearing 相当の datetime ブロック
  - `DesiredreservationDate` (TEXT 型): 予約希望日（フリーワード、日付正規化しない）→ text hearing
  - `reservationDate2` (TEXT 型): 予約日日時補助。director 判断（基本は使わない）
- **その他お問い合わせは RAG 不使用**: 「かしこまりました」と返すのみで `question` へそのまま格納。回答要不要確認で yes→患者情報聴取、no→終話。
- **終話アナウンス1種**: 「…を受付いたしました。3営業日以内に折り返し電話もしくはSMSにてご連絡…」共通。`{classification}` 埋め込み。
- **status**: 必須項目全充足=`1`、代表案内=`2`、それ以外=`0`。smsFlag も同様。

### 施設固有の特殊ルール

- **救済語（generate_by_OpenAI の params.prompt に埋め込む）**:
  - 肯定: 「肺」「いいです」
  - 否定: 「が今」「PHVます」
  - 診察: 「印刷」「定期」「新薬」「再診」「最新」「禁忌」
  - 予約: 「薬」「お薬」
  - 新規: 「電気」「でんき」
  - 変更: 「人工」「変更」
  - 幼児: 「雇用時」「幼児」
  - 受診内容: 「シンナイヨウ」
- **予約日の年補完**:
  - 月日のみ発話 → 入電年を自動付与（YYYY-MM-DD 00:00:00）
  - 10 月以降入電で 1〜9 月の日付を言われた場合 → **翌年**に補正
  - → `reservationDate` ブロックは `addCurrentDate=Yes` 相当 + プロンプトで翌年ルール明記
- **復唱（echo_back）は3項目のみ、各1回限り**:
  - 現在の予約日: 「現在の予約日は、{reservationDate}でよろしいですか。」
  - 生年月日: 「生年月日は、{patientDateOfBirth}でよろしいですか。」
  - 連絡先電話番号: 「ご連絡先は、{additionalPhoneNumber}でよろしいですか。」
  - 不明確・否定でも再聴取せずそのまま次へ（PhoneNumber だけは例外、下記）
- **電話番号正規化 (PhoneNumber サブフロー)**:
  - 先頭 `090/080/070/060` → 携帯（11桁）、それ以外 → 固定（10桁）
  - 桁不一致時は再入力を **最大2回**、それでも失敗なら次工程へ
  - Incoming_phone_number が携帯なら「今おかけいただいている番号でよろしいですか」で確認、固定/非通知なら手動聴取（`<dtmf/>`付き）
- **変更の二択分岐**: `details`（変更内容確認）で「受診内容」or「日程変更」
  - 受診内容 → `Change` (受診内容テキスト) → 共通聴取へ
  - 日程変更 → `DesiredreservationDate` → 共通聴取へ
- **キャンセルの二択分岐**: 再受診希望 yes/no
  - yes → `DesiredreservationDate` → 共通聴取へ
  - no → `another`（その他質問）→ 共通聴取へ
- **固定文出力後の復帰**: 「ご質問いただいた内容はAI電話ではご対応できませんので、折り返しでご確認させていただきます。」出力後は中断したヒアリング項目へ**必ず復帰**。ただし `question`（問い合わせ）項目では本固定文を出力禁止。

### director が見落としやすいポイント

- **`reservationDate` (DATE) と `DesiredreservationDate` (TEXT) を一本化しない**。型も役割も違う。director はつい一本にまとめがちだが、別ブロックで定義すること。
- **救済語はプロンプトに埋め込む**（generate_by_OpenAI の params.prompt）。scaffold は自動付与しないので director が設計書 YAML の該当モジュールに明記。
- **複数情報一括発話対応**: 「5月10日に内科で予約しているヤマダタロウですが、変えたいです」→ 用件（変更）/患者名/予約日を一括記録、不足のみ再質問、既取得は絶対再確認しない。これは OpenAI プロンプトのハードルール。
- **問い合わせ項目では「ご対応できませんので…」固定文を出してはならない**。director 判断で question ブロックのプロンプトに明記。
- **patientName はカタカナ強制正規化**: ユーザーに「カタカナで」と指示禁止、AI が内部で変換。苗字のみなら「下のお名前も…」と追問。PatientName サブフロー側で対応。
- **重厚版（766行）の全文を director プロンプトに渡さない**。元資料 `customer_docs/鹿児島厚生連病院_健診.md` を正とし、function スキーマ全フィールド・全 TTS 文言・プロンプト原文は director が直接読む。
- **`classification` の enum**: `予約` / `変更` / `キャンセル` / `その他お問い合わせ` / `""`。function スキーマでは空文字列も許容。director 判断で空文字デフォルト値を設計に含めるか決定。
- **`smsFlag` の enum**: `"0"` / `"1"` / `"2"`（元資料より）。description は「status=1→1、代表案内→2、それ以外→0」。director 判断。

## 検出事項

### W-001: DTMF 8桁処理の不要性（予約日プロンプト）
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_予約日_変更 / OpenAI_予約日_キャンセル
- **問題**: 
  - 設計書では `dtmf_max_length: 4` で4桁入力に制限されている（hearing_items order 8, 12）
  - プロンプト側では STEP3（line 723-730 / STEP3（line 961-967））で8桁YYYYMMDD処理も記載されている
  - IVRの dtmf_max_length 制約により、実運用では STEP3 は到達不可能なコード
  - ドキュメント上の混乱につながる
- **修正方針**: 
  - プロンプト STEP3 の説明冒頭に「注：本フローでは DTMF_max_length=4 のため、このステップは実質上到達不可能です」と注釈を追加、または STEP3 をコメントアウト
  - または、設計書と整合させるため、8桁対応を明示的にサポート（dtmf_max_length: 8）に拡張

### W-002: 種類_確認の「問い合わせ」キーワード曖昧性
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_種類_確認
- **問題**: 
  - 種類_確認の「その他の健診」カテゴリに「問い合わせ」キーワードが含まれている（line 304）
  - フロー設計上、問い合わせルートが用件_確認で分岐（line 338）するため、種類_確認に到達しない
  - 設計書 step_details（line 931）にも同様の mapping が記載されているため、意図的な仕様だと推定される
  - しかし可読性上、実運用では到達しない分岐が含まれていることが明確でない
- **修正方針**: 
  - プロンプト STEP3「その他の健診」カテゴリの「問い合わせ」キーワードの直後に「注：このキーワードは、用件確認で『問い合わせ』が選択された場合には到達しないため、エラーハンドリング目的で含まれています」と注釈を追加

### W-003: YES/NO判定プロンプトとフロー conditions の対応非対称性
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_企業分岐 / OpenAI_胃検査希望有無
- **問題**: 
  - 企業分岐・胃検査希望有無プロンプトでは、Yes/Noの両判定値（「本人」「企業担当」 / 「あり」「なし」）が出力される
  - しかし、フロー設計書の conditions では、Yes判定は明示的に match キーで処理（「本人」/ 「あり」）されるが、No判定は「other」で処理される設計
  - つまり、プロンプト出力値と フロー match 値が直接対応していない（非対称）
  - 実運用上は問題ありませんが、プロンプト側での出力値が、フロー側でどう処理されるかが明確でない
- **修正方針**: 
  - プロンプトの「# 判定アルゴリズム」セクション末尾に、フロー対応表を追加：
    ```
    ### フロー連携
    - 出力値「本人」 → フロー conditions match: "本人"
    - 出力値「企業担当」/ その他 → フロー conditions match: "other"
    ```

### I-001: 日付年補完ロジックのドメイン適用性
- **修正担当**: — （ドメイン要件確認推奨）
- **対象モジュール**: OpenAI_予約日_変更 / OpenAI_予約日_キャンセル
- **問題**: 
  - プロンプト（line 719 / 956）の年補完ロジック：「基準日の (MM)月(DD)日」より前の日付になる場合は翌年を補完
  - 例：基準日が2026年5月1日、ユーザー入力が「3月1日」→ 2027年3月1日に補完
  - 医療予約ドメインでは、通常「未来日のみ」というビジネスルールが存在する可能性
  - このプロンプトでは「過去日への変更」も許容される設計になっている
  - 要件として、「3月1日」は「2026年3月1日（過去）」ではなく「2027年3月1日」が正しいか、ドメイン確認が必要
- **確認方針**: 
  - 医療予約要件シートで「変更対象の予約日は、現在の予約日より未来日のみか、過去日も許容か」を確認
  - 過去日が許容されないなら、プロンプトに「入力日付が基準日より前の場合は NO_RESULT」という制約を追加検討

### I-002: 複数日付検出の複雑ケース
- **修正担当**: — （仕様確認推奨）
- **対象モジュール**: OpenAI_予約日_変更 / OpenAI_予約日_キャンセル
- **問題**: 
  - プロンプト（line 762）で「複数日付が含まれる発話は NO_RESULT」と定義されている
  - しかし、「M月D日」と「YYYY年M月D日」の混在（例：「2026年と6月1日」）など、より複雑な統語的混在に対して、OpenAI の text 処理で確実に検出できるか検証が不十分
  - また、「MMDD形式の4桁」と「音声日付」の同時存在検出精度も確認が必要
- **テスト推奨**: 
  - tester.py で、複数日付混在の入力例をテストケースに追加
  - 例：「6月1日と7月15日」「2026年5月1日あたり」「6月1日で、今年ですよ」など

### 移管方針
- main.yaml の流れ: opening → API_DRJOY → 時間判別（acceptance_times）→ 2月アナウンス（一時運用）→ 個人・企業判別 → 用件聴取 → 内容聴取 → 予約日（変更/キャンセル時）→ 受診者名 subflow → 生年月日 subflow → 連絡先 subflow → 用件確認分岐 → クローズ TTS → termination
- **個人・企業判別 → 用件聴取の 2 段ヒアリング**: 個人ルートは用件 4 分岐（新規 / 変更 / キャンセル / 問い合わせ）、企業ルートは用件 2 分岐（新規 / 問い合わせ）。`用件.yaml` で `用件の確認へ`（個人 4 択）/ `用件の確認へ_1`（企業 2 択）の TTS が分かれる
- 個人企業判別の前に企業団体名・部署担当者名 subflow を企業側だけ通すため、Pattern A（OpenAI 直後分岐）→ 個人/企業 の context_match_router → 企業のみ `企業団体名・部署担当者名` subflow を経由してから用件聴取に合流
- 用件 = キャンセルのみ「予約キャンセルの流れ（その他診療科）」→ 終話、その他は「クローズ_1（携帯あり）」「クローズ（携帯なし）」で締め
- 受診者名 / 生年月日 / 連絡先は Gen3 標準サンプル subflow を完コピ。受診者名は患者氏名 PatientName 1 枠に充当
- リトライ上限到達系（用件失敗 / 内容失敗 / 生年月日失敗 / 企業失敗 / 企業個人失敗 / 予約日失敗）は **すべて `3回失敗` TTS（代表 0570-010-986 を 2 連呼）→ 切断**に集約。Gen3 では各 hearing/subflow の error_jump を共通 termination 1 本に集約

### 施設固有の特殊ルール
- **受付時間 (時間判別.yaml)**: 月曜は終日休み（祝日扱い）、火〜金 11:30〜15:15、土 8:30〜11:15、日祝・年末年始休。Gen3 では acceptance_times に 6 系統（平日 11:30 前 / 平日時間内 / 平日 15:15 以降 / 土曜 8:30 前 / 土曜時間内 / 土曜時間外）を圧縮、月曜と祝日は閉鎖時間として扱う
- **「2月アナウンス」一時運用**: 「こちらのお電話でのご予約は2月27日まで、それ以降は 0570-010-986 へ」を**毎回**冒頭で読み上げる（受付時間内ルート全件）。期間限定告知のため、現時点では director 判断で除外し customer_doc に番号変更の事実だけ残す（運用が現役なら復活）
- **「予約受付不可（6/25〜）」一時運用**: 7 月の予約受付終了告知 (`現在、お電話、及びご予約が大変混みあっており〜`)。これも期間限定 → 除外
- **「稼働前」アナウンス**: 「この電話番号はご利用開始前です」サンプル → 本番稼働後は除外。director 判断で flowname に組み込まない
- **代表電話 0570-010-986**: 「予約確定のお電話発信元」「3回失敗時のかけ直し先」「クローズ TTS の SMS 発信元」の 3 箇所で同一番号。すべて termination ブロックの TTS 内に固定埋め込み
- **協会けんぽ 加入事業所は FAX 受付**: `予約受付不可（6/25～）` 内に「協会けんぽ 加入事業所からのご予約はお電話ではなく FAX にてお願いいたします」あり。一時運用文言だが現行も同方針 → 企業ルート完了 TTS の末尾に必ず付記
- **携帯チェックで締め文言切替**: 倉敷・蘇生会・飯塚と同パターン。クローズ_1（携帯あり「お電話、もしくはショートメールにて」）/ クローズ（携帯なし「お電話にて」）。連絡先 subflow 戻り値 phonetype を context_match_router で分岐
- **キャンセル専用 TTS**: 「再予約をされない方は、お送りした資料、検体一式は破棄をお願いいたします」を保持。文言短縮しない
- **個人・企業判別の「上記以外ON」フォールバック**: 否定/肯定のいずれにも該当しない発話 → 再質問（最大 3 回）→ 失敗時 `企業個人失敗` → 3回失敗 TTS。Gen3 では individual hearing の retry_count=3 + error_jump 共通 termination で吸収

### director が見落としやすいポイント
- 用件分類は `個人：用件` と `企業：用件` 2 種のカスタムエンティティを併用 → Gen3 では 1 つの OpenAI 用件分類に統合し、前段の個人/企業 context を引き継いで個人時は 4 択、企業時は 2 択を SKILL_A プロンプトで切り替える設計（director 判断）
- 「もしもし」検知は他施設健診と同じ countもしもし パターン（1 回目「はい、聞こえております」、2 回目以上「『もしもし』と聞こえております」）→ 用件 hearing の救済語扱い
- `現在の予約日/予約希望日.yaml` と `現在の予約日/予約希望日_1.yaml` の 2 本セット: キャンセル時は予約日 1 回、変更時は現予約日 → 希望日 の 2 回連続聴取（_1 が現予約日、無印が希望日）→ Gen3 では同名の datetime hearing を変更ルート時のみ 2 回、キャンセル時は 1 回呼ぶ
- `内容フリー聴取.yaml` は **新規/変更/問い合わせ全用件で必ず通る**（main.yaml `用件 → 内容フリー聴取 → 現在の予約日/予約希望日` の固定ルート）→ Gen3 でも自由文 hearing として共通配置
- `企業団体名・部署担当者名.yaml` は企業ルート専用 → 個人ルートでは subflow ブロック自体を通さない設計
- `END_切断` の入力待ち=あり は Gen3 では terminal なので不要 → そのまま削除
- 「セリフ: none」の振り分け箱（用件の確認 / 携帯電話番号チェック / 携帯電話番号チェック_1 / 完了フラグ / 個人：用件分岐 / 電話切断へ）は独立ブロック不要 → context_match_router の一部か前段の next 書き換えで吸収
- Gen1 動的タグ `<? $予約日の読み ?>` `<? $入電番号 ?>` 等は subflow 内 echo_back に既に集約済み、メインフローの `<? ... ?>` 直接参照は無し。残骸を見つけたら `<% var %>` 構文へ
- **モジュール名・termination 名は ASCII または安全な日本語のみ**。丸数字（①②③⑩等）禁止
- **ContextMatchRouter.reference_module は subflow 参照時 `jump_{step名}`**（scaffold prefix）
- **CMR conditions は必ず最後に `match: "other"` + next 明示**
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止

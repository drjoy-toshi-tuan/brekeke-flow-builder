- **モジュール名**: `script_最後の質問分岐`

- **問題**: 現在のスクリプトは `$runner.getModuleResult("OpenAI_用件_個人")` で用件を取得している。しかしリトライ超過（`リトライ_用件_個人` No more → `saveCtx_用件_問い合わせ_個人`）の場合、`OpenAI_用件_個人` の最終結果は `"NO_RESULT"` や `"TIMEOUT"` であり `"問い合わせ"` と一致しない。結果として `result = "other"` となり `Jump_RAG_終話前` が実行される。設計意図「classification=問い合わせ → 最後の質問スキップ」に反する。

  var cls = $runner.getModuleResult("OpenAI_用件_個人");

### W-003: 予約日復唱リトライ超過時に未確認日付でフロー継続（設計判断）

- **ファイル**: `output/json/prompted_聖マリアヘルスケアセンター_健診.json`
- **修正担当**: 人間確認
- **対象モジュール（5件）**:

  | リトライモジュール | No more 先 | 意味 |
  |---|---|---|
  | `リトライ_復唱_予約日_変更` | `変更内容_聴取` | 未確認の予約日で変更内容聴取に進む |
  | `リトライ_復唱_予約日_キャンセル` | `個人共通後段_氏名` | 未確認の予約日で氏名収集に進む |
  | `リトライ_復唱_予約日_変更_企業` | `Jump_氏名聴取_企業変更` | 同上（企業） |
  | `リトライ_復唱_予約日_キャンセル_企業` | `Jump_氏名聴取_企業キャンセル` | 同上（企業） |
  | `リトライ_復唱_予約日_二次受診` | `再検査内容_聴取` | 未確認の受診日で次へ進む |

- **問題**: 予約日の復唱確認リトライが上限に達した場合、`saveCtx_予約日` で日付は保存されているが復唱確認が取れていない状態のまま次のステップに進む。誤認識の日付が `reservationDate` に格納されたまま折り返し電話が発生するリスクがある。
- **設計書の記述**: 設計書にリトライ超過時の明示的な挙動定義がなく、generator が設計上の継続ルートをデフォルトで選択したものと推定される。
- **判断指示**: 以下のいずれかを設計として確定し、必要であれば generator に修正指示を出すこと:
  - **A（現状維持）**: 2回リトライで確認できなければ未確認日付のまま継続（折り返し電話で担当者が確認する前提）
  - **B（代表案内）**: 予約日が確定しない場合は代表電話に誘導（`完了フラグ_代表案内` に遷移）

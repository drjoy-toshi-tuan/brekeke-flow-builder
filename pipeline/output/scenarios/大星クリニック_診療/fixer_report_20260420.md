## 検出事項

### C-001: 用件確認 — NO_RESULT が JSON conditions で未対応
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_用件確認
- **問題**:
  - プロンプトの「STEP3：それ以外」で NO_RESULT を出力する設計
  - JSON conditions には以下の4つの分岐のみ: 診療 → jump_氏名聴取 / 予防接種 → jump_氏名聴取 / その他 → jump_氏名聴取 / 対象外 → END_FAQなし案内
  - NO_RESULT を出力した場合、JSON側で対応する next: が定義されていない
  - retry_failure: end_failure は「リトライ上限時」の処理であり、プロンプト内部で NO_RESULT が出力された場合の処理ではない
- **修正方針**:
  - プロンプト内で NO_RESULT を出力する条件を STEP2 の語句一致判定で捕捉し、「上記いずれにも該当しない → 対象外 を出力」に変更する
  - または、JSON conditions に NO_RESULT を処理する分岐を追加（next: を定義）
  - **推奨**: NO_RESULT 条件を「対象外」の定義に組み込み、プロンプトで NO_RESULT を出力せず「対象外」のみ出力に統一

### C-002: 診療科 — NO_RESULT 出力と JSON no_result_default の齟齬
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_診療科
- **問題**:
  - プロンプトの「STEP3」で NO_RESULT を出力する設計
  - JSON の no_result_default: 登録なし により、NO_RESULT が「登録なし」に自動マッピングされると仮定
  - ただし、generate_by_OpenAI モジュールが NO_RESULT → 登録なし の変換を行うか明示的に記載がない
  - プロンプト内で「NO_RESULT」を明示的に出力する設計のため、OpenAI の出力値として「NO_RESULT」がAPIレスポンスに含まれるリスク
- **修正方針**:
  - プロンプト内で「リスト外診療科（当院にない科）」を「登録なし」として出力するに変更
  - 無音・フィラー・意味不明の場合も「登録なし」を出力
  - プロンプト内に「NO_RESULT」という文字列を含めない
  - つまり、STEP3 を「リスト外や無音の場合は『登録なし』を出力」に修正

### W-001: 複合意図への防御文言欠落
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_用件確認、OpenAI_診療科、OpenAI_希望ワクチン
- **問題**:
  - ユーザーが複数の選択肢を同時に発話した場合（例：「診療と予防接種の両方」「脳神経外科か皮膚科か迷ってるんですが」）への防御が明記されていない
  - 現プロンプトは「複数一致した場合は上から優先」という「上から順」の優先順位ルールのみで、複合意図を排除していない
  - 「意味推測は禁止」「類義語拡張は禁止」という絶対ルールで間接的にカバーされているが、複合意図の明示的な拒否文言がない
- **修正方針**:
  - プロンプトインジェクション対策セクションに「複合意図（複数の選択肢の同時指定）への防御」を明示的に追加
  - 例：「複数の選択肢が含まれる場合（『診療と予防接種両方』『〇科か〇科か迷っている』等）は、上から優先順位の高い1つのみを採用する。『どちらがメインですか？』という反問は行わない」

### W-002: 出力制約「1語のみ」と実際の出力値の矛盾
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_診療科、OpenAI_希望ワクチン
- **問題**:
  - **診療科**: 出力制約は「1語のみを出力すること」（L136）だが、実際の出力値には「登録なし」（2語）が含まれる
  - **希望ワクチン**: 出力制約は「1語のみを出力すること」（L274）だが、実際の出力値はすべて複数語（「インフルエンザワクチン」「子宮頸がんワクチン」等）
  - この矛盾により、プロンプト実装者や保守者が「実際は何語まで許容されるのか」を誤解する可能性あり
  - 出力安全性の観点からも、制約が曖昧では OpenAI の出力が逸脱するリスクが増加
- **修正方針**:
  - **案1**（推奨）: 出力制約を「以下の複合語のいずれか1つのみを出力すること」に修正
  - **案2**: 医療用語として「診療科名」「ワクチン名」は複合語が標準であることを明示し、「複数の診療科名を並列出力しない」等の明確な制約に変更
  - 実装側が従うべき制約を「単語数」ではなく「リストからの単一選択」に焦点を当て直す

### W-003: ワクチン選択で NO_RESULT 時の処理が不明確
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_希望ワクチン
- **問題**:
  - プロンプトの STEP3 で NO_RESULT を出力する設計だが、JSON conditions では「match: default → jump_診察券番号聴取」のみ定義
  - NO_RESULT が出力された場合、JSON が default に分類するのか、別の処理に遷移するのか明示されていない
  - 設計書の notes では「リスト外ワクチンの場合は『当院に〇〇はございませんが、〇〇でしょうか？』と近いワクチンをレコメンド」と記載だが、レコメンド後の肯定応答をどう処理するか（再度このモジュールを呼び出すのか、default で次へ進むのか）が不明確
- **修正方針**:
  - プロンプト内で明示的に「リスト外ワクチン」を検出し、『当院にございません』と出力する仕様に変更
  - または、JSON の retry_failure: skip に対応して、リトライせずに「ワクチン: 未確認」等を記録して次へ進む設計に明示
  - レコメンド機能の実装詳細（肯定応答の判定、再分岐の有無）を設計書と同期させる

### C-002: 年推定ロジックの曖昧さ（過去日判定）
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_変更_予約日, OpenAI_キャンセル_予約日
- **重大度**: Critical
- **問題内容**:
  - STEP3「年の推定」（lines 418-423, 684-687）で、月日のみの入力について：
    ```
    発話された月日が、基準日の月日より過去 → 翌年（基準日の翌年）を採用
    ```
    という指示があるが、**「翌年が妥当か、現在の予約日が妥当か」を判断するための曖昧さがある**
  - 例：基準日 2026-05-01、入力「3月4日」
    - 3月4日 < 5月1日（過去）→ 翌年 2027-03-04 確定
    - しかし、ユーザーの意図は「来年の3月4日」なのか、「今年の5月以降で3月に関連する別の予約」なのか不確定
  - STEP4「有効範囲チェック」（lines 427-431）で「基準日から12ヶ月後の同日以内」とあるが、**「過去日は無効」という明示的な条件がない**
    - 現在の計算では、基準日前の日付が確定後の有効範囲内に含まれる可能性あり
  - 設計書（line 497）の簡潔な注記「過去日なら +1 年で確定」との齟齬がある
- **修正方針**:
  ```
  STEP3 の「年の推定」セクションを以下のように明確化：
  
  【月日のみの入力における年推定ルール】
  入力に年が含まれない場合（パターン B / C / E）：
  1. 発話された月日が、基準日の月日と同じか以降 → 基準日と同じ年を採用
  2. 発話された月日が、基準日の月日より過去 → 翌年を採用（基準日の次の同一月日）
  
  【例示】
  基準日: 2026-05-01（5月1日）
  - 入力「5月15日」→ 2026-05-15（同年）
  - 入力「6月1日」→ 2026-06-01（同年）
  - 入力「3月4日」→ 2027-03-04（翌年、なぜなら 3月4日 < 5月1日）
  - 入力「4月30日」→ 2027-04-30（翌年、なぜなら 4月30日 < 5月1日）
  
  STEP4 の「有効範囲チェック」に以下を追加：
  - 確定した日付が基準日より過去になっていないことを確認
    （翌年推定でも過去になる場合は NO_RESULT）
  ```

### W-001: 複合意図への防御の曖昧さ
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_用件確認
- **重大度**: Warning
- **問題内容**:
  - line 46 に「複数のラベルが同時にキーワードマッチした場合 → 上から優先」と記載されているが、**「複数の異なる用件が同時に述べられた場合」への明示的な対応がない**
  - 例：入力「予約をキャンセルして、新しい予約をしたいです」
    - STEP2 または STEP3 で「キャンセル」と「予約」（新規）の2つのキーワードがマッチ
    - 現在のルールでは「上から優先」で最初の1つのみ採用だが、この複合意図が正しく判定されるか検証不可能
  - echo_back=true の仕様（汎用 #data# 復唱）を考えると、ユーザーが複数意図を述べた場合、復唱後に「これだけでいいですか？」確認がないまま分岐する可能性がある
- **修正方針**:
  ```
  「# 判定アルゴリズム」セクションに以下の「複合意図処理」を追加：
  
  【複合意図の処理】
  入力に複数の異なる用件キーワードが含まれた場合：
  - STEP2（数字モード）で判定できる場合（例：「1番」「2番」の明示）→ その数字のみ有効、他キーワードは無視
  - STEP3（語句一致）で複数ラベルがマッチした場合 → 上から順に最初にマッチしたラベルを採用し、他は無視
  - 例：「キャンセルして新規予約」→「キャンセル」は STEP3 でマッチ → 「予約キャンセル」を確定
       「新規予約」は無視（新規予約は本フローでは受け付けないため）
  
  ユーザーに対しては、復唱時点で「確認できたご用件は【予約キャンセル】ですね」と 
  **判定結果のみを復唱し、他の用件は応答対象外**であることを自動案内する（TTS側で実装）
  ```

### W-002: DTMF 数字モード と語句一致モードの遷移ルール曖昧性
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_用件確認
- **重大度**: Warning
- **問題内容**:
  - lines 45, 61-106 で「一度一致したら即終了」という指示と「部分マッチ方針」（line 105）が定義されているが、**STEP2 と STEP3 の評価順序が曖昧**
  - 例：入力「1番でお願いします」
    - STEP2: 「1番」に部分マッチ → 「予約変更」判定か、
    - STEP2 完了後、STEP3 は実行されないのか、それとも STEP3 も確認するのか？
  - 実装上、以下2パターンが考えられるが、プロンプト内に明示がない：
    1. **STEP2 マッチ → 即終了**（STEP3 スキップ）
    2. **STEP2 マッチなし → STEP3 実行**（STEP2 と STEP3 は排他的）
- **修正方針**:
  ```
  「# 判定アルゴリズム」セクションに以下を明記：
  
  【STEP2 と STEP3 の関係】
  - STEP2（数字モード判定）で数字パターンが部分マッチした場合
    → その判定結果を即座に確定し、STEP3 へは進まない（早期終了）
  
  - STEP2 で数字パターンがマッチしない場合のみ
    → STEP3（語句一致判定）を実行
  
  - 例：
    入力「1番」 → STEP2 で「1番」マッチ → 「予約変更」確定 → STEP3 実行せず
    入力「変更」 → STEP2 でマッチなし → STEP3 で「変更」マッチ → 「予約変更」確定
    入力「1番で変更」 → STEP2 で「1番」マッチ → 「予約変更」確定 → STEP3 の「変更」は評価しない
  ```

### W-003: 相対日付計算ロジックの対応パターン不明確
- **修正担当**: prompter  <prompter>

    - 「再来週の月曜日」（「来週」のみサポート、「再来週」は不明）
    - 「来月の1日」「先月」（月指定の相対表現）
  - ユーザーが「今週の木曜日」と言った場合、基準日によっては過去日になるため NO_RESULT となるべきだが、プロンプト内にこのルール記載がない

  ```
  - 「明日」→ 基準日 + 1日
  - 「明後日」→ 基準日 + 2日
  - 「来週の{曜日}」（例：来週の火曜日）→ 基準日から次に来る該当曜日
  
  対応しない表現（NO_RESULT）：
  - 「今週の{曜日}」（基準日による過去判定のため）
  - 「再来週」（曜日指定なし、または「再来週の{曜日}」）
  - 「来月」「先月」「今月末」（月単位の相対表現は複雑なため）
  - 「3週間後」等（期間指定）
  ```

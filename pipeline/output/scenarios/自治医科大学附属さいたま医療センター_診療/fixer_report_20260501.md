### 3.3 予約日_聴取での有効範囲チェック問題

**Line 1388-1392:**
```
## 【STEP4：有効範囲チェック】
抽出した日付が以下の条件をすべて満たさない場合は NO_RESULT とする。
- 日付がカレンダー上に実在すること（うるう年考慮）
- 基準日と同じ、または基準日より未来の日付であること（過去日は採用しない）
- 基準日から3ヶ月後の同日以内であること
```

**問題**: 変更・キャンセルルートでは、ユーザーが「既に確定している過去の予約」を変更・キャンセルする必要があります。
現在のロジックでは過去日付を NO_RESULT として排除するため、**既存予約の変更・キャンセルが不可能**になる可能性があります。

例:
- 今日: 2026年5月1日
- 既存予約: 2026年4月28日
- ユーザー: 「4月28日の予約を変更したい」
- 結果: NO_RESULT（過去日排除）→ **予約変更失敗**

→ **分岐判断の正確性に致命的な欠陥**

### C-002: 予約日_聴取で過去日付が排除され、既存予約の変更・キャンセルが不可能
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_予約日_聴取
- **重大度**: Critical
- **問題**:
  - STEP4の「基準日と同じ、または基準日より未来の日付であること（過去日は採用しない）」ロジックでは、
    既に成立している予約（必ず過去日またはシステム日以前）の変更・キャンセルができない。
  - 例: システム日=5月1日、既存予約=4月28日 → ユーザーが「4月28日を変更したい」と発話 → NO_RESULT
  - これはフロー設計の変更・キャンセルルートと矛盾。
- **修正方針**:
  1. 設計書を確認し、予約日_聴取が「再診/初診ルート（予約を新規作成）」と「変更/キャンセルルート（既存予約を参照）」で異なるロジックが必要か検証
  2. 変更/キャンセルルートの場合：過去日付を許容し、合理的な範囲（例：過去3ヶ月以内）で受け付ける
  3. 新規予約ルートの場合：未来日付のみ（現在実装）でよい
  4. 必要に応じてプロンプトを2つに分割（predict_date_reserve / predict_date_modify_cancel）

### C-001: enum モジュール全体の NO_RESULT 出力値ミスマッチ <prompter>

- **修正担当**: prompter
- **対象モジュール**: 
  - OpenAI_個人情報同意確認
  - OpenAI_連絡方法聴取
  - OpenAI_用件確認
  - OpenAI_受診歴確認
  - OpenAI_予約_紹介状確認
  - OpenAI_予約_医師指定確認
  - OpenAI_小児科_ワクチン確認
  - OpenAI_小児科_インフル確認

- **問題**: 
  - 出力仕様に「NO_RESULT」が記載されているが、設計書の hearing_items では output_labels に NO_RESULT が定義されていない
  - JSON の conditions ブロックが「NO_RESULT」を認識しないため、OpenAI がこの値を出力した場合、分岐先が未定義となる可能性がある
  - つまり、無音・フィラーのみ等の場合に NO_RESULT を出力すると、IVR フローが不正状態に陥る

- **修正方針**:
  1. 各プロンプトの「出力仕様（厳守）」セクションから「NO_RESULT」を削除
  2. 代わりに、判定不可時の動作を明確に記述：
     - enum 系: 「一致しない場合は NO_RESULT を出力しない。代わりに、★確定できない場合は、設計書の no_result_default に従い以下の値を出力すること」と記述
     - 各モジュールの no_result_default 値を【STEP4/STEP5：それ以外】に明記
  3. 例:
     ```
     - 個人情報同意確認: 「無音・フィラーのみ → 否定 出力（scaffold で default → END_同意拒否 へ遷移）」
     - 連絡方法聴取: 「無音・無関係発話 → 空文字 を出力（scaffold で 電話 相当に変換）」
     - 用件確認: 「無音・フィラーのみ → NO_RESULT は出力禁止。scaffold の context_match_router で default → END_汎用代表案内 処理へ」
     ```

### C-002: 診療科プロンプト「登録なし」と「NO_RESULT」の出力値曖昧性 <prompter>

- **修正担当**: prompter
- **対象モジュール**: 
  - OpenAI_新規_診療科判定
  - OpenAI_再診_診療科聴取
  - OpenAI_変更_診療科聴取
  - OpenAI_キャンセル_診療科聴取

- **問題**:
  - 出力仕様に「登録なし」と「NO_RESULT」が並記されている
  - 設計書では no_result_default="登録なし" が指定されているが、プロンプトが「NO_RESULT」を出力する場合の処理が不明
  - ユーザーが「わかりません」「不明」と言った場合、
    - 設計書期待: 「登録なし」を clinicalDepartment に格納
    - プロンプト実装: 「NO_RESULT」を出力する可能性がある
  - この不整合により、JSON の conditions で「登録なし」を期待しているのに、実際には「NO_RESULT」が入る場合がある

- **修正方針**:
  1. 出力仕様から「NO_RESULT」を完全に削除
  2. STEP2・STEP3・STEP4 を整理し、「判別できない全てのケース → 登録なし」に統一
  3. 各診療科プロンプトのSTEP4「登録なし判定」セクションを強化：
     ```
     ### 登録なし判定
     以下のいずれかに該当する場合 → 登録なし を出力（NO_RESULT は一切出力しない）
     
     - わからない
     - 分からない
     - わかりません
     - 不明
     - どこだろう
     - 決まっていない
     - STEP2・STEP3 いずれにも一致しない入力（無音・フィラーのみ・意味不明・複数診療科含む）
     
     ### 出力値確認ルール
     【最終確認】
     必ず以下のいずれか一つのみを出力すること：
     1. 25診療科のいずれか（STEP2 完全一致 または STEP3 語句一致）
     2. 登録なし（STEP4 判定に該当）
     3. 他の値を出力してはいけない。特に NO_RESULT は絶対禁止。
     ```

### 抽出した要件サマリー

- **冒頭ブロック**: incoming-classifier で MOBILE/FIXED/OTHER 分岐。OTHER → 非通知拒否アナウンス（status=7）。MOBILE/FIXED → 通常冒頭アナウンス
- **聴取順序固定**: 続柄 → 氏名 → 生年月日 → 連絡先電話番号 → 用件、の順序が要件
- **個人情報サブフロー**: 氏名・生年月日・電話番号は標準サブフロー（PatientName / DateOfBirth / Phone）を 1 枠ずつ使用
- **電話番号聴取の特殊分岐**: MOBILE は incoming_phone_number 確認型（Yes/No）/ FIXED は通常聴取 → 復唱。標準 Phone サブフロー側で吸収する想定
- **用件分岐**: 4 分類（診療予約・予約変更・キャンセル・予約日時確認）+ 「診療予約」の場合は更に「診察予約 / ワクチン予約」で再分岐（2 段 OpenAI）
- **5 用件別ルートの聴取項目厳守**: 各 classification で聴取する項目が厳密に決まっており、他項目の混入は禁止
- **「わからない」格納値の正規化**: medicalCardNumber=「わからない」/ clinicalDepartment=「登録なし」/ institution=「わからない」/ doctorOrclinic=「希望無し」
- **status 値**: 0/1/7/8/9 の 5 値割当（customer_doc 仕様。Gen3 では 0 不可だが「途中切断」相当は status 未確定のまま終話しなければ作業中扱いになるので運用上問題なし）
- **smsFlag 値**: 0/1/2/3/4/5 の 6 値割当（各終話パターンごとに異なる）
- **復唱項目**: 予約日 / 生年月日 / 連絡先電話番号 の 3 項目のみ
- **2 種類の診察予約終話**: 「受診歴無し」と「受診歴有り」で別文言 — context_match_router で Medical_history を見て分岐
- **FAQ 確認**: customer_doc §10 にルールはあるが FAQ ナレッジ未提供 — 当面 RAG サブフロー無しで構築

## 5. directorの判断ログ

| # | 判断箇所 | 選択肢 | 採用した判断 | 根拠 |
|---|---|---|---|---|
| 1 | 用件分岐 (4 値) | Pattern A（OpenAI 直後分岐） vs Pattern B（後段 ContextMatchRouter） | **Pattern A** | OpenAI の出力で直接分岐先が一意に決まるため。`conditions` に 3 値明示 + `match: "other"` で 4 値目（予約日時確認）を捕捉 |
| 2 | 診察_ワクチン確認 (2 値) | Pattern A vs Pattern B | **Pattern A** | classification を上書きするのが目的だが、分岐自体は OpenAI 直後で完結 |
| 3 | 受診歴 → 終話分岐 | OpenAI 直後 vs 後段 ContextMatchRouter | **Pattern B (context_match_router)** | 受診歴聴取から終話判定まで複数ステップ（紹介状確認・診療科・複数診療科・医師名・都合が悪い日・症状）を経由するため、終話判定は後段の Medical_history 参照で実施。`reference_module: "OpenAI_受診歴"` |
| 4 | 医療機関名「わからない」分岐 | output_format: text + 後段 router vs output_format: enum 直接 | **enum 直接（Pattern A）** | OpenAI に「わからない」検出と医療機関名抽出の両方を担当させ、conditions で `わからない` を分岐、`other` を 診療科聴取に流す。institution への格納は OpenAI 出力をそのまま反映 |
| 5 | 続柄 (identityConfirm) | enum vs text | **text** | customer_doc は「姉です」をそのまま格納と明記（「姉」のみへの省略禁止）。enum 化すると元発話の保持が困難。output_format: text + 用件キーワード防御は prompter 側で OpenAI プロンプトに反映 |
| 6 | 診療科の no_result | no_result_default 設定 vs リトライ | **no_result_default: "登録なし"** | customer_doc に「診療科の聴取で回答内容が確認できなかった場合（聴取失敗）、clinicalDepartment へ「登録なし」と格納すること」と明記。モジュール選定ガイド 2.5 のパターンに一致 |

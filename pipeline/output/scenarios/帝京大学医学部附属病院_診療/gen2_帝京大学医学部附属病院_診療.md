### 移管方針

- **Pattern B**（OpenAI直後分岐ではなく後段 ContextMatchRouter 利用）。用件分類（変更/キャンセル/確認/用件聴取失敗）に加え、終話時に `phone_type`（mobile/landline/ip050/withheld）での二重分岐が必要なため、複数 context 組合せで ContextMatchRouter が必須。
- **Date of Call Classifier は不要**（acceptance_times による営業時間分岐の明示記載なし。単一シナリオ=診療1 のみ）。ただし冒頭に「**当日確認ゲート**」が入るため、opening 直後に Yes/No 判定（generate_by_OpenAI Yes_No）→当日=代表案内終話、非当日=用件ヒアリング、という特殊分岐ブロックが必要。director 判断でフロー配置すること。
- **サブフロー必須**: 患者名（氏名）/ 生年月日（DOB）/ 連絡先電話番号（Phone）/ 診察券番号は全て既存サブフローを Jump to Flow で呼び出す。メインへインライン展開禁止。
- 用件ごとに「診察券番号→診療科→予約日→（希望日 or 理由 or 内容確認）→共通ヒアリング」という順序が厳守されており、**診察券番号が先・診療科が後**（通常シナリオと逆）。director はここを見落とさない。
- **終話チェーンは 8 種類**: 予約変更(携帯/固定), 予約キャンセル(携帯/固定), 予約確認(携帯/固定), 聴取失敗, 代表案内(当日/新規共通)、加えて回答不可切断終話。termination_patterns に全 status/smsFlag/checkpoint を明示すること。

### 施設固有の特殊ルール

- **当日確認ゲート（最優先）**: 冒頭アナウンス直後に「本日がご予約日となっているご予約でしょうか」を必ず先に聞く。「はい」「肺」「初診」「初めて」「新規」等→代表案内終話（status=2, smsFlag=0, checkpoint=2）。「いいえ」「違う」「pH」「DA」「DeNA」等→用件聴取へ。2回目も判定不能なら「ご回答の確認ができませんでした」で切断（status=0, checkpoint=0）。(customer_doc `## 1. 初回アクション`〜`## 4. 2回目も判定不能な場合の処理` 行809-847)
- **救済変換**: 「肺」「φ」→はい／「pH」「DA」「DeNA」→いいえ／「予約塩昆布」「エンコード」→予約変更／「406人」「よやくひゃくにん」→予約確認／「要約」「規約」「良薬」「意訳」「唯一」→予約（行770-783）。OpenAI prompt に救済マッピングを記載すること。
- **歯科口腔外科の誤認識補正**: 診療科質問の直後に限り「2日」「5日」「二日」「五日」等→歯科口腔外科に確定（月・曜日・相対日付が含まれる場合は除外）。診療科名を音声で復唱しない（行301-315）。
- **「新規」「初診」「しょしん」等のキーワード**は、診察券番号聴取中・用件聴取中のいずれに現れても即座に代表案内へ強制遷移（status=2, smsFlag=0）。retryCount 対象外（行498-508, 852-873）。
- **救急科を選択した場合**は変更/キャンセルフローで「**終話（聴取失敗）**」（END_聴取失敗_終話, smsFlag=4）に強制合流（行1063, 1096）。director は「救急科→聴取失敗終話」のエッジを忘れずに引く。
- **期間表現は「3診療日以内」**（他施設の「3営業日」と異なる。終話文言で原文厳守）。
- **status/smsFlag 最終値の確定タイミング**: 終話アナウンス（「それでは失礼いたします。」で終わる文言）を実際に音声出力した直後でなければ値を書き換えない。それ以外は status=0, smsFlag=0, checkpoint=0, endpoint=途中切断（行34-51, 607-611）。termination 以外の generate_by_OpenAI プロンプトに「終話文言の出力前は絶対にstatusに1/2を入れない」の制約を埋め込むこと。
- **phone_type=withheld（非通知）も終話は「固定電話用」**（行376-378）。振分は mobile/ip050→携帯用終話、landline/withheld→固定用終話の2択。
- **追加の質問アナウンス直後切断は必ず途中切断扱い**（smsFlag=0, status=0, checkpoint=0, endpoint=途中切断）。終話アナウンスが出ていないため通話完了扱いにしない（行1229-1237）。
- **固定発話厳守**: 用件聴取「ご用件を、次の3つのうちのいずれかでお話しください。予約変更、予約キャンセル、予約確認、それではお話しください。どうぞ。」、追加の質問「他の診療科に関するお問い合わせにつきましては、改めてAI電話にておかけ直しください。他にご質問やお伝えしたいことがございましたらお話しください。なければ「ありません」とお話しください。」等、6章の文言は**厳密一致**で gen_properties に渡すこと（行1151-1256）。
- **連絡先電話番号の確認フロー**: incoming_phone_number を「今おかけいただいている番号でよろしいですか」で確認→「はい」→確定／「いいえ」→新規番号聴取→「{}でよろしいですか」で再確認（1回限定）→phone_type 再判定。incoming の phone_type を流用してはいけない（行520-522, 1216, 1223）。

### director が見落としやすいポイント

- **function スキーマに注意**: `classification` の enum に `再診` が残っている（行1653-1663）が、シナリオ内に再診フローの実装は無い。これは旧仕様の残滓の可能性だが、function スキーマは **customer_doc の #function ブロックを全件そのまま転記**すること（勝手に削除しない）。
- **smsFlag enum は `0〜5`** が定義されている（行1801-1808）が、使用されるのは 0/1/2/3/4 のみ。5 は予備。enum は原文通り。
- **classification の enum に空文字 `""` が含まれる**（行1661）。途中切断時の初期値用。termination_patterns の `endpoint="冒頭切断"` / `途中切断` パターンで classification="" を出力すること。
- **status と smsFlag の対応表（終話種別ごと）**:
  - 予約変更(携帯/固定): status=1, smsFlag=1
  - 予約キャンセル(携帯/固定): status=1, smsFlag=2
  - 予約確認(携帯/固定): status=1, smsFlag=3
  - 聴取失敗: status=1, smsFlag=4
  - 代表案内(当日/新規): status=2, smsFlag=0, checkpoint=2
  - 回答不可切断: status=0, checkpoint=0（smsFlag 指定なし＝0 保持）
- **冒頭切断（opening 直後に無発話で切断）**: endpoint="冒頭切断", status=0, smsFlag=0, checkpoint=0。当日確認ゲート以降の切断は endpoint="途中切断"。
- **「変更」「キャンセル」「確認」三択以外が 3 回目まで続いた場合、4回目を待たずに「かしこまりました。それでは担当者から詳細を確認いたしますので、いくつか質問させていただきます。」で classification="用件聴取失敗" に強制遷移→診療科から順に聴取**（行933-957）。3 回リトライのループ制御が必要。
- **「変更を希望する診療科」と「確認したい診療科」など、用件ごとに質問文言が微妙に違う**。共通サブフロー使用時も呼び出し前 TTS（前置き）で分岐する設計にすること（行1157-1167）。
- **改訂履歴**: 冒頭に「2026/04/20 10:40 携帯電話・固定電話の終話アナウンス分岐させる」の記載あり（行3-4）。phone_type による終話分岐が **今回新規追加された要件**。この分岐を絶対に落とさない。

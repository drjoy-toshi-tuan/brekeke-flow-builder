### W-001: 山城聞き取り - "看護師" キーワードマッチングの曖昧性

**修正担当**: prompter  <prompter>

**対象モジュール**: OpenAI_山城聞き取り

**問題**:

山城聞き取りプロンプト（lines 387-400）では、山城諒子（看護師）の判定キーワードに以下が含まれています：

```
### 山城諒子
- 山城諒子
- 諒子
- りょうこ
- リョウコ
- ryouko
- ryoko
- 看護師        ← キーワード
- 看護師さん     ← キーワード
- 看護師の山城   ← キーワード
- 看護師の山城さん
- ナース         ← キーワード
- 山城諒子さん
```

プロンプトの STEP2「語句一致判定」では `部分一致可` を明記しており、判定アルゴリズムは「上から順に機械的に評価」「一度一致したら即終了」としています。

**潜在的な問題シナリオ:**

1. **ユーザーが「看護師ではない人」と言った場合:**
   - 入力: "看護師ではない人"
   - マッチング: "看護師" を含む → 山城諒子に判定される
   - **結果: 誤判定** （ユーザーは諒子を選びたくない意思）

2. **ユーザーが「看護師以外」と言った場合:**
   - 入力: "看護師以外"
   - マッチング: "看護師" を含む → 山城諒子に判定される
   - **結果: 誤判定**

**設計上の想定:** 
設計書 line 335 に「ユーザーは下の名前（碧生 / 諒子）、氏名フルネーム、または「看護師」という役職で回答します」と明記されており、想定される応答は肯定的な選択（「看護師の人」や「諒子」）です。負の応答（「看護師ではない」）は想定外です。

しかし、実際の通話では以下のような応答が起こる可能性があります：
- ユーザーの言い間違い
- 音声認識エラー（"諒子" → "ろうしではない" など誤認識）

**推奨改善案:**

1. **STEP1 正規化に否定形フィルタを追加:**
   - 「ではない」「以外」「じゃない」を含む入力を事前に検出 → NO_RESULT に振り分け

2. **代替: マッチングを逆にする:**
   - 山城碧生をまず判定して、それ以外（または看護師キーワード）で諒子に分岐

3. **代替: 文脈的な判断:**
   - 否定形を含む複合的な入力は NO_RESULT として再確認を促す

**重大度判定:**
- 攻撃性: 低（ユーザーの誠実な誤言語化が原因）
- 発生可能性: 低～中（多くのユーザーは直接的に選択肢を選ぶ）
- 影響度: 中（誤転送が発生する可能性）

→ **WARNING 判定：改善を推奨するが、3回リトライで大多数は正常に分岐するため CRITICAL ではない**

### I-001: 英字キーワードの大文字小文字正規化に関する UX 案

**修正担当**: —（参考情報）

**対象モジュール**: OpenAI_大城聞き取り, OpenAI_下地聞き取り, OpenAI_山城聞き取り

**提案**:

プロンプトの STEP1「入力正規化」セクションでは、以下のように記載されています：

```
3. 全角英数字（Ａ-Ｚａ-ｚ０-９）→ 半角（A-Za-z0-9）
```

しかし、**大文字と小文字の統一が明記されていません**。

例えば：
- minoru（小文字）はマッチするが
- MINORU（大文字）でマッチするかは不明確

実際の STT エンジン（AmiVoice）の出力仕様を確認の上、以下のいずれかを採用することを推奨します：

**選択肢 A: 大文字に統一**
```
3. 全角英数字（Ａ-Ｚａ-ｚ０-９）→ 半角（A-Za-z0-9）
4. 英文字を大文字に統一（a-z → A-Z）
5. カタカナはひらがなに変換...
```

**選択肢 B: 小文字に統一**
```
3. 全角英数字 → 半角
4. 英文字を小文字に統一（A-Z → a-z）
5. カタカナはひらがなに変換...
```

AmiVoice が「minoru」を出力する傾向があれば、**選択肢 B（小文字統一）** が推奨です。

**重大度**: INFO（現状でも STT が一貫した形式を出力していれば実害なし）

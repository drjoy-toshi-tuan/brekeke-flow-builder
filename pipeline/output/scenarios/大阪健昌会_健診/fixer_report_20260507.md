### W-001: 波ダッシュ（～）文字の符号化一貫性確認不足
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_希望時間帯（行 663）
- **問題**: 
  - 行 663 で「※出力は『波ダッシュ ～』を含む正確なラベル文字列で出力すること。」と明記されています
  - 出力ラベル「8時～9時台」に含まれる波ダッシュが、設計書 YAML（行 487-493）と同じ Unicode ポイント（U+301C）であることが確認されていません
  - 波ダッシュの亜種（小チルダ U+301E ～ や全角チルダ U+FF5E ～）との混在は、後続の JSON パースやマッチング処理で不一致を引き起こします

- **具体的なリスク**:
  ```
  設計書 YAML: value: "8時～9時台"（U+301C: WAVE DASH）
  プロンプト: 出力: "8時～9時台"（U+301E: SMALL TILDE だった場合）
  
  → JSON マッチング失敗
  → コンテキスト更新時に値不一致
  ```

- **修正方針**: 
  - テスト時に以下コマンドで文字コードを検証してください：
    ```bash
    # プロンプト内の波ダッシュを確認
    grep -a "8時～9時台" prompts_大阪健昌会_健診.md | od -An -tx1
    
    # 設計書 YAML の波ダッシュを確認
    grep -a "8時～9時台" 設計書_大阪健昌会_健診.yaml | od -An -tx1
    
    # 両者が同じバイト列（... e3 80 9c ...）で一致することを確認
    ```
  - UTF-8 で "～"（U+301C WAVE DASH）の byte sequence は `e3 80 9c` です

### W-002: 出力制約の曖昧性（「1語」の定義）
- **修正担当**: prompter  <prompter>
- **対象モジュール**: 全列挙型プロンプト（問合せ先施設名、用件確認、受診票有無確認、希望時間帯、オプション追加変更有無、予約日確定有無）
- **問題**: 
  - 複数のプロンプトで「以下のいずれか1語のみを出力すること」と記載されています
  - しかし、出力値に「8時～9時台」「12時台以降の午後」など複数文字の複合ラベルが含まれています
  - 日本語の「1語」は通常「1つの単語」を意味しますが、このコンテキストでは「1つの出力値オプション」を意味しており、初読時に混同される可能性があります

- **具体的な混同リスク**:
  ```
  読み手が理解する可能性のある2つの解釈：
  
  解釈1（正しい）: 「8時～9時台」全体を1つの値として出力
    出力: 8時～9時台
  
  解釈2（誤り）: 「8時～9時台」を複数に分割して出力
    出力: 8時 / 9時台（複数の「語」？）
  ```

- **修正方針**: 
  - より正確に以下のように言い換えてください：
    ```markdown
    # 出力仕様（厳守）
    以下のいずれか1つの値のみを出力すること：
    - 8時～9時台
    - 10時～11時台
    - 12時台以降の午後
    - 希望なし
    - NO_RESULT
    
    複合語や特殊文字を含むラベルは、文字列全体を1つの値として扱い、分割出力しないこと。
    解説・理由・句読点・記号・改行は一切出力しない。
    ```

### W-003: 複数キーワードマッチ時の優先度評価（具体例欠落）
- **修正担当**: prompter  <prompter>
- **対象モジュール**: OpenAI_問合せ先施設名（STEP3 行 106-142）、OpenAI_用件確認（STEP3 行 248-288）
- **問題**: 
  - 絶対ルール（複数箇所で行 40-41 など）で「複数のラベルが同時にキーワードマッチした場合 → 上から優先（評価順を厳守）」と明記されています
  - しかし、具体的なテストケースがドキュメントに含まれていません
  - 例えば、施設名プロンプトの「近畿」キーワード（行 114-120）に「禁忌」「金ピ」「現金」等の音似表現・類似文字が含まれており、複数マッチのリスクがあります

- **具体的な複合マッチシナリオ**:
  ```
  STEP3 キーワードマッチ評価順:
  1. 【うめだ】（行 106-110）
  2. 【近畿】（行 112-122）← 「禁忌」「金ピ」「現金」等が含まれる
  3. 【淀川】（行 124-133）
  4. 【福島】（行 135-142）
  
  ユーザー発話: 「禁忌だから近畿がいい」
  マッチ分析:
    - 「禁忌」が STEP3 の「近畿」キーワードに含まれる → マッチ候補
    - 「近畿」が STEP3 の「近畿」キーワードに含まれる → マッチ候補
    - 同じ「近畿」ラベルへの複合マッチだが、判定ロジックが不透明
  ```

- **修正方針**: 
  - 各プロンプトの STEP3 末尾に「複合マッチ判定例」セクションを追加してください：
    ```markdown
    ### 複合マッチ判定例（評価順優先）
    
    【例1】
    入力: 「禁忌だから近畿で」
    分析: 「禁忌」と「近畿」のキーワードが混在
    判定: 「近畿」キーワード（行 114）が最初にマッチ → 出力: 近畿
    
    【例2】
    入力: 「うめ近畿」
    分析: 「うめ」（うめだ）と「近畿」が混在
    判定: 【うめだ】（行 106）が評価順上位 → 出力: うめだ
    
    【例3】
    入力: 「金ピ福島」
    分析: 「金ピ」（近畿の類似表現）と「福島」が混在
    判定: 【近畿】（行 112）が評価順上位 → 出力: 近畿
    ```
  - tester が上記の複合マッチケースを検証カバレッジに追加してください

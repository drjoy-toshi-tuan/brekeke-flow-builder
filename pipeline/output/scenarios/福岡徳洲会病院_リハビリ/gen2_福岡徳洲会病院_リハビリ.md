### 移管方針
- Gen3 scenario_flow ブロック構造に直接マッピング。フローは極めて浅く、氏名→用件→終話の3ステップのみ。
- **Pattern 判定: Pattern A**（OpenAI 直後分岐なし）。分岐はデータ格納の status 判定のみで、終話前の`script`ブロックで`patientName`/`reason`の空文字チェックを行い END_成功 / END_失敗 に振り分ける構成。
- **Date of Call Classifier: 不要**。営業時間外・曜日別分岐の記載なし、受付時間制御は IVR 側（`acceptance_times`）で吸収する想定（customer_doc に記載なし → 浜口にヒアリング要）。
- **サブフロー化**: 氏名聴取は標準の`hearing_name`サブフロー（Jump to Flow）を使用。リハビリ特有聴取（用件）は本フロー内 hearing ブロックでよい。DOB/電話/診察券の聴取は customer_doc では **聴取していない**（function schema には存在するが空文字許容）ので、サブフロー呼び出し不要。
- 8ブロック型: `opening` → `hearing`(氏名) → `hearing`(用件) → `script`(status判定) → `termination`×2。generator 不要、scaffold 一発で出る想定。

### 施設固有の特殊ルール
1. **リハビリシナリオ初導入**: customer_doc の function schema に `classification`（欠席連絡/変更/お迎え時間/体験利用・新規契約申し込み/担当者会議/無回答）・`Preferred_date`・`Medical_history`・`company_name`・`company_check`・`checking_details`・`Fax_number`・`change_item` 等、**通所リハビリ特有の項目**が並ぶ。**通常の診療・健診テンプレートの項目をそのまま流用しないこと**。classification は customer_doc の聴取フローでは取得していない（AI 側で reason から判定する余地がある）が、本版は「超シンプルシナリオ」のため **reason にまるごと格納** する方針で良い。
2. **復唱・再確認の完全禁止**: §II, §1-8。氏名復唱もしない。Yes/No 確認ブロックは一切入れない。
3. **再質問は各項目1回のみ**。超過時は聞き取れたテキストをそのまま保存して次項目へ強制遷移（§1-4-1、YAML の `リトライ超過時: 用件聴取` 参照）。scaffold の retry 既定値と揃っているか確認。
4. **再質問文言は完全一致の固定文**。`{tts_g:申し訳ありません。うまく聞き取れませんでした。再度、お名前をお話しください。}` 等。**言い換え・意訳禁止**（§Ⅰ, §1-9）。氏名再質問のみ「ゆっくり話すとAIが…」の長文版を使用（customer_doc の §1-7 + §V の合成版、YAML 箱`患者氏名_再質問`のセリフを正として使う）。
5. **明示的出力禁止文字列**（§Ⅰ）: 「聞き取りませんでした」「もう一度お願いします」「〜でしょうか」「〜ですか」等。prompter はこれらを含むフォールバック文を書かないこと。疑問終助詞は使わず「お話ください」で統一。
6. **患者氏名はカタカナ変換**（§1-7）: generate_by_OpenAI の正規化プロンプトでカタカナ出力を強制。不一致時は聞き取れた表記で保存（再質問1回のみ）。
7. **数字表記ルール**（§1-5）: 「よん/し」→4、「ご/ごお/ごー」→5。"for"/"C"/"号"/"go" 禁止。正規化プロンプトで明示。
8. **status 判定**: `patientName` or `reason` のいずれか非空なら `1`、両方空なら `0`、代表案内時 `2`（customer_doc §Ⅲ と function schema description で微妙に揺れあり→ function schema 側の3値定義を採用）。
9. **着信元電話番号の先頭0保持**（§1-3）。`incoming_phone_number` 取得時に零サプレス禁止。
10. **口調**: 「ありがとうございます」「お気軽に」禁止、「おっしゃってください」→「お話ください」（§1-1）。終話も「失礼いたします」で締める。

### director が見落としやすいポイント
- function schema は **リッチ**（16フィールド必須）だが **聴取は2項目のみ**（patientName / reason）。残りは空文字で保存する前提。`required` に並んでいるからといって聴取ブロックを追加しないこと。
- 終話セリフは **成功/失敗とも同一文言**「お申込みを受付しました。失礼いたします。」。違いは `status` の値のみ。終話前に `script` ブロックで status を確定させる構造にする。
- customer_doc 内で「冒頭切断/時間外/電話転送/電話案内/途中切断/通話完了」の `endpoint` enum が定義されているが、本シンプル版では **時間外/転送は範囲外**。`通話完了` / `途中切断` / `冒頭切断` のみ現実的に発生する。director は endpoint 設定モジュール配置を忘れずに。
- 「FAQ対応はしない」（§1-2）: RAG サブフロー・FAQ 分岐は**一切組み込まない**。
- 改訂履歴の明示記載はないが、ファイル冒頭日付 `2026年1月28日 11:45 超シンプルシナリオ` とある。以前のリッチ版があった可能性あり → **現行 customer_doc のシンプル版のみを正**とし、function schema の未使用フィールドは空文字保存に留める。
- `classification` enum に `""`（空文字）が含まれる → 無聴取シナリオを許容する設計意図。director は enum 値をいじらず function schema をそのまま transplant する。

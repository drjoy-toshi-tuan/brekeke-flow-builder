## 1. 初回セットアップ

### 1-1. Git のインストール確認

ターミナル（コマンドプロンプトまたはPowerShell）を開いて、以下を実行します。

```bash
git --version
```

`git version 2.xx.x` のように表示されれば OK です。  
表示されない場合は [Git公式サイト](https://git-scm.com/) からインストールしてください。

---

### 1-2. 名前とメールアドレスの設定

GitHub に表示される自分の名前・メールを登録します。**初回1回だけ** 行います。

```bash
git config --global user.name "あなたの名前（例: Taro Hamaguchi）"
git config --global user.email "あなたのメールアドレス（GitHubに登録したもの）"
```

---

### 1-3. SSH鍵の設定（GitHub認証）

**初回1回だけ** 行います。SSH鍵を使うと、毎回パスワードを入力したり認証が切れたりしません。

**① 既存の鍵を確認する**

```bash
ls ~/.ssh/id_ed25519.pub
```

ファイルが表示されれば鍵はすでにあります。**② をスキップして③に進んでください。**  
`No such file or directory` と表示された場合は② へ。

---

**② SSH鍵を生成する（鍵がない場合のみ）**

```bash
ssh-keygen -t ed25519 -C "あなたのGitHubメールアドレス"
```

`Enter file in which to save the key` と聞かれたら **そのままEnter**（デフォルト場所に保存）。  
パスフレーズは任意です（空のままEnterでも可）。

---

**③ 公開鍵を GitHub に登録する**

公開鍵の内容を表示します。

```bash
cat ~/.ssh/id_ed25519.pub
```

表示された `ssh-ed25519 AAAA...` から始まる1行をコピーします。

次に GitHub にアクセスして登録します。

1. GitHub → 右上のアイコン → **Settings**
2. 左メニュー **SSH and GPG keys** → **New SSH key**
3. Title に `PC名 など任意の名前` を入力
4. Key に先ほどコピーした内容を貼り付け
5. **Add SSH key** をクリック

---

**④ 接続テスト**

```bash
ssh -T git@github.com
```

```
Hi TS-dong-nc! You've successfully authenticated, but GitHub does not provide shell access.
```

上記のように表示されれば設定完了です。

---

### 1-4. リポジトリのクローン（ローカルにコピー）

プロジェクトをパソコンに取得します。**初回1回だけ** 行います。  
**SSH形式のURL（`git@github.com:...`）を使用してください。**  
`https://github.com/...` 形式は使わないでください（認証切れが起きます）。

```bash
git clone git@github.com:TS-dong-nc/gen_flow.git
cd gen_flow
```

クローン後は `voicebot-flow-builder` フォルダが作成され、その中で作業します。

## 2. 案件作業の流れ

案件を1件担当するたびに、以下の **STEP 1〜5** を行います。

---

### STEP 1: 最新の状態を取得する

**作業を始める前に必ず実行してください。**  
他のメンバーの変更を取り込み、手元の環境を最新にします。

```bash
git pull origin master
```

> **ポイント:** pull を忘れると、古い状態でブランチを作ることになり、後で競合（コンフリクト）が起きやすくなります。

---

### STEP 2: ブランチを作成する

案件ごとに「ブランチ」という作業スペースを作ります。  
**master ブランチで直接作業してはいけません。**

```bash
git checkout -b feature/{施設名}_{フロー名}
```

**例:**

```bash
git checkout -b feature/藤田胃腸科病院_診療
git checkout -b feature/琉球大学病院_健診
```

> **命名ルール:**  
> `feature/` の後に `{施設名}_{フロー名}` をつなげます。  
> スペースや括弧（）は使わず、アンダーバー `_` で区切ってください。

ブランチが作成されたか確認するには:

```bash
git branch
```

現在のブランチの先頭に `*` が表示されます。

```
  master
* feature/藤田胃腸科病院_診療   ← 今いるブランチ
```

---

### STEP 3: 作業する

**顧客資料の配置**

受け取った資料（PDF・CSV等）を所定のフォルダに置きます。

```
output/scenarios/{施設名}_{フロー名}/reference/
```

例: `output/scenarios/藤田胃腸科病院_診療/reference/` に PDF を置く

**パイプラインの実行**

Claude Code で `@pl` を呼び出し、設計〜ビルドまで自動実行します。

```
@pl
```

---

### STEP 4: 変更をコミットする

作業が完了したら、変更内容を Git に記録（コミット）します。

**① 変更ファイルを確認する**

```bash
git status
```

追加・変更されたファイルの一覧が表示されます。

**② コミットするファイルを選択する（ステージング）**

今回の案件フォルダをまとめてステージングします。

```bash
git add output/scenarios/{施設名}_{フロー名}/
```

例:

```bash
git add output/scenarios/藤田胃腸科病院_診療/
```

> **注意:** `git add .` や `git add -A` は使わないでください。  
> 意図しないファイル（他の案件の作業中ファイル等）が含まれてしまうことがあります。

**③ コミットメッセージを書いて確定する**

```bash
git commit -m "feat: {施設名} {フロー名} 新規作成"
```

例:

```bash
git commit -m "feat: 藤田胃腸科病院 診療フロー 新規作成"
git commit -m "fix: 琉球大学病院 診療フロー モジュール名修正"
```

---

### STEP 5: リモートにプッシュする

自分のパソコン上の変更を GitHub（リモート）に送ります。

```bash
git push -u origin feature/{施設名}_{フロー名}
```

例:

```bash
git push -u origin feature/藤田胃腸科病院_診療
```

> `-u` は最初のプッシュ時だけ必要です。2回目以降は `git push` だけで OK です。

---

## 3. コミットのルール

### コミットメッセージの書き方

先頭に以下のプレフィックスをつけます。

| プレフィックス | 使う場面 | 例 |
|---|---|---|
| `feat:` | 新規案件の作成 | `feat: 恵佑会札幌病院 診療フロー 新規作成` |
| `fix:` | 既存フローの修正 | `fix: 恵佑会札幌病院 診療フロー 終話文言修正` |
| `docs:` | ドキュメントの更新 | `docs: 恵佑会札幌病院 確認レポート更新` |

### コミットの単位

- **1案件 = 1コミット** を基本とします
- 途中で大きな変更が発生した場合は複数コミットに分けて OK です
- 「作業中」「一時保存」のようなメッセージは避けてください

---

## 4. プルリクエストの作成

プッシュが完了したら、マネージャーにレビューを依頼するために **プルリクエスト（PR）** を作成します。

### GitHub のウェブサイトから作成する方法

1. [リポジトリのページ](https://github.com/TS-dong-nc/gen_flow) を開く
2. 黄色いバナー「**Compare & pull request**」が表示されたらクリック  
   （表示されない場合は「**Pull requests**」タブ → 「**New pull request**」）
3. 以下を入力して「**Create pull request**」をクリック

| 項目 | 内容 |
|---|---|
| タイトル | `feat: {施設名} {フロー名} 新規作成` |
| 本文 | 作業内容の概要、特記事項があれば記載 |
| Assignees | 自分の名前を選択 |

4. マネージャーにレビューを依頼します

> PR が master にマージされると作業完了です。

## 5. よく使うコマンド一覧

| やりたいこと | コマンド |
|---|---|
| 今いるブランチを確認 | `git branch` |
| ブランチを切り替える | `git checkout feature/xxx` |
| 現在の変更状況を確認 | `git status` |
| 変更内容の差分を確認 | `git diff` |
| 最新状態を取得 | `git pull origin master` |
| ファイルをステージング | `git add output/scenarios/xxx/` |
| コミット | `git commit -m "feat: xxx"` |
| プッシュ（初回） | `git push -u origin feature/xxx` |
| プッシュ（2回目以降） | `git push` |
| コミット履歴を確認 | `git log --oneline -10` |

## 6. 困ったときのQ&A

### Q. `git pull` したら「コンフリクト」が出た

他の人と同じファイルを編集していた場合に起きます。  
**自分で解決しようとせず、すぐにマネージャーに連絡してください。**

---

### Q. 間違ったファイルをステージングしてしまった

`git add` を取り消せます。

```bash
git restore --staged {ファイルパス}
```

---

### Q. まだプッシュしていないコミットを取り消したい

直前のコミットを取り消し、変更内容はそのまま残します。

```bash
git reset --soft HEAD~1
```

> **注意:** `--hard` オプションは変更内容も消えてしまうので使わないでください。

---

### Q. 間違えて master でコミットしてしまった

**プッシュする前に** マネージャーに連絡してください。  
プッシュ後は自分では対応できません。

---

### Q. 現在どのブランチにいるかわからない

```bash
git branch
```

`*` マークがついているのが現在のブランチです。  
master になっている場合はすぐに作業を止めて、ブランチを切り直してください。

---

### Q. プッシュしたら「rejected」と出た

```
! [rejected] feature/xxx -> feature/xxx (non-fast-forward)
```

リモートに自分がまだ取り込んでいない変更があります。  
`git pull origin feature/{ブランチ名}` を実行してから再度プッシュしてください。

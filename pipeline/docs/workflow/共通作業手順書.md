# 共通作業手順書 — ボイスボットフロー設計ガイドライン

## 1. フロー設計の基本原則

### 1.1 患者体験を最優先する
- 通話時間は **3分以内** を目標とする
- 1回の発話で聴取する情報は **1つだけ**
- 選択肢は **最大4つ** まで（5つ以上は分割）
- 「はい/いいえ」で答えられる確認を優先する

### 1.2 エラーに強いフローを作る
- 全てのSTT入力に **リトライ（2回）** を設定
- リトライ上限で **終話_失敗** に遷移
- タイムアウトは **30秒** を標準とする
- 終話_失敗には **「再度おかけ直しください」** ガイダンスを必ず含める

### 1.3 命名規則を厳守する

| 要素 | 規則 | 例 |
|---|---|---|
| フロー名 | `{施設名}${機能名}` | `兵庫県立こども病院$診療-Demo` |
| モジュール名（TTS） | `{ガイダンス内容}` | `冒頭ガイダンス`, `終話ガイダンス分岐` |
| モジュール名（STT入力） | `入力_{対象}` | `入力_確認_症状`, `入力_確認_診療科` |
| モジュール名（リトライ） | `リトライ_{対象}` | `リトライ_確認_症状` |
| モジュール名（分岐） | `分岐_{条件}` | `分岐_初診再診`, `終話ガイダンス分岐` |
| モジュール名（確認） | `確認_{対象}` | `確認_性別`, `確認_紹介元病院名` |
| モジュール名（遷移） | `移動_{遷移先}へ` | `移動_個人情報聴取_2へ` |
| サブモジュール | `save-history{番号}` | `save-history1`, `save-history2` |

## 2. モジュール設計テンプレート

### 2.1 標準的な聴取フロー（1項目分）

```
[TTS: ガイダンス発話]
    ↓
[STT: 音声入力受付]
    ├─ success → [次の聴取 or 確認]
    ├─ timeout → [リトライカウンタ]
    ├─ error → [リトライカウンタ]
    └─ no_result → [リトライカウンタ]
                      ├─ true（リトライ可） → [STT: 再度入力]
                      └─ false（上限到達） → [終話_失敗]
```

### 2.2 確認フロー（はい/いいえ）

```
[TTS: 「〇〇でよろしいですか？」]
    ↓
[STT: はい/いいえ入力]
    ├─ はい → [次のステップ]
    ├─ いいえ → [再入力へ戻る]
    ├─ timeout → [リトライカウンタ]
    └─ error → [リトライカウンタ]
```

### 2.3 Jump to Flow（フロー間遷移）

```json
{
  "type": "@General$Jump to Flow",
  "params": {
    "flowname": "drjoy^{施設名}${遷移先フロー名}",
    "properties": "プロンプト設定\nAmiVoice設定\nAPI設定"
  }
}
```

properties の設定項目:
- `{モジュール名}.prompt=` — 遷移先のTTSプロンプトを上書き
- `amivoice.*` — AmiVoice共通設定
- `office_id=` — 施設ID
- `pbx.db.name=` — DB名
- `context.settings.url=` — コンテキスト設定API
- `acceptance_times.url=` — 受付時間確認API
- その他API URL設定

## 3. TTS ガイダンス作成ガイド

### 3.1 基本フォーマット

```
{tts_g: ガイダンステキスト}
```

### 3.2 良いガイダンスの例

```
{tts_g: お電話ありがとうございます。こちらは、〇〇病院の自動応答サービスです。}
{tts_g: ご用件をお話しください。診察の予約、予約の変更、その他のお問い合わせ、のいずれかでお答えください。}
{tts_g: 申し訳ございません。うまく聞き取りができませんでした。再度、ご用件をお話しください。}
{tts_g: 大変恐れ入りますが、再度おかけ直しください。}
```

### 3.3 避けるべき表現

| 避けるべき | 推奨 |
|---|---|
| `1を押してください` | `いちを押してください` |
| `SMS` | `ショートメッセージ` |
| `TEL` | `お電話番号` |
| `etc.` | 具体的に列挙 |
| `〜等` （曖昧） | 具体的に列挙 |
| 1文60文字超 | 40文字以内に分割 |

## 4. profile_words 登録ガイド

### 4.1 フォーマット

```
単語 よみがな
```

1行1エントリ、スペース区切り。

### 4.2 長音揺れ対応

日本語の長音は複数の発音パターンがあるため、主要な揺れを全て登録する:

```
紹介状 しょうかいじょう
紹介状 しょーかいじょう
紹介状 しょうかいじょー
紹介状 しょーかいじょー
```

### 4.3 必須登録カテゴリ

- 診療科名（内科、外科、小児科...）
- 医療行為（診察、検査、手術...）
- 施設固有の用語
- 日付・時間の読み（ついたち、ふつか...）
- よくある患者の発話パターン

## 5. バリデーション基準

`schemas/validator.py` で以下を自動検証する:

1. **必須フィールド存在**: name, start, modules
2. **遷移先整合性**: nextModuleName が modules 内に存在
3. **孤立モジュール検出**: どこからも参照されないモジュール
4. **TTS形式検証**: prompt が `{tts_g: ...}` 形式
5. **パラメータ型検証**: timeout_ms が数値文字列、etc.
6. **リトライ構造検証**: STTモジュールにリトライが対応

## 6. Git 運用ルール

- **ブランチ**: `feature/{施設名}/{機能名}` で作業
- **コミットメッセージ**: `[generate] {施設名} 初稿作成` / `[review] {施設名} 校閲完了`
- **output/**: 最終成果物のみコミット（draft_* は .gitignore）
- **PR**: 校閲レポートをPR本文に記載

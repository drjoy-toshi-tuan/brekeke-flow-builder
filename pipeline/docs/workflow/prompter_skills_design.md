# Prompter Skills 設計

> prompterが1モジュールのプロンプトを作成するたびに、対応するパターンのSkillファイルを読み込んで使う。
> 毎回ゼロから生成するのではなく、検証済みのベースプロンプトを起点に施設固有の部分だけを差し替える。

---

## 設計思想

### 問題
- prompter.md内にテンプレート（構造の骨格）はあるが、**実際のプロンプト完全版**は含まれていない
- 20モジュール分のプロンプトを書く際、毎回prompter.md全体をコンテキストに入れるのはコスト面で非効率
- テンプレートだけでは判定ステップの具体的な書き方（完全一致リストの網羅性、STEP間の遷移記述）にブレが出る

### 解決策
- パターン別に **Skillファイル** を分離する
- 各Skillには **検証済みの完全なプロンプト実例** を1-2件含める
- prompterは1モジュールを書くたびに、該当パターンのSkillだけを Read して参照する
- Skillの実例を「ベース」として、施設固有の部分（選択肢、診療科名、TTS文言等）を差し替える

---

## ファイル配置

```
docs/ai/skills/
├── SKILL_A_classification.md      # 分類型（N択判定）
├── SKILL_B_yes_no.md              # はい/いいえ判定型
├── SKILL_C_date.md                # 日付変換型
├── SKILL_D_normalization.md       # 正規化型（リスト照合・診療科等）
├── SKILL_E_free_text.md           # 自由テキスト型（要約・FAQ等）
└── SKILL_F_phone.md               # 電話番号型（個人情報サブフロー用・参考）
```

---

## 各Skillファイルの構造

全Skillファイルは以下の共通構造を持つ:

```markdown
# Skill {パターン名} — {用途}

## このSkillの使い方
（prompterがどういう場合にこのSkillを読むか）

## 差し替えポイント
（ベースプロンプトのどこを施設固有に書き換えるか）

## ベースプロンプト（検証済み完全版）
（そのままコピペして差し替えれば使えるレベルの完全なプロンプト）

## 差し替え例
（具体的な差し替え前→後の例）

## 注意事項
（パターン固有の落とし穴・よくあるミス）
```

## Skill A: 分類型（N択判定）

### ソース
sample_flow.json の `openAI_確認_用件`（5択・DTMF+音声）

### ベースプロンプトに含まれるもの
- Role定義（N択選択判定エンジン）
- Context（TTS発話文言）
- プロンプトインジェクション対策
- 出力仕様（選択肢リスト + NO_RESULT）
- 絶対ルール
- STEP1: 入力正規化
- STEP2: 数字モード判定（DTMF対応・完全一致リスト付き）
- STEP3: 語句一致判定（部分一致・キーワードリスト付き）
- STEP4: STT誤変換補正（既知パターン）
- STEP5: それ以外→NO_RESULT

### 差し替えポイント
| 箇所 | 差し替え内容 |
|---|---|
| Context内のTTS文言 | 設計書セクション5のアナウンス文言 |
| 選択肢リスト（出力仕様） | 設計書セクション4の出力ラベル |
| STEP2の数字↔選択肢マッピング | 選択肢の数に応じて増減 |
| STEP3のキーワードリスト | 選択肢に対応するキーワードに差し替え |
| STEP4のSTT誤変換パターン | 設計書に記載があれば追加、なければ空 |

### DTMF入力なしの場合
- STEP2の数字モード判定で「DTMF半角数字のみ」の記述を除去
- 漢数字パターン（一、二、三…）は音声入力用に残す

## Skill B: はい/いいえ判定型

### ソース
sample_flow.json の `openAI_予約_通院歴`（初診/再診）、`openAI_確認_紹介状有無`（有/無）

### ベースプロンプトに含まれるもの
- Role定義（○○判定エンジン）
- Context
- プロンプトインジェクション対策
- 出力仕様（正ラベル / 負ラベル / NO_RESULT）
- 絶対ルール
- STEP1: 入力正規化
- STEP2: 完全一致判定（正ラベル）— 10パターン以上
- STEP3: 完全一致判定（負ラベル）— 10パターン以上
- STEP4: それ以外→NO_RESULT

### 差し替えポイント
| 箇所 | 差し替え内容 |
|---|---|
| Context内のTTS文言 | 設計書の質問文 |
| 正ラベル / 負ラベル | 設計書の出力ラベル（有/無、肯定/否定、充足/不足 等） |
| STEP2/3の完全一致リスト | ラベルに応じた発話パターン |

### 完全一致リストのバリエーション
Skill内に以下のセットを用意しておき、ラベルに応じて選択する:

| 判定種別 | 正ラベル用パターン例 | 負ラベル用パターン例 |
|---|---|---|
| 所持有無（有/無） | ある, あります, 有る, 持ってます, 持っています... | ない, ありません, 持ってない, 持っていません... |
| 同意（肯定/否定） | はい, はいそうです, よろしいです, 大丈夫... | いいえ, 違います, やめます, 結構です... |
| 経験（再診/初診） | ある, あります, 受診したことある, 通院中... | ない, ありません, 初めて, 初診です... |
| 充足（充足/不足） | 足りる, 足ります, 大丈夫, 間に合います... | 足りない, 足りません, 不足, 間に合わない... |

## Skill C: 日付変換型

### ソース
sample_flow.json の `openAI_確認_現在の予約日`

### ベースプロンプトに含まれるもの
- Role定義（日付抽出・正規化AI）
- Context
- プロンプトインジェクション対策
- システム日付の取り扱いルール（「今日はyyyy年mm月dd日」の自動挿入への対応）
- 出力仕様（yyyy/mm/dd形式 or NO_RESULT）
- 正規化ルール（相対日付、和暦変換、DTMF数字列、月日のみ等）

### 差し替えポイント
| 箇所 | 差し替え内容 |
|---|---|
| Context内のTTS文言 | 設計書の質問文 |
| 日付種別（Roleの記述） | 「現在の予約日」「変更希望日」「受診希望日」等 |

> このパターンは差し替え箇所が少ない。ベースプロンプトをほぼそのまま使える。

## prompter.mdからの参照方法

prompter.mdの「作業手順 Step 5: プロンプトを記述する」を以下のように変更する:

```markdown
### Step 5: プロンプトを記述する

**1モジュールごとに、以下の手順で記述する。**

1. Step 4で選定したパターン（A〜F）に対応するSkillファイルを Read する
   ```
   docs/ai/skills/SKILL_{パターン}_{名前}.md
   ```

2. Skillファイルの「ベースプロンプト」をコピーする

3. 「差し替えポイント」に従い、施設固有の情報（設計書セクション4・5から取得）で差し替える:
   - TTS文言 → Context セクション
   - 出力ラベル → 出力仕様セクション + 判定アルゴリズムのSTEP
   - DTMF有無 → 数字モード判定の有無
   - 選択肢リスト → STEP2/3の判定リスト

4. 差し替え結果を確認し、出力ラベルがnext配列のconditionと一致していることを検証する

5. 次のモジュールへ進む（Step 1に戻ってSkillを読み直す）
```

### 移管方針
- Gen1「箱」構造 → Gen3 9 ブロック型に変換。多摩総合医療センター_診療と類似の構造（用件 6 分岐 + 診療科 × 用件マトリクス + 紹介状あり初診の追加聴取）。
- 冒頭は opening + 冒頭_アナウンス（2 ブロック 1 セット必須）。挨拶 TTS は「お電話ありがとうございます。湘南藤沢徳洲会病院の、予約専用、AI 電話です。診療科によっては、こちらの番号でお受けできない場合もございます。また、当日の予約は承っておりませんので、ご了承ください」。
- 用件分岐は **Pattern A**（用件聴取 OpenAI 直後に 5 分岐: 初診予約 / 再診予約 / 予約変更 / 予約キャンセル / 予約確認）。各分岐に echo_back（用件の復唱、「ご予約の変更でよろしいですか？」等）を直配置。**用件は echo_back 必須**。
- 診療科分岐は **Pattern B（後段 ContextMatchRouter）併用必須**。診療科 × 用件のクロス分岐があり、診療科だけでは終話先が決まらない（例: 産科は初診のみ代表案内、変更/キャンセルは通常ルート）。診療科聴取 → 用件で分岐 → 各 NG メッセージ という二段構成を維持。
- サブフロー（患者氏名 / 生年月日 / 電話番号 / 診察券）は Jump to Flow 経由。診察券は「再診以降のみ」聴取（受診歴有相当）→ 初診ルートと再診ルートで分岐後合流。
- 時間外判別.yaml（祝日チェック → 平日/土曜の時間判定）→ date_of_call_classifier ブロック。**addCurrentDate: Yes 自動付与対象**。
- entity_classification.md / entity_stt_patch.json / entity_openai_synonyms.md は scaffold/properties が自動取り込み。director は転記不要。

- **DTMF 受付**: 用件聴取（1=診察予約 / 2=変更 / 3=キャンセル / 4=確認）と Yes/No 確認（1=肯定 / 2=否定）で全面 DTMF 並走。WEB か電話か（1=WEB / 2=電話）も DTMF。
- **WEB フォーム選択肢**: 携帯入電（入電番号取得不可、転送できない）→ `WEBか電話か` サブフロー → WEB 選択時は SMS 送信 + 終話。電話選択時は通常フローへ合流。**入電番号判定が分岐の起点**になる点に注意。
- **非通知発信**: 「先頭に 186 を付けてかけ直し」案内を流して切断（固定 TTS、`非通知→発信番号通知でかけ直し案内`）。
- **携帯/固定で終話 TTS 分岐**: 4 用件（診療予約 / 予約変更 / 予約キャンセル / 予約確認・聴取失敗時）すべてで携帯/固定の終話 TTS が別建て。携帯時のみ「ショートメール」を併記。
- **3 回失敗 + 受付時間内**: `END_2回聞き取り失敗時：予約センター案内`（予約センター 0466-35-1350 を 3 回繰り返し）→ termination_patterns に集約。
- **復唱（echo_back）対象**: 用件（5 種）/ 診療科 / 予約日 / 紹介状有無の Yes/No 系。氏名・生年月日・電話・診察券はサブフロー内で完結。

### director が見落としやすいポイント
- `最初の箱: API_DRJOY_CONTEXT_SETTINGS` → `挨拶` → `API_DRJOY` の 3 箱は Gen3 では opening 冒頭処理に吸収。明示ブロック化不要。
- 「セリフ: none」かつ振り分けのみの「箱」（`用件の確認 / _1〜_3`、`用件分岐`、`用件分岐_1`、`診療科分岐`、`救急総合診療部分岐`、`産科分岐`、`分岐：側弯症`、`紹介状有無：分岐`、`循環器科初診：分岐`、`対象外診療科：用件分岐`、`通常ルートへ / _1`、`診察券有無の確認`、`携帯電話分岐 / _1 / _2`、`完了フラグ`、`時間外へ` `時間内へ`、`氏名聴取へ`、`次へ_5`、`【用件再セット】キャンセル→変更へ`、`SMS送信フラグ` 等多数）は Gen3 で独立ブロック不要 → 前段の next 書き換え or context_match_router 一部として吸収。
- `判定: エンティティ含む` で参照される値（味覚嗅覚外来・側彎症・各診療科グループ・肯定/否定/分からない・DTMF 等）→ entity_classification.md / entity_stt_patch.json / entity_openai_synonyms.md にすべて自動分類済み。director は転記不要。
- 用件聴取.yaml に **2 段階の確認構造**あり: 1 回目の聞き取り（`用件の確認`）→ 用件確定後 echo_back（「予約変更でよろしいですか?」）→ Yes なら確定 / No なら再聴取。さらに `初再診確認`（過去受診歴あり Yes/No）と `新患確認`（当院初めて Yes/No）の派生確認も持つ。**初診/再診の判定は「初めて？」Yes/No に集約**するのが Gen3 流（director 判断、6 分岐圧縮 vs 多摩方式どちらか）。
- `用件の確認` で「健診発話」を拾うと `これから？` フロー（健診/人間ドックを受診される方か Yes/No）→ Yes なら人間ドック案内 / No なら氏名聴取に流す **健診回避ルート**あり。順天堂等 FAQ 系とは別の単純分岐。
- **予約日聴取の DTMF 4 桁機能**: シャープ/アスタ + 4 桁数字を受け付け、過去日なら +1 年で確定する仕様（`予約日セット（DTMF）` `日付比較` `プラス1年後で確定`）。Gen3 で SKILL_C 自由テキストで吸収するか、別途 dtmf 並走させるかは **director 判断**。
- 終話直前に **携帯/固定 ×4 用件 = 8 種類の終話 TTS** を持っている。Gen3 では携帯/固定の context_match_router 1 個 → 用件別 termination ブロックの組合せで集約。各用件で「日曜・祝日を除く 3 日以内に折り返し」の文言は共通、携帯時のみ「ショートメール」を併記。
- **「予約希望時期」サブフローの後で用件をキャンセル→変更に書き換える奇妙な遷移**（`【用件再セット】キャンセル→変更へ` 箱）あり。元資料の挙動: 予約キャンセル選択 → `再予約希望確認` で「再度予約希望」Yes → 用件を「予約変更」に書き換えて変更フローに合流。**Gen3 では context 操作（context_match_router で reset するか、prompter 側で再分類するか）director 判断**。
- `側弯症の確認` `睡眠時無呼吸確認` `味覚嗅覚外来確認` の 3 つは **診療科特定後の追加 Yes/No 確認サブフロー**。すべて同パターン（Yes 系/No 系/DTMF 1or2）→ subflow ブロックで 3 本立てる必要あり。または個別 hearing ブロックに展開（director 判断）。
- 救急総合診療部に流れた場合、`<? $診療科の読み ?>` を含む案内文を読み上げる仕様 → Gen3 では `<% var %>` 構文に変換。
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止。

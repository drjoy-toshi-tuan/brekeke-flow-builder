### 移管方針
- Gen1「箱」構造 → Gen3 scenario_flow 9 ブロック型へ変換。

  - `企業名.yaml` → 契約ルート専用 hearing。**新規ルートでも個人/企業支払いの「企業」分岐を踏むと企業名サブフロー再呼び出し**（`企業名_1` 経由）
  - `予約関する用件.yaml` → 新規ルート専用 hearing 3 分岐（**新規 / 変更 / キャンセル**、direct DTMF 1/2/3）+ 各 echo_back。main の用件①では拾わない変更/キャンセルがここで救済される

  - `変更項目.yaml` → 変更ルート hearing
  - `家族内容.yaml`（自由聴取）→ 新規ルート専用「そのほか予約を希望される方がいらっしゃる場合はお名前・生年月日をお話ください、いない場合は『いない』」自由文 hearing（output_format: text）

### 施設固有の特殊ルール
- **受付時間**: 平日 8:00〜20:00（土日祝・年末年始休み）。年末年始休業は令和8年1月4日（日）まで、1月5日（月）8:00 から再開。`入電日時判断.yaml` 内 `1/31以降` 箱 + `祝日チェック` サブフロー。Gen3 では acceptance_times + 年末年始 Script で代替（director デフォルトに集約）
- **代表電話番号**: 0776-28-8513（健診センター直通）。**Brekeke Blind Transfer ではなく `代表転送` サブフロー（音声案内 + 切断）型** → call_transfer ブロック不要、termination + 番号読み上げで完結
- **当日予約は AI 非対応**: 用件「当日予約」分岐 → 「当日に関するお問い合わせは、健診センターへ転送いたします。少々お待ちください。」→ `代表転送` サブフロー → 切断。混雑時タイトル箱「只今お電話が込み合っております。0776-28-8513 へおかけなおしください」が代替アナウンス
- **用件は main で 4 分岐 + 新規ルート内で 3 分岐の 2 段構造**: main `用件の分岐`（当日予約 / 新規 / 問い合わせ / 契約）→ 新規選択時のみ `予約関する用件` サブフローで再度（新規 / 変更 / キャンセル）3 分岐。Gen3 では用件①直後の主分岐 4 本 + 新規ルート段の予約関する用件 hearing 3 本の **2 段 hearing 構造**を保持
- **個人/企業支払い Yes/No**: 「健康保険組合などからの補助はございますでしょうか？」 → Yes（補助あり = 企業負担）なら企業名再聴取、No（個人負担）なら直接問合せ内容へ
- **家族内容**（**新規限定**）: 「そのほか予約を希望される方がいらっしゃる場合はお名前・生年月日をお話ください。いない場合は、『いない』とお話ください」→ 自由文聴取 1 本。Gen3 では output_format: text の hearing で実装、対象人数は context に蓄積する想定
- **追加問い合わせ**: 「日中折り返し希望の時間帯やお伝えしたいことがあれば、お話ください。無いようでしたら『特にありません』のようにお話下さい」→ 全ルート最終段の自由文聴取 1 本
- **携帯チェック 3 文言切替**: 携帯あり/なし で**用件×携帯**の 3 種を使い分け（main の `締めあいさつ` / `締めあいさつ_1` / `締めあいさつ_2` の差異）:
  - 締めあいさつ（問い合わせ/変更/契約）: 「土日、祝日を除く、翌営業日までに担当者から折り返し電話、もしくは、SMSにてご連絡いたします」
  - 締めあいさつ_1（キャンセル）: 「必要がある場合のみ、担当者から折り返し電話、もしくは、ショートメールにてご連絡いたします」
  - 締めあいさつ_2（新規）: 「予約の確定については、土日、祝日を除く、翌営業日までに担当者から折り返し電話にてご連絡いたします」
- **「もしもし」検知**: 用件①入口で `countもしもし` 計数。1 回目「はい、聞こえております」、2 回目以上「はい、『もしもし』と聞こえております」→ 用件再聴取
- **3 回聞き取れず**: 用件 3 回失敗で `END_用件取得できず` → 代表電話案内 termination

### director が見落としやすいポイント
- 冒頭 `API_DRJOY_CONTEXT_SETTINGS` / `API_DRJOY` / `API_DRJOY_1` 系は Gen3 では opening 冒頭の標準処理に吸収 → 明示ブロック化不要
- 「次へ」「最後」（"続いて、" "最後に、"）は短い繋ぎ TTS → 次段ブロック先頭 TTS にマージ
- セリフ none 振り分けのみの箱（用件の分岐 / _1〜_6、完了フラグ / _1、携帯電話番号チェック / _1 / _2、ブレケケのフラグ設定、着信入電/IVR / _1 / _2、完了フラグ「20」設定、切断、切断へ）は独立ブロック不要
- `用件の分岐_2` で「契約 → 企業名」直行する経路があり**契約ルートは生年月日経由しない代替もある**（main yaml 注意深く読む）→ director は契約と新規/問い合わせの個人情報聴取順序を取り違えない
- `タイトル_1` 箱（"大変申し訳ございません"）→ `タイトル`（混雑時案内 + 0776-28-8513 へかけ直し）→ `切断` の経路は **3 回聴取失敗 + 混雑用フォールバック**。Gen3 では代表電話案内 termination に集約
- `予約関する用件.yaml` の `用件の復唱（新規）_1` `用件の復唱（変更）_1` `用件の復唱（キャンセル）_1` は END_戻る で main に戻ってから分岐再判定（DTMF 直接ルート）。Gen3 では echo_back の中で完結
- 「<? $予約日の読み ?>」「<? $連絡先電話番号（ハイフン区切あり） ?>」等の Gen1 動的タグは Gen3 では `<% var %>` 構文へ
- **モジュール名・termination 名は ASCII または安全な日本語のみ**。丸数字（①②③⑩等）等の環境依存文字は禁止（用件①.yaml の "①" は箱名としては保持されているが Gen3 ではモジュール名 `youken1` 等にリネーム必須）
- **ContextMatchRouter.reference_module は subflow 参照時 `jump_{step名}`**（scaffold prefix）を使う
- **CMR conditions は最後に `match: "other"` + next 明示**
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止

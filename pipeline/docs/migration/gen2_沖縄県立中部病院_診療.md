### 移管方針
- 冒頭は opening + 冒頭_アナウンス の2ブロック1セット。冒頭アナウンス直後は入力待ち無しで「通院確認」Yes/No 分岐（Pattern A: OpenAI直後分岐）に接続。No→END_通院中以外受付不可（status=9 / smsFlag=0 / endpoint=電話案内）、Yes→個人情報聴取へ。
- 着信電話番号判定は Date of Call Classifier ではなく **incoming_phone_number 型の3分岐**（非通知 / 携帯=070/080/090/**060** / 固定=上記以外の 0 始まり＋050 11桁）。非通知→冒頭アナウンス前に END_非通知拒否 で即切断（status=8）。
- 連絡先電話番号は **「今おかけいただいている番号でよろしいですか？」の Yes/No 確認**を先にし、否定時のみ自己申告→復唱確認ルートへ合流（Pattern B: 後段合流）。携帯/固定共通で同じ聴取ブロックを使い回す（成田赤十字のように mobile 限定ではない点に注意）。
- 氏名／DOB／電話番号／診察券番号は **サブフロー経由（Jump to Flow）**。DTMF付きや復唱・桁数再ヒアリングはサブフロー側に閉じ込める。メインにインライン展開禁止。患者氏名のみのためサブフローは PatientName 1本で足りる。
- 用件分岐は **classification 4択（再診予約の変更／予約日の確認／予約のキャンセル／予約の取り直し）** で Pattern A。健診ルートは無い（成田赤十字と違い診療単一シナリオ）。
- 「予約のキャンセル」→「キャンセル時案内（DTMF 1/2）」で **次回予約希望あり→ classification を「再診予約の変更」へ上書きして予約希望日ブロックへ合流**（Pattern B 的な classification 書き換え合流）。終話は元の classification ではなく合流後の END_再診予約の変更 が走る点に注意。
- 「予約の取り直し」→ 前回の予約日を日付正規化し **3か月判定で分岐**（3か月以上前 → END_取り直し受付不可（status=7）／3か月以内・フリーワード → 予約希望日へ合流）。Date of Call Classifier ではなく **script ブロックで月差判定**（director 判断）。
- 終話 status は 0/1/2/7/8/9、smsFlag は 0/1/2/3 のみ。**元資料 3.2 に「終話（予約の取り直し）→ smsFlag=4」の記述があるが yaml は取り直し完了時 END_再診予約の変更 に合流しており smsFlag=1**。矛盾は yaml 優先（director 判断、合流仕様側に揃える）。
- RAG は不要。FAQ は「question」フリーテキスト格納のみ（yaml に FAQ 確認ブロックが無い＝実装上はスキップ仕様）。

### 施設固有の特殊ルール
- **通院確認ゲート**：全通話で本人確認より前に「当病院に通院中ですか？」の Yes/No を通過させる。No は status=9。この分岐を個人情報聴取の前に必ず配置すること。
- **携帯プレフィックスに 060 を含む**：`070/080/090/060` の 11 桁ちょうどを「携帯」と判定。050 は固定側（11 桁）に含める。成田赤十字（050＝mobile扱い）と逆なので注意。
- **連絡先電話番号の聴取は "着信番号確認→自己申告→復唱" の3ブロック構成**。ケース①（着信番号でOK）→ additionalPhoneNumber=Incoming_phone_number、ケース②（否定）→ 再聴取→復唱、ケース③（肯定否定以外で別番号を発話）→ 復唱、ケース④（無効）→ リトライ。成田のような「mobile 限定の番号確認」ではなく全種別共通で着信番号確認が先。
- **数字 4/5/9/0 の誤認識救済**：4=「しー/C/Sea/よー」、5=「いつ/Go」、9=「Q/きゅー」、0=「れい/まる/おー」。電話番号・診察券番号の prompt 側で対処（prompter 担当）。※発話時は別リスト（1-7）の標準読みのみ使用。
- **氏名＝カタカナ強制変換**：patientName は全角カタカナのみ。STT が漢字を返しても音ベースでカタカナに強制補正し transcript も書き換える。PatientName サブフロー内で復唱禁止＆漢字混入不受理のプロンプト注記必須。
- **診療科のフリーワード扱い**：元資料本文6章「診療科はフリーワード。再聴取は実施しない」だが yaml は「わからない」時 `登録なし` 格納 & リトライ上限2 & 復唱確認あり。**display_type=DEPARTMENT 正規化に委任し、復唱確認は別ブロック化**。診療科リスト 31 種は function enum をそのまま列挙（director 判断）。
- **status=2（聴取失敗）／smsFlag=0 専用の強制切断エンドポイント**：復唱後否定の再聴取失敗・無効回答による聴取失敗の両方が同一文言「代表電話番号 098-973-4111 案内」で終話。termination_patterns に END_無効回答による聴取失敗（status=2）を1本定義して使い回す。
- **「肺→はい」「PH がいます→いいえ」「PD→いいえ」救済**：Yes/No 判定時のみ適用（生年月日／連絡先電話番号／キャンセル時案内／予約の取り直し冒頭確認等）。Yes/No 系 generate_by_OpenAI の prompt に注記必須。
- **生年月日ヒアリング時のみ「社長は→昭和」「修正→平成」**：DOB サブフローのプロンプトに限定して記載。
- **FAQ確認は実装上スキップ**：本文8章にはルールがあるが yaml の箱一覧に FAQ 確認箱が無い。設計書上も「question」は function の required に入るが聴取無しで空文字のまま終話する想定。FAQ ブロックを追加しない（director 判断：yaml 優先）。
- **更新日時 2026/03/03**：改訂履歴セクションは無いが「1768行目（3/3）依頼対応：氏名に関するルール制御」と冒頭に注記あり。patientName カタカナ強制ルール（10章）がこの改訂分。既に取り込み済み前提。

### director が見落としやすいポイント
- 『**追加の連絡先ではなく、連絡先電話番号は着信番号確認を必ず最初に通す**』。yaml の「連絡先電話番号_着信番号確認」箱がその実装。いきなり「ご希望の連絡先電話番号をお話しください」から始めないこと。
- 『**一度入電者から聴取した内容は、いかなる理由があっても再度聞き直さないでください**』（本文1章）／『**一度に複数情報が来た場合：既出情報は復唱・確認しない**』（本文3章）。全 generate_by_OpenAI プロンプトに反映。
- 『**復唱可能な聴取項目は "生年月日" と "連絡先電話番号" の2つのみ**。氏名・診察券番号・診療科等は復唱禁止』。復唱ブロックを誤って他項目に付けないこと。
- 『**氏名（patientName）聴取後に「○○とカタカナで記録いたします」等の再読み上げは絶対禁止**』。氏名サブフローのプロンプト注記で明示。
- **キャンセル時案内の DTMF**：「予約変更をご希望の方は1を、完全キャンセルをご希望の方は2を押してください。`<dtmf digit="1"/>`」。DTMF タグは元資料どおり保持。1→再診予約の変更へ classification 書き換え合流、2→END_予約のキャンセル。
- **取り直し受付判定は "月単位での差分判定"**（本文7章補足）。日単位の厳密計算ではなく月差で「3か月以上前」を判定する。
- 『**予約で過去日付と思われる指定は、今日より未来のみ受け付ける**』（本文1-4）。予約希望日の日付正規化プロンプトに反映。
- **checkpoint 概念は無し**。status と smsFlag の2フラグのみ。endpoint は 冒頭切断／時間外／電話転送／電話案内／途中切断／通話完了 の 6 種 enum。
- **代表電話番号 098-973-4111**：聴取失敗時の案内文言で 3 回繰り返し出現。TTS では `<speak type="telephone">` タグで番号部分を囲む想定（本文 1-8、10 章以外にも電話番号発話のルールあり）。

### 移管方針
- Gen1 の「箱」構造 → Gen3 scenario_flow 9 ブロック型に変換。主な対応:
  - `API_DRJOY_CONTEXT_SETTINGS` / `API_DRJOY` / `挨拶` → opening + 冒頭_アナウンス（2 ブロック 1 セット）。冒頭文言「お電話ありがとうございます。こちらは蘇生会総合病院、健康増進センター予約専用ダイヤルです。」を流す。倉敷と異なり incoming-classifier の前段分岐（IVR_TRANSFER 経由 vs 上記以外）は無し → director デフォルト（非通知/海外/携帯/固定の 4 値分岐 or 単一経路）に従う
  - `入電日時チェック` → `時間判断`（lessEqual 12/30日前 / 12/31～1/4）→ 倉敷は acceptance_times + Script で休業判定だったが、**蘇生会は年末年始も通常フローを継続し冒頭で注意喚起 TTS のみ流す**設計。Gen3 では opening 直後に Script（年末年始判定）+ announcement（注意喚起）を配置し、その後 `企業確認` 直前の `人間ドック、健康診断、精密検査確認` へ合流。受付時間外カットは別途 acceptance_times で対応（director 判断: 月～土 9:00〜16:00 を採用、根拠は `代表番号（電話番号取得失敗時）` TTS）
  - `人間ドック、健康診断、精密検査確認` → hearing（Yes/No、サブフロー: 否定肯定確認.yaml）。Yes → 用件聴取へ、No → `人間ドック、健康診断、精密検査以外のお問い合わせ` → 代表番号（0570-071-101）案内 → 終話
  - `用件聞き取り.yaml` → hearing ブロック（用件分類 generate_by_OpenAI）。**5 分岐**: 新規（健診予約）/ 変更 / キャンセル / 問い合わせ / 企業。各分岐直後に echo_back（用件の復唱）を 5 本配置 → Pattern A。DTMF 1=新規 / 2=変更 / 3=キャンセル / 4=企業 / 5=問い合わせ も併用（max_dtmf_length: 1）
  - 氏名 / 生年月日 / 連絡先電話番号（電話番号区切り含む）→ subflow ブロックで Gen3 標準サンプル JSON 完コピ。氏名サブフローのプロンプトは用件で「受診される方」「ご連絡いただいている方」「ご予約されていた方」と差し替えるが、**Gen3 では PatientName サブフロー 1 枠に充当**（director 判断、複数枠化は不要）
  - 予約日聞き取り / 受診希望日フリー / 受診内容フリー / 受診内容フリー_1（キャンセル用）/ 受診歴フリー / 変更内容フリー / 連絡事項フリー / 問い合わせフリー → 各 hearing ブロック（OpenAI 正規化、output_format は datetime / text / enum を director が判断、addCurrentDate: Yes は予約日聞き取り / 受診希望日フリー の前段 OpenAI に scaffold 自動付与）
  - 企業ルート（`企業名確認.yaml` → `担当者名聞き取り.yaml` → `企業問合せ内容.yaml`）→ 3 段 hearing で順次配置（subflow 化はしない、施設固有のため main 内 hearing 直書き）
  - `携帯電話番号チェック` → 連絡先 subflow 戻り値 phonetype を context_match_router で分岐、`締めあいさつ`（携帯あり: 電話 or ショートメール）/ `締めあいさつ_1`（携帯なし: 電話のみ）の 2 系統 termination ブロックぶら下げ（SUBFLOY_RETURN_CONTEXTS 経由）
  - `終話` → 「お電話ありがとうございました。それでは失礼いたします。」共通文言 → END_切断（status=1, smsFlag=director デフォルト）
- 用件別の合流先（main.yaml `用件の分岐` / `用件の分岐_1` / `用件の分岐_2` から読み取り）:
  - 新規: 受診歴フリー → 受診内容フリー → 受診希望日フリー → 連絡事項フリー → 完了フラグ → 用件の分岐_1（最終確認） → 氏名 → 生年月日 → 連絡先 → 締めあいさつ
  - 変更: 予約日聞き取り → 用件の分岐_2（変更されるのは予約日ですか？ Yes/No）→ Yes: 受診希望日フリー / No: 受診内容フリー_1 → 連絡事項フリー → 完了フラグ → 氏名 → 生年月日 → 連絡先 → 締めあいさつ
  - キャンセル: 「キャンセルの申し込みを受付いたしました。必要がある場合のみ、担当者よりご連絡させていただきます。」→ 終話（個人情報聴取スキップ）
  - 問い合わせ: 問い合わせフリー → 問い合わせ確認 → 連絡事項フリー → 完了フラグ → 氏名 → 生年月日 → 連絡先 → 締めあいさつ
  - 企業: 企業名確認 → 担当者名聞き取り → 企業問合せ内容 → 完了フラグ → 氏名 → 生年月日 → 連絡先 → 締めあいさつ

### 施設固有の特殊ルール
- **受付時間（director 判断）**: 月〜土 9:00〜16:00、日祝休。根拠は `代表番号（電話番号取得失敗時）` TTS「日曜、祝日を除く、月曜から土曜の9時から16時に、健康増進センター予約専用ダイヤルへおかけなおしください」。Gen1 main 内に時間外カット箱は存在しない → Gen3 では acceptance_times で時間外を冒頭判定し、時間外用 termination + 0120-621-388 案内へ
- **年末年始は休業せず注意喚起のみ**: `時間判断` の 12/31〜1/4 は `年末年始アナウンス`「12月31日より1月4日まで当センターは、休診となっております。折り返しのご連絡は1月5日月曜日より、始めさせていただきます。」を流した上で **そのまま `企業確認` → 通常フロー継続**。倉敷の「休日 → 用件絞り込み」とは異なり、フロー分岐は無く案内のみ。Gen3 では opening 直後 Script + announcement 1 本で吸収
- **代表番号は 2 種類**:
  - **0570-071-101**: 人間ドック・健康診断・精密検査以外の問い合わせ向け（「電話番号が分かる場合はそのままお切りください。分からない場合はご案内いたしますのでメモをご準備ください。電話番号は0570‐071‐101です。繰り返します。…」を 3 回繰り返す）
  - **0120-621-388**: 電話番号取得失敗時 / 受付時間外の予約専用ダイヤルへの再案内（同様に 3 回繰り返す）
  - 両方 termination ブロック内の TTS で 3 回繰り返し読み上げ。Gen3 でも文言維持（短縮しない）
- **「人間ドック、健康診断、精密検査確認」Yes/No が用件聴取の前段に必須**: 倉敷とは異なり用件分類の前に必ず Yes/No で「今回は人間ドック、健康診断、精密検査について、でよろしいでしょうか？」を確認 → No なら 0570-071-101 案内へ即離脱。`否定肯定確認.yaml` を Yes/No 共通サブフロー化（生年月日変更確認の `次へ_1` でも再利用）
- **用件 5 分岐の復唱必須**: 用件聞き取り内に「用件の復唱（新規/キャンセル/変更/問い合わせ）+ 企業 復唱」が定義済み → Gen3 でも各分岐直後に echo_back 5 本必須。再質問は「用件を確認」箱で否定 / 肯定 / 用件再発話を吸収（Pattern A）
- **「もしもし」検知**: 用件聞き取り入口で `countもしもし` 箱で計数。1 回目「はい、聞こえております。恐れ入りますが、」→ 用件再聴取、2 回目以上「はい、『もしもし』と聞こえております。恐れ入りますが、」→ 用件再聴取、4 回以上で `聞き取り失敗` → END_用件取得できず。Gen3 では用件 OpenAI のプロンプトで救済語扱い（director 判断、独立 hearing は不要）
- **DTMF 1〜5 対応**: 用件聞き取り内に `DTMF判別` 箱あり → Gen3 では DTMF AmiVoice STT Input + max_dtmf_length: 1 で並列化（音声入力と DTMF 入力の両対応）
- **連絡先携帯チェックで締め文言切替**: 倉敷と同じ運用。携帯あり → `締めあいさつ`「日曜、祝日を除く、翌営業日以内に担当者から折り返し電話、もしくは、ショートメールにてご連絡いたします」、なし → `締めあいさつ_1`「日曜、祝日を除く、翌営業日以内に担当者から折り返し電話にてご連絡いたします」。Gen3 では context_match_router で phonetype を分岐、各分岐に termination ブロックを 2 本ぶら下げる
- **企業ルートは個人ルートと聴取項目が別**: 企業名（自由記述）→ 担当者名（人名 STT、`氏名聞き取り.yaml` ではなく専用 `担当者名聞き取り.yaml`、文末「ス」削除のクリーンアップ含む）→ 企業問合せ内容（自由記述）。Gen3 では 3 段 hearing 直書き、サブフロー化しない（施設固有プロンプト）
- **キャンセルは個人情報聴取スキップ**: キャンセル分岐のみ「必要があれば担当者から折り返し」と告げて即終話。氏名 / 生年月日 / 連絡先は聴取しない。倉敷も同様だが、倉敷はキャンセルでも氏名 / 生年月日 / 連絡先まで取る → 蘇生会固有の差分

### director が見落としやすいポイント
- `最初の箱: API_DRJOY_CONTEXT_SETTINGS` と `API_DRJOY` 系 1 箱はすべて Gen3 では opening 冒頭の標準処理に吸収 → 明示ブロック化不要
- 「次へ」「次へ_2」（セリフ「続いて、」）は短い繋ぎ TTS → 次段ブロック先頭 TTS にマージ（director デフォルト）。「次へ_1」のみ Yes/No 分岐があるため独立 hearing 化
- 「セリフ: none」かつ振り分けのみの箱（`用件の分岐` / `用件の分岐_1` / `用件の分岐_2` / `完了フラグ` / `入電日時チェック` / `携帯電話番号チェック` / `問い合わせ確認` / `時間判断` / `企業確認` / `タイトル` / `タイトル_1` 等）は Gen3 では独立ブロック不要 → 前段の next 先書き換えか context_match_router の一部として吸収
- 「冒頭NG」 → 「代表番号（電話番号取得失敗時）」のパスは入電番号取得失敗時の代表番号案内 → Gen3 では incoming-classifier の error_jump 先 termination に集約
- 「電話番号聞き取りNG」 → 「代表番号（電話番号取得失敗時）」も同様に termination 集約
- `年始年始` 箱（none → END_切断）は誤参照の名残 → 削除して `年末年始アナウンス` から直接 `企業確認` 通常フローへ繋ぐ
- 「キャンセル」分岐は **個人情報聴取スキップで即終話** → `携帯電話番号チェック` を経由しない。Gen3 設計書でも明示的に分岐させること（倉敷との差分、director 見落としやすい）
- 転送先は **Brekeke Blind Transfer ではなく「代表番号 0570-071-101 / 0120-621-388 にかけ直し」案内型**。担当者直結の Blind Transfer は無し → call_transfer ブロック不要、termination + 番号読み上げ（3 回繰り返し）で完結（director デフォルト 6「Blind Transfer デフォルト」は適用しない、本案件は番号案内型）
- 予約日聞き取り.yaml は単純な日付聴取（DTMF 補助なし、フリーワード「4月1日」形式）→ Gen3 では Brekeke 標準の datetime hearing + addCurrentDate: Yes で過去日補正は OpenAI 正規化のプロンプトで吸収
- 連絡先電話番号聞き取り.yaml の `番号確認_1`「ご連絡先のお電話番号は、今おかけいただいている <? $入電番号 ?> でよろしいですか？」は **入電番号 = 連絡先候補時の確認** → Gen3 では電話番号サブフロー（復唱あり）の標準ロジックで吸収済み、メイン側で別途実装不要
- `否定肯定確認.yaml` は用件確認用（人間ドック・健康診断・精密検査確認 / 変更内容の予約日確認）の **共通サブフロー** → Gen3 では Yes/No 用 hearing を都度配置（共通 subflow 化はしない、scaffold が固定プロンプト埋め込み）
- 「<? $連絡先電話番号（ハイフン区切あり） ?>」「<? $入電番号（ハイフン区切りあり） ?>」等の Gen1 動的タグは Gen3 では `<% var %>` 構文へ。subflow 内 echo_back に集約済みのため、メインフローの直接参照は不要
- 受診歴フリー（「これまでに外来受診を含め、当院を受診されたことはありますか？」）は新規ルートの先頭で 1 回のみ聴取 → 来院歴 context（history）に保存。倉敷には無い項目
- entity_classification.md によると **R4_OPENAI_SYNONYM が 91 件と多め**（用件の同義語が 65 件で大半）→ 用件分類 OpenAI プロンプトに「キャンセル/企業/新規/変更/問い合わせ」の同義語列挙が必須（特に「眼瞼すんの」「人間独」「人工診断」「全身」等の音声誤認識救済語を含む）
- **モジュール名・termination 名は ASCII または安全な日本語のみ**。丸数字（①②③…⑩）等の環境依存文字は禁止
- **ContextMatchRouter.reference_module は subflow を参照する場合 `jump_{step名}`（scaffold が付ける prefix）を使う**。subflow ブロックの step 名そのものではダメ
- **CMR conditions は必ず最後に `match: "other"` + next 明示**
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止

### 移管方針
- Gen1「箱」構造 → Gen3 9 ブロック型への変換:
  - `挨拶`（箱、「お電話ありがとうございます…医療福祉相談室です」）→ **opening + 冒頭_アナウンス（2 ブロック 1 セット必須）**
  - `入電時間判断`（サブ yaml、祝日チェック → 2026年GW日付分岐 → 平日/土曜時間分岐）→ **date_of_call_classifier**（addCurrentDate: Yes 自動付与対象）
  - `0307：看護師復唱→看護師 へ`（サブ yaml、担当者名聴取 → 個人特定 → 看護師 or 担当者転送）→ **担当者名 hearing ブロック群 + ContextMatchRouter**
  - 担当者個人特定の 2 段階（姓 → 個人名特定）は **Pattern B（後段 ContextMatchRouter で組合せ判定）** が妥当。同姓複数（大城/下地/山城）の枝分かれは hearing で姓を取得 → 続く hearing で個人特定 → CMR で確定転送先を決定
  - 「分かりません」入力時 → 固定電話（098-923-2647）転送 → call_transfer ブロック
  - 受付時間外 / 年末年始 / GW → announcement（時間外案内）→ termination
  - 3 回失敗 / 聞き取り不可 → 固定電話転送 → call_transfer
- 復唱（echo_back）: 担当者名の最終確認（「担当者は、◯◯、でよろしいですか?」）→ 既に Gen1 で実装済みのため hearing 内 echo_back に組込み

### director が見落としやすいポイント
- `API_DRJOY_CONTEXT_SETTINGS` / `API_DRJOY` / `API_DRJOY_1` は Gen3 では opening 冒頭の標準処理に吸収 → 明示ブロック化不要
- 「セリフ: none」かつ振り分けのみの箱（`コンテキストセット`, `担当者読み取得`, `担当者読みセット`, `完了フラグ`, `完了フラグ（時間外）`, `完了フラグ_1`, `転送先番号セット`, `担当/看護師判別` 等）は Gen3 では独立ブロック不要 → 前段 next の書き換えまたは ContextMatchRouter に吸収
- `判定: エンティティ含む` の意味: STT/OpenAI で抽出された担当者エンティティ（カスタム定義）にマッチするかの判定。Gen3 では generate_by_OpenAI の output_format（enum）で吸収するか、後段 ContextMatchRouter で値マッチさせるか **director 判断**（entity_classification.md の R4_OPENAI_SYNONYM 列を確認）
- `count上記以外` / `count再質問` 系の箱（リトライカウント）は Gen3 では **hearing ブロックの retryCount プロパティ**で吸収 → 独立ブロック不要
- `テスト用リセット` / `テスト` 箱はテスト用の入口、Gen3 では **削除**
- 「医療福祉相談室シナリオ」は診療系と異なり、**患者氏名・DOB・診察券・電話番号サブフローは不要**。聴取項目は「担当者名」のみ。PatientName 等のサブフロー Jump to Flow は組み込まない（director 判断で正式確認）
- `祝日チェック` はサブシナリオ呼び出し → Gen3 ではカレンダー Script モジュール必要（docs/brekeke/モジュール選定ガイド_v2.md §3.8 参照）。addCurrentDate: Yes 自動付与対象
- 転送失敗時のリトライ（`転送失敗_1` → END_切断）はそのまま call_transfer の失敗パス → termination に流す
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止

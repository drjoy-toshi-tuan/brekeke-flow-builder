### シナリオ概要

株式会社健康予防医学協会グループ 6 クリニック共通の「健診予約 統合コールセンター」。受診希望者がエリア→施設を選択した上で、健診の予約・変更・キャンセル・その他用件を申し込む電話。**厚木単体のシナリオではない**（厚木は 6 施設のうちの 1 つ）。

### 設計方針

- 構造は「上流: エリア選択 DTMF（神奈川=1 / 新宿・渋谷=2 / 東京・品川=3）→ 施設選択 DTMF（厚木｜ヒロオカ｜渋谷ウエスト｜秋葉原｜鉄鋼ビル丸の内｜みなと健診）」→ 6 施設合流 → 「下流: 用件 4 分岐（n4: 予約/変更/キャンセル/その他、DTMF 併用）→ 個人情報聴取 → 終話」。施設確定後は全施設が同一下流に合流するため 1flow ＋施設名 context 保存で収まる見込みだが、subflow 分割要否は director 判断。
- 入電者は受診希望者本人想定。標準サブフローは PatientName（n13）/ DateOfBirth（n14、復唱あり=echo_back）を使用可。Phone は「固定電話入電のみ番号確認（n15）・携帯はスキップ」という分岐つきのため、標準 Phone サブフローが流用できるかは director 判断。MedicalCard は図に登場しない（不使用）。
- 用件分岐: n4 は DTMF 併用 4 択で Pattern A（OpenAI 直後分岐）が第一候補。ただしコース選択 n6 は 4 択 DTMF の外に「雇用時健診」を発話のみで検知する設計のため分類値域に注意。echo_back・複数 context 組合せが必要になるなら Pattern B（後段 ContextMatchRouter）。director 判断。
- Date of Call Classifier: キャンセル経路の n10 は「現在の予約日を西暦 8 桁 DTMF 入力」であり入電日分類ではないため、不要とみられる（director 判断）。

### 施設固有の特殊ルール

- 冒頭ガード: 非通知入電→「186」案内で終話。国際入電→代表番号 046-227-1131 案内で終話。
- 遅刻ガード（**配線確定 2026-06-10 浜口**）: 冒頭ガード後に「ご用件は明日以降のご予約に関することでしょうか」（yxE1OzltzCGTDaTNNFiJ-1）を本線挿入。**はい→エリア選択**。いいえ→「遅刻か変更キャンセルか」（-4）。-4 の**遅刻側→エリア選択へ進み、施設確定後は用件確認(n4)をスキップして氏名聴取(n13)へ直行**（注記ノード yxE1OzltzCGTDaTNNFiJ-32）。-4 の**変更・キャンセル等側（=当日かつ遅刻連絡でない）→ 受付せず終話**（当日の変更・キャンセルは AI では受け付けない方針。終話文言は図・元資料に無い→BLOCKER）。
- 雇用時健診（**解釈確定 2026-06-10 浜口、出典: 旧【要件定義】ヘルスケアクリニック厚木御中.pdf**）: 「定期健診」（n6 DTMF 3 番）は**通常進行**（n13 へ）。「雇用時健診」は DTMF に無く**発話検知のみ**で n2 の HP 申込みフォーム誘導へ → 携帯入電には SMS でフォームリンク送信して終話（固定入電は案内のみで終話）。-6（その他確認）→ n2 の無ラベルエッジは**雇用時健診発話時のジャンプ**として配線（旧要件定義「問い合わせ内容で雇用時健診関連を検知したら雇用時健診へ」と同型）。-6 で「わからない」は course にセットして n13 氏名へ（折返しでスタッフ対応）。n2 見出し「『定期健康診断』の発話」は誤記扱い（旧要件定義はコース正規化で「定期健診」と「雇用時健診」を別値とし、雇用時健診のみ HP 誘導と明記）。
- 当日・市の健診・特定健診（**意味確定 2026-06-10 浜口**）: n2→SMS 終話エッジのラベル「当日/市検診/特殊」=「当日」「市の健診」「特定健診」の 3 ケースが終話へ流れることを指す。「当日」は上記 -1/-4 の冒頭切り分けで処理。「市の健診」「特定健診」は **n4 用件確認で該当発話を検知したら受付不可として終話へ**。受付不可の案内文言は図・元資料に無い→BLOCKER。
- status / sms_flag 暫定値（**TODO 降格 2026-06-10 浜口、出典: 旧要件定義**）: 通常終話 n16 = **status 1:未処理 ＋ sms_flag 1**、雇用時健診 SMS 終話 -31 = **status 7:HP案内 ＋ sms_flag 2**。ただし旧値は厚木単体シナリオのもの。6 施設統合 CC で体系が変わる可能性があるため確認レポートで顧客確認（TODO）。SMS の URL・文面は placeholder 継続（プロパティで後差し替え、顧客確定待ち）。旧要件定義の変換 MD: output/scenarios/ヘルスケアクリニック厚木_統合CC/reference/要件定義_ヘルスケアクリニック厚木御中.md
- 変更: 詳細は聞かず n9「折返し電話の際にスタッフが伺う」アナウンス→氏名聴取へ（折返し回収方式）。
- キャンセル: 現予約日 DTMF 8 桁（n10）→ 再受診希望 Yes/No（n11）→ 氏名へ。
- その他用件（n12）: FAQ 発動なし。用件を自由発話で聴取し受付で折返し対応。
- 連絡先確認は固定/携帯の発信者番号種別で分岐: 固定=今おかけの番号でよいか確認（違う時のみ DTMF 入力）、携帯=「今おかけの番号へ折返し」と案内のみ。
- 終話 n16 は用件（予約/変更/キャンセル）を差し込む共通文言＋折返し電話 or SMS 連絡。

### director が見落としやすいポイント

#### 解消済み（2026-06-10 浜口確定 — 再 BLOCKER 化しない。詳細は「施設固有の特殊ルール」参照）

- -1「明日以降の予約か」の「はい」側未描画 → **はい=エリア選択へ本線挿入で確定**。
- -4「遅刻か変更キャンセルか」の変更・キャンセル側出エッジ無し → **当日かつ遅刻でないため受付せず終話で確定**（エリア選択合流ではない）。
- n2 見出し不一致／定期健診と雇用時健診の関係 → **定期健診(3 番)=通常進行、雇用時健診=発話検知のみで n2、見出しは誤記扱いで確定**（旧要件定義裏付け）。
- 「当日/市検診/特殊」ラベル → **当日=冒頭切り分け、市の健診・特定健診=n4 で検知し受付不可→終話で確定**。
- sms_flag・status 番号 → **旧要件定義の暫定値を採用し TODO 降格**（通常終話=1:未処理+SMS1 / 雇用時健診=7:HP案内+SMS2。顧客確認は確認レポートで）。
- -6→n2 無ラベルエッジ → **雇用時健診発話時のジャンプとして配線で確定**。-6→n13 の座標解決エッジは本線（コース名称聴取後 氏名へ）。

#### 残 BLOCKER / 要確認（確認レポートに必ず載せる）

- **当日系の終話文言**: -4 変更・キャンセル側の「受付せず終話」のアナウンス文言が図・元資料に無い。推測禁止 → BLOCKER。
- **市の健診・特定健診の受付不可案内文言**: 同上、文言未確定 → BLOCKER。
- **SMS URL・SMS 文面**: placeholder のまま（HP 申込みフォーム URL 顧客確定待ち）→ プロパティ後差し替え前提で BLOCKER（URL）/ TODO（文面はテンプレ可なら）。
- **⚠黄ノード（n6 コース選択 / n12 お問い合わせ内容）**: 顧客側で要確定の素案段階。図の文言で構築し、確認レポートに「顧客確定待ち」明記。
- n9 の「押し返し電話」は「折り返し」の誤記とみられる。原文ママ TTS 化しない（確認レポートに記載）。
- 浮きエッジ pf45N7fBwcwUXuKzfR7y-24（→n9）は作図残骸。設計に取り込まない。

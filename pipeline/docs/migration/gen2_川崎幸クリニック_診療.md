# 川崎幸クリニック 診療 移管ノート

- 施設名: 川崎幸クリニック / 第二川崎幸クリニック
- シナリオ名: 診療
- 環境: 本番
- 元資料: `docs/reference/customer_docs/【診療1】：川崎幸クリニック.md`
- 移管日: 2026-04-20
- 移管者: 浜口
- 備考: 同一施設の別シナリオ「健診」ブランチが並走中。氏名/生年月日/電話番号/診察券番号のサブフロー構造は健診と共通化前提。classification/診療科リスト/終話分岐は診療独自。

### 移管方針
- Gen3 の 9 ブロック型で再構築。`generate_by_OpenAI` モジュール + scenario_flow ブロック体系、TTS は `{tts_g:...}`（全小文字）・SSML 禁止。
- 氏名・生年月日・連絡先電話番号・診察券番号はすべて **Jump to Flow のサブフロー化**（健診側の同名サブフローと流用／差異は TTS のみ）。メインにインライン展開しない。
- 分岐パターンは **Pattern A**（classify 直後の3分岐: キャンセル / 確認 / 内容聴取）。echo_back も複数 context 組合せもなく、後段 ContextMatchRouter 不要。
- Date of Call Classifier は **必要**。キャンセルフロー内「現在の予約日」ヒアリング後に「今日 / それ以外」の 2 分岐が発生するため DoC 必須（当日キャンセル不可 → `status=2` 電話案内終話）。
- RAG / FAQ サブフローは今回は **不要**（customer_doc 1-2 に FAQ 記述はあるが、フロー側に FAQ 聴取ステップは設計されていない。RAG ブロック起こさない）。
- 電話転送は **なし**（customer_doc 11: 「転送は行いません」）→ call_transfer ブロック不要。
- 連絡先電話番号は **着信番号 type により分岐**する。`incoming-classifier`（090/080/070/060/050 = 携帯 / 非通知 / その他0始まり = 固定）をメイン冒頭に配置し、連絡先電話番号サブフロー内で「携帯 → 今おかけの番号でよいか確認 (Yes→確定 / No→ dtmf 再聴取)」「それ以外 → 最初から dtmf 聴取」に分ける。

### 施設固有の特殊ルール
- **受付対象**: 「診療予約の**確認**」と「**明日以降のキャンセル**」のみ。本日の予約・新規予約・変更は一切受付不可。
- **当日キャンセル**: 予約日=今日なら `status=2` で電話案内終話（電話番号「044-511-2112」を3回反復読み上げ）。
- **用件 enum**: `キャンセル / 確認 / 内容聴取 / ""`（4値）。それ以外は「内容聴取」扱いで details にフリーテキスト収集。
- **終話パターン**:
  - キャンセル完了 → status=1, smsFlag=**1**
  - 確認完了 → status=1, smsFlag=**2**
  - 内容聴取完了 → status=1, smsFlag=**2**
  - 当日キャンセル → status=**2**（smsFlag 更新なし=初期値0）
- **用語変換テーブル**（customer_doc 1, 用語の変換と表記統一）: 「基礎研修員→基礎健診」「印刷/定期/新薬/再診/最新/禁忌→診察」「薬/お薬→予約」「電気/でんき→新規」「日々か→耳鼻咽喉科」「ないか→内科」「予約を返答→予約を変更」。正規化プロンプトに必須。
- **肯定判定の特殊語**: 「いいです」「肺」を **肯定**として扱う（customer_doc 4 冒頭）。
- **復唱禁止の徹底**: 氏名 / 生年月日 / 診察券番号 / 診療科 / 予約日 / 確認内容 は **復唱禁止**。**電話番号のみ復唱許可**（`{tts_g:...}` で `電話番号ですね` の形で、ただし SSML `<speak type="telephone">` は Gen3 では使用不可 → TTS 側で「...でよろしいですか」相当の平文に置換）。
- **氏名カタカナエラー**: カタカナ以外で格納された場合「うまく聞き取りができませんでした、ゆっくり話すとAIが聞き取れません。普通のスピードではっきりと発話してください。」で再聴取ループ（最大リトライ設計は properties で）。
- **氏名が苗字のみ**: 「下のお名前もお伺いしてもよろしいですか？」で下の名前を追加聴取するサブステップが必要。
- **生年月日**: 西暦指定だが和暦入力も許容（AI で西暦変換）。dtmf 併用 (`<dtmf digit="8"/>` 相当を Gen3 dtmf パラメータで表現)。
- **連絡先電話番号**: 10桁/11桁のみ受付。11桁は先頭3桁が 090/080/070/060/050 のどれか必須。9桁/10桁で頭0欠落は先頭0補完。`<dtmf/>` 必須。
- **診察券番号**: 「わからない」を許可（必須聴取ではない）。数字含まないときは再ヒアリング。C→4 等の誤認識救済テーブルあり（customer_doc 1-9）。
- **予約日正規化**: 月日のみ発話 → 着信年を補完。10月以降に入電し1〜9月を言われた場合は **翌年**。過去日付は未来日付のみ受付。予約日（reservationDate=日付型）と予約希望日（フリーワード）を混同しない。

### director が見落としやすいポイント
- **改訂履歴 (2026/04/14)**: 「アナウンス文言修正」「生年月日にdtmf適用」。旧版資料を参照すると dtmf が漏れる可能性 → 生年月日ステップには必ず dtmf パラメータを入れる。
- **冒頭アナウンス直後のマイク制御**: customer_doc 2 に「冒頭アナウンスの発話後、次の質問を出力するまではマイクを置くことを厳に禁止する」との特記あり。opening ブロックで barge-in 抑制／次のヒアリングまで連続発話する設計に。
- **smsFlag=0 の初期化タイミング**: 冒頭アナウンスで `status=0, smsFlag=0` を更新値として設定（customer_doc 2 「smsFlag=0を格納する」）。script ブロックで忘れず初期化。
- **hearing 順序の固定**: 氏名 → 生年月日 → 連絡先電話番号 → 診察券番号 → 用件 の順。customer_doc 4-1「すでに聞いている項目は飛ばしてください」は AI 側の発話スキップであり、フロー構造は上記順序を堅持。
- **診療科リストが極めて大量（50以上）**: scaffold の正規化プロンプトに全件反映必要（director は customer_doc §3 を直接コピー）。類似科レコメンド（「当院に○○はございませんが、○○でしょうか？」）は正規化後の分岐で処理。
- **引用**: customer_doc 1-5「"○○でしょうか？"と回答を待つような返答をした場合は、返答があるまでは次の質問はしない」→ 近似診療科レコメンドのステップは **Yes/No ヒアリング待ち**を明示的に挟む必要がある。
- **FAQ を RAG 化しない**: customer_doc 1-2 に FAQ 記述はあるが「AI電話ではご対応できませんので、折り返しでご確認」で必ず逃がす設計。RAG サブフロー起こさず、内容聴取フローに合流。
- **電話番号復唱の TTS 表現**: customer_doc は `<speak type="telephone" breakc="300ms">` を使っているが Gen3 では SSML 禁止 → TTS は通常文 `{tts_g:...}` で「...でよろしいですか」化。
- **健診サイドとの差異注意**: 健診ブランチは classification が「予約/変更/キャンセル/確認/問い合わせ」等で別構造。診療は 3 値のみ・「新規予約なし」「変更なし」が明確な差異。サブフロー流用時に classification 分岐コピペ流用しないこと。

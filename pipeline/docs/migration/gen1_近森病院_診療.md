### 移管方針
- Gen1「箱」構造 → Gen3 9 ブロック型に変換。`挨拶`(「お電話ありがとうございます。 近森病院の予約に関するAI電話です」)→ opening + 冒頭_アナウンス(2 ブロック 1 セット必須)。挨拶直後に `折り返し連絡後確定`(「このお電話で承る内容は、担当者からの折り返しのご連絡により、確定となります」)が常時流れる → opening の announcement TTS 末尾に統合 or announcement ブロックを追加するかは director 判断。
- `24h稼働:稼働前その他例外休業時対策` サブシナリオ → date_of_call_classifier ブロック(addCurrentDate: Yes 自動付与対象)。但し**営業時間帯判定は無く**、4/22 稼働開始の日付ガード(4/21 以前 / 当日 8 時前 → END_営業前 → 「この番号は、4 月 X 日、朝 8 時半より利用開始となります」案内)と祝日チェックのみ。Gen3 では本番投入時点で日付ガードは不要なら削除、祝日チェックは Script モジュールで残す(director 判断)。
- 用件分岐は **Pattern A**(用件 OpenAI 直後に 4 分岐: 変更 / キャンセル / 予約(診療)/ その他確認)。各分岐に echo_back(変更復唱 / キャンセル復唱 / 診察予約復唱 / その他確認復唱)を直配置。
- **メインフローの聴取順序**: 用件聞き取り → 診察券番号 → 受診者名 → 生年月日 → 診療科聞き取り → **用件 4 種分岐**(用件を判断箱)→ 用件別フロー → 連絡先電話番号 → 最後の質問(伝えたいこと) → **携帯電話番号有無分岐**(用件を判断_1 / _2)→ 用件 4 種 × 携帯有無 = 8 通りの終話 TTS。
- 用件別フロー:
  - 変更: 予約日 → 連絡先電話番号
  - キャンセル: 予約日 → 内容聴取(キャンセル理由)→ 連絡先電話番号
  - 予約(診療): 予約希望日(受診を希望されている時期、自由テキスト)→ 連絡先電話番号
  - その他確認: 内容聴取(当院に伝えたいこと)→ 連絡先電話番号
- サブフロー(受診者名 / 生年月日 / 電話番号 / 診察券)は Jump to Flow 経由。`電話番号区切り.yaml` は `連絡先電話番号聞き取り.yaml` 配下の読み上げ整形用 → Gen3 サブフローテンプレ内蔵で完結、director は触らない。
- 終話は携帯有無で 2 文言(共通文の前段に各用件の文言):
  - 携帯あり: 「(用件名)の申し込みを受付いたしました。後ほど、担当者から折り返し電話、もしくはショートメールにてご連絡いたします」
  - 携帯なし: 「(用件名)の申し込みを受付いたしました。後ほど、担当者から折り返し電話にてご連絡いたします」
  - Gen3 では context_match_router 1 個に集約可能(director 判断)。

### 施設固有の特殊ルール
- **24 時間受付ボット**: 通常の「受付時間外アナウンス(平日 / 土曜の時間帯判定)」は元資料に無い。**祝日カレンダー**のみ Script で実装(addCurrentDate: Yes 自動付与対象)。「日曜祝日休診」のような時間外案内が必要かは customer_doc 確認、director 判断。
- **初回稼働日(2025/4/22)の日付ガード**: 元資料には「4/21 以前 / 当日 8 時前 → END_営業前 → 『4 月 X 日、朝 8 時半より利用開始』」案内 → 切断のロジックがある。**移管時点(2026-05-01)では既に稼働済みなので削除可**だが、Gen3 でも「将来の停止期間用」に残すかは director 判断。
- **DTMF 1-4 受付**: 用件聞き取りで 1=変更 / 2=キャンセル / 3=診療予約 / 4=確認。Gen3 では SKILL_A 用件分類プロンプトで吸収するか、ContextMatchRouter 並走するかは director 判断。
- **DTMF 4 桁日付**: 予約日(変更 / キャンセル時)で「シャープ / アスタ + 月 2 桁 + 日 2 桁」を受付。比較日より前なら +1 年で確定する仕様。Gen3 でどう扱うかは director 判断(SKILL_C 自由テキスト + DTMF 並走 / OpenAI プロンプト吸収)。
- **予約希望日(新規予約のみ)は自由テキストで「時期」を聞く軽量サブ**: 「受診を希望されている時期をお話ください。希望がなければ、ありません、のようにお話ください」。日付パース不要、内容聴取と同等の軽量 hearing で十分(director 判断)。
- **内容聴取は 2 用途で文言が変わる**:
  - キャンセル時: 「差し支えなければ、キャンセルの理由をお話ください。特に理由がない場合は、『特にない』、のようにお話ください」
  - その他確認時: 「ご確認内容について、簡潔にお話下さい」
  - 最後の質問サブ: 「当院にお伝えになりたいことがあればお話ください。特になければ『ありません。』とお話ください」
  - Gen3 では同一テンプレ hearing ブロックを 3 回配置 or 1 個 + 動的文言は director 判断。
- **3 回失敗時**: 用件・診療科・予約日 各サブで count 再質問(5 / 9 以上で escalate)→ 3 回不可で `END_聞き取れず` → Gen3 では termination_patterns に集約。**代表電話番号案内のアナウンス文言は元資料に無い** → customer_doc で確認、無ければ「お電話が遠いようで」標準文言で切断(director 判断)。
- **当日予約 NG / 健康診断 NG / 救急 NG / 紹介状有無確認**: いずれも元資料に無い(西宮渡辺・さやま と異なる)。director は誤って組み込まないこと。
- **復唱(echo_back)対象**: 用件 4 種 + 診療科 + 予約日 + 生年月日(生年月日サブ内蔵)のみ。受診者名・電話・診察券はサブフロー内で完結、メインに echo_back を出さない。

### director が見落としやすいポイント
- `最初の箱: API_DRJOY_CONTEXT_SETTINGS` → `API_DRJOY` → `挨拶` の 3 箱は Gen3 では opening 冒頭処理に吸収。明示ブロック化不要。
- 「セリフ: none」かつ振り分けのみの箱(`用件を判断 / _1 / _2 / _3` / `携帯電話番号チェック` / `完了フラグ` / `電話切断へ` / `分岐なし診療科セット` / `分岐あり診療科セット` 等多数)→ Gen3 で独立ブロック不要、前段の next 書き換え or context_match_router の一部として吸収。
- 短い繋ぎ TTS(`次へ`「ありがとうございます。次に、」/ `次へ_1`「かしこまりました。では次に、」/ `次へ_2`「ありがとうございます。続いて、」/ `次へ_3`「かしこまりました。続いて、」/ `次へ_4`「ありがとうございます。次に、」/ `診療聴取へ`「かしこまりました。それでは、」/ `連絡先電話番号聞き取りへ`「ありがとうございます。最後に、」/ `接続詞`「最後に、」/ `用件へ`「まず初めに、」)→ 次段 hearing/subflow ブロックの導入文言にマージするか、announcement ブロックで個別配置するかは director 判断(省略しすぎると自然さが落ちる)。
- 用件を判断箱が 3 つ(**用件を判断 / _1 / _2 / _3**)あるが役割が違う:
  - `用件を判断`: 用件 4 種を別ルートに分岐(変更 / キャンセル → 予約日、確認 → 内容聴取、予約 → 予約希望日)
  - `用件を判断_3`: キャンセル時の予約日聴取後「キャンセル → 内容聴取(理由)」「その他 → 予約希望日(誤り？)」分岐 ※ 元資料の挙動が怪しい、director は customer_doc で確認
  - `用件を判断_1` / `_2`: 終話前、携帯有無 × 用件 4 種で 8 通りの終話 TTS を選択
- `用件を判断_3` 箱は元資料で「変更 / キャンセル後の合流箱」だが、デフォルトが `予約希望日` になっている → **キャンセル以外(変更)は予約希望日に行ってしまう挙動**。Gen1 の元バグの可能性があり、Gen3 ではキャンセル → 内容聴取 / 変更 → 連絡先電話番号 に **director が修正**する。
- **用件聞き取りサブ**で `もしもし` エンティティ + countもしもし リトライあり(北里大・多摩総合と同系統)。OpenAI プロンプトに「もしもし」誤検知排除を入れる必要あり(SKILL_A)。
- 診療科聞き取りサブには **分岐あり診療科**(呼吸器 / 消化器 / 脳神経 → 内科 / 外科 の二段聴取)と **分岐なし診療科**(一般外科 / 循環器内科等 多数)がある。Gen3 では診療科 hearing ブロック内で OpenAI プロンプトが二段聴取制御するか、後段 ContextMatchRouter で分岐させるかは director 判断(SKILL_E + Pattern B 推奨)。
- 診療科確認後の `コンテキスト変換診療科抽出` 箱で「糖尿病・リウマチ科」だけ特殊変換あり → Gen3 では context 値の正規化ステップとして OpenAI プロンプト内で吸収。
- `予約日.yaml` は「日にち先行発話 / 月先行発話 / 数字のみ / DTMF 4 桁」の 4 系統 + 0 桁詰め + 大正/昭和/平成/令和の元号判定(実質予約日では使わないが流用)+ 比較日より前なら +1 年確定。Gen3 では date OpenAI(YYYY-MM-DD 出力)+ DTMF 並走で代替。
- `生年月日.yaml` 冒頭で「受診される方の生年月日をお話下さい」→ サブフロー内で完結、メインに echo_back を出さない。
- 受診者名サブのセリフは「受診される方のお名前を、フルネームでお話下さい」(**患者氏名 = 入電者ではなく、受診される方**)。Gen3 では PatientName サブフロー 1 枠に充当。
- **モジュール名・termination 名は ASCII または安全な日本語のみ**。丸数字(①②③…⑩)等の環境依存文字は禁止。
- **ContextMatchRouter.reference_module は subflow を参照する場合 `jump_{step名}`(scaffold が付ける prefix)を使う**。subflow ブロックの step 名そのもの(例: `電話番号聴取`)ではダメ。
- **CMR conditions は必ず最後に `match: "other"` + next 明示**。
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止。`<? $診療科の読み ?>` `<? $予約日の読み ?>` 等の Gen1 変数記法は Gen3 では `<% var %>` に変換。
- entity_classification.md / entity_stt_patch.json / entity_openai_synonyms.md にエンティティは自動分類済み。director は転記不要、scaffold/properties が自動取り込み。

### 移管方針
- **Pattern 判定**: 用件分類（診療／予防接種／その他）で**最初の分岐が発生する前に** 患者名・生年月日を先行聴取する構造（customer_doc §5 「共通ヒアリング項目」より）。用件分類後の分岐は単純な enum 一致であるため **Pattern A（OpenAI直後分岐）** を採用。
- **Date of Call Classifier**: 不要。受付時間帯判定は customer_doc に記載なし（24時間AI応答、時間外分岐なし）。
- **サブフロー化方針**:
  - 個人情報聴取（患者名／生年月日）: サブフロー化。**健診シナリオと共通化候補**。生年月日は復唱あり（和暦/西暦をそのまま）なので復唱ブロック込みで共通化可能。
  - 診察券番号: サブフロー化（Jump to Flow）。
  - 連絡先電話番号: サブフロー化(着信電話種別で分岐するロジックを内包)。健診シナリオと共通化候補。
  - RAG/FAQ: customer_doc にナレッジ本体は未記載だが、「その他」用件で FAQ 案内モードを使う明記あり。**`その他内容確認` ステップ内で RAG サブフロー呼び出し** を想定（FAQ未マッチ時は `ありがとうございます` のみで次へ）。
- **着信番号による連絡先スキップ**: Brekeke 側で着信番号先頭3桁を取得し、携帯（070/080/090/060）なら連絡先聴取を完全スキップ。ContextMatchRouter（Pattern B）で判定。
- **TTS形式**: `{tts_g:...}` 小文字。SSML タグ（`<break>`, `<speak type="telephone">`, `<dtmf>` 等）は customer_doc に多数登場するが **Gen3 では禁止**。`<dtmf/>` は Brekeke 側 DTMF 受付設定、電話番号復唱は telephone 読み上げ専用モジュールで代替。
- **smsFlag**: status=1 のとき自動的に 1、それ以外 0。終話直前の script ブロックで機械的に設定。

### 施設固有の特殊ルール
1. **復唱禁止ルール（極めて厳格）**: 生年月日と連絡先電話番号（新規口頭発話時のみ）**以外は一切復唱禁止**。患者名・診察券番号・診療科・ワクチン・予約希望日時の echo_back を **絶対に生成しない**。scaffold の `echo_back: false` を明示指定すること。
2. **「診療」用件の内容確認は記録のみ**: 「今日はどうされましたか？」の回答に対し、FAQ ヒット時も案内文を返さない(記録優先モード)。RAG 呼び出し禁止。
3. **「その他」用件のみ FAQ 案内モード**: `その他内容確認` ステップだけ RAG/FAQ 案内を許可。FAQ 無し時は「ありがとうございます」のみ。
4. **携帯電話着信時の連絡先スキップ**: 070/080/090/060 開始の着信は連絡先質問を完全省略し、即終話アナウンスへ。
5. **診療科・ワクチンのレコメンド**: リストにない場合「当院に○○はございませんが、○○でしょうか？」と類似候補を提示。回答待ちで次へ進まない。recommend 用の generate_by_OpenAI プロンプトが必要。
6. **「診察」vs「予約」判定語彙**: `印刷／定期／新薬／再診／最新／禁忌` → 診察、`薬／お薬` → 予約。用件分類プロンプトに明記必要。
7. **予約希望日時は日付変換しない**: フリーワードそのまま保存（`output_format: text`）。相対日付（今日／明日／明後日）のみ着信日時から換算。過去日付は未来確認を求める。
8. **「月の」等の繋ぎ言葉許容**: 「5月の20日」→「5月20日」として受理。
9. **classification の enum**: `診療／予防接種／その他／""`。用件聴取プロンプトで「変更・キャンセルも その他」と明示。
10. **代表電話案内時**: status=2、endpoint=電話案内。FAQ 外質問時の `END_FAQなし案内`（03-6426-5933）で到達。
11. **checkout フィールド**: 途中切断時の最後の未回答項目名を格納。質問前切断なら「冒頭切断」。script ブロックで現在ステップ名をトラッキング。

### director が見落としやすいポイント
- **改訂履歴（2026/3/23）の重要変更**:
  > 「復唱確認ルールを変更（生年月日と連絡先電話番号以外は一切の復唱を禁止）」
  > 「診療科（1-3）とワクチン（1-7）の特定時における復唱指示の矛盾を修正し、無言で次へ進むよう徹底」
  
  過去版に引きずられて echo_back を入れないよう注意。scaffold での `echo_back: false` 徹底。
- **聴取順序が用件別で異なる**:
  - 診療: 患者名 → 生年月日 → **診療科 → 内容確認** → 診察券番号 → 予約希望日時 → 連絡先
  - 予防接種: 患者名 → 生年月日 → **希望ワクチン** → 診察券番号 → 予約希望日時 → 連絡先
  - その他: 患者名 → 生年月日 → **その他内容確認** → 連絡先（**診察券番号・予約希望日時は聞かない**）

  「その他」で診察券番号を聞かないのが非自明。customer_doc §5 に明記:
  > 「*診察券番号や、予約希望日時は聞いてはいけない*」
- **患者名の下の名前追加聴取**: 苗字のみ発話時「下のお名前もお伺いしてもよろしいですか？」で再聴取。復唱確認ではない（再質問は許容）。
- **連絡先電話番号の3条件分岐**: 携帯（完全スキップ）／固定（現番号でよいか確認→違う場合のみ新番号聴取）／不明（直接聴取）。固定電話の「はい」応答は復唱なしで終話へ。
- **endpoint enum**: 冒頭切断／時間外／電話転送／電話案内／途中切断／通話完了 の6値。「時間外」は customer_doc 本文に分岐記載なし（enum のみ）が、現状フロー上は到達しない可能性あり。director 側で director 判断として termination_patterns から除外してよい。
- **患者名は再質問禁止**: 「名前の確認は一度だけ行い、再度名前を聞かない」。苗字のみ→下の名前追加は許可だが、それ以外の聴取済み項目は一切再質問不可。
- **AI内部思考の発話禁止**: 「スキップします」「確認は不要です」等を**絶対に発話しない**。prompter に明示制約として入れる。

### 健診シナリオとの共通化
- 冒頭アナウンスは**施設名が同じでもシナリオ名が違う**ため別セリフ（診療: "予約のお電話ありがとうございます"）。共通化不可。
- **患者名聴取サブフロー**: 共通化候補（同じ文言・同じ苗字のみ分岐）。
- **生年月日聴取+復唱サブフロー**: 共通化候補。
- **連絡先電話番号サブフロー**: 共通化候補（携帯スキップロジック含む）。
- 健診シナリオ側の customer_doc も参照して、細部の文言差異を director が最終確認すること。

### 移管方針
- Gen1「箱」構造 → Gen3 9 ブロック型へ変換。主な対応:
  - `挨拶`（「医事課外来係をお呼び出ししましたが、回線が混みあっておりお繋ぎできませんでした。AI電話がご用件をお伺いし、3診療日以内に担当者から折り返し電話連絡いたします。」）→ opening + 冒頭_アナウンス（2ブロック1セット必須）
  - `診察券番号聞き取り`（サブyaml）→ PatientID サブフロー。**最初に聴取 → 結果（診察券有/無）で分岐**する点が特殊（通常順序と逆）
  - 診察券「無」→ `次へ`（「それでは、」TTS）→ `患者氏名聞き取り` サブフロー → API_DRJOY_1 → `生年月日聞き取り` サブフロー → `連絡先電話番号聞き取り`
  - 診察券「有」→ `連絡先電話番号聞き取り` に**直接ジャンプ**（氏名/DOB を skip）→ `携帯電話番号チェック` → 問い合わせ①
  - `連絡先電話番号聞き取り` + `電話番号区切り` → Gen3 標準 PhoneNumber サブフロー（桁数判定・ハイフン整形は subflow 内蔵）
  - `携帯電話番号チェック` → 携帯番号取得済かどうかの簡易 CMR。後段は同じ `問い合わせ①` に流すため、Gen3 では context_match_router の小ブロックか省略
  - `問い合わせ内容①聞き取り`（サブyaml）→ hearing ブロック（自由文、output_format=text）。**用件分類による分岐は無し**、内容を記録するだけ
  - `追加用件有無確認`（サブyaml、Yes/No）→ Yes/No hearing ブロック（SKILL_B）
  - 追加用件「有」→ `問い合わせ内容②聞き取り` → `完了フラグ` → `予約変更の流れ`（実体は折り返し案内 TTS）→ `通話完了` → `終話`
  - 追加用件「無」→ `完了フラグ` に直行
- **本フローに用件分類分岐は無い**（①も②も自由文記録のみ）→ Pattern A/B いずれも不要、scenario_flow は基本直線 + 診察券有無の 1 分岐 + 追加用件有無の 1 分岐
- entity_classification.md は 5 yaml 集計だが、**用件分類で使われていない**点に注意（次節参照）

### 施設固有の特殊ルール
- **callback ボット**: 通常 inbound ではない。acceptance_times 不要、call_transfer 不要、date_of_call_classifier 不要（時間帯/曜日/年末年始の分岐箱が yaml に一切無い）。最後は終話のみ
- **聴取順序が特殊**: 診察券番号 → 氏名 → 生年月日 → 電話番号 → 用件（通常は氏名→DOB→診察券→電話→用件の順）。**診察券有無で氏名/DOB を skip する設計**。Gen3 でこの順序を尊重するか、director の標準順序に並べ替えるかは **director 判断**（推奨: 元資料順を尊重、診察券先頭で API_DRJOY → 患者特定して以降の聴取を省略する設計意図がある）
- **API_DRJOY 2 回呼び出し**: opening 直後（`API_DRJOY`）と氏名聴取後（`API_DRJOY_1`）の 2 箇所。Gen3 では 1 回目は opening 標準処理に吸収、**2 回目は明示 script ブロック化が必要**（氏名で患者特定 → DB 照合の意図、director 判断）
- **3 回聞き取り失敗時の代表案内 TTS が独特**: `3回聞き取りに失敗した場合` 箱で「代表電話の音声ガイダンスで『患者さん、また、ご家族の方は』の1番 → 『その他については』の4番を押してください」と DTMF ナビゲーション付きで案内（代表番号自体は TTS 中に明示されていない、director は元資料文言を完コピで使用）
- **折り返し案内文言**: `通話完了`（共通）「なお、代表電話は大変こみあっております。当院からの折り返し電話に出られなかった場合でも、必ずこちらから再度ご連絡をしますのでお待ちくださいますようお願い申し上げます。」+ `予約変更の流れ`（追加用件後）「ご用件を承りました。3診療日以内に担当者から折り返し電話連絡いたします。」→ Gen3 では termination 直前の announcement 2 段に分けるか 1 つに統合するか director 判断
- **携帯電話番号チェック**: `携帯電話取得あり` 一致でも next が `次へ_1`（同じく問い合わせ①）→ 実質分岐なし。元 yaml の意図不明だが、Gen3 では **context_match_router 不要、PhoneNumber サブフローから問い合わせ① へ直結**で良い（director 判断）
- **47 診療科の entity 値は使われていない**: main.yaml に診療科リスト（47 診療科）はあるが、実フロー内に診療科聴取箱が無い。entity_classification.md R1 48 件は **STT 辞書には積まないか、用件 hearing 補助の use_template 指定のみ**（director 判断、デフォルトは積まない）

### director が見落としやすいポイント
- `最初の箱: API_DRJOY_CONTEXT_SETTINGS` → `挨拶` → `API_DRJOY` の冒頭 3 箱は Gen3 で opening 冒頭処理に吸収。明示ブロック化不要
- 「セリフ: none」の振り分け／ジャンプ専用箱（`診察券有無`, `診察券有り`, `診察券無し`, `次へ_1〜_4`, `携帯電話番号チェック`, `完了フラグ`, `終話へ`）は Gen3 では **独立ブロック不要**、前段の next 書き換え or context_match_router 一部に吸収
- `次へ` 系の短い繋ぎ TTS（「それでは、」「続いて、」「では次に、」「はじめに、」）は Gen3 で **announcement の繋ぎ TTS 単独配置**するか hearing 導入文言に統合（director 判断、省略しすぎると自然さ低下）
- 用件聴取（問い合わせ①／②）の聴取内容チェックは「無言9（5秒）+ご用件なし → 再質問」「上記以外2（約1秒）+ご用件あり → 完了」のパターン。Gen3 では SKILL_C（自由テキスト）+ retry_count=2 で吸収
- **追加用件有無確認の Yes/No 分岐後**: 「有」→ 問い合わせ② → 完了フラグ、「無」→ 完了フラグ。**問い合わせ②と問い合わせ①は yaml 内容が完全同一**（コピー）→ Gen3 でも別 hearing ブロックを 2 本立てるか、1 つの hearing をループさせるかは director 判断（推奨: ブロック 2 本で素直に並べる、ループは scaffold が苦手）
- 患者氏名は PatientName サブフロー 1 枠に充当（受診者本人 1 名前提、入電者=受診者の callback 設計）
- 電話番号 subflow の retry_count=2 を超えた失敗 next 先（連絡先聞き取り不可時）は元資料に明示なし。**サブフロー終端の error 経由で `通話完了` 相当に合流させる**のが妥当（director 判断、病診連携の連絡先聞き取り不可 announcement と同等の救済を入れるかどうかも判断）
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止。動的値は `<% var %>` 構文（`<? $... ?>` 等の Gen1 構文は変換必須）
- ContextMatchRouter.reference_module は subflow 参照時に `jump_{step名}` 形式、conditions 末尾に `match: "other"` + next 明示

### 移管方針
- Gen1「箱」構造 → Gen3 9 ブロック型へ変換。主な対応:
  - `挨拶`（「薬剤部をお呼び出ししましたが…AI電話がご用件をお伺いし、本日中に担当者から折り返し」）→ opening + 冒頭_アナウンス（2ブロック1セット）
  - `API_DRJOY_CONTEXT_SETTINGS` / `API_DRJOY` → opening 標準処理に吸収（明示ブロック化不要）
  - `患者関連の問い合わせか確認`（サブyaml、Yes/No） → hearing ブロック（output_format=enum、SKILL_B Yes/No 判定）または generate_by_OpenAI 直後 Pattern A 分岐の起点
  - `問い合わせ内容①聞き取り`（自由文） → hearing ブロック（output_format=text、用件自由聴取、分岐に使わない）
  - `問い合わせ内容②聞き取り`（患者関連分岐後の追加自由文聴取） → 薬局ルートの hearing ブロック（output_format=text、独立 context に保存）
  - `薬局名・担当者名`（合成自由文：薬局名 + 担当者氏名を 1 発話） → 1 ブロックの hearing（病診連携の「施設名・部署・担当者名」と同じ運用、推奨は 1 自由文 context で担当者運用に委ねる）
  - 患者氏名 / 生年月日 / 診察券番号 / 連絡先電話番号 → Gen3 標準サブフロー（PatientName / DOB / PatientID / PhoneNumber）に Jump to Flow 経由で接続
  - `追加用件有無確認`（サブyaml、Yes/No、「その他、追加のご用件はございますか?」） → hearing ブロック（output_format=enum、SKILL_B）。Yes 時の合流先は `完了フラグ → 予約変更の流れ` の TTS（「ご用件を承りました…」）→ 終話
  - `連絡先電話番号聞き取り不可` → 専用 announcement「お電話番号を聞き取ることができませんでした。後ほど、担当者にて確認させていただきます。」→ 患者関連かチェックへ合流（病診連携と異なり終話に直行しない、後段の患者関連分岐に入る点に注意）
- **分岐構造（Pattern A 推奨）**: 用件①hearing → 患者関連 Yes/No 分岐
  - **患者関連=Yes** ルート: 用件②hearing → 患者氏名 sub → 診察券有無 Yes/No 分岐
    - 診察券=有: 診察券番号 sub → 次へ_2 経由で生年月日 sub → 連絡先電話番号 sub
    - 診察券=無: 直接 生年月日 sub → 連絡先電話番号 sub
  - **患者関連=No** ルート: 薬局名・担当者名 hearing → 連絡先電話番号 sub
  - 連絡先後に `患者関連かチェック` で再分岐（既取得の患者関連 context で振り分け）→ Yes 側は `次へ_3` 経由 診察券へ、No 側は `次へ` 経由 追加用件確認へ … というのが原資料の流れだが、Gen3 では先頭の患者関連分岐の context を使い回すだけで良く、**後段の再分岐は不要**（director 判断、推奨は Pattern A の単一分岐に統合）
  - 最終合流: 追加用件確認 → 完了フラグ → 予約変更の流れ TTS → 通話完了 TTS（「代表電話は大変こみあっております…」）→ 終話

### 施設固有の特殊ルール
- **callback ボット**: 病診連携と同じく薬剤部代表に繋がらなかった電話を AI が一次受け → acceptance_times 不要、call_transfer 不要、incoming-classifier から opening へ直結
- **call_transfer / date_of_call_classifier なし**: 転送箱・受付時間判定箱がフロー内に一切存在しない。両ブロックを生成しない
- **2 系統分岐**: 患者関連 Yes/No で必要 hearing が大きく変わる（患者ルートは 4 sub + 用件②、薬局ルートは 薬局名・担当者名のみ）。director は Pattern A で先頭分岐を作り、共通末端（連絡先電話番号 + 追加用件確認 + 終話）に合流させる
- **連絡先電話番号聞き取り不可の合流先**: 病診連携と異なり「終話直行」ではなく「患者関連かチェック」への合流。Gen3 では先頭分岐の context を再利用して、Yes ルートは追加用件確認へ、No ルートは追加用件確認へそれぞれ合流（携帯番号スキップ）
- **携帯電話番号の追加聴取**（連絡先電話番号聞き取り.yaml l.397–468「緊急時の連絡方法として、携帯電話の番号をお伺いしています。」「携帯電話をお持ちの場合は番号を、お持ちでない場合は『持っていない』と」） → 病診連携と同じ仕様。PhoneNumber サブフロー標準仕様外なので、サブフロー後段に追加 hearing ブロックを置くか、PhoneNumber を 2 連で呼ぶ（director 判断）
- **47 診療科 entity は不使用**: main.yaml に 47 診療科リストが定義されているが、薬剤部フロー内に診療科聴取箱が存在しない（病診連携と同様）。STT 辞書には積まない（director 判断、entity_classification.md の R1 48 件は use_template 参照にとどめる）
- **DTMF 救済語の有無**: 用件①/②聞き取り yaml には DTMF 入力の記載は無し（さやま診療と異なり数字入力分岐なし）
- **追加用件有無確認の Yes ハンドリング**: yaml 内部では Yes 時 `END_追加有` → main の `完了フラグ` 経由 `予約変更の流れ` TTS に合流する設計だが、実フロー上は Yes/No いずれも同じ終話 TTS に流れる（追加要件があっても受付内容として記録するのみで再ヒアリングはしない）→ Gen3 では Yes/No 分岐を作らず単一終話で良い（director 判断）
- **エンティティ「予約以外」「予約変更」「予約キャンセル」「定期受診」**: main.yaml に定義があるが、薬剤部フロー内で実際に分岐に使われている箱は無い（用件①/② はいずれも自由文聴取で分類分岐しない）→ 不採用、OpenAI 正規化に積まない

### director が見落としやすいポイント
- **問い合わせ内容①と②は 2 回別々に聴取**する設計（①= 初回用件、②= 患者関連と判明した後の追加聴取）。1 つの context にまとめず、別 context（例: yoken_text_1 / yoken_text_2）に保存する。**②は患者ルートでのみ呼ばれる**点に注意
- **「次へ」「次へ_1〜_6」**（セリフ「それでは、」「では次に、」「続いて、」「はじめに、お伺いします。」） → 短い繋ぎ TTS の announcement ブロックとして個別に配置するか、隣接 hearing の冒頭文に統合（director 判断、省略しすぎると自然さが落ちる）
- **「セリフ: none」の振り分け箱**（`診察券有無`, `携帯電話番号チェック`, `患者関連かチェック`, `完了フラグ`, `診察券有`, `診察券無` 等）は **独立ブロック不要**、前段の next 先書き換えまたは ContextMatchRouter の一部として吸収
- **`連絡先電話番号聞き取り不可` 箱**: 病診連携では終話直行だが、薬剤部では `患者関連かチェック` に合流する設計。電話番号 subflow の retry_count 超過 next 先として配置し、その後の合流処理に注意
- **薬局名・担当者名の聴取は 1 ブロック合成聴取**（薬局名 + お名前を 1 発話で受ける）。2 ブロックに分けない。元 yaml の聴取文「お電話いただいている方の、薬局名、お名前をおっしゃってください。」をそのまま使う想定。output_format は text、保存先 context は 1 自由文 context に集約（推奨、病診連携と同方針）
- **「予約変更の流れ」TTS**（「ご用件を承りました。当日中に担当者から折り返し電話連絡いたします。」）と **「通話完了」TTS**（「なお、代表電話は大変こみあっております…必ずこちらから再度ご連絡をしますのでお待ちください…」）は終話直前の 2 連 announcement として残す（折り返し前提の callback ボットなので運用上重要）
- **patient 1 名前提**: 入電者 = 受診者 / 患者本人 or 家族の代理問い合わせ前提。PatientName サブフロー 1 枠で充当（feedback_patientname_subflow_allocation.md 準拠）
- **副作用・処方・症状等の問い合わせ内容**: yaml 上は自由文聴取のみで分類分岐していないため、Gen3 でも分類しない。プロンプターは用件①/② の OpenAI プロンプトを「自由文要約・記録用」設計とする（SKILL_E 自由テキスト系）
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止

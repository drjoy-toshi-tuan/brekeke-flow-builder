### 移管方針
- 入電番号からの自動取得 → 施設名確認 → ヒット時はそのまま状態聴取へ／ヒット失敗 or 否定で施設名聴取からの通常ルートへ合流。
- メインの分岐起点は「救急搬送希望 / 直接来院」の二択（custom entity）。Pattern A で hearing 直後分岐。
- 搬送ルート: 個人情報 → 施設名 → 住所 → 状態 → バイタル（血圧/体温）→ 患者氏名 → 連絡先 → 緊急手術確認 → Blind Transfer（2 系統順次）。
- 直接来院ルート: 選定療養費（紹介状無し時 9,900 円）アナウンス → 患者氏名 → 個人情報 → 症状聴取 → 連絡先 → 終話（はしか流行注意のアナウンス付き）。
- 転送系統は `転送1(080-9923-2523)` → `転送２(080-7031-1207)` の順次フォールバック（Blind Transfer 2 段、各々の前に「ただいまお繋ぎしております」アナウンス）。
- 終話時の聞き取り失敗フォールバック: 「入電番号へ折り返し」または「施設名・住所から調べて折り返し」の 2 文言を分けて維持。

### 施設固有の特殊ルール
- **緊急時オーバーライド（緊急時変更）**: ベット逼迫時に「公設救急優先」の固定アナウンスを流して切断する分岐が存在。Gen1 では「緊急時対応ノードへ繋ぐ場合はここから」という入口があり、現状は `中村さん作成：搬送or来院` サブフロー経由。Gen3 では運用フラグ（acceptance_times 的な切替）で出し分けられる構成にしておく必要あり。詳細仕様は元資料 main.yaml 行 301-310 と `中村さん作成：搬送or来院.yaml` を必ず確認。
- **選定療養費アナウンス（直接来院時）**: 「紹介状持参がない場合、選定療養費 9900 円をお支払いいただきます」という固定文言を直接来院ルートに必須挿入（搬送ルートには出さない）。
- **はしか流行アナウンス**: 終話文言（搬送/直接来院いずれも）に「国内ではしか流行確認、発疹/発熱/海外渡航歴の方は受付申し出」の固定文言を含める。季節要因なので Dr.JOY 側で差し替え可能な扱い。
- **バイタル聴取**: 血圧・体温を聴取するサブフロー（旧バイタル）。診療予約には無いノード。`血圧や体温の聴取（旧バイタル）.yaml` 参照、output_format は数値 + 単位の自由テキスト想定。
- **転送先 2 系統の順次フォールバック**: `転送1(080-9923-2523)` → `転送２(080-7031-1207)`。Blind Transfer の `on_failure_announcement` で 1 段階デフォルトのところ、本施設は **2 段階順次トライ**が必須仕様。Brekeke IVR UI 側で手動追加が必要（フロー設計書には call_transfer 1 ブロックのみ置き、N-* で確認項目として明示）

### director が見落としやすいポイント
- メインの主分岐は `搬送or来院` サブフローの戻り値（救急搬送希望 / 直接来院）で、そこから `搬送or直接来院判断_1` で Pattern A 分岐。`搬送or直接来院判断_2`（連絡先取得後の完了フラグ確認）も同 entity を再利用しているので、context を 1 本で持ち回す（重複定義しない）。
- `CP*NG`（CP個人情報NG / CP施設名NG / CP住所NG / CP詳細NG / CPバイタルサインNG / 搬送or来院 CP / 緊急手術確認 CP）はセリフ none の振り分けのみ → 独立ブロック化禁止、`3回失敗⇒終話へ（番号OK）` に集約。`CP連絡先NG` のみ「施設名と住所で調べて折り返し」の別文言なので独立保持。
- 緊急手術確認 → 転送 の流れは `転送へ（完了フラグ7）` で「電話を転送いたします、少々お待ちください」アナウンス → 転送1 → 失敗時に「ただいまお繋ぎしております」→ 転送2 のフォールバック
- `中村さん作成：搬送or来院.yaml`（緊急時運用版）と `搬送or来院.yaml`（通常版）が併存。現行どちらが本番で動いているかは元資料 main.yaml の `緊急時対応ノードへ繋ぐ場合はここから` の遷移を確認（現状デフォルトが `中村さん作成：搬送or来院` を指している点に注意）

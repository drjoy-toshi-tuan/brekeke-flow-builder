### 移管方針
- Gen1「箱」構造 → Gen3 9 ブロック型へ変換。挨拶 TTS は「お電話ありがとうございます。真生会富山病院、コロナ、インフルエンザ、予防接種専用、AI電話です」で opening + 冒頭_アナウンス 2 ブロック 1 セット。
- 用件分岐は **Pattern A**（用件聞き取り OpenAI 直後に 4 分岐: 新規予約 / 変更 / キャンセル / 問合せ）。**ただし「初再診確認」は予防接種では新規/再診ルート切替に直結せず**、新規予約復唱 → 用件確定で初再診確認（用件聞き取り.yaml `初再診確認`）に進むのは **DTMF 1=肯定 / 2=否定 で初診/再診を出すための分岐のみ**。Gen3 では subflow 内で吸収するか別 hearing で分けるかは director 判断。
- echo_back（用件復唱）4 種は診療と同様: 「予防接種のご予約で、よろしいですか？」/「ご予約の変更で、よろしいですか？」/「ご予約のキャンセルでよろしいですか？」/「その他問い合わせ、でよろしいですか?」。
- 用件別フロー分岐（main.yaml `用件を判断` から director 読取）:
  - 新規予約: ワクチン種類聴取 → 接種費用案内（インフル/コロナ/同時で TTS 切替）→ 予約希望日 → API_CHECK_BOOKING → 空き枠数で分岐（0 件 → 再聴取、1〜4 件 → 復唱 or 希望時間聴取、5+ 件 → 希望時間自由聴取）→ API_CHECK_FRAME_AND_LOCK → 復唱確認（ワクチン・予約日時）→ 肯定否定 → API_BOOKING → 患者氏名 → 生年月日 → 連絡先電話（入電番号自動セット）→ 終話「予約完了 + 内容復唱 + 当日注意事項」
  - 変更: 予約日（既存予約日）→ ワクチン種類 → 予約希望日 → 以下新規と同じ → 終話「変更完了」
  - キャンセル: 予約日（既存予約日）→ 患者氏名 → 生年月日 → 終話「キャンセル完了」
  - 問合せ: 接種後の体調か確認（肯定否定）→ Yes なら **代表案内ルート（0766-52-2156）で切断**、No なら問合せ内容の聞き取り → 患者氏名 → 生年月日 → 終話「3 日以内折り返し or SMS」
- ワクチン / 予約日 / 患者氏名 / 生年月日 / 連絡先電話 / 問合せ内容は全て **subflow ブロック (Jump to Flow)** で組む。`電話番号区切り.yaml` は連絡先電話 subflow 内ヘルパーで吸収。
- 時間判断.yaml（祝日チェック呼び出し + 平日 9-15 時 / 土曜 9-11 時 + 1/4 まで休業）→ acceptance_times に集約（**addCurrentDate: Yes 自動付与対象**）。
- entity_classification.md: STT use_template 9 種 + ワクチン種類 (R1_STT_PROPER_NOUN_GROUP 37 件) は additional_words 直書き、OpenAI 正規化 158 件。

### 施設固有の特殊ルール
- **受付時間**: 日祝・年末年始（〜1/4）除く、月〜金 9:00〜15:00、土曜 9:00〜11:00（**診療側 (8:30〜17:00 / 8:30〜12:00) と異なる別ルール**。受付時間外アナウンス文言もこれに従う）。
- **予防接種専用窓口** → 診療科聞き取りは無く、ワクチン種類聞き取り (`予防接種の種類聞き取り.yaml`) で **インフルエンザ / 新型コロナ / 同時接種** の 3 区分。同時接種は STT 辞書に「あわせて / どちらも / 二つとも / 一緒に」等 OpenAI 同義語多数（director は entity_openai_synonyms に委譲）。
- **接種費用案内 TTS**（main.yaml `インフルエンザ接種費用` / `新型コロナ接種費用` / `同時接種　接種費用`）: 射水市・富山市の接種券持参者は接種券記載金額、一般料金は **インフル 4,000 円 / コロナ 15,300 円 / 同時はインフル 4,000 円・コロナ 15,300 円**。ワクチン分岐後の announcement で **必ず** 案内する（料金トラブル防止、director は省略禁止）。
- **接種日は病院側で限定**（main.yaml `希望日の聞き取りへ` の TTS は「接種は午後に実施しています。接種日は、1月26日月曜日です。ご都合の良い日を、お話ください」）→ Gen1 当時は接種日 1 日のみ表記だった。**Gen3 移管時の最新接種日は director が customer_doc または運用担当に確認必須**（日付ハードコード TTS のためここを誤ると本番で破綻）。
- **稼働期間制限**: 「9/24 まで稼働前」「1/26〜終了アナウンス（今年度の AI 電話予約は終了）」(main.yaml `1/26～終了アナウンス設定` / `稼働前アナウンス`)。**通年稼働ではなくワクチン接種シーズン限定**。Gen3 では acceptance_times の年内範囲設定 or 別 announcement で「○月○日まで／○月○日から」アナウンスにするかは director 判断。
- **1 通話 1 名分のみ**（予防接種の種類聞き取り.yaml `冒頭アナウンス` の TTS）→ ワクチン種類聴取の **直前に必ず announcement で告知**（家族複数名一括予約のクレーム回避目的）。
- **空き枠数別の TTS 分岐ロジック**（API_CHECK_BOOKING 戻り値）:
  - total = 0 → 「<予約希望日>にご予約できる時間帯はございません」→ 予約希望日再聴取
  - 1 ≤ total ≤ 4 → 全候補時間を読み上げて選択 (`<? $start_time_list ?>、です。難しい場合は「いいえ」、とお答えください`)
  - total ≥ 5 → 「ご希望のお時間をお話ください」→ 希望時間 OpenAI 抽出 → API_CHECK_TIME で再判定
  - **時間枠ロック失敗（API_CHECK_FRAME_AND_LOCK = false）→ 「ご希望の時間帯は予約が取れないため、別の時間帯をご返答ください」→ API_CHECK_BOOKING 再実行**
  - これらは scenario_flow で context_match_router + script + 再 jump ループとして組む（director 判断、Brekeke 側の API モジュール仕様確認必須）
- **予約日 (`予約日の聞き取り.yaml`、変更・キャンセル時の既存予約日) には DTMF 4 桁入力対応**（シャープ・アスタ + MMDD 4 桁、過去日なら +1 年自動補正）→ Gen3 では SKILL_C 自由テキスト+DTMF 並走か OpenAI YYYY-MM-DD 出力 + 過去日補正で対応（director 判断）。
- **3 回失敗時のフォールバック**: 健診センター（**0766-52-2473**）案内 TTS で 3 回繰り返して切断（main.yaml `3回失敗アナウンス（健診センター案内)`）。**診療側の「アイセンター 0766-52-6979」とは別番号**（director は混同禁止）。
- **代表案内（0766-52-2156）は「接種後の体調不良問合せ」専用パス** → 用件聞き取り.yaml `END_接種後の体調→代表案内へ` から到達。一般 3 回失敗ルートの健診センター（0766-52-2473）と **完全に別の終話**として実装（director 判断、混同禁止）。
- **DTMF 入力**: 用件聞き取り（DTMF 1=予約 / 2=変更 / 3=キャンセル / 4=確認・その他）、初再診確認（1=肯定/初診 / 2=否定/再診）、ワクチン種類聞き取り（DTMF 1=肯定 / 2=否定）等 多数。Gen3 では generate_by_OpenAI のプロンプト側 or ContextMatchRouter で吸収（SKILL_A 参照、director 判断）。
- **非通知判定**: main.yaml `入電番号確認` で非通知 → 「通知設定するか先頭に 186 を付けてかけ直し」案内で切断。**入電番号は連絡先電話番号として自動セット**（`入電番号を連絡先電話番号にセット` 箱）→ 連絡先電話番号サブフロー呼び出し不要のはず（director は流れの分岐確認、非通知時のみ subflow 呼び出すか否か customer_doc 要確認）。
- **復唱（echo_back）対象**: 用件 4 種 / ワクチン種類 3 種 / 予約希望日 / 予約日（変更・キャンセル時）/ 復唱確認（ワクチン・予約日時セット）/ 初再診確認（DTMF or 発話）。氏名・生年月日・電話はサブフロー内で完結、メインに echo_back 出さない。
- **終話 TTS は 3 種**: 新規予約「予約完了 + 内容復唱 + マスク着用 + 接種券持参」/ 変更「変更完了 + 内容復唱 + 注意事項」/ キャンセル「キャンセル完了」/ 問合せ「3 日以内折り返し or SMS」。**新規予約と変更の終話で同じ「補足（注意事項）」TTS を共有**（main.yaml `補足` 箱）。

### director が見落としやすいポイント
- **モジュール名・termination 名は ASCII または安全な日本語のみ**。元資料に `count再質問_1〜_7` `count上記以外_1〜_2` `次へ_1〜_3` 等の連番ノードが多数あるが、Gen3 では意味のある日本語に置換すべき（例: `END_健診センター案内` `END_代表案内_接種後体調` `END_時間外` `END_稼働前` `END_今年度終了` 等）。**丸数字（①②③…）等の環境依存文字は禁止**。
- **ContextMatchRouter.reference_module は subflow を参照する場合 `jump_{step名}` (scaffold が付ける prefix) を使う**。subflow ブロックの step 名そのもの（例: `予約希望日` `予防接種の種類聞き取り`）ではダメ。
- **CMR conditions は必ず最後に `match: "other"` + next 明示**。
- `最初の箱: API_DRJOY_CONTEXT_SETTINGS` → `挨拶` → `API_DRJOY` → `新エンジン移行後アナウンス(完了フラグ：7)` → `稼働前分岐 (9/24 まで)` → `9/25分岐` → `1/26～終了アナウンス` → `時間判断` の冒頭チェーンは、**Gen3 では opening 冒頭処理 + acceptance_times に吸収**。新エンジン関連の 9/25 / 1/26 切替は **Gen3 移管時点で前提が変わっている可能性大**（毎年の接種シーズン日程は変動）→ director は customer_doc / 運用担当に **接種シーズン開始日・終了日・接種実施日** を必ず確認。
- `用件の確認` (main.yaml 共通) と `用件を判断` (main.yaml 共通) は **役割が違う**。前者は echo_back 後の最終分岐（4 用件 → 各サブフロー先）、後者は **問合せルートの分岐（接種後体調 vs 通常問合せ）+ 変更/キャンセルでの予約日聴取分岐 + 新規予約の予防接種の種類聴取分岐**。Gen3 では context_match_router 2 個に分けるのが素直。
- `判定: エンティティ含む` で参照されるエンティティ値（ワクチン種類 / 肯定 / 否定 / 初診 / 再診 / 用件 / 接種後の体調 / もう一度 / もしもし / 分からない 等）は **entity_classification.md / entity_stt_patch.json / entity_openai_synonyms.md にすべて自動分類済み** → director は転記不要、scaffold/properties が自動取り込み。
- 用件聞き取り.yaml の `count再質問_1〜_7` / `count上記以外_1〜_2` / `count再質問_4〜_5（DTMF専用）` は Gen3 では **hearing.retry_count=2 に統合**。ただし `count再質問_7` は「否定回答後の用件再聴取」、`count再質問_4` は「DTMF 失敗 2 回後の終話」と意味が違う → director 判断（過剰統合で文脈崩壊しないこと）。
- 予約希望日.yaml の `日付チェック_1〜_3` `月チェック` `日チェック` `0桁詰め` `数字を日付に` `日付の判定`（大正〜令和の年号レンジ判定）等の DTMF 4 桁正規化ロジックは **Gen3 では subflow 内 OpenAI に YYYY-MM-DD 出力 + 過去日 +1 年補正で対応**（director 判断、SKILL_C 参照）。年号レンジ全網羅は predefined_dictionary でなく datetime hearing で吸収。
- API_CHECK_BOOKING / API_CHECK_TIME / API_CHECK_FRAME_AND_LOCK / API_BOOKING の **4 つの API 呼び出しは Brekeke 側のカスタムモジュール**で、Gen3 移管時にも同等の Dr.JOY 連携 API が必要。**director は Dr.JOY 担当に予防接種枠管理 API の有無を要確認**（無ければ予約完結フローは実装不可、案内のみフローへ縮退する判断が必要）。
- `補足` 箱の TTS（新規予約・変更の終話で共通）は **「予約内容復唱 → 注意事項 → もう一度予約内容復唱」と二重に予約内容を読み上げる長文** → Gen3 でもこの 2 重復唱を維持するか、注意事項のみに簡略化するかは director 判断（高齢者向け配慮で 2 重維持推奨）。
- **接種日が予約希望日 TTS にハードコード**されているため、**移管時は director が customer_doc 確認の上で TTS テンプレートを動的化（例: `{接種日_1}{接種日_2}...` プロパティ参照）するか確認が必要**。プロパティ管理を gen_properties に委ねるなら scaffold 側で TTS placeholder を残す。
- 患者氏名は PatientName サブフロー 1 枠（受診者 = 入電者前提、家族代理通話は元資料に明示なし → director は subflow 仕様デフォルトで OK、ただし「1 通話 1 名分」の運用上、複数名分の予約は不可と TTS で明示済み）。
- TTS は `{tts_g:...}` 小文字、SSML 禁止。Gen1 の `<? $予約希望日の読み ?>` `<? $clinical_department ?>` `<? $booking_frame_time ?>` `<? $start_time_list ?>` 等の変数記法は Gen3 では `<% var %>` に変換（`clinical_department` は予防接種では「インフルエンザワクチン」「新型コロナワクチン」「同時接種」のいずれかが入る変数のため、変数名は `vaccine_type` 等にリネームすべきか director 判断）。

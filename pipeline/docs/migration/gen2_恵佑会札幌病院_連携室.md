### 移管方針
- 冒頭アナウンスは「明日以降の予約か？」の Yes/No 判定が直結する設計。Yes→用件確認、No→当日案内（status=2/smsFlag=0で代表電話案内終話）。冒頭ブロックの直後に Yes/No 判定の generate_by_OpenAI を置き、否定で代表案内 termination に分岐させる **Pattern A** 構成。
- 用件確認は4分岐（予約 / 変更 / キャンセル / その他問い合わせ）+ DTMF "1〜4" 一致。OpenAI で classification を抽出 → ContextMatchRouter で4分岐に振り分ける **Pattern B** が自然。
- 予約フローは「受診歴 → (初診なら紹介状確認 → 紹介時診療科 / 紹介状なしなら診療科) / (再診なら診療科 → 予約希望日 → 診察券番号)」の入れ子分岐。受診歴と紹介状の2段の OpenAI 分岐 + ContextMatchRouter で構築。
- **歯科口腔外科は全診療科聴取モジュールから即座に専用電話案内へ離脱**（status=2/smsFlag=0、050-1725-8755 を案内して終話）。診療科ブロック内で歯科口腔外科を別 next にする必要あり。
- 氏名／生年月日／連絡先電話番号／診察券番号は **PatientName / DateOfBirth / PhoneNumber / MedicalCardNumber サブフローを Jump to Flow** で接続。インライン展開禁止（feedback_subflow_connection_rule）。
- 生年月日は **2026/03/15 改訂で復唱確認が追加**。サブフロー側に echo_back を入れる必要あり（DateOfBirth サブフローの復唱バージョン採用）。
- 連絡先電話番号は incoming_phone_number を見て3分岐（携帯=利用確認 / 固定=新規聴取 / 非通知=新規聴取＋非通知文言）。PhoneNumber サブフローの incoming 分岐対応版を使用。
- 終話は smsFlag で2系統 × 携帯/固定で4系統:
  - 一般（予約/変更/その他問い合わせ）: status=1/smsFlag=1（固定 / 携帯で文言違い）
  - キャンセル: status=1/smsFlag=2（固定 / 携帯で文言違い）
  - 当日案内: status=2/smsFlag=0
  - 歯科口腔外科離脱: status=2/smsFlag=0
- Date of Call Classifier は不要（営業時間の言及が customer_doc に無く、冒頭で「明日以降か」を聞く運用で代替している）。

### 施設固有の特殊ルール
- **歯科口腔外科ガード**: あらゆる聴取項目で「歯科口腔外科」関連発話を検知したら status=2/smsFlag=0 で 050-1725-8755 案内終話。診療科聴取・紹介時診療科聴取の OpenAI に enum で「歯科口腔外科」を含め、ContextMatchRouter で歯科口腔外科 termination に飛ばす（FAQ への誤波及禁止）。
- **その他確認事項の再質問は1回のみ**（2026/03/15 改訂）: 「他にお問い合わせはございますか？」を1回挟んだ後は次フローへ強制遷移。再ループ禁止。
- **生年月日復唱**（2026/03/15 改訂）: 「生年月日はyyyy年m月d日でよろしいですか？」を西暦で復唱。否定なら再聴取、それ以外は次へ。
- **氏名のカタカナ判定**: 1回だけリトライ可。再聴取後は内容そのまま patientName へ格納（空でも次へ）。聴取できなかった場合も先へ進む。
- **連絡先電話番号の桁数チェック**: 10桁または11桁。先頭0欠落（"90..."）は0補完。再聴取は1回のみ。FAQ 参照禁止（桁数のみで判定）。
- **incoming_phone_number 分岐**: 090/080/070/060 → 着信番号利用確認、050/その他固定 → 新規聴取、anonymous/非通知/unknown/空 → 「お電話番号が非通知のため」と前置きして新規聴取。
- **smsFlag = 2 はキャンセルのみ**。一般終話は smsFlag=1。
- **終話文言は固定文言**: 「ありがとうございます」「お気軽に」禁止、「お話ください」（「おっしゃってください」NG）。
- **数字キーワード辞書 / 同音異義置換**（「印刷→診察」「電気→新規」「歯医者→歯科」等）が customer_doc に大量にあるため、診察券番号・診療科・用件確認の各 generate_by_OpenAI prompt にキーワードリストを反映する必要あり。director は元資料の該当ブロックを全件抽出すること。
- **予約日の年補完**: 月日のみ発話なら入電年を補完。10月以降に1〜9月の日付が来たら翌年。**「予約日」と「予約希望日」は別変数**で、予約日のヒアリングがない場合に予約希望日の値を予約日に流用してはならない。
- **タグ保護**: `<dtmf />`, `<speak>`, `<forward>` を含むテキストは加工せず保持（ただし TTS には `<% var %>` 以外の動的タグはリテラル化されるリスクあり、E-15 / gen_properties strip 対象）。
- **status=0 厳守**: 終話トリガー文言を発話し終えるまで status は 0。途中切断は必ず 0 で dialogue_completed。

### director が見落としやすいポイント
- 連携室シナリオなのに患者本人向け予約フロー: 「地域連携室」という箱名だけで医療機関職員向けと誤解しやすい。冒頭文「恵佑会札幌病院の、診察予約申し込み電話です」（170行目）が証拠。
- 生年月日の復唱は **2026/03/15 改訂で追加された**（44-46行目）。旧仕様（復唱なし）で起草しないこと。
- 「その他確認事項の再聴取は1回のみ」も同改訂で追加（83-84行目, 467-472行目）。无限ループ防止のため、追加質問 → デフォルト終話分岐へ強制合流させる。
- 用件確認 DTMF: 「`<dtmf digit="1"/>`」と本文中に出現するが、これは "1〜4 のいずれか" を DTMF 入力可能とする意味（335行目）。OpenAI 抽出 + DTMF 一致 routing を両方備える。
- 紹介状あり時のみ「紹介時診療科聴取」の文言が変わる（402行目: 「紹介状の封筒に記載されている診療科名を…」）。紹介状なしの「診療科聴取」（405行目）と TTS を取り違えないこと。
- キャンセル終話は **smsFlag=2**（492行目）。一般3用件は smsFlag=1。終話 termination ブロックを smsFlag 別に分けるのを忘れない。
- function スキーマには checkout / endpoint / reservationType も required で含まれる（506-525行目）。required 全充足のためのデフォルト値設定が必要。
- **問い合わせ内容**（その他問い合わせフロー）と**その他確認事項**（全用件共通の最後）は別物。両方とも reason / others に格納し、再質問1回ルールが適用される（83-87行目）。
- 診療科 enum と診療科キーワード辞書が両方ある（180-267行目）。OpenAI prompt の enum と同義語注釈は一致させる。

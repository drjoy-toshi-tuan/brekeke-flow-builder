### シナリオ概要

患者（または家族）が「都立豊島病院の予約専用AI電話」に電話し、当日受診・翌日以降の予約新規/変更/キャンセル/問い合わせを行う。診察予約が中心で、紹介状の有無・選定療養費（7,000円）確認が特徴的なフロー。産婦人科は妊婦初診専用の複雑な分岐を持つ。

### 設計方針

- **1フロー（multiflow 不要）**: 用件4択はすべて1フロー内で分岐可能。
- **入電者**: 患者本人または家族。PatientName / Phone / MedicalCard / DateOfBirth サブフローはすべて利用可。
- **用件分岐方式**: DTMF 4択（1=予約/2=変更/3=キャンセル/4=その他）→ Pattern A（DTMF プッシュ直後に分岐）。OpenAI は不要。
  - ただし「ご用件をお話しください」の自由発話フォールバックが設計書に存在（用件失敗時）→ hearing ブロック1枠で自由発話受付し、generate_by_OpenAI で意図分類するか要判断。director 判断。
- **Date of Call Classifier**: 「本日の受診 vs 翌日以降」の DTMF 2択（1=当日/2=翌日以降）が冒頭にある → date_of_call_classifier ブロックではなく DTMF script ブロックで分岐。director 判断。
- **診察券番号**: DTMF 入力（# でスキップ / * で再入力）→ MedicalCard サブフロー（card_number モジュール）を使用。
- **WebRTC 用バリアント**: 冒頭アナウンスと終話が別文言（スライド15）。WebRTC 用 opening は「緊急症状の誘導なし」バージョン。director が確認レポートで WebRTC 分岐の実装方針を BLOCKER 化すること（別フロー / 同フロー内 flag / 現時点は demo なので pending）。

### 施設固有の特殊ルール

1. **冒頭 DTMF 2択**: 本日受診=1 / 翌日以降=2。本日受診ルートと翌日ルートでは後続のフローが大きく異なる。
2. **紹介状確認 + 選定療養費**: 本日受診 かつ 受診歴あり かつ 紹介状なし → 「7,000円かかる場合がございます、受診を希望する場合は1」。受診取りやめ(2) → 終話②（お断り終話）。
3. **産婦人科の予約の種類分岐**: 予約の種類=3（妊婦初診）のときのみ「出産予定日」「和通分娩の希望」「お住まいの区市町村」「既往歴」を追加聴取する。他の診療科では不要。
4. **救急科（×）**: AI電話対応不可。診療科ヒアリングシートで「×」設定。患者が救急科を発話した場合は「救急科はAI電話では対応していません。代表電話へおかけなおしください」→ 終話②相当の処理が必要。転送先番号は未記載 → BLOCKER。
5. **FAQ ブロック**: ヒアリングシートに約23件のFAQ登録済み（診察券紛失・紹介状・最寄駅・駐車場・送迎バス・診療時間等）。generate_by_OpenAI の FAQ 照合ブロックで実装。
6. **終話の3パターン**:
   - 終話①: 「受付のみ、確定ではない、折り返し電話あり」→ 予約新規受付完了
   - 終話②: 「お手続きを終了」→ 取りやめ・対象外時
   - 終話③: 「受付のみ、確定ではない、折り返し電話 or SMS（翌営業日以降になる場合あり）」→ 産婦人科妊婦初診など
7. **追加の質問**: 終話前に「その他、ご不明な点がございましたらお話しください。ない場合は1を押してください」→ 追加発話があれば FAQ 照合 or 用件失敗時アナウンス → 終話へ。
8. **連絡先電話番号の2分岐**: 通知あり→「今おかけいただいている番号でよろしいですか（1/違う）」、非通知・固定電話→「市外局番からお話しください、ダイヤルプッシュでも可」。Phone サブフローの通常パターン。

### director が見落としやすいポイント

- **稼働時間が「検討中」**: ヒアリングシートの運用ヒアリングシート（sheet5）で稼働曜日・時間がすべて「検討中」。`call_reception_hours` は BLOCKER として確認レポートへ。
- **スライド10～12（アナウンス一覧1/2/2）が空**: PPTX→MD変換でスライド10～12の表が未変換。TTS 文言の詳細はスライド14・15に記載あり（抽出済み）が、未収録のアナウンスがある可能性。確認レポートで「アナウンス一覧スライドの表が変換できていないため、設計書に記載のないTTS文言は追加確認が必要」と明記すること。
- **スライド8（修正依頼内容）が空白**: 0７/07 デモ時点の修正内容が表で提供されているが変換不可。「スライド8の修正内容は設計書確認後に人間が補完」と BLOCKER 化。
- **産婦人科の「区の健診・予防接種（2）」**: 予約の種類=2 のときのフロー（用件の確認→何を聴取するか）がスライドに明示されていない。director 判断で「用件の確認→終話②（AI非対応）」か BLOCKER 化。
- **診察券番号の聴取タイミング**: スライド9のフロー図では「診察券番号」の位置が用件確認後に来ている。PatientName（氏名）の前か後かを元資料フロー図（スライド9）を見て確定すること（変換後テキストには順序不明）。
- **用件失敗時の TTS**: 「お問い合わせ内容をお話しください」→ これは DTMF 失敗時の自由発話フォールバックと解釈できるが、FAQ へルーティングするか終話②へ行くかが不明 → director 判断 or BLOCKER 化。
- **年齢聴取**: スライド14に「患者さんの年齢をお話しください」が記載されているが、スライド9のフロー図には「年齢」ブロックが見当たらない。最終フロー図（スライド9修正後）での位置を確認すること。PPTX 表変換の欠落可能性あり → BLOCKER 化推奨。

### 移管方針
- Gen1「箱」構造 → Gen3 scenario_flow 9 ブロック型へ変換。主な対応:
  - `API_DRJOY_CONTEXT_SETTINGS` / `API_DRJOY` / `挨拶` → opening + 冒頭_アナウンス（2 ブロック 1 セット）。冒頭文言「お電話ありがとうございます。飯塚病院 予防医学センター専用の、AI電話です。これより、お話いただいた内容は、当院の個人情報保護方針に基づき適切に管理致します。」
  - `用件.yaml` → hearing（generate_by_OpenAI、4 分岐: 新規 / 変更 / キャンセル / 問い合わせ）。直後 echo_back 4 本 → Pattern A。DTMF 1〜4 並列対応（max_dtmf_length: 1）
  - `企業分岐.yaml` → 新規ルートのみ通る Yes/No hearing（「ご連絡をいただいている方は健康診断を受診されるご本人様でしょうか？」）。Yes（患者）→ 氏名へ、No（企業）→ 「企業健診のご担当者さまですね」announcement → そのまま氏名へ合流
  - `種類.yaml` → 新規ルート専用 hearing 3 分岐（人間ドック / 協会けんぽ / その他の健診）+ 各 echo_back
  - `胃検査希望有無.yaml` → 新規ルート Yes/No hearing。Yes → `胃検査内容.yaml`（胃透視 / 胃カメラ 2 値 + 各復唱）→ 合流、No → 直接合流
  - `予約日.yaml` → 変更/キャンセル/問い合わせ ルートで使用。datetime hearing + addCurrentDate: Yes scaffold 自動付与
  - `変更内容.yaml` → 変更ルート hearing 2 分岐（受診内容 / 日程変更）+ 各 echo_back
  - 氏名 / 生年月日 / 連絡先（電話番号区切り 同梱）→ subflow ブロック Gen3 標準 JSON 完コピ。氏名は患者氏名 1 枠 PatientName に充当
  - `問合せ内容.yaml`（自由聴取）/ `ラスト質問.yaml` → 自由文 hearing（output_format: text）として配置
  - `携帯電話番号チェック` → 連絡先 subflow 戻り値 phonetype を context_match_router で分岐
  - `終話` → 「お電話ありがとうございました。それでは失礼いたします。」共通文言 → END_切断
- 用件別の合流先:
  - **新規**: 種類 → 問合せ内容 → 胃検査希望有無 → （Yes時 胃検査内容）→ ラスト質問 → 完了フラグ → 企業分岐 → 氏名 → 生年月日 → 連絡先 → 締めあいさつ
  - **変更**: 予約日 → 変更内容 → 問合せ内容 → ラスト質問 → 完了フラグ → 氏名 → 生年月日 → 連絡先 → 締めあいさつ
  - **キャンセル**: 予約日 → ラスト質問 → 完了フラグ → 終話（**個人情報聴取スキップ**、main.yaml `キャンセル` 箱は `終話` 直結）
  - **問い合わせ**: 予約日 → 問合せ内容 → ラスト質問 → 完了フラグ → 氏名 → 生年月日 → 連絡先 → 締めあいさつ

### 施設固有の特殊ルール
- **受付時間**: main.yaml 内に時間外カット箱は存在しない → director 判断。代表番号や転送案内 TTS が一切無いため、acceptance_times の時間外用 termination は「後ほど担当者よりご連絡いたします」型で統一（番号案内不要）
- **転送無し / 代表番号案内も無い**: Brekeke Blind Transfer も代表番号読み上げも無し。**全ルート最終段でショートメール送信フラグ立て + 折返し約束で完結**（main.yaml `締めあいさつ` の「お電話終了後に送信されるショートメールから内容確認と修正ができます」が要旨）
- **企業分岐は新規ルート後段 Yes/No**: 用件分類で「新規」を選んだ後、種類 → 問合せ内容 → 胃検査ルート → ラスト質問 → 完了フラグ後の合流地点で「健康診断を受診されるご本人様でしょうか？」を確認 → No なら「企業健診のご担当者さまですね」announcement のみ流して通常の氏名 / 生年月日 / 連絡先聴取に合流（個別フロー分岐は無し、ラベル付けのみ）
- **「種類」3 分岐**: 人間ドック / 協会けんぽの健診 / その他の健診。DTMF 1〜3 併用。**新規ルートのみ通過**
- **胃検査ルート 2 段**: 「胃検査のご希望はありますか？」Yes/No → Yes なら「胃透視検査 / 胃カメラ検査」2 値で更に確認。新規ルート専用
- **DTMF 1〜4 / 1〜3 / 1〜2 が箇所ごとに揃う**: 用件 4 分岐、種類 3 分岐、変更内容 2 分岐、胃検査内容 2 分岐 すべてに DTMF 並列入力ロジックあり
- **「もしもし」検知**: 用件聞き取り入口で `countもしもし` 箱で計数。1 回目「はい、聞こえております」、2 回目以上「はい、『もしもし』と聞こえております」→ 用件再聴取
- **携帯チェックで締め文言切替**: 倉敷・蘇生会と同パターン。携帯あり → 電話 or ショートメール、携帯なし → 電話のみ。ただし飯塚は「ショートメールから内容確認と修正ができる」自社ポータル誘導文言が独自 → 文言短縮しないこと
- **キャンセルは個人情報聴取スキップ**: 蘇生会と同等。main.yaml `キャンセル` → `終話` 直結
- **生年月日の和暦範囲チェック**（大正〜令和）: `予約日.yaml` 内 `生年月日の判定` 箱の流用残骸は予約日聴取では使われない誤参照 → そのまま無視
- **診療科エンティティは予防医学センター用には未使用**: main.yaml「診療科」カスタムエンティティに 47 科目羅列があるが、予防医学センターシナリオ内で診療科聴取 hearing は無い → Gen3 では取り込まない

### director が見落としやすいポイント
- 冒頭の `API_DRJOY_CONTEXT_SETTINGS` と `API_DRJOY` 系 1 箱は Gen3 では opening 冒頭の標準処理に吸収 → 明示ブロック化不要
- 「次へ」/「次へ_1〜_10」は短い繋ぎ TTS → 次段ブロック先頭 TTS にマージ（director デフォルト）
- セリフ none 振り分けのみの箱（用件の分岐 / _1〜_3、完了フラグ、携帯電話番号チェック、タイトル）は独立ブロック不要
- `タイトル` 箱（「上手く聞き取りができませんでした。それでは、」→ `問合せ内容` 接続）は **聴取失敗フォールバックの中継**
- `問い合わせ` / `新規` / `変更` / `キャンセル` 各箱（用件確定後の announcement 文言）は **echo_back の代替告知** → Gen3 では echo_back 直後の announcement として残す
- `胃検査希望有無.yaml` の `企業か患者聞き取り` 箱（最初の箱名が誤り、実態は胃検査希望聴取）は Gen1 の流用名前バグ
- `胃検査内容.yaml` 「セットした変更内容の分岐」「変更内容のセット」「コースの復唱（健診内容）」「用件の復唱（日程変更）」等の箱名は **変更内容.yaml からの流用残骸**（実態は胃透視 / 胃カメラ確認）→ Gen3 では胃検査用に正しい名前へリネーム
- `ラスト質問` サブフロー指定があるが flow_dir に独立 yaml ファイルが無い → main.yaml 内インライン定義 or 流用想定 → director 判断で問合せ内容と同形式の自由文 hearing を新規定義
- `問合せ内容` も同様に独立 yaml 無し → main.yaml では呼び出しのみ → director 判断で自由文 hearing を新規定義
- 「<? $予約日の読み ?>」「<? $連絡先電話番号（ハイフン区切あり） ?>」「<? $入電番号（ハイフン区切りあり） ?>」等の Gen1 動的タグは Gen3 では `<% var %>` 構文へ
- **モジュール名・termination 名は ASCII または安全な日本語のみ**。丸数字（①②③…⑩）等の環境依存文字は禁止
- **ContextMatchRouter.reference_module は subflow を参照する場合 `jump_{step名}`**（scaffold が付ける prefix）を使う
- **CMR conditions は必ず最後に `match: "other"` + next 明示**
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止

### 移管方針
- 冒頭は opening + 冒頭_アナウンス の2ブロック1セット。冒頭アナウンス文言は元資料原文「お電話ありがとうございます。筑波総合クリニックの、予約専用、AI電話です。」をそのまま使用。
- 非通知着信は **incoming-classifier の withheld 分岐** → `END_非通知案内`（status=0 固定、Gen2 の冒頭アナウンス+非通知文言を結合した原文のまま）で即終話。regular_next に合流しない。
- 非通知以外は **5段階基本フロー**（①患者名 → ②生年月日 → ③診察券番号 → ④連絡先電話番号 → ⑤用件）を共通の hearing チェーンで固定。用件ブロック（DTMF 4択）で classification を確定し、予約/変更/キャンセル/確認の4サブルートへ分岐する **Pattern A（OpenAI直後分岐）**。
- 予約ルート内は **受診歴 Yes/No**（Pattern A）で「新規」「再診」に再分岐：新規→紹介状→（紹介元）→診療科 ／ 再診→症状→同じ症状なら診療科、違う症状なら紹介状→（紹介元）→診療科（再診でも紹介状は必須）。
- 氏名／生年月日／診察券番号／連絡先電話番号は **サブフロー経由（Jump to Flow）**。DTMF 付き聴取・桁数再ヒアリング・復唱（`<speak type="telephone">` / `<speak type="digits">`）・カタカナエラー再聴取などの定型部品はサブフロー側に閉じ込める。メインにインライン展開しない。
- 連絡先電話番号サブフロー内で **incoming_phone_number 型の phone_type 分岐**：mobile（070/080/090/060 で始まる11桁）→「今おかけの番号でよいか」Yes/No 確認ブロック、other（固定/IP/anonymous）→ 直接 DTMF 手動聴取へ。mobile No 応答も手動聴取に合流（Pattern B: ContextMatchRouter 合流）。
- 診療科で「一般内科」「内科」「何か」「なにか」判定 → `END_一般内科案内`（代表電話 029-877-1221 を3回繰り返す原文維持、status=2）。ただし **キャンセルルートでは一般内科判定を行わない**（分岐禁止）。
- 終話パターンは `END_非通知案内`（status=0）／ `END_一般内科案内`（checkpoint=1, status=2, smsFlag=0, endpoint=電話案内）／ `END_エラー`（代表案内、リトライ超過用）／ `END_予約完了`（予約・変更・確認: status=1, smsFlag=1）／ `END_キャンセル完了`（checkpoint=7, status=1, smsFlag=2）の5系統を termination_patterns に定義。
- TTS は全て `{tts_g:...}`（小文字）。`<speak type="telephone">` `<speak type="digits">` `<dtmf/>` `<dtmf digit="1"/>` タグは元資料どおり該当ブロックの TTS にそのまま保持（SSML 禁止ルールの例外として原文維持。director 判断）。
- RAG / FAQ：元資料に RAG 利用の記載なし → **RAG 設置なし**。FAQ 相当の自由質問は `END_エラー`（代表電話案内の固定文言）で受ける。

### 施設固有の特殊ルール
- **数字誤認識救済**：「for/C/Sea/イオン」→4、「Go/号」→5、「新居」→2、「Q」→9、「まる/おー/れい」→0。用件 DTMF・電話番号・診察券番号・生年月日の prompt に共通変換リスト（1-8）を必ず列挙（prompter 担当）。
- **用件 DTMF は数字キーワード単独でも確定**：入電者が「1」「イオン」だけ発話しても classification を確定し、再質問禁止。
- **「一般内科」「内科」＝代表電話案内（status=2）**：**予約・変更のみ**。キャンセルと確認では絶対に行わない（用件 classification で事前分岐してから診療科判定を配置すること）。
- **診療科リスト揺れ**：display_type=DEPARTMENT の正規化に委任。prompt では function enum をそのまま列挙。下位キーワード（「のうげ」「じびか」「ヒビインコウガ」等）は DEPARTMENT レイヤで吸収されるため scenario_flow には持ち込まない。
- **呼吸器は内外科問い返し必須**：「呼吸器」単独→「内科でしょうか？外科でしょうか？」と確認。症状単独発話（例：胸痛）からの診療科推定は禁止、症状は無視してそのまま診療科を聞き直す。
- **紹介状は再診でも必須**（前回と違う症状の場合）。classification="再診" でも紹介状確認をスキップする設計にしないこと。
- **新規では診療科1件のみ**：clinicalDepartment2 は新規では常に空文字。変更／キャンセルのみ「2つ目以降の診療科」聴取あり。
- **患者名カタカナ判定**：苗字のみ判定時に限り「下のお名前もお伺いしてもよろしいですか？」で再質問を許可（他項目の再質問は禁止）。PatientName サブフロー側で対応。
- **checkpoint ルール**：用件確定=0、予約/変更/キャンセル/確認の補足完了=1、キャンセル理由受領=7、一般内科案内=1（status=2）。checkpoint は AI 非発話・非出力の内部フラグ。
- **smsFlag ルール**：status=1 かつ classification∈{予約・変更・確認}→smsFlag=1、checkpoint=7（キャンセル完了）→smsFlag=2、それ以外（status=2 代表案内・status=0 非通知エラー等）→smsFlag=0。
- **改訂履歴（2026/04/01）**：①「ありがとうございます」発話の絶対禁止（`END_一般内科案内` や `END_エラー` など固定文言内の「お電話ありがとうございました」は例外として原文維持）、②予約（新規・再診）フローで「予約希望日」聴取禁止（予約希望日は変更ルート専用）。prompter の全 generate_by_OpenAI に織り込み。
- **改訂履歴（2026/03/31）**：誤認識救済に「新居」（=2）「イオン」（=4）追加、事前発話でのフロースキップ許可＋順序維持（下記参照）、紹介状・紹介元の聴取漏れ防止強化。

### director が見落としやすいポイント
- 『**事前取得によるフロースキップは許可するが、ヒアリング順序（1.患者名→2.生年月日→3.診察券番号→4.連絡先番号→5.用件）は絶対に崩さず、未取得項目があれば必ずこの順番通りに質問する**』『**必須項目が【全て取得】できるまでは、完了フローや終話に勝手に進んではいけない**』。5段階フロー全 hearing と generate_by_OpenAI の prompt に Chain of Thought 形式で注記必須。
- 『**患者名以外は「○○ですね？」と必ず復唱する**』『**患者名の復唱は禁止**』。PatientName サブフローの save2db は echo_back=false、生年月日・診察券番号・電話番号は echo_back=true で該当ブロックに復唱 TTS を生成。
- 『**10月以降で1月から9月の日付を言われたときは、翌年にする**』『**過去日付は受付不可、未来日のみ**』の年度補完ルール。予約日関連の正規化 prompt（reservationDate）に必須。
- 『**予約日（reservationDate）は日付形式、予約希望日（DesiredreservationDate）はフリーワード**』。変更ルートで両方別項目として聴取、格納フォーマットが異なる（display_type=DATE vs TEXT）。
- 『**入電者からの明示的な指示がない限り、AI側から勝手に dialogue_completed を出力してはいけない**』『**終話文を発話する前に dialogue_completed を呼び出してはならない**』。save2db の発火タイミングは終話ブロック内で TTS 後に固定。
- 『**内部管理項目（source／reason／smsFlag／status／checkpoint／clinicalDepartment2／phone_type／incoming_phone_number）は AI 出力に含めない**』。generate_by_OpenAI の output schema から除外、または空文字デフォルト。
- 『**5段階フロー各項目のガン無視ルール**（患者名聴取中に「キャンセルで」等と言われても用件として記録しない）』→ ただし 2026/04/01 改訂で「事前取得は許可（フロースキップ可）」に緩和。両者の整合は「**現在質問中の項目が取得できるまでは他項目のみの発話は無視、複合発話なら両方記録しフロースキップ可**」と解釈（director 判断）。
- 患者名聴取の **カタカナ正規表現 `^[ァ-ヶー]+$` による苗字判定**。カタカナ未判定→カタカナエラー再聴取。PatientName サブフロー側で対応。
- 連絡先電話番号は mobile 確認ブロックで「はい」→ `additionalPhoneNumber: {Incoming_phone_number}` に自動代入。`incoming_phone_number` は内部参照用で dialogue_completed には出力しない。
- `<dtmf/>` タグは削除禁止（別システムで音声化される）。メタ情報ではないため TTS 原文にそのまま残す。

### 移管方針
- 冒頭は opening + 冒頭_アナウンス の2ブロック1セット。冒頭アナウンス直後は入力待ち無しで「当日翌日確認」に接続する。
- 「当日翌日確認」は Yes/No 分岐（Pattern A: OpenAI直後分岐）。肯定→代表案内（0476-22-2311）で終話（checkpoint=2 / smsFlag=0 / endpoint=電話案内）、否定→次へ。
- 連絡先電話番号は Date of Call Classifier ではなく **incoming_phone_number 型による3分岐**（mobile=070/080/090/050、landline、anonymous）。mobile のみ「今おかけいただいている番号でよろしいですか？」の Yes/No 確認ブロックを挟み、No または landline/anonymous は DTMF 付き再ヒアリングブロックに合流（Pattern B: 後段ContextMatchRouter 的な合流）。
- 氏名／DOB／電話番号／診察券番号は **サブフロー経由**（Jump to Flow）。DTMF 付き聴取・桁数再ヒアリング・復唱確認などの定型部品はサブフロー側に閉じ込める。メインにはインライン展開しない。
- 用件①（診療 / 健診）で subject を保存、用件②（予約・変更・キャンセル・確認）で分岐。subject × classification の組み合わせ 2×4 = 8 ルートになるため、用件①で診療サブフロー／健診サブフローを Jump to Flow で呼び分ける構成が素直（director 判断）。
- 終話パターンは **予約受付完了**（予約/変更/確認系: smsFlag=1）／**キャンセル受付完了**（smsFlag=0、文言は本文末尾の「必要がある場合のみ折り返し」）／**代表案内**（当日翌日・救急科: checkpoint=2, smsFlag=0）の3系統を termination_patterns に定義。
- TTS は全て `{tts_g:...}`（小文字）で発話。`<speak type="telephone">` `<speak type="digits">` `<dtmf />` `<dtmf digit="1" />` タグは**電話番号復唱サブフロー・診察券サブフロー・用件ブロック**等のプロパティにそのまま保持（SSML 禁止ルールの例外として元資料どおり残す。director 判断）。
- RAG は元資料に記載なし → 不要。FAQ 的な問い合わせは「その他問い合わせ」→ question / details に自由文格納で受ける。

### 施設固有の特殊ルール
- **「肺→はい」救済**：Yes/No 判定で「肺」「いいです」は肯定として扱う（『以下の単語は「肯定」と判断します：「いいです」「肺」』）。Yes/No 系 generate_by_OpenAI の prompt に注記必須。
- **数字 4/5 の誤認識救済**：「"for"や"C"」→ 4、「"号"や"go"」→ 5。診察券番号・電話番号ブロックの prompt 側で対処（prompter 担当）。
- **当日翌日予約は AI 不可**：最初の質問で肯定→代表案内②で終話。**連絡先番号・氏名・DOB 等を一切聴取せずに終話する**点に注意（通常フローで DOB を必ず取る設計にしないこと）。
- **診療サブフロー・予約ルートで『救急』が発話されたら代表案内**：『救急科の場合：代表案内②で終了』。診療科①ブロックの分岐で「救急/きゅうきゅう/きゅうげ」含有→ END_代表案内_救急。
- **subject enum 値**："診療" / "健診"（ただし健診側の発話キーワードは「健康診断」）。内部格納値と発話キーワードが一致しないので prompt に両方明記。
- **status ルール**：予約日か予約希望日まで確認完了で status=1、代表案内時 status=2、それ以外 0。checkpoint の値をそのまま status に写す形で director は指示。smsFlag は status=1→1、それ以外→0（キャンセル時も 1 と本文下部にあるが、yaml の『END_キャンセル受付完了』の案内文言は「必要がある場合のみ折り返し」で smsFlag=0 相当。director 判断で揃える。迷えば smsFlag=1 に寄せる）。
- **診療科リストの揺れ**：function enum と yaml diff（例: "感染症科"の下位キーワードに「感染科」等）。診療科マッチは **display_type=DEPARTMENT の正規化に委任**し、prompt では function enum をそのまま列挙。
- **健診「希望コース」の4択**：人間ドック / 協会けんぽ / 事業所健診・法定健診 / 特定健診。course は自由文格納（enum なし）だが、発話は4択 DTMF（dtmf digit="1"）で聴取する。
- **改訂履歴（2026/04/17）**：冒頭アナウンスの修正のみ。冒頭アナウンス文言は元資料本文の原文をそのまま使用すること（生成時に要約しない）。

### director が見落としやすいポイント
- 『**予約日と予約希望日は別なので、予約日のヒアリングがなかった場合に予約希望日の中身を予約日に入れてはいけない**』。診療_変更／キャンセル系では **両方を別項目として聴取**。reservationDate は日付正規化、DesiredreservationDate はフリーワードのまま格納（format違い）。
- 『10月以降で1月から9月の日付を言われたときは、翌年にする』の年度判定ルール。予約日関連の正規化プロンプトに必須。
- 『**入電者が一度に複数の必要情報を発話した場合、すでに伝えられた内容については絶対に復唱・確認を行わず、不足している項目のみを一つずつ質問する**』が共通ルールとして随所に繰り返し記載。generate_by_OpenAI の全プロンプトに反映。
- 『**checkpoint / smsFlag は内部フラグ**。AI は絶対に発話しない』。TTS 文言生成時の混入防止。
- 健診_変更の **変更内容①**「予約日の変更でしょうか？」で Yes/No 分岐：肯定→予約希望日、否定→変更内容②（自由文）。診療側と分岐構造が異なる点に注意。
- 電話番号聴取ブロックは『10桁または11桁以外は再ヒアリング』『11桁時は先頭3桁が 090/080/070/050 以外なら再ヒアリング』。聴取サブフロー側でリトライ設定（Max Retry 2 等）を検討。
- 患者名は**苗字のみ判定時のみ**下の名前再質問を許可（他項目の再質問は一切禁止）。氏名サブフロー側で対応。

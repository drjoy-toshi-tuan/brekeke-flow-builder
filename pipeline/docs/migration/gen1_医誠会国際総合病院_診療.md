### 移管方針
- Gen1「箱」構造 → Gen3 9 ブロック型に変換。`挨拶` → opening + 冒頭_アナウンス（2 ブロック 1 セット必須）。挨拶 TTS は「ここからは、予約専用 AI 電話が応対いたします。担当者より折り返しご連絡いたしますので、ガイダンスに従ってお話ください」。
- `入電時間判別` サブシナリオ → date_of_call_classifier ブロック（addCurrentDate: Yes 自動付与対象）。時間外（受付時間外）は `時間外アナウンス（Flag3）`「受付時間は土日祝日・年末年始を除く月曜から金曜の 8 時 30 分から 17 時」案内 → termination。
- `用件（非通知判断追加）` サブフローは **非通知判別 → はじめに or 直接用件確認** の入口分岐を内蔵 → Gen3 では入電番号取得済 / 未取得を context として保持し、後段の個人情報聴取順序に反映させる（メインの `入電番号取得確認 / _1 / _2 / _3` で再判定する設計）。
- 用件は **5 分類**: `診察予約（再診/新規統合）` `予約変更` `予約キャンセル` `その他（=確認内容の聞き取り）`。診察予約（=再診/新規）に進んだ場合、サブフロー内で「当院を受診されたことはございますか？」 → 肯定=再診セット / 否定=新規セット で context 確定。Pattern A（用件 OpenAI 直後 4 分岐 + 復唱）+ Pattern B（再診/新規再判定は後段 CMR）の組合せ。
- メイン分岐: **その他用件 → 確認内容の聞き取りサブフロー**（変更/キャンセル理由を自由テキスト聴取）→ `END_理由聞き取れずCP` で離脱、それ以外は `END_新規予約・予約変更・キャンセル・定期受診へ` 経由で診療科聴取に合流。
- 個人情報聴取（番号取得時）: 診療科 → 予約日 → 確認内容 → 患者氏名 → 生年月日 → 連絡先電話番号確認（取得済ならスキップ、未取得は連絡先電話番号聞き取りサブフロー）。
- 個人情報聴取（非通知時）: 用件確定後 →（非通知ルート）患者氏名 → 連絡先電話番号 → 診察券番号 → 生年月日 と順序が違う点に注意。

### 施設固有の特殊ルール
- **代表電話 0570-099-166**: 時間外/聞き取り失敗時の案内で「番号は、0570-099-166 です」を **3 回繰り返し**（`番号案内 → くりかえし → END_回数カウント` ループ、3 回到達で終話）。
- **入電番号取得済 / 未取得（非通知）の二系統運用**: 入電番号があれば「担当者より折り返しご連絡後、確定となります」案内 + 通常順序、非通知なら「折り返しの連絡先をお話ください」追加案内 + 連絡先聴取（フリー）サブフロー経由の順序で個人情報を聴取。Gen3 では context_match_router で 4 系統（時間内番号あり/時間内非通知/時間外番号あり/時間外非通知）を集約。
- **時間内番号あり**: 「順番に折り返し対応いたしますのでお待ちください」（連絡先聴取スキップ）。
- **時間外番号あり**: 「外来時間終了のため翌診療日に折り返し対応いたします」（連絡先聴取スキップ）。
- **時間内/時間外非通知**: 「折り返しの連絡先をお話ください」 → `連絡先聴取（フリー）` サブフロー必須。
- **「もしもし」エンティティはなし**（北里大・多摩総合と異なる）。OpenAI プロンプトで「もしもし」誤検知排除を入れる必要は薄いが、`用件（非通知判断追加）.yaml` の `もしもし` カスタム entity は空配列なので無視 OK。
- **3 回失敗 / 聞き取れず**: `失敗あつまれ → 2 回失敗：終話へ`「大変申し訳ございません。うまく聞き取ることができませんでした」 → 時間判別再実行 → 入電番号取得確認_2/_3 → 番号案内ルートに合流。氏名/診療科/生年月日/連絡先のいずれの失敗もここに集約。

### director が見落としやすいポイント
- 最初の `API_DRJOY_CONTEXT_SETTINGS` → `API_DRJOY` → `入電時間判断へ` → `入電時間判別` → `挨拶` の 5 箱は Gen3 では opening 冒頭処理に吸収。明示ブロック化不要。
- 「セリフ: none」かつ振り分けのみの「箱」（`次へ` `次へ_1〜_3` `用件の確認` `用件を判断` `完了フラグ` `現在の時間取得` `現在の時間取得_1` `時間判別` `時間判別_1` `入電番号取得確認 / _1 / _2 / _3` `失敗あつまれ` `氏名聞き取れずCP` `診療科聞き取れず` `生年月日聞き取れず` `連絡先番号聞き取れず` `年末年始アナウンス終話へ` 等）→ Gen3 で独立ブロック不要、前段 next 書き換え or context_match_router に吸収。
- `次へ`（「では次に、」）/ `次へ_1`（「かしこまりました。」）/ `次へ_2`（「それでは、」）/ `次へ_3`（「続いて、」）/ `生年月日へ`（「それでは、」）/ `用件聴取へ`（「ありがとうございます。それでは、」）/ `非通知：患者名聴取`（「まず初めに、」）/ `連絡先電話番号聞き取りへ`（「最後に、お電話番号をお伺いします」）等の **短い繋ぎ TTS は次段 hearing/subflow ブロックの導入文言にマージ**。
- `用件を判断` の 5 分岐（再診/変更/キャンセル/新規/その他）→ 「再診/変更/キャンセル/新規」は同一 `END_新規予約・予約変更・キャンセル・定期受診へ` に集約、「その他」のみ `確認内容の聞き取りへ` に分岐。Gen3 では用件 OpenAI 出力 enum を 5 値（診察予約/変更/キャンセル/その他=確認/聞き取れず）にして CMR 1 個で集約。
- `END_新規予約・予約変更・キャンセル・定期受診へ` は終了箱だが「セリフ: それでは、」を持ち、内部から診療科サブフローに繋がる扱い → Gen3 では announcement ブロック「それでは、」+ subflow（診療科）の 2 段に展開。
- `用件（非通知判断追加）.yaml` 内の `セットした用件の分岐_1`（「当院を受診されたことはございますか？」）は **診察予約用件のみ通る分岐**（再診/新規セット）。Gen3 では subflow 内で完結させ、メインに通院歴 hearing を出さない（director 判断、PatternName 1 枠と混同しない）。
- 患者氏名は PatientName サブフロー 1 枠（patient 1 名のみ、入電者=受診者前提）。
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止。Gen1 動的タグ（`<? $用件 ?>` 等）は Gen3 では `<% var %>` に変換。
- ContextMatchRouter.reference_module は subflow 参照時 `jump_{step名}`（scaffold 付与 prefix）を使う。CMR conditions は最後に `match: "other"` + next 明示。
- モジュール名・termination 名は ASCII または安全な日本語のみ。丸数字（①②③⑩等）禁止。
- entity_classification.md / entity_stt_patch.json / entity_openai_synonyms.md にエンティティは自動分類済み。director は転記不要、scaffold/properties が自動取り込み。

### 移管方針

- **入電者は医療機関スタッフ側**（患者側ではない）。用件分類(classification)は通常の診療シナリオと異なり `初診/再診/変更/キャンセル/問い合わせ/無回答` と定義されているが、実質は「自由用件ヒアリング + 担当者/施設名記録」が主体。OpenAI による意図分類分岐は最小限にとどめ、Pattern A 分岐は「至急確認 Yes/No」の1回のみに絞る。
- **Date of Call Classifier は不要**。営業時間外アナウンスは customer_doc に明示的記述なし。ただし「至急の場合の代表電話案内」で営業時間（土日祝除く月〜金 8:30〜17:00）を読み上げる TTS が存在する点に注意（分岐条件ではなく文言）。
- **サブフロー化**: 電話番号確認→直接聴取→復唱の流れは標準の Phone サブフロー（Jump to Flow）に集約。患者名・施設名は hearing ブロックでインライン聴取（サブフロー化不要、DOB/診察券の聴取はシナリオ上発生しない）。
- **聴取順序が非標準**: 用件 → 病棟・担当者 → 患者名 → 施設名 → 連絡先 → 確認事項。患者名より先に病棟・担当者（MSW 名）を聞く点に注意。
- **echo_back なし**（電話番号のみ復唱）。`{tts_g:...}` 小文字・SSML 禁止に合わせ、元資料の `<speak type="telephone">` は generate_by_OpenAI 出力の電話番号読み上げ用プロンプトへ読み替える（TTS 本文には SSML を残さない）。

### 施設固有の特殊ルール

- **絶対的な聞き直し禁止**: 一度聴取した内容は不明確でも再質問・復唱しない。hearing ブロックの Retry は「無言時のみ再プロンプト」に限定し、ContextMatch で再確認ループを作らない。
- **複数情報の一括発話**: 初回の用件発話で `inquiry / patientName2 / patientName / company_name` が同時に埋まる可能性あり。OpenAI 抽出後の不足項目のみスキップではなく「未 fill の項目だけ順送りに質問」する挙動を prompter に明示。
- **電話番号のみ復唱許可**: `<speak type="telephone" breakc="300ms">{}</speak>でよろしいでしょうか？` は電話番号専用復唱フォーマット。Gen3 では TTS プレーンテキスト＋間ポーズ指定は tts_g のテンプレに準拠（SSML 非表出）。
- **担当者/病棟名の格納先が `patientName2`**（通常の「担当者」用フィールドではなく患者名フィールドの 2 番目を流用）。scaffold YAML の保存先マッピングで patientName と patientName2 の取り違え注意。
- **用件種類の想定ラベル**: 「患者・家族・医療機関・福祉施設・行政」（customer_doc 冒頭フロー例記載）。ただし function の enum は診療系のものが流用されている。**enum をそのまま採用**し、分類分岐は組まない方針。
- **無言エラー共通箱**: 「大変申し訳ございません。ご回答が確認できませんでした。再度お伺いいたします。」を Retry 時の標準 TTS として全 hearing ブロックで統一。
- **status / smsFlag のルールが同一**: 必須全埋→`1`、代表案内→`2`、それ以外→`0`。scaffold script ブロックで両方同時セット。
- **endpoint enum に「時間外」あり**だが時間外分岐自体が customer_doc に無い。値としては未使用で保持（将来拡張用）。

### director が見落としやすいポイント

- **改訂履歴**: customer_doc に明示的な改訂履歴セクションは無し。ただし冒頭の「復唱禁止」指示が後半の電話番号復唱許可と一見矛盾するため、**電話番号のみ例外**であることを明文化すること。
- **用件分類を OpenAI に投げない**: inquiry は自由テキスト格納（display_type: TEXT）。A 系テンプレで分類しようとしない。
- **至急 Yes 分岐の終話文言が極端に長い**（代表電話番号を3回繰り返す）。TTS を1発話にまとめず、読点で自然な間を入れる設計が必要。
- **「施設名・所属・お名前をお話しください」は company_name 1 フィールドに格納**。氏名を PhoneName 等に分けない（customer_doc 指定どおり company_name 一括）。
- **確認項目聴取（reason）が「最後の問い合わせ」も兼ねる**二重用途。通話終盤の問い合わせ追加質問と同一フィールド。重複ブロックを作らないこと。
- **患者側シナリオ（健診・診療）の移管ノートをコピペしない**。診察券番号・生年月日・予約希望日などのヒアリングは本シナリオでは **発生しない**。function には定義されているが埋めない前提。

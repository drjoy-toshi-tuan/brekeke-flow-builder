### 移管方針
- 冒頭は opening + 冒頭_アナウンス の 2 ブロック 1 セット。冒頭アナウンス直後は入力待ち無しで「申し込み方法確認」に直結する（歯科口腔外科予約申し込み電話として応答）。
- **最上段に「申し込み方法確認」分岐**：1/ネット → SMS サブフロー（`additionalPhoneNumber` 聴取 → END_SMS送信案内で status=8 / smsFlag=3）、2/電話 → 通常の用件確認フロー。Pattern A（OpenAI 直後分岐）で素直に処理。再質問は **1 回のみ**、2 回目失敗で END_代表案内_申し込み方法（011-863-2105 を **3 回連呼**して終話、status=0）。
- 連絡先電話番号は Date of Call Classifier ではなく **incoming_phone_number 型による 3 分岐**（MOBILE / FIXED / OTHER）。MOBILE のみ「今おかけいただいている番号でよろしいですか？」Yes/No 確認ブロックを挟む。FIXED / OTHER は固定アナウンス違い（OTHER 系は「お電話番号が非通知のため」を冠する）。Pattern B 的な合流で復唱→終話判定へ集約。
- 氏名／DOB／電話番号／診察券番号は **サブフロー経由（Jump to Flow）** で固定。DTMF 付き聴取・復唱・桁数再ヒアリング・生年月日エラー文言などはサブフロー側に閉じる。メインにインライン展開しない。
- 用件 classification は **予約／変更／キャンセル／その他問い合わせの 4 分岐**。予約ルートは受診歴で「初診（紹介状有無→医師名 or 希望医師→患者名…）／再診（予約希望日→診察券番号→患者名…）」に枝分かれ。サブフロー化／1 枚フロー内分岐は director 判断可。
- 終話パターンは **4 用件 × FIXED/MOBILE の 8 END** ＋ **END_受付不可診療科**（status=2 / smsFlag=0、050-1726-5776 案内）＋ **END_SMS送信案内**（status=8 / smsFlag=3）＋ **END_代表案内_申し込み方法**（status=0）の計 11 系統を termination_patterns に定義。MOBILE 側は「ショートメッセージをお送りしますので必ずご確認ください」の追加文を含む。
- TTS は基本 `{tts_g:...}`（小文字）。ただし元資料は `<speak type="telephone" breakc="300ms">`、`<dtmf digit="...">`、`<dtmf2/>`、`<break time="...">`、`<sub alias="...">`、`<prosody rate="...">` を多用。**電話番号復唱／申し込み方法／生年月日エラー／受付不可診療科 等のプロパティは原文のままタグを保持する**（director 判断で SSML 禁止ルールの例外扱い）。
- RAG は元資料記載なし → 不要。FAQ 的問い合わせは「その他問い合わせ」ルートで `reason` / `details` に自由文格納。

### 施設固有の特殊ルール
- **受付不可診療科 → 代表案内（歯科口腔外科以外は全部 NG）**：総合内科／消化器外科／呼吸器外科／乳腺外科／消化器内科／腫瘍内科／泌尿器科／耳鼻咽喉科・頭頸部外科／形成外科／放射線診断科／IVR センター／放射線治療科／麻酔科／緩和ケア内科／心療内科・サイコオンコロジー外来／循環器内科／病理診断科／婦人科 のキーワードを **全ヒアリング項目で常時監視** し、検出したら即 END_受付不可診療科（status=2）。用件確認の yaml 例にも「総合内科」キーワード一致で該当 END へ飛ばす条件あり。director は generate_by_OpenAI プロンプトに反映。
- **status / smsFlag 値は特殊**：status は 0/1/2/8、smsFlag は 0/1/2/3。SMS 案内のみ status=8 / smsFlag=3、キャンセル終話のみ smsFlag=2、受付不可のみ status=2 / smsFlag=0 を保持。
- **数字 4/5 の誤認識救済**：「救急 / Go」→ 数字（連絡先電話番号聴取時）。診察券番号・電話番号ブロックの prompt 側で対処（prompter 担当）。
- **診察券番号 3 ケース分岐**：数字含む → そのまま格納／「わからない」 → `"わからない"` 格納／無関連 → 再聴取 1 回のみ、2 回目はどんな回答でも無言で格納して次へ（聞き直し・復唱・「〜を記録します」禁止）。
- **医師名ブロックの強制進行**：聞き返しがあってもそのまま `DoctorName` に格納、聞き直さない。希望医師は「なし」回答許可。
- **患者名**：苗字のみ判定時のみ再質問（`patientName_first` に下の名前を格納）、それ以外の名前再質問は禁止。必ずカタカナ変換して格納。
- **生年月日復唱は必須**：「yyyy 年 m 月 d 日でよろしいでしょうか」。否定時は専用エラー文言（`<sub alias="イチ キュウ ナナ ゼロ">…1970…<dtmf digit="8"/>`）で再聴取。予約日／予約希望日のエラー時にこの文言を流用禁止。
- **予約日 vs 予約希望日の格納形式差**：`reservationDate` = DATE 型（例 `2016-01-02 00:00`）、`Preferred_date` = TEXT 型（フリーワードそのまま）。director は変換しない。

### director が見落としやすいポイント
- 『**申し込み方法確認**』の再質問は **1 回だけ**。2 回目失敗で `END_代表案内_申し込み方法`（011-863-2105 を **3 回連呼**する特殊文言）へ。status=0 保持のまま終話。
- SMS 用フロー（ルート 1）では **MOBILE でも「今おかけいただいている番号で…」確認 Yes/No を挟んでから END_SMS送信案内** へ。否定時は固定の `SMS_連絡先電話番号_固定` ブロック（"URL を送付するために、ご連絡先の携帯番号をお伺いします…"）に合流。通常フローの連絡先聴取と **似て非なる**ので混同注意。
- 『**連絡先電話番号の確認を 3 回できなければ**「ご回答の確認ができませんでしたのでこちらからお電話失礼させていただきます」で終話。ただし anonymous / 空の場合は 10 桁 / 11 桁になるまで繰り返し』。サブフロー側のリトライ上限設計に反映（Max Retry 2 等）。
- 電話番号聴取は『10 桁または 11 桁以外は再ヒアリング』『11 桁時は先頭 3 桁が 090/080/070/050 以外なら再ヒアリング』『10 桁時は先頭 3 桁が 090/080/070/050 なら誤認識 → 再ヒアリング』（桁と prefix の両方向チェック）。
- 生年月日の年判定が特殊：`Year(NOW())-110` より古い年は無効、3 桁年（900-999 → 19XX）、2 桁年（00-07 / 08-31 / 32-63 / 64-99 で元号確認分岐）、百の位 0 → 千の位 2 補正、百の位 9 → 千の位 1 補正。DOB サブフローの正規化プロンプトに必須。
- `incoming_phone_number` は受信値そのまま保持、先頭 0 削除禁止、`+81` のみ `0` 変換、anonymous / unknown / 空文字列はそのまま保持。**桁数判定（ちょうど 10 桁／11 桁）を先に行ってから prefix 判定**（順序厳守）。
- 『**入電者が一度に複数の必要情報を発話した場合、すでに伝えられた内容については絶対に復唱・確認を行わず、不足している項目のみを一つずつ質問する**』が共通ルールとして繰り返し記載。generate_by_OpenAI の全プロンプトに反映。
- 『**status / smsFlag / checkout / endpoint は内部フラグ**、AI は絶対に発話しない』。TTS 文言生成時の混入防止。
- FAQ 検索ルート：回答有 →「ほかに何かお問い合わせはございませんか？」で二段構え／回答無 →「問い合わせ内容を承りました。担当からの折り返しの際に回答をさせていただきます」で即次項目へ（追加質問確認しない）。通話終了直前の `finalQuestion` ヒアリングも **1 回だけ**、二度聞き禁止。
- **改訂履歴（2026-03-24 / 2026-03-26）**：3/24 で申し込み方法の電話番号聴取項目追加、3/26 で診察券番号の制御追加（3 ケース分岐）。いずれも本文に反映済みなので、文言・ロジックを原文尊重で使用。
- 変更／キャンセルは「予約日（現在の予約日 = `reservationDate`）」の聴取がある（変更は「予約希望日」も別途聴取、キャンセルは「キャンセル理由 = `reason`」）。**予約希望日と予約日を混同して同じ箱に入れない**。

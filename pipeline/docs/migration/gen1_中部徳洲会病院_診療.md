### 移管方針
- Gen1「箱」構造 → Gen3 9 ブロック型に変換。`挨拶` → opening + 冒頭_アナウンス（2 ブロック 1 セット必須）。挨拶 TTS は「お電話ありがとうございます。中部徳洲会病院の予約専用、AI 電話です」。
- `時間判断` サブシナリオ → date_of_call_classifier ブロック（addCurrentDate: Yes 自動付与対象）。受付時間外は `受付時間外の案内CP`「受付時間は土曜日曜祝日を除く、平日の 13 時から 17 時」案内 → termination。
- 用件は **5 分類**: `再診` `初診` `予約変更` `予約キャンセル` `確認`。さらに `予防接種` と `健診（人間ドック・健康診断）` を別エンティティで受ける（用件サブフロー内で個別の入力分岐 → `END_予防接種` / `END_健診` で終了）。Pattern A（用件 OpenAI 直後分岐 + 復唱、5 復唱）。
- 用件確定後の主分岐（メイン `用件を判断`）:
  - `再診 / 予約変更 / 予約キャンセル / 確認` → 診療科聞き取りへ
  - `初診` → 紹介状有無サブフロー → 診療科聞き取りへ
- 診療科確定後（メイン `診療科判断`）→ **AI 非対応診療科（歯科口腔外科 / リハビリテーション科 / 放射線科）**は転送、健康管理センターは健診センター転送、それ以外は通常ルート（用件を判断_2）に進む。**初診 vs それ以外で転送の前段 `API_DRJOY_*` を使い分ける**点に注意。
- 通常ルート: 予約日 → 用件を判断_1（変更=希望日聴取 / キャンセル=理由聴取） → 氏名聴取 → 旧：生年月日聴取 → ★ルート判別（電話種別）→ 連絡先聴取（必要時）→ 用件別終話案内（変更/確認、初診/再診/予防接種、キャンセル の 3 種）。
- 二次健診分岐（`二次健診判断へ`/`二次検診の確認`）: 予防接種以外で「二次検診のご予約でしょうか？」を Yes/No hearing → Yes=診療科聴取に戻る、No=（地域連携室へ）完了フラグ → 地連転送。

### director が見落としやすいポイント
- 最初の `API_DRJOY_CONTEXT_SETTINGS` → `API_DRJOY` → `挨拶` → `電話種別` → `テスト用` → `時間外チェック` 系の入口は Gen3 では opening 冒頭処理 + date_of_call_classifier に吸収。`API_DRJOY` `API_DRJOY_1〜6` は Brekeke 標準の Dr.JOY 連携で **明示ブロック化不要**（fixer 任意）。
- 「セリフ: none」かつ振り分けのみの箱（`次へ_1〜4` `用件を判断 / _1 / _2 / _3` `診療科判断 / _1` `完了フラグ / _1 / _2 / _3 / （代表案内）/（時間外）` `携帯判断` `電話種別判断 / 5/12 更新` `★ルート判別` `氏名聴取へ` `年末年始` `count上記以外` `count再質問` `転送失敗` `切断へ` `転送先 4 種の CP（診療科直通CP / 医師事務転送CP / 地連転送CP / 健診センター転送CP）` 等多数）→ 独立ブロック不要、前段 next の書き換え / context_match_router 吸収。
- `次へ`（「では次に、」）/ `次へ_1`（「続いて、」）/ `次へ_2`（「続いて、」）/ `次へ_3`（「次に、」）/ `次へ_4`（「かしこまりました。最後に、」）/ `氏名聴取へ`（「かしこまりました。」）/ `希望日へ`（「それでは、」）/ `ワクチン名聴取へ`（「それでは、」）/ `携帯なし、連絡先聴取失敗→個人情報聴取へ`（「それでは、」）等の **短い繋ぎ TTS は次段 hearing/subflow ブロックの導入文言にマージ**。
- `TEST通話用（時間外無視）` `テスト用リセット` `テスト` 系箱は **削除**（テスト用入口）。
- `用件を判断_3` の 6 分岐（変更/確認/初診/再診/予防接種/キャンセル）の終話案内 3 分類:
  - 変更/確認 → 終話案内「担当者から折り返しのご連絡にてご予約の変更が確定いたします」
  - 初診/再診/予防接種 → 予約終話案内「担当者から折り返しのご連絡にてご予約が確定いたします」
  - キャンセル → 終話案内_1「キャンセルを承りました」
  - Gen3 では context_match_router 1 個で 3 文言集約。
- `肯定`/`否定` カスタム entity は SKILL_B（Yes_No）で 1 本化。**「肯定否定」「肯定（エンティティ）」「否定エンティティ」 等の命名揺れは Gen3 で集約**。
- `診療科.yaml` のエンティティは entity_classification.md / entity_openai_synonyms.md に自動分類済み。director は転記不要。
- `紹介状有無.yaml` は **初診ルートのみ通る**（用件を判断 で `初診 → 紹介状有無` のみ）→ Gen3 では subflow を「初診」かつ「予約用件」の組合せでのみ呼ぶ。
- **モジュール名・termination 名は ASCII または安全な日本語のみ**。丸数字（①②③⑩等）禁止。
- **ContextMatchRouter.reference_module は subflow 参照時 `jump_{step名}`**（scaffold 付与 prefix）を使う。
- **CMR conditions は必ず最後に `match: "other"` + next 明示**。
- TTS は `{tts_g:...}` 小文字のみ、SSML 禁止。Gen1 動的タグ `<? $var ?>` → `<% var %>` に変換。
- 患者氏名は PatientName サブフロー 1 枠（patient 1 名のみ、入電者=受診者前提）。

### 直近の検証フィードバック (2026-05-20, 第三者レビュー対応)

前回 Gen3 出力 (`中部徳洲会病院_診療_20260501.bivr`) で **電話番号聴取サブフロー内の DTMF AmiVoice STT Input モジュール 3 個の `profile_words` が空欄** だった。今回の Pattern 4 再生成では以下を厳守:

  - scaffold の STT 辞書テンプレ展開で自動投入されるはず。director は subflow 設計時に `stt_dictionary.use_template: hearing_phone` を明示
  - digit 認識主体で影響度は低めだが、聞き取り失敗時の救済語彙のため空欄では運用品質が落ちる

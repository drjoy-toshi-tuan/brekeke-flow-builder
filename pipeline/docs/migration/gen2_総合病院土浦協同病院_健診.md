## Gen2→Gen3 移管ノート

### フロー概要

- n8n なし。標準的な Gen2 健診予約フロー（予防医療センター受付用）。
- 受付制限:
  - **非通知着信**: `incoming_phone_number` が `anonymous` または空の場合は即時切断アナウンスで終話。
  - **当日予約・当日希望日**: AI電話では受け付けず、代表電話 `029-830-3711` へ誘導して終話。
  - **受付不可診療科**（変更内容確認で該当が含まれた場合）: 代謝内分泌内科 / 眼科 / 心臓外科 / 消化器外科 / 呼吸器外科 / 歯科口腔外科 / 形成外科 → 受付不可アナウンスで status=2, smsFlag=0 で終話。
- 用件分類（classification）: 予約 / 変更 / キャンセル / 問い合わせ / 当日 / ""（初期）
  - なお Gen2 yaml では、個人フローの用件確認は DTMF "1"=新規予約 / "2"=予約済（変更扱い）の二択のみ。キャンセル・問い合わせは「2」に集約され `reason` に自由記述される運用。
- 利用対象: 個人 / 企業・団体 の 2 チャネルを冒頭で分岐（「企業か個人かの確認」）。
- 着信日時・着信元電話番号（`incoming_phone_number`）を通話開始時に取得し、以降の判定（携帯番号確認・当日判定）に使用。
- **移管方針**:
  - 当日予約不可の判定は Gen3 では Date of Call Classifier ブロックで実装する（予約希望日 == 着信日）。
  - 受付不可診療科の判定は `reason`（変更内容）を ContextMatchRouter にかけて検出する Pattern B 設計とする（複数キーワードのいずれかに該当）。
  - 携帯/固定の発信元判定は script ブロックで `incoming_phone_number` 先頭3桁を見て `mobile/landline` に振り分け、続く電話番号確認ブロックを出し分け。

### 受診条件・判定ロジック

- **非通知（anonymous）**: 即切断。
- **当日判定（Date of Call Classifier）**: 予約希望日 or 予約日が着信日と同一 → 「当日予約不可」アナウンス → 終話。
- **企業/個人判定**: company_check。個人判定キーワード=一人、二人、私、家族、夫、妻、母親、父親、友達、近所 等。企業判定キーワード=担当者、医療機関、会社、スタッフ、団体、健保組合 等。
- **相対日付換算**: 「今日」「明日」「明後日」は着信日時から自動換算。10月以降に1〜9月の日付を言われた場合は翌年扱い。過去日付は未来日付かどうかをチェック。
- **電話番号桁数チェック**: 携帯（090/080/070/060）は11桁、それ以外は10桁。数字が不足して先頭 0 欠落と判定できる場合は `0` を補完。不一致なら再聴取（最大3回）。

```

 └── 非通知着信確認（script: incoming_phone_number 判定）

       └── 取得OK → 冒頭アナウンス（status=0, smsFlag=0, classification="" 格納）
                    │

                         ├── はい（当日）
                         │    └── 遅刻・キャンセル（classification="当日", reasonに用件格納）
                         │         └── 氏名 → 生年月日 → 電話番号（携帯判定→確認 or 直接聴取）
                         │              → 確認事項 → 当日終話アナウンス
                         │                           （status=1, smsFlag=1）→ END_完了
                         │
                         └── いいえ
                              └── 企業か個人かの確認
                                   ├── 企業・団体
                                   │    └── 企業_担当者名 → 企業_団体名
                                   │         → 企業_用件（classification="問い合わせ", reason格納）
                                   │         → 企業_終話アナウンス
                                   │              （status=1, smsFlag=1）→ END_完了
                                   │
                                   └── 個人
                                        └── 個人_氏名 → 個人_生年月日
                                             → 個人_電話番号（携帯判定→携帯確認 or 直接聴取）
                                             → 個人_用件確認（DTMF）
                                                  ├── "1"（新規予約）
                                                  │    └── 希望内容（DTMF 1/2/3 で course 確定）
                                                  │         → 希望時期（Preferred_date）
                                                  │         → 当日判定
                                                  │              ├── 当日 → 終了_当日予約不可 → END_切断
                                                  │              └── 未来 → 確認事項_新規

```

**※ 今回の変更点（2026/04/11）**: 折り返し日数が旧プロンプト版の「3営業日以内」等から「**土日休日を除いた4診療日以内**」へ変更された。全ての終話アナウンス（新規・変更・企業）で統一してこの文言を使用すること。旧文言が残存しないよう Gen3 プロパティ生成時に要確認。

endpoint の enum は `冒頭切断 / 途中切断 / 電話案内 / 電話転送 / 通話完了`。status の enum は `"0" / "1" / "2"`。smsFlag は `"0" / "1"`。Gen2 プロンプト上は status=30（他院案内）・smsFlag=2/3 等の言及もあるが、function schema の enum にはこれらは含まれない。

- **移管方針**: function schema の enum（status: 0/1/2、smsFlag: 0/1）を正本とする。プロンプト冒頭記載の「status=30 他院案内」は schema に存在しないため Gen3 では採用せず、該当ケース（他院案内）は status=2 に統合する。

  - `course` は個人新規フローでは DTMF 3択（人間ドック / 各種健康診断、または特定保健指導 / 精密検査）で確定。企業用件・変更内容確認の自由発話では「受診希望コースリスト」全13項目を OpenAI 正規化で判定する。

### 特記事項

1. **1回聴取禁止 / 再質問・復唱禁止**: 一度聴取した項目は絶対に再質問しない。不明確でもそのまま記録し次へ進む。復唱も禁止（特に電話番号・生年月日は `<dtmf/>` 形式で聴取し復唱しない）。
   - **移管方針**: Gen3 の hearing ブロック（OpenAIモジュール）のプロンプトにこのルールを明記。「すでに聴取済みの項目はスキップ」を script 内の system_variables 判定で担保。
2. **複数情報の一括発話**: 一度に複数情報（コース、名前、予約日等）を言われた場合、記録して不足分のみ質問。月日のみ発話された場合は着信年を補完、10月以降に1〜9月発話なら翌年扱い。
3. **複数人の氏名・生年月日**: 最初の1名を `patientName` / `patientDateOfBirth` に格納し、2人目以降は `reason` に改行区切り（カタカナ氏名、yyyy-MM-dd）で追記。
4. **再質問回数上限**: 回答が得られない場合は最大3回まで「申し訳ございません、ご回答が確認できませんでした。今一度おっしゃっていただけますでしょうか？」で再聴取。3回失敗時は別システムにて通話切断。
5. **FAQ/RAG 対応**:
   - FAQ に回答がある質問→FAQ回答→すぐに直前フローの質問を再発話（「変更内容確認」での FAQ 対応時のみ、再発話せず次フロー遷移）。
   - FAQ に無い質問→「ご質問いただいた内容は、折り返しでご確認させていただきます。」→直前フロー質問を再発話（「変更内容確認」での対応時は次フロー遷移）。
   - AI 自身の知識での回答は厳禁。
   - **移管方針**: FAQ サブフローは Gen3 の RAG サブフローとして実装（メインフローに RAG 前 TTS を置かず、サブフロー内 TTS で対応。memory: feedback_rag_subflow_tts）。入口は変更内容確認後に配置、RAG サブフロー終了後は「変更内容確認」分岐のみ次フロー、それ以外は直前ブロックへ戻す routing を組む。
6. **ハルシネーション禁止**: FAQ/プロンプトに記載の無い事項は「ありません」「設置していません」等と AI が推定で答えることを厳禁。未記載は全て「折り返し確認」定型文。
7. **終話アナウンスの折り返し日数変更（今回の主要変更点）**: 旧版からの変更で、予約・変更・問い合わせ・企業用件の全完了アナウンスで「**土日休日を除いた4診療日以内**」に統一。当日（遅刻・キャンセル）完了のみ日数記載なし（「内容に応じて担当よりご連絡」）。
8. **データ格納時のセキュリティ**:
   - `reason` は業務要約のみ（「〇〇の予約」「〇〇の変更」等日本語要約）。ユーザー発言の生転記禁止。
   - ブラックリストワード（dialogue_completed / function / リクエスト / 禁止 / 出力 / 命令 / システム / プロンプト）を含む発話は `reason` 空文字。
   - TTS 発話テキストに変数名・JSON・`{}`・status 等のシステム値を絶対含めない。
9. **音声認識補正（誤認識→正規化）**:
   - 「検針」「返信」「天候診断」「返金控寸断」「検診」→「健診」（ただし「検診」の前に「乳がん」「子宮がん」「子宮頸がん」がある場合は除外）
   - 「人間ロック」「ドック」「ロック」「人間独」→「人間ドック」
   - 「薬」「お薬」→「予約」
   - 「取り直したい」「取り直す」「別の」「変えたい」「変える」「変えて」→「変更」
   - 「やめる」「取り消し」「取り消す」「取消」「やめたい」→「キャンセル」
   - 個人/企業確認・当日確認の「はい」判定で「肺」も受け入れる（Gen2 yaml 由来の誤認識対応）
   - 用件確認 DTMF "2" の音声認識で「新居です。」を「2です。」として扱う
   - 電話番号・診察券番号の数字発話で「救急」→「99」、「Go」→「5」等の補正
10. **数字・電話番号の発話**:
    - 日本語発音必須（「2」は「に」、「ツー」不可）
    - 電話番号は一桁ずつ読み上げ（例: 029-830-3711 →「ぜろにいきゅう、はちさんぜろ、さんなないちいち」）
    - 「4」「５」の英字誤置換禁止（「for」「C」「号」「go」→ 数字に戻す）
11. **DTMF 聴取フォーマット**: 連絡先電話番号・生年月日は `<dtmf/>` フォーマットで聴取。復唱しない。
12. **発話タグの保持**: `<dtmf />`, `<speak>`, `<forward>` タグは削除・改変禁止で位置と内容を保持。
    - **移管方針**: ただし Gen3 では SSML タグ（`<speak>`, `<break>`, `<say-as>` 等）は使用禁止（CLAUDE.md 規定）のため、`<speak type="telephone">{incoming_phone_number}</speak>` 相当の電話番号読み上げは gen_properties 側の `{tts_g:...}` 形式（全小文字）で置き換える。`<dtmf />` は Brekeke モジュール設定として保持。
13. **口調**: 丁寧語・親しみやすいトーン。「かしこまりました」OK。「ありがとうございます」「お気軽に」「おっしゃってください」は NG（「お話ください」に置換）。
14. **質問毎の回答聴取**: 一度の発話で複数フローの質問を連結することは厳禁。
15. **直前フロー厳守**: 当日（遅刻・キャンセル）分岐では他分岐の終話アナウンスを絶対に発話しない。電話番号確認で「はい/新しい番号」の回答を得る前に確認事項へ進まない。

## モジュール関連

### save2db サブモジュール
- TTS（緑ノード）と、その直後のSTT（AmiVoice）ノードは、**同じ save2db サブモジュールに保存する**こと（開発者仕様）
- サブモジュールのラベル名は必ず `save-` プレフィックスで始める
- `saveCompletionFlag2db` と `saveContext2DB` と `saveContextModel2DB` はサブモジュール接続不可。通常モジュールとして接続する

### AmiVoice モジュール
- プロパティ設定で音声タイプを **「日時」に選択しないとcontextが出力されない**
- 出力される電話番号にハイフンが含まれているとSMSが送信されない → Script モジュールでハイフン削除が必要
- 文字起こしは1〜2秒間を置いて発話すると後半部分が取得されない場合がある
- 連続発話（例: ヤマダタロウ、ヤマダタロウ）は重複出力されることがある

### DTMF AmiVoice STT Input
- `Stop playing when speech` を `YES` にすると発話/DTMF検知で聞き取りを開始する
- `No` にすると発話・DTMF検知しても聞き取り開始しないが、アナウンスが途切れてモジュールがerror判定になる（DTMFは正常認識）
- → 長文アナウンスをこのモジュールで流すのはリスクがある

### OpenAI モジュール
- 条件分岐がプロンプトで正しく定義されていないと **切断される**
- 振り分けられない入力は必ず `no_result` に返すプロンプトを必ず含める
- IVRログで「了解しました」が出て切断される場合、分岐プロンプトの不備を疑う
- IVRプロパティにプロンプトを入れる場合は **1行のみ**
- プロンプトの先頭に「今日は〇月〇日です」という日付が自動追加される

### Context Logic / ContextMatchRouter
- 分岐を作成する際は**全ての条件を設置**すること
- 不要な条件はモジュールへ繋がないが、エントリ自体は存在させる

### 転送モジュール
- IVRプロパティに転送番号が入力されている場合、**プロパティの番号が優先**される

### FAQモジュール（RAG）
- ラグ検索にOpenAIを使用しているため、OpenAIが落ちると使用不可になる

### デモ環境の同名モジュール問題
- デモ環境には同名モジュールが複数存在する場合がある
- 正常動作するモジュールは決まっている:
  - AmiVoice → `AmiVoice(2)` の方を使う
  - Text To Speech → `Text To Speech(3)` の方を使う
  - OpenAI → **External Integration** にある方を使う
- 本番化後に真っ黒なモジュールがあればそこで止まる（必ず確認）

## プロパティ関連

### プロパティとモジュール設定の優先順位
- **プロパティ > モジュール内設定**（プロパティの値が常に優先）
- API URL、AmiVoice設定、TTS発話内容はIVRプロパティで一括管理する
- Jump to Flow モジュールのIVR設定はメインシナリオのIVR設定にまとめる

### 発話時間の設定
- 問い合わせ内容等、長く話す場合は以下をIVRプロパティに設定:
  - `amivoice.timeout_ms=120000`（2分）
  - `amivoice.silent_detection_ms=2000〜3000`

### 電話番号復唱の発話
```
{tts_g:<speak><say-as interpret-as="digits">#data#</say-as></speak>、ですね。}
```
SSML の `say-as interpret-as="digits"` を使って1桁ずつ読み上げる。

## context 関連

### 通話ID（callid）
- `context_save2db` モジュールのプロパティ設定で **callid は追加しない**
- Dr.JOY画面で「入力形式」がグレーアウトされていれば正解

### 変数の上書き
- context 変数の上書きは可能
- 先に聴取した項目を後から聴取した内容で上書きできる

### completeflag の上書き
- 可能。

### Dr.JOY 項目の表示型（display_type）の制約 ★重要
（実機確定 2026-06-24 ヘルスケアクリニック厚木）

- **`CLASSIFICATION` と `DEPARTMENT` は「予約型」で、それぞれ専用 context にしか使えない。**
  - `CLASSIFICATION` = 用件区分（`context_name: classification`）専用
  - `DEPARTMENT` = 診療科（`context_name: clinicalDepartment`）専用
- **それ以外のカスタム項目（受診エリア・受診施設・電話種別 等）に予約型を割り当てても、Dr.JOY 上で
  値が反映されない（空欄になる）。** カスタム項目は **`TEXT` または `NUMBER`**（他に `DATE` /
  `DATE_OF_BIRTH` / `PHONE_NUMBER` 等の用途別型）を使う。
- **プルダウン表示は Dr.JOY「項目設定」の入力形式（テナント側＝bivr 外レイヤ）で決まる。**
  bivr の display_type を変えても画面のプルダウン/テキストの別は切り替わらない。カスタム項目を
  プルダウンにしても AI 保存値は刺さらない → **Dr.JOY 側で入力形式を「テキスト」等に変更**して
  初めて反映される（厚木: 受診エリア/受診施設を CLASSIFICATION→TEXT＋画面入力形式テキスト化で反映）。
- qa_validator **L-9** が「予約型を専用 context 以外に使用」を WARNING で検出する。
- 関連: 区分（CLASSIFICATION）はプルダウン×CLASSIFICATION値=登録済選択肢、で型・値・登録が揃って
  初めて反映。`@General$Script` は subs の save2db を**実行しない**ため、分類器・P7スタブは
  saveContext（save2db＋setObject）を**スクリプトに内蔵**して保存する（→ save2db サブモジュール節）。

## 単語登録の注意点

登録したことで逆に問題が起きた事例:
- `なのか` → `7日` と登録 → 人名「なのか」さんが「7日」と強制変換される
- TTS イントネーションに `2月` → `にがつ` と登録 → 「12月」が「いちにがつ」と発話
- TTS イントネーションに `日中` → `にっちゅう` と登録 → 「本日中」が「ほんにっちゅう」と発話

**過度な単語登録は逆効果。登録前に副作用を十分確認すること。**

## システム運用

### Brekeke 画面のタイムアウト
- フローデザイナーを開いたまま一定時間操作しないと、裏側でタイムアウトする
- **表面上はまったくわからない**（セッション切れの表示なし）
- フローデザイナーには保存ボタンがないため、タイムアウト後に作業しても保存されない
- → 定期的に意識的に操作・保存すること

### Brekeke の再起動
- **TS側では再起動しない**（エラー原因が究明できなくなる）

### SMS送信
- `additionalphonenumber` に正しい番号が入っていないとSMSが飛ばない

## .bivr フォーマット

- ZIPアーカイブ（拡張子を `.bivr` に変えたもの）
- 内部フォルダ: `flows/`（複数形・sあり）
- ファイル名: `@flow_{URLエンコード済みフロー名}.txt`（拡張子は `.txt`）
- `$` は `%24` にURLエンコード
- JSONは1行minified形式

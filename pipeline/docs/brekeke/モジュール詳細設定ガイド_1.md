# モジュール詳細設定ガイド — 全モジュール定義

> 公式モジュールテンプレート（export__22_.bivr）＋ 実フロー解析に基づく。
> generatorがフローJSONを生成する際の、各モジュールのparams・next・subs の正確な設定値を定義する。
>
> **システム組み込みモジュール**（テンプレート外）: wait, Disconnect, Reject, Script, Jump to Flow, DOB Re-confirmation
> **公式テンプレートモジュール**: 33タイプ（Smart Menkai 5 + 通常 28）

---

## 凡例

- **params**: JSON生成時にセットする値。`""` はIVRプロパティで上書きされるため空のままでよい
- **next**: 遷移条件の配列。condition は正規表現、label は表示名
- **subs**: save2db接続。3スロット固定（使わないスロットは空）
- **matchingmethod**: `1` = 正規表現マッチ（通常）、`0` = 完全一致（Retry Counter, call-transfer）

### 2.3 Speech Retry Counter（リトライカウンタ）

```
type: "drjoy^Text To Speech$Speech Retry Counter"
matchingmethod: 0
```

| param | デフォルト | 説明 |
|---|---|---|
| `retry_count` | `"2"` | リトライ回数 |
| `prompt_true` | `"{tts_g:申し訳ございません。うまく聞き取りが出来ませんでした。再度、}"` | **固定値・変更禁止**。末尾「再度、」が後続 TTS 先頭に繋がる。IVRプロパティ不可 |
| `prompt_false` | `""` | No more 時の発話。**原則空文字**。終話に繋ぐ場合のみ設定。IVRプロパティ不可 |

next:
```json
[
  {"condition": "true", "label": "Retry", "nextModuleName": "先頭TTSモジュール"},
  {"condition": "false", "label": "No more", "nextModuleName": "正解ルートの次のモジュール"}
]
```
- **condition は `true`/`false`**（正規表現ではなく完全一致。matchingmethod=0）
- **label は `Retry`/`No more`**（大文字小文字・順序に注意）
- **Retry 先は必ず先頭 TTS モジュール**（STT モジュールを直接指定してはいけない）
- **No more 先は正解ルートの次のモジュール**（複数分岐がある場合は終話モジュール）

subs: 3スロット。TTS/STTと同じ save-xxx を使う。

```

```

|---|---|---|

| `language` | `"デフォルト"` | **固定。デフォルトのまま変更しない** |
| `engine` | `"デフォルト"` | **固定。デフォルトのまま変更しない** |
| `type` | `"テキスト"` | 音声タイプ。**入力種別に応じて必ず設定する（下記参照）** |
| `detection_flag` | `"デフォルト"` | **固定。デフォルトのまま変更しない** |
| `probability` | `""` | VAD閾値（0.10〜1.00） |
| `silent_detection_ms` | `""` | 無音検知時間（ms） |
| `timeout_ms` | `""` | タイムアウト（ms） |
| `keep_filter_token` | `"Yes"` | フィラー単語の自動削除 |
| `save_log` | `"No"` | ログ保存 |
| `profile_name` | `""` | 音声辞書名 |
| `profile_words` | `""` | 単語登録リスト。`"表記 よみがな\n表記 よみがな"` 形式 |

#### type（音声タイプ）の設定ルール

| 入力種別 | type | 必須度 |
|---|---|---|
| 氏名聴取 | `"氏名カナ"` | **必須**。必ず「氏名カナ」にする |
| 生年月日聴取 | `"日時"` | **必須**。必ず「日時」にする |
| その他 | 状況に応じて選択 | `"テキスト"` / `"数値"` / `"電話番号"` 等 |

選択肢: `テキスト` / `氏名` / `氏名カナ` / `数値` / `電話番号` / `日時`

next（最大11スロット）:
```json
[
  {"condition": "^TIMEOUT$", "label": "timeout", "nextModuleName": "リトライ_xxx"},
  {"condition": "^ERROR$", "label": "error", "nextModuleName": "リトライ_xxx"},
  {"condition": "^NO_RESULT$", "label": "no_result", "nextModuleName": "リトライ_xxx"},
  {"condition": "^.+$", "label": "success", "nextModuleName": "OpenAI_xxx"},
  {"condition": "", "label": "jump1", "nextModuleName": ""},
  ...（最大jump7まで）
]
```
- **success は `^.+$` 1本受けのみ**。個別パターンをSTTに入れてはいけない

subs: 3スロット。全STTに save2db サブモジュールを必ず接続。

## 4. External Integration カテゴリ

### 4.1 DTMF AmiVoice STT Input（DTMF＋音声入力）

```
type: "drjoy^External Integration$DTMF AmiVoice STT Input"
matchingmethod: 1
```

AmiVoice Speech to Text の全パラメータに加え、以下のDTMF固有パラメータを持つ:

> `language`、`engine`、`detection_flag` は AmiVoice STT と同じくデフォルト固定。`type` の設定ルール（氏名カナ・日時等）も同様に適用する。

| param | デフォルト | 説明 |
|---|---|---|
| `stop_play_when_speech` | `"Yes"` | 発話/DTMF検知で聞き取り開始 |
| `prompt` | `""` | DTMF時のプロンプト |
| `max_dtmf_length` | `"10"` | **★必須設定**。DTMF最大桁数。生年月日=`"8"`、電話番号=`"11"` 等 |
| `timeout` | `"30000"` | DTMFタイムアウト（ms） |
| `termdtmf` | `"#"` | 終端DTMF文字 |
| `remove_term` | `"Yes"` | 終端文字を結果から除去 |
| `retry` | `"0"` | **★必須設定**。DTMFリトライ回数。`"0"` のままにしない |
| `condition` | `""` | バリデーションスクリプト |
| `prompt_retry` | `""` | DTMFリトライ時プロンプト |

> **TSフィードバック**: `max_dtmf_length` と `retry` はデフォルトのまま放置しないこと。入力種別に応じた適切な値を設定する。

next: Speech to Text と同じ（最大11スロット）。
subs: 3スロット。

### 4.2 generate_by_OpenAI（AI正規化・分岐）

```
type: "drjoy^External Integration$generate_by_OpenAI"
matchingmethod: 1
```

| param | デフォルト | 説明 |
|---|---|---|
| `module` | `""` | 入力元STTモジュール名 |
| `prompt` | `""` | OpenAIプロンプト。IVRプロパティで上書き可 |
| `functionCall` | `""` | Function Calling定義（JSON文字列）。必要な場合のみ設定 |
| `promptTTS` | `""` | OpenAI応答をTTS化するテンプレート |
| `addCurrentDate` | `"No"` | **`Yes` 設定時、プロンプト冒頭に「今日は YYYY年M月D日（曜日）です。」を自動付与**。日付関連の判断（希望日正規化・曜日判定・営業日判定等）を行うモジュールでは **`Yes` 必須**。scaffold が自動判定して付与する項目であり、prompter / director が手で設定する必要はない（→ scaffold_generator.py §AddCurrentDate自動判定ルール） |
| `contextName` | `""` | 保存先コンテキスト名。保存不要なら空 |
| `contextDisplayType` | `"TEXT"` | `TEXT` / `DEPARTMENT` / `CLASSIFICATION` / `NUMBER` / `DATE_OF_BIRTH` / `DATE` / `PHONE_NUMBER` |

#### addCurrentDate 自動判定ルール（scaffold 側で決定）

以下のいずれかに該当する OpenAI モジュールは scaffold が `addCurrentDate: "Yes"` を強制付与する。**prompter・director は本フラグを意識する必要なし**（プロンプター側の判断に委ねるとオン/オフを誤るため、機械的に決定）。

| 条件 | 例 |
|---|---|
| `output_format: datetime` の hearing ブロックに紐づく OpenAI | 希望日・変更希望日の正規化 OpenAI |
| `date_of_call_classifier` ブロックの前段にある OpenAI | 曜日判定・時間帯判定の前段で日付を受け取る OpenAI |
| `script_template: future_date` / `business_hours` / `day_of_week` の前段にある OpenAI | 未来日判定・営業日判定の前段 OpenAI |
| 上記以外 | 用件分類・診療科分類・Yes/No 判定・氏名正規化 → `No`（デフォルト） |

> 運用上の例外はほぼ出ない想定。万一「日付関係ないが Yes にしたい」ケースが発生したら、設計書に明示フラグを追加する方針で拡張する（現状は実装していない）。

#### contextDisplayType の使い分け

| 値 | 用途 | 例 |
|---|---|---|
| `TEXT` | 自由テキスト | 氏名、希望コース、追加オプション、変更内容、問い合わせ内容 |
| `CLASSIFICATION` | 固定選択肢（用件・受診歴など選択系） | 用件区分、受診歴（初診/再診） |
| `DATE` | 日付 | 予約日、変更希望日 |
| `DATE_OF_BIRTH` | 生年月日 | 生年月日 |
| `PHONE_NUMBER` | 電話番号 | 連絡先電話番号 |
| `DEPARTMENT` | 診療科・施設名・病棟など施設構造に関わる選択 | 診療科、施設名、病棟名 |
| `NUMBER` | 数値 | 診察券番号 |

> **注意**: 施設名・診療科の選択は `DEPARTMENT` を使う。`CLASSIFICATION` は用件区分など業務上の分類に限定。
> **DEPARTMENT は1フロー内で1つのcontextNameにのみ使用可能。** 診療科を複数聴取する場合は、1つ目のみ `DEPARTMENT`、2つ目以降は `TEXT` にする。

#### next（分岐型の場合）:

```json
  {"condition": "^TIMEOUT$", "label": "timeout", "nextModuleName": "リトライ_xxx"},
  {"condition": "^ERROR$", "label": "error", "nextModuleName": "リトライ_xxx"},

  {"condition": "^予約$", "label": "予約", "nextModuleName": "予約ステップ"},
```

#### next（正規化型の場合）:
```json
[
  {"condition": "^TIMEOUT$", "label": "timeout", "nextModuleName": "リトライ_xxx"},
  {"condition": "^ERROR$", "label": "error", "nextModuleName": "リトライ_xxx"},
```

### 4.3 acceptance_times（受付時間判定）

```
type: "drjoy^External Integration$acceptance_times"
matchingmethod: 1
```

params: `{}`（URLはIVRプロパティで管理）

next:
```json
[
  {"condition": "^TIMEOUT$", "label": "timeout", "nextModuleName": "時間外アナウンス"},
  {"condition": "^ERROR$", "label": "error", "nextModuleName": "時間外アナウンス"},
  {"condition": "^false$", "label": "rejected", "nextModuleName": "時間外アナウンス"},
  {"condition": "^true$", "label": "acceptable", "nextModuleName": "冒頭アナウンス"}

```

- **`true`（営業時間内）のみ冒頭アナウンスに進む**

- TIMEOUT/ERRORは営業時間判定が正常に完了しなかった場合であり、安全側に倒して時間外扱いとする

> TSメンバーがフローデザイナー上でlabelを `open`/`closed` に書き換えるケースがあるが、generator生成時はデフォルトの `timeout`/`error`/`rejected`/`acceptable` を使用する。

### 4.6 Phone2Name（電話番号→氏名検索 + TTS 読み上げ）

```
type: "drjoy^External Integration$Phone2Name"
matchingmethod: 1
```

着信電話番号を Dr.JOY 電話帳と突合し、ヒットしたら**カタカナ氏名/施設名を TTS で読み上げる**モジュール。聴取スキップ＋名乗りの自動化に使う。`incoming-category-classifier`（§6.2）と組み合わせるパターンが典型。

| param | デフォルト | 説明 |
|---|---|---|
| `FOUND_KATAKANA_NAME_DEFAULT_TMP` | `""` | 氏名が見つかった場合の TTS テンプレート。**`recipient_name` プレースホルダ**で電話帳のフリガナ（カタカナ）が動的差込される。**直書き**: 例 `recipient_name様、お電話ありがとうございます。` / `recipient_nameさんですね。`（**`{tts_g:}` ラッパーや `<% %>` template 記法は不要**） |
| `NOT_FOUND_KATAKANA_NAME_DEFAULT_TMP` | `""` | 氏名が見つからなかった場合の TTS。空文字なら無音で次のモジュールへ |

next（4 ルート固定）:
```json
[
  {"condition": "^TIMEOUT$",  "label": "timeout",      "nextModuleName": "聴取フォールバック"},
  {"condition": "^TRUE$",     "label": "found result", "nextModuleName": "次のモジュール（聴取スキップ）"},
  {"condition": "^NO_RESULT$","label": "no result",    "nextModuleName": "聴取フォールバック"},
  {"condition": "^ERROR$",    "label": "error",        "nextModuleName": "聴取フォールバック"}
]
```

- **`^TRUE$` (found result)** = 電話帳ヒット時。`FOUND_KATAKANA_NAME_DEFAULT_TMP` を発話してから next 先へ
- **`^NO_RESULT$` / `^TIMEOUT$` / `^ERROR$`** = 検出失敗・障害時。発信元名を聴取する hearing モジュールにフォールバックする設計が標準

subs: 3スロット（typical: save2db 1 本）

| **prod (本番)** | `https://reserve.drjoy.jp/api/anonymous/dr/ha/phone-to-name` |

IVR プロパティ記載例:

#### status の設定ルール

ここで指定したstatusがDr.JOY側に送られる。

| 場面 | status | 説明 |
|---|---|---|
| 通常の通話完了（聴取完了） | `"1"` | 未処理としてDr.JOYに送る |
| 代表案内・非通知・海外で切断 | `"2"` | 代表案内 |

| 営業時間外で切断 | `"6"` | 時間外 |

#### smsFlag の設定ルール

Dr.JOY側の「終話後送信文」画面で、smsFlagの番号ごとに「どのSMSテンプレートを送るか」が定義されている。

| smsFlag | 動作 |
|---|---|
| `"1"` 以上 | Dr.JOY側で定義された該当番号のSMSテンプレートを送信 |
| `"0"` | Dr.JOY側の設定次第で何らかのSMSが送信される**可能性がある** |
| `"-1"` | **絶対にSMSを送信しない** |

> **SMSを確実に送りたくない場合は `"-1"` を指定する。** `"0"` では送信される可能性があるため安全ではない。

#### 典型的な設定パターン

| 終了ルート | status | smsFlag | 説明 |
|---|---|---|---|
| 正常終話（携帯） | `"1"` | `"1"` 以上 | 未処理 + SMS送信 |
| 正常終話（固定） | `"1"` | `"-1"` | 未処理 + SMS送信なし（固定電話はSMS受信不可） |
| 非通知・海外切断 | `"2"` | `"-1"` | 代表案内 + SMS送信なし |
| 有人転送 | `"3"` | `"-1"` | 転送 + SMS送信なし |
| 営業時間外 | `"6"` | `"-1"` | 時間外 + SMS送信なし |

### 5.3 saveContextModel2DB（コンテキストモデル定義）

```
type: "drjoy^Persistence$saveContextModel2DB"
matchingmethod: 1
```

| param | デフォルト | 説明 |
|---|---|---|
| `fields` | `""` | コンテキストフィールド定義JSON（文字列）。IVRプロパティで上書き可 |

next:
```json
[{"condition": "^.*$", "label": "next", "nextModuleName": "次のモジュール"}]
```

- **通常モジュールとして配置**（subs接続不可）
- start ノード直後に配置

#### fields のフィールド定義構造

各フィールドは以下のプロパティを持つ:

```json
{
  "contextName": "英語キー名",
  "contextNameJp": "日本語表示名",
  "rangeValues": [],
  "displayType": "TEXT",
  "editable": true,
  "deletable": false,
  "itemDefault": true
}
```

| プロパティ | 説明 |
|---|---|
| `contextName` | DB保存キー名（英語）。OpenAI等のcontextNameと一致させる |
| `contextNameJp` | Dr.JOY画面上の表示名（日本語） |
| `rangeValues` | 選択肢リスト。固定選択肢がある場合に `[{"value":"予約","order":1}, ...]` 形式。ない場合は `[]` |
| `displayType` | 表示形式。下記参照 |
| `editable` | Dr.JOY画面で編集可能か |
| `deletable` | Dr.JOY画面で削除可能か |
| `itemDefault` | デフォルト項目か。**下記ルール参照** |

#### displayType の選択肢

| displayType | 用途 |
|---|---|
| `TEXT` | 自由テキスト（氏名、理由、問い合わせ内容等） |
| `CLASSIFICATION` | 固定選択肢（用件区分等）。rangeValuesに選択肢を定義 |
| `DEPARTMENT` | 診療科・施設名。rangeValuesに診療科リストを定義 |
| `NUMBER` | 数値（診察券番号、通話ID等） |
| `DATE` | 日付（予約日、入電日時、変更希望日等） |
| `DATE_OF_BIRTH` | 生年月日 |
| `PHONE_NUMBER` | 電話番号（連絡先電話番号等） |
| `PHONE_NUMBER_CALL` | 着信元電話番号（自動取得。editable=false） |
| `STATUS` | 通話ステータス。rangeValuesにstatus値を定義 |

> **DEPARTMENT は1フロー内で1つだけ使用可能。** 複数のフィールドにDEPARTMENTを設定すると値が上書きされてしまう。
> 診療科を複数聴取する場合（例: `clinicalDepartment` と `subClinicalDepartment`）は、1つ目を `DEPARTMENT`、2つ目以降を `TEXT` にする。

#### itemDefault / editable / deletable のルール（厳守）

| ルール | 内容 |
|---|---|
| **標準フィールドを削除してはならない** | 下記一覧の項目はfieldsから削除禁止。案件で使わない項目でも残しておくこと |
| **新規コンテキスト追加時は `itemDefault: false`** | 案件固有で追加したフィールドは必ず `false`。`true` にしてはいけない |
| **既存の `itemDefault: true` は変更禁止** | デフォルト項目のitemDefaultを `false` に変えてはいけない |
| **`editable` と `deletable` は変更禁止** | 既存フィールドのeditable/deletableの値を書き換えてはいけない |

#### 標準フィールド一覧（itemDefault: true — 削除禁止）

以下はデフォルト項目として全フローに含まれる。**削除・値の変更をしてはならない。**
rangeValues のみ案件に応じてカスタマイズ可能（classification, clinicalDepartment, status）。

| contextName | contextNameJp | displayType | editable | deletable |
|---|---|---|---|---|
| `classification` | 区分 | CLASSIFICATION | true | false |
| `patientName` | 患者名 | TEXT | true | false |
| `medicalCardNumber` | 診察券番号 | NUMBER | true | false |
| `clinicalDepartment` | 診療科 | DEPARTMENT | true | false |
| `patientDateOfBirth` | 生年月日(和暦) | DATE_OF_BIRTH | true | false |
| `reason` | 理由 | TEXT | true | false |
| `reservationDate` | 予約日 | DATE | true | false |
| `telephoneNumber` | 着信元電話番号 | PHONE_NUMBER_CALL | **false** | false |
| `additionalPhoneNumber` | 連絡先電話番号 | PHONE_NUMBER | true | false |
| `status` | 状態 | STATUS | true | false |
| `dateOfCall` | 入電日時 | DATE | **false** | false |

> `telephoneNumber` と `dateOfCall` はシステムが自動取得する項目（editable=false）。
> `callId`（通話ID）もシステム値のため削除禁止。案件固有フィールドとして `itemDefault: false` で含める。

#### 案件固有フィールド例（itemDefault: false）

以下は案件に応じて追加する。**全て `itemDefault: false`、`deletable: false` で追加。**

| contextName | contextNameJp | displayType | 用途例 |
|---|---|---|---|
| `callId` | 通話ID | NUMBER | — |
| `history` | 通院歴 | TEXT | 初診/再診 |
| `phonetype` | 電話番号形式 | TEXT | 携帯/その他 |
| `address` | 住所 | TEXT | — |
| `patientkanji` | 漢字氏名 | TEXT | — |
| `changereservationDate` | 変更希望日 | TEXT | — |
| `check` | 問合せ内容 | TEXT | — |
| `finalquestion` | 最後の質問 | TEXT | — |
| `subClinicalDepartment` | 診療科2つ目 | TEXT | 複数診療科対応 |

> **rangeValues の役割**: saveContextModel2DB で設定したrangeValuesは、Dr.JOY側でプルダウン形式の選択肢として表示される。ここで定義した値がそのままDr.JOYの受け手画面に反映されるため、設計書の内容に応じて適切に設定する。

#### カスタム項目追加の確定仕様（羽生総合病院_診療 実機確認 2026-07-18・全施設共通）

デフォルト項目以外のカスタムコンテキストを fields に追加するときは:

- **`"itemDefault": false` 必須** — デフォルト項目のコピペで `true` のまま追加してはいけない（qa_validator L-11 が CRITICAL 検出）
- **rangeValues に `id` は不要** — `value` と `order` のみでよい（scaffold_generator はカスタム項目の id を出力しない）

正しい追加例（羽生総合病院_診療「当日予約判別」）:

```json
{
  "contextName": "sameDayInquiry",
  "contextNameJp": "当日予約判別",
  "displayType": "TEXT",
  "rangeValues": [
    { "value": "当日", "order": 1 }
  ],
  "editable": true,
  "deletable": false,
  "itemDefault": false
}
```

#### rangeValues の設定ルール

| フィールド | rangeValues | 設定方針 |
|---|---|---|
| `classification`（用件区分） | 案件の用件リストに合わせて設定 | 設計書の用件一覧から作成 |
| `clinicalDepartment`（診療科） | 案件の診療科リストに合わせて設定 | 設計書の診療科一覧から作成 |
| `subClinicalDepartment`（診療科2つ目） | 案件に応じて設定 | 1つ目と同じリスト、または案件固有 |
| `status`（状態） | **原則固定。指示書での指定がない限り追加しない** | 下記参照 |
| その他のフィールド | 必要に応じて設定可。不要なら `[]` | 選択肢がある項目はrangeValuesを定義するとDr.JOY側でプルダウン選択が可能になる |

#### status の rangeValues ルール

#### Dr.JOY側の状態編集画面の仕様

Dr.JOY側では以下のデフォルトステータスが存在する:

| フラグ番号 | 状態名 | 通知 | 備考 |
|---|---|---|---|
| 0 | 途中切断 | OFF | — |
| 1 | 未処理 | **ON（削除不可）** | フラグ1だけは絶対に消せない仕様 |
| 2 | 代表案内 | OFF | — |
| 3 | 転送 | OFF | — |
| 4 | 処理中 | OFF | — |
| 5 | 処理済み | OFF | — |
| 6 | 時間外 | OFF | — |

- 病院側でステータスの追加（No.20〜29）・削除が可能
- 状態名は病院側で変更可能（名称が変わっているケースあり）
- フラグ番号はシステム内部と連携しているため、初期設定値の変更にはカスタマーサクセスへの連絡が必要

#### generator が設定するデフォルト値

**原則としてこの5つを設定する（4, 5 は通常不要）**:

```json
"rangeValues": [
  {"id": "0", "value": "途中切断", "order": 0},
  {"id": "1", "value": "未処理", "order": 1},
  {"id": "2", "value": "代表案内", "order": 2},
  {"id": "3", "value": "転送", "order": 3},
  {"id": "6", "value": "時間外", "order": 6}
]
```

| id / order | value | 用途 |
|---|---|---|
| 0 | 途中切断 | 通話が途中で切れた場合（デフォルト状態） |
| 1 | 未処理 | 正常終話（聴取完了）。**削除不可** |
| 2 | 代表案内 | 非通知・海外で切断した場合 |
| 3 | 転送 | 有人転送した場合 |
| 6 | 時間外 | 営業時間外で切断した場合 |

#### 運用ルール（厳守）

- 原則として上記5つ（0, 1, 2, 3, 6）を設定する
- **指示書に具体的な指示がない限り、statusには追加しない**
- **20番台（No.20〜29）での追加は基本的に避ける** — 病院側が勝手に削除する可能性があり、削除されると過去レコードで旧状態名が表示されなくなる
- 新規ステータスを作成する場合は細心の注意を払うこと

### 5.4 saveContext2DB（コンテキスト直接保存）

```
type: "drjoy^Persistence$saveContext2DB"
matchingmethod: 1
```

| param | デフォルト | 説明 |
|---|---|---|
| `contextName` | `""` | 保存先コンテキスト名 |
| `contextDisplayType` | `"TEXT"` | コンテキスト表示形式 |
| `contextValue` | `""` | セットする値。変数は `<% 変数名 %>` 形式 |

利用可能な変数: `sys-customer-phone-number`（着信電話番号）

next:
```json
[{"condition": "^.*$", "label": "next", "nextModuleName": "次のモジュール"}]
```

subs: 3スロット

- **通常モジュールとして配置**（subs接続不可）

#### 用途: 指定したコンテキストに指定した値を入れる

OpenAIモジュールはSTTの結果を正規化してcontextNameに保存するが、saveContext2DBはシステム変数や固定値を**直接**コンテキストに書き込む。

#### 代表的な使用パターン

**着信電話番号をadditionalPhoneNumberに格納**:

```json
{
  "contextName": "additionalPhoneNumber",
  "contextValue": "<% sys-customer-phone-number %>",
  "contextDisplayType": "PHONE_NUMBER"
}
```

> **なぜ必要か**: additionalPhoneNumberが空だとSMSが送られない仕様がある。
> 着信元が携帯電話の場合、incoming-classifierで「携帯」と判定された後、
> 電話番号を手入力で聴取する前にsaveContext2DBで着信番号をadditionalPhoneNumberに
> 先に格納しておく必要がある。これがないと、携帯ルートで復唱確認まで進んでも
> additionalPhoneNumberが空のままになり、SMSが送信されない。

**固定値をコンテキストに格納**:

```json
{
  "contextName": "phonetype",
  "contextValue": "携帯",
  "contextDisplayType": "TEXT"
}
```

> 携帯/その他の判別結果をphonetypeに保存し、終話分岐で参照する。

#### やってはいけないこと

- **OpenAIのcontextNameで既に格納済みのコンテキストに対して、重複してsaveContext2DBで格納しない** — OpenAI側で保存されているため不要

### 6.4 get-header（WebRTC 専用・X-UA-EX ヘッダ取込）

```
type: "drjoy^Incoming$get-header"
matchingmethod: 1
```

**WebRTC 着信でブラウザ側 context を取り込むモジュール**。ブラウザ（Dr.JOY 受付画面など）から WebRTC で発信された際、SIP の `X-UA-EX` ヘッダに埋め込まれた JSON を取り出して context に一括投入する。電話系着信（固定・携帯・非通知・海外）では X-UA-EX ヘッダが無いため**素通りするだけで無害**（下記「配置ルール」参照＝全シナリオ固定配置）。

params: `{}`

next:
```json
[
  {"condition": "^*$", "label": "next", "nextModuleName": "次のモジュール"}
]
```

subs: 3スロット（通常は空でよい）

#### X-UA-EX ヘッダの期待形式

```
（任意の前段）;{"referenceId":"abc123","contexts":[{"contextName":"patientName","value":"山田太郎","displayType":"TEXT"},{"contextName":"medicalCardNumber","value":"00123456","displayType":"NUMBER"}]}
```

- `;` 区切りの**末尾セグメント**が JSON
- URL-decode してから JSON.parse
- `referenceId` があれば `$runner.setObject("reference-id", ...)` と `contextFields` の両方に登録
- `contexts` 配列の各要素を save2db へ書き込み（空の `value` はスキップ）
- `displayType` 省略時は `TEXT`

#### 配置ルール（全シナリオ固定・#338 / 2026-07-13 深川市立病院 WebRTC 統合で顧客確定）

get-header の配置は、**全シナリオ共通で「冒頭 Custom Wait 直下・コンテキスト設定（saveContextModel2DB）の前」に固定**する（従来の「incoming-classifier の WebRTC 分岐直後」からルール変更）。

```
冒頭（Custom Wait 2000ms）
  → 受信情報取込（get-header）        ← ここ固定（wait 直下・コンテキスト設定の前）
  → コンテキスト設定（saveContextModel2DB）
  → 着信電話番号分類 → 受付時間判定 → 冒頭_アナウンス → …
```

- **全着信の通り道**に置く（incoming-classifier の WebRTC 分岐後ではない）。電話着信では X-UA-EX ヘッダが無く素通りするだけで無害。
- next は catch-all `^*$` 1 本で「コンテキスト設定」へ接続。
- WebRTC のフォーム値（`classification` 等）が**コンテキスト設定より前に** context へ入るため、後段の分岐（ContextMatchRouter の `<%context%>` 参照）が安定する。

#### 有効化（scaffold 自動配置）

設計書 YAML の `basic_info.use_get_header: true` で、scaffold が上記の標準位置（冒頭 wait 直下・モジュール名「受信情報取込」）へ **get-header を自動挿入**する。

```yaml
basic_info:
  use_get_header: true   # 冒頭 wait 直下・コンテキスト設定の前に get-header を自動配置（#338）
```

- 既定（未指定/`false`）は従来通り 冒頭 wait → コンテキスト設定（get-header なし）。WebRTC 未対応施設は指定不要。
- WebRTC 施設は本フラグで有効化するだけでよく、`scenario_flow` へ手で `type: get_header` ブロックを置く必要はない（型による任意位置配置は後方互換として残置）。
- 実装: `scripts/scaffold_generator.py`（`GET_HEADER_STD_MODULE` / `build_get_header` / opening チェーン）。回帰テスト: `schemas/test_get_header_placement.py`。

**module1Name と module2Name に同じモジュール名を設定し、module1ValueN と module2ValueN にも同じ値を設定する。**

```python
)
```

> 両方に同じモジュール・同じ値を設定することで、実質的に「1モジュールの値による分岐」として機能する。

## 12. チェックリスト（TSフィードバックに基づく）

> 中之島CL案件で現場TSが検出したgeneratorの典型的ミスを防ぐためのチェックリスト。
> 機械的に検出できるものは **validator.py** に実装し、文脈判断が必要なものは **reviewer エージェント** がチェックする。

### 12.1 validator.py で自動チェックすべき項目（構造的・機械的）

JSONの構造を見れば判定できるもの。IVRプロパティファイルが同時に渡された場合はクロスチェックも行う。

#### 接続の完全性

- [ ] **全終話TTS（next が空のTTS）の後に Disconnect または Reject が存在するか** — TTSで終わってはいけない
- [ ] **waitモジュールにsave2dbサブモジュールが接続されているか**
- [ ] **TTS/STT/Retry Counter に save2db サブモジュールが接続されているか**
- [ ] **saveCompletionFlag2db の status が空でないか**
- [ ] **saveCompletionFlag2db の smsFlag が空でないか** — 空ではなく `"-1"` または正の数値が必要
- [ ] **status=0 の saveCompletionFlag2db が存在しないか** — status=0はシステム自動送信のため不要

#### パラメータの必須値チェック

- [ ] **DTMF AmiVoice STT Input の max_dtmf_length がデフォルト `"10"` のままでないか** — 入力種別に応じた値が必要
- [ ] **DTMF AmiVoice STT Input の retry が `"0"` のままでないか**
- [ ] **STT の type が氏名聴取モジュールで `"氏名カナ"` になっているか** — モジュール名に「氏名」を含む場合に検査
- [ ] **STT の type が生年月日聴取モジュールで `"日時"` になっているか** — モジュール名に「生年月日」を含む場合に検査
- [ ] **電話番号聴取の save2db に contextName=`"additionalPhoneNumber"` が設定されているか**
- [ ] **氏名聴取の OpenAI に contextName=`"patientName"` が設定されているか**
- [ ] **Phone Normalization モジュールが電話番号聴取フロー内に存在するか**
- [ ] **DEPARTMENTのcontextDisplayTypeが1フロー内で2つ以上使われていないか**

#### IVRプロパティ ↔ フローJSON クロスチェック

- [ ] **JSON内の全 Text to speech モジュール（promptが空）に対応する `{モジュール名}.prompt=` がIVRプロパティに存在するか** — 漏れ検出
- [ ] **IVRプロパティの `{モジュール名}.prompt=` に対応するモジュールがJSON内に存在するか** — ダブり・孤立プロパティ検出
- [ ] **Re-confirmation node data と Speech Retry Counter のpromptがJSON内のparams に直接定義されているか** — これらはIVRプロパティではなくJSON内に記載すべき
- [ ] **IVRプロパティ内に `# [WARNING]` コメントが存在しないか** — 存在する場合は未設定のTTSがある

### 12.2 reviewer エージェントがチェックすべき項目（文脈的・意味的）

フローの意味を理解しないと判定できないもの。

#### 終了ルートの網羅性

- [ ] **全ての終了ルート（時間外、非通知、海外、正常終話等）にsaveCompletionFlag2dbが配置されているか** — フロー全体のパスを追跡して確認
- [ ] **acceptance_times の遷移先が正しいか** — `true`（営業時間内）のみ冒頭アナウンスへ。TIMEOUT/ERROR/`false`（時間外）はすべて時間外アナウンスへ分岐すること

#### 不要なモジュールの検出

- [ ] **OpenAI の contextName で既に格納しているコンテキストを、後段の saveContext2DB で重複格納していないか** — OpenAI側で保存済みなら追加は不要

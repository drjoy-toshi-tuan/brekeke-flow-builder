# モジュール選定ガイド — 「こういう場面でこのモジュールを使う」

> 実フロー2種の完全解析に基づく。
> - **HC厚木型**（旧型）: 1フローに全モジュール147個を詰め込む構成
> - **VN_TS型**（新型）: メイン16モジュール + サブフロー6本に分割する構成
>
> 設計書を書く際に「この場面ではどのモジュール・どの構成が必要か」を判断するためのガイド。

---

## 0. フロー構成の選択：1フロー型 vs サブフロー分割型

### 0.1 構成パターンの比較

| 項目 | HC厚木型（1フロー型） | VN_TS型（サブフロー分割型） |
|---|---|---|
| メインフローのモジュール数 | 147 | 16 |
| サブフロー数 | 1（Jump遅刻のみ） | 6 |
| フロー間遷移 | `@General$Jump to Flow`（2箇所） | `Custom Jump to Flow`（5箇所） |
| 個人情報聴取 | メインフロー内に直書き | `個人情報汎用` サブフローに集約 |
| RAG検索 | メインフロー内にRAGモジュール配置 | `RAG検索` サブフローに分離 |
| 電話番号の固定/携帯判別 | 着信分類のみ（1回） | 個人情報汎用サブフロー内で詳細分岐 |
| 復唱確認 | OpenAI分岐で肯定/否定判定 | 専用モジュール（Re-confirmation, DOB Re-confirmation）使用 |

### 0.2 どちらを選ぶか

#### 必須ルール（新規作成・第2世代→第3世代マイグレーション時）

以下は**必ずサブフローに切り出す**。メインフローに直書きしない。

| 対象 | サブフロー | 理由 |
|---|---|---|
| 患者氏名 | 氏名サブフロー | 汎用パターン。施設間で共通 |
| 生年月日 | 生年月日サブフロー | DOB Re-confirmation含む汎用パターン |
| 電話番号 | 電話番号サブフロー | 携帯/固定判別・Phone Normalization・復唱含む複雑な処理 |
| 診察券番号 | 個人情報汎用サブフロー等 | 汎用パターン |
| RAG検索（FAQ） | RAG検索サブフロー | Dr.JOY側のFAQリストを検索する独立処理 |

> **例外**: 既存フローのシナリオ修正（既に1フロー型で稼働しているフローの部分修正）の場合は、既存構成を維持する。

#### 1フロー型が許容されるケース

- 既存フローのシナリオ修正（構成を変えずに内容だけ修正する場合）

### 0.3 サブフロー分割型のフロー間遷移モジュール

| モジュール | type | 用途 |
|---|---|---|
| **Custom Jump to Flow** | `drjoy^Custom Module$Custom Jump to Flow` | サブフローを呼び出す。完了後にメインフローの次モジュールへ戻る |
| Jump to Flow（旧型） | `@General$Jump to Flow` | HC厚木型で使用。Custom Jump to Flow の前身 |

**Custom Jump to Flow の使い方**:
```
メインフロー:
  患者_氏名 [Custom Jump to Flow]
    flowname = "drjoy^VN_TS SUB FLOW$氏名"
    → (^.*$ | success) → 患者_生年月日   ← サブフロー完了後の戻り先
```

## 1. フローの骨格：必ず登場するモジュール群

### 1.1 開始チェーン（フロー冒頭に必ず配置）

| 順序 | モジュール | type | いつ使うか |
|---|---|---|---|
| ① | 冒頭（wait） | `Custom$wait` | **全フロー必須**。着信直後の安定待機（2秒） |
| ② | コンテキスト設定 | `saveContextModel2DB` | **全フロー必須**。Dr.JOY画面の項目定義。waitの直後 |
| ③ | 着信電話番号分類 | `incoming-classifier` | **全フロー必須**。非通知・海外をここで弾く |
| ④ | 営業時間チェック | `acceptance_times` | **営業時間制御が必要なフローで使う**。不要なら省略し、③から直接⑤へ |
| ⑤ | 冒頭アナウンス | `Text to speech` | **全フロー必須**。施設名と自動応答である旨を案内 |

#### 開始チェーンのフロー図

```
冒頭(wait)
  → コンテキスト設定(saveContextModel2DB)
  → 着信電話番号分類(incoming-classifier)
      ├─ 非通知 → 非通知案内(TTS) → フラグ(saveCompletionFlag2db) → 切断
      ├─ 海外   → 海外案内(TTS) → フラグ(saveCompletionFlag2db) → 切断
      └─ 固定/携帯/その他 → [営業時間チェック →] 冒頭アナウンス → 用件確認へ
```

- 非通知・海外は**コンテキスト設定の直後、営業時間チェックより前**に弾く
- 固定・携帯・その他は区別せずそのまま下のフローに進む
- 携帯/固定の判別は、後段の電話番号聴取ステップで改めて行う（セクション3.5参照）
- WebRTC 分岐は `incoming-classifier` に常時生成されるが、WebRTC 未採用施設では固定・携帯と同じく通常フローに合流させる（セクション1.3 参照）

```
判断基準（営業時間チェック）:
- 24時間対応 → 不要（省略）
- 受付時間がある → 必要
- 受付時間は Dr.JOY 側で制御 → 不要（省略可。ただし時間外アナウンスを自前で出したい場合は必要）
```

**設計書で明記すべきこと**:
- 営業時間チェックの有無

### 1.3 WebRTC 着信対応（Pattern 5・WebRTC 専用モジュール）

ブラウザ（Dr.JOY 受付画面など）から WebRTC 経由で発信されたコールを扱う場合に使う特殊モジュール群。電話系着信（固定・携帯・非通知・海外）しか扱わない施設では使用しない。

#### 使用モジュール

| モジュール | type | 用途 |
|---|---|---|
| `get-header` | `drjoy^Incoming$get-header` | SIP の `X-UA-EX` ヘッダからブラウザ側の context（氏名・診察券番号等）を取り出し save2db に投入 |

#### 配置ルール

```
incoming-classifier
  ├─ 非通知 → 非通知案内 → 切断
  ├─ 海外   → 海外案内 → 切断
  ├─ WebRTC → get-header → （個人情報聴取サブフロー省略して本題へ）
  └─ 固定/携帯/その他 → 営業時間チェック → 冒頭アナウンス → 本題
```

- `incoming-classifier` の **WebRTC ラベル直後**に `get-header` を 1 個だけ置く
- `get-header` の後は「個人情報聴取サブフロー（氏名・生年月日・電話番号・診察券番号）を省略して用件聴取に進む」構成が典型（ブラウザ側で既に入力済みのため）
- ブラウザ側が送る JSON の `contextName` / `value` / `displayType` は個別施設の設計で合意すること（必須項目の合意が必要）

#### いつ選ぶか

- **採用する**: WebRTC 発信元（ブラウザ UI）から context を引き継ぎ、聴取の一部または全部を省略したい施設
- **採用しない**（デフォルト）: 電話系着信のみの施設。`incoming-classifier` の WebRTC 分岐は `regular_next` に合流で OK

> 2026-04 時点で採用施設はまだない（Pattern 5 は構想段階）。施設が決まった時点で、scaffold_generator 側の自動配置を有効化する。詳細仕様は `docs/brekeke/モジュール詳細設定ガイド_1.md §6.4 get-header` 参照。

### 1.4 電話帳ベースの送信元分類（incoming-category-classifier + Phone2Name）

着信電話番号を Dr.JOY 電話帳と突合し、**発信元の所属カテゴリで分岐 + 名乗りを自動 TTS** したいケースで使う。`incoming-classifier`（§1.1 ③）が「非通知/海外/固定/携帯」という回線種別で分岐するのに対し、こちらは「定型取引先・薬局・連携施設など、ドメイン側のカテゴリ」で分岐する。

#### いつ選ぶか

- **採用する**:
  - 薬局からの疑義照会受付（薬局名を発信元番号から判定して聴取スキップ）
  - 医療連携室への他施設からの紹介・問合せ（紹介元施設を判定）
  - 定型取引先からの問合せ（取引先カテゴリで応対を分ける）
  - 共通条件: **発信元の電話番号が Dr.JOY 電話帳に事前登録される運用が成り立つ**こと（少数の定型先からの入電が多数を占める業務）
- **採用しない**（デフォルト）:
  - 患者本人からの入電が主体のシナリオ（電話番号と本人の紐付けが不確実、登録運用が回らない）
  - 発信元が事前登録できない不特定多数からの入電

#### 使用モジュール

| モジュール | type | 用途 |
|---|---|---|
| `incoming-category-classifier` | `drjoy^Incoming$incoming-category-classifier` | 電話帳マッチで「リスト1〜5/ブラックリスト/NO_RESULT」を返す |
| `Phone2Name` | `drjoy^External Integration$Phone2Name` | ヒットしたエントリのフリガナを TTS で読み上げる（`recipient_name` プレースホルダ） |

#### 配置ルール

```
冒頭(wait)
 → コンテキスト設定(saveContextModel2DB)
 → incoming-classifier
     ├─ 非通知/海外 → 既存どおり切断
     └─ 通常着信 → [acceptance_times（任意）]
        → 冒頭_アナウンス
        → incoming-category-classifier
           ├─ ^リスト1$ → Phone2Name（カテゴリ A 向け TTS）→ 担当者名聴取
           ├─ ^リスト2$ → Phone2Name（カテゴリ B 向け TTS）→ 担当者名聴取
           ├─ ^BLACKLIST$ → 拒否案内 → 切断（任意）
           └─ ^NO_RESULT$ / ^TIMEOUT$ / ^ERROR$ → 発信元名聴取（hearing）→ 担当者名聴取
```

- `incoming-classifier` で**回線種別の非通知/海外を弾いた後**に `incoming-category-classifier` を配置する（順序固定）
- 通常時は acceptance_times の後ろ、冒頭アナウンスの後ろが推奨
- 検出失敗（NO_RESULT / TIMEOUT / ERROR）は hearing で発信元名を聞く**フォールバック経路を必ず残す**こと（電話帳の登録漏れ・障害時の経路）
- 「○○薬局さんですね。」のような名乗り発話は **Phone2Name の `FOUND_KATAKANA_NAME_DEFAULT_TMP`** で `recipient_name` プレースホルダ経由で出す。リテラル文字列で薬局名を入れた TTS モジュールを薬局数だけ並べる方式は推奨しない（電話帳変更で TTS 群の更新が必要になる）

#### Dr.JOY 側設定（人間作業）

- 対象先（薬局・連携施設等）の電話番号・氏名・フリガナ・所属リスト（リスト1〜5 のどれか）を **Dr.JOY 電話帳に登録**する
- カテゴリの粒度（リスト1 を何用に使うか）は施設側との合意事項

#### 参考実装

- 実装例の bivr: `docs/reference/incoming_category_examples/sample_松本協立病院相談室.bivr`（Phone2Name の `施設確認` モジュール、2 フローに登場）
- 電話帳 CSV 見本: `docs/reference/incoming_category_examples/sample_phonebook_相談室.csv`
- 解説: `docs/reference/incoming_category_examples/README.md`

#### 設計書 YAML 記述（scaffold 対応済、2026-05-11〜）

`scenario_flow` 内のブロック型として `incoming_category_classifier` と `phone2name` を**直接書ける**（scaffold_generator が `drjoy^Incoming$incoming-category-classifier` / `drjoy^External Integration$Phone2Name` を生成）:

```yaml
- step: 薬局カテゴリ判定
  type: incoming_category_classifier
  conditions:
    - match: "リスト1"        # 薬局カテゴリにヒット
      next: 薬局名読み上げ
    - match: "BLACKLIST"
      next: END_拒否案内
    - match: "NO_RESULT"     # 電話帳未登録
      next: 薬局名聴取        # フォールバック
    - match: "TIMEOUT"
      next: 薬局名聴取
    - match: "ERROR"
      next: 薬局名聴取

- step: 薬局名読み上げ
  type: phone2name
  found_template: "recipient_name様、お電話ありがとうございます。"
  not_found_template: ""           # 空ならスキップして遷移
  next_found: 担当者名聴取          # ^TRUE$ ヒット時
  next_failure: 薬局名聴取          # TIMEOUT/NO_RESULT/ERROR フォールバック
```

`incoming_category_classifier` の `conditions` の match 値（リスト1 / BLACKLIST 等）は Dr.JOY 電話帳と Brekeke のラベル仕様に依存するため、実機検証で必要に応じて調整する。`phone2name` の `recipient_name` は Brekeke が電話帳のフリガナを動的差込する固定プレースホルダ（変更不可）。

> **migration note**: 2026-05-11 より前は `type: augment` + `augment_pattern: new_module` で placeholder Script を生成 → Brekeke UI で手動置換していた。現在は専用ブロック型として scaffold が直接生成するため、手動置換不要。

### 1.2 終了チェーン（フロー末尾に必ず配置）

新規作成・マイグレーション時は、携帯と固定/その他で終話TTS・SMS送信を分ける。

#### 終了チェーンのフロー図

```
[最後の聴取ステップ完了]
    ↓
completeflag_SMSflag [saveCompletionFlag2db: status=1, smsFlag=1]
    ↓
終話分岐_電話番号 [ContextMatchRouter: 「電話番号聴取」モジュールを参照（モジュールが context 保存した phonetype を読む）]
    ├─ 携帯（phonetype=1） → 終話分岐用 [Script: 用件を判定]
    │     ├─ 予約     → 終話_新規予約(TTS) → 切断  ← SMS送信あり
    │     ├─ 変更     → 終話_変更(TTS) → 切断
    │     ├─ キャンセル → 終話_キャンセル(TTS) → 切断
    │     └─ 確認     → 終話_確認(TTS) → 切断
    │
    └─ 固定/その他（phonetype=2） → 終話分岐固定電話 [Script: 用件を判定]
          ├─ 予約     → 終話_固定予約(TTS) → 切断  ← SMS送信なし
          ├─ 変更     → 終話_固定変更(TTS) → 切断
          ├─ キャンセル → 終話_固定キャンセル(TTS) → 切断
          └─ 確認     → 終話_固定確認(TTS) → 切断
```

#### 構成のポイント

| 要素 | 説明 |
|---|---|
| `saveCompletionFlag2db` | 通話完了ステータス(status)とSMSフラグ(smsFlag)をセット |
| `終話分岐_電話番号` | ContextMatchRouter で「電話番号聴取」モジュール（subflow ステップ名）を `module1Name` に指定し、Brekeke がそのモジュールの context 保存値（phonetype）を読んで携帯 or 固定/その他に分岐。**`reference_module` には必ずモジュール名を書く。context 名（`phonetype` 単体等）の指定は禁止**（director.md 参照） |
| `終話分岐用` / `終話分岐固定電話` | Scriptモジュールで用件（予約/変更/キャンセル/確認）を判定し、対応するTTSに遷移 |
| 携帯用TTS | 「SMSで詳細をお送りします」等の案内を含む |
| 固定用TTS | SMSに言及しない案内（固定電話はSMSを受け取れないため） |

#### 携帯 vs 固定でのTTS・SMS出し分け

| 項目 | 携帯ルート | 固定/その他ルート |
|---|---|---|
| TTS発話 | 「ショートメッセージで詳細をお送りします」 | SMS送信に言及しない案内 |
| SMS送信 | あり（smsFlag=1） | なし（固定はSMS受信不可） |
| 終話TTS名 | `終話_{用件名}` | `終話_固定{用件名}` |

> **重要**: 固定電話・その他はSMSを受信できないため、終話ガイダンスでSMS送信に言及してはいけない。
> phonetype は個人情報聴取サブフロー内の incoming-classifier + saveContext2DB で設定される（セクション3.5参照）。

## 2. 会話ステップ：聴取項目ごとに登場するモジュール群

### 2.1 基本構成（5モジュール1セット）

```
TTS（質問） → STT（音声入力） → OpenAI（正規化/分岐） → Retry（リトライ） → save2db（録音保存）
```

サブフロー分割型では、このセットがサブフロー単体として切り出される。

### 2.2 復唱確認が入る場合の拡張構成

**生年月日の復唱**（DOB Re-confirmation使用）:
```
TTS → STT → DOB Re-confirmation → 復唱STT → 復唱OpenAI → Retry
              │
              ├─ success → 復唱STTへ（生年月日を読み上げて確認）
              ├─ INVALID → リトライへ
              └─ timeout/error → リトライへ
```

**電話番号の復唱**（Phone Normalization + Re-confirmation node data使用）:
```
TTS → STT → Phone Normalization → Re-confirmation → 復唱STT → 復唱OpenAI → Retry
                                     │                                        │
                                     └─ 電話番号を読み上げて確認 ←────── 否定で戻る
```

### 2.3 STTモジュールの選択

| 場面 | 使うモジュール | type |
|---|---|---|
| ダイヤル入力を受け付ける項目 | DTMF AmiVoice STT Input | `drjoy^External Integration$DTMF AmiVoice STT Input` |
| 音声のみの項目 | AmiVoice Speech to Text | `drjoy^AmiVoice$Speech to Text` |

```
判断基準:
- 生年月日 → DTMF（8桁数字入力あり）
- 電話番号 → DTMF（数字入力あり）
- 診察券番号 → DTMF（数字入力あり）
- 復唱確認（はい/いいえ + DTMF 1/2） → DTMF
- 用件確認（「1を押してください」方式） → DTMF
- 用件確認（「予約とおっしゃってください」方式） → AmiVoice
- 氏名、診療科 → AmiVoice
- その他自由発話 → AmiVoice
```

### 2.4 OpenAIモジュールの種別選択

#### (A) 分岐型 — 回答によって次のステップが変わる

**使う場面**:
- 用件確認（予約/変更/キャンセル/...）
- はい/いいえ確認（肯定→次へ、否定→やり直し）
- 選択肢確認（受診歴、変更内容、コース確認など）
- 特定キーワード検出（遅刻・雇用時健診などの特殊ルート振り分け）
- 電話番号の復唱確認（肯定/否定）

#### (B) 正規化型 — 回答をフォーマット変換して保存するだけ

**使う場面**:
- 氏名のカタカナ変換
- 生年月日の日付変換（DOB Re-confirmationが正規化を担当する場合はOpenAIは復唱確認のみ）
- 電話番号の正規化（Phone Normalizationが正規化を担当する場合はOpenAIは復唱確認のみ）
- 日付の変換（予約日、変更希望日）
- 自由テキストの要約（問い合わせ内容）
- 診察券番号の正規化

### 2.4.1 出力フォーマット（output_format）の判定ルール

聴取項目ごとに、コンテキストに保存する値のフォーマットを以下の3分類で判定する。この分類により、OpenAIモジュールの要否と役割が機械的に決まる。

| output_format | コンテキスト保存形式 | OpenAI の要否・役割 | next 分岐 |
|---|---|---|---|
| **datetime** | `YYYY-MM-DD HH:MM`（半角） | 必須: 日時正規化 | 不要（単線） |
| **text** | 自由テキスト | 不要（そのまま保存） | 不要（単線） |
| **enum** | 定義済み選択肢のいずれか | 必須: 分岐判定 | 必要（多分岐） |

#### 判定基準: TTSアナウンス文から判断する

| TTSの問いかけ | 入力例 | output_format |
|---|---|---|
| 「日付や曜日をお話しください」 | 4月15日、来週の火曜日 | **datetime** |
| 「何時ごろがよろしいですか」 | 10時、午前中 | **datetime** |
| 「希望時期を教えてください」 | 来月上旬、なるべく早く | **text** |
| 「症状をお話しください」 | 頭が痛い、咳が出る | **text** |
| 「診療科をお話しください」 | 内科、整形外科 | **enum** |
| 「ご用件をお話しください」 | 予約、変更、キャンセル | **enum** |

**原則**: 具体的な値に変換できる問いかけ → 正規化が必要（datetime/enum）。曖昧な回答を許容する問いかけ → テキスト保存（text）。

#### Gen2→Gen3 移管時の判定

Gen2 の function スキーマから判定可能:
- `enum` フィールド → **enum**
- 日付・時刻を格納するフィールド（reservationDate 等）→ フィールドの description と TTS 文言から **datetime** か **text** を判断
- 自由記述フィールド（reason, symptoms, question 等）→ **text**

> **datetime の制約**: `YYYY-MM-DD HH:MM` 形式でないとコンテキストに入力できない。「来月の上旬」等の曖昧表現は datetime では受け付けられないため、曖昧入力を許容する場合は必ず text にする。

### 2.5 診療科の no_result ハンドリング（特殊パターン）

診療科を聴取する STT モジュールおよび OpenAI モジュールにおいて、**no_result**（認識結果なし）が発生した場合、`saveContext2DB` で値を `"登録なし"` にセットしてからリトライカウンターに遷移する。

```
TTS_診療科 → STT_診療科 → OpenAI_診療科 → リトライ_診療科
                │ no_result      │ no_result        ↑
                ↓                ↓                  │
          saveContext2DB     saveContext2DB ─────────┘
          (value="登録なし")  (value="登録なし")
```

**背景**:
- Dr.JOY 側で「ノーリザルト」という出力値を判定できないため、そのまま出力されると「削除レコード」に分類されてしまう
- error / timeout の場合は Dr.JOY 側で自動的に「登録なし」相当に分類されるため、対処不要
- **no_result のみ**がこのパターンに該当する
- 通常はリトライが介在するため no_result がそのまま出力されることは稀だが、ゼロではない

> **適用範囲**: 診療科の聴取ステップのみ。他の聴取項目（用件確認、予約日、氏名等）には不要。

### 2.6 リトライ失敗時のフォールバック先

#### デフォルト動作: 正解ルートの次のモジュールに進む

リトライの `No more`（false）先は、**そのステップの正解ルートで次に進むはずだったモジュール**と同じ遷移先にする。つまり、聴取に失敗しても「聴取できなかった」状態のままフローを先に進める。

```
例: 氏名ステップ
  正解ルート: 入力-氏名 → OpenAI-氏名 → (success) → 生年月日
  リトライNo more先: → 生年月日（正解ルートの次と同じ）

例: 希望コース
  正解ルート: OpenAI-希望コース → (通常) → 追加オプション
  リトライNo more先: → 追加オプション（正解ルートの次と同じ）
```

**generatorへの指示**: 設計書に特別な記載がない限り、全てのリトライの `No more` 先は「そのステップの正解ルートで次に遷移するモジュール」に設定する。

#### 例外: 設計書（指示書）で別途定義されている場合

設計書や修正指示書で「リトライ失敗時は〇〇へ」と明示されている場合のみ、デフォルトと異なるフォールバック先を設定する。

よくある例外パターン:

| パターン | 動作 | 具体例 |
|---|---|---|
| 切断 | リトライ失敗で通話終了 | 「用件が聴取できなければ通話を継続する意味がない」と定義された場合 |
| 着信番号自動格納 | saveContext2DBで着信番号を保存して次へ | 「電話番号聴取に失敗したら着信番号を使う」と定義された場合 |
| 特定ステップへジャンプ | デフォルトと異なるモジュールへ | 設計書で明示的に指定された場合 |

> **重要**: これらの例外はあくまで設計書の指示に従うもの。generatorが独自判断で「この項目は致命的だから切断にしよう」と判断してはいけない。

### 3.1.1 氏名聴取の分岐ルール（PatientName サブフローは1枠のみ）

**PatientName サブフロー（氏名聴取サブフロー）は 1 フロー当たり最大 1 本**。複数の氏名聴取項目がある場合は以下の優先順位で 1 本の PatientName サブフローに充当し、残りはインラインの hearing ブロックで処理する。

**充当ルール**:

1. **患者氏名がある場合**
   - 患者氏名 → PatientName サブフロー（`type: subflow` + `flowname: "{group_name}$氏名聴取_{date}"` + `target: "氏名聴取"`）
   - (a) 他に氏名聴取項目（入電者氏名・受診者氏名・担当者名・付き添い者名など）があれば **hearing ブロック**（`type: hearing` + `save_to: callerName / guardianName / staffName` 等）で対応
2. **患者氏名がない場合**
   - (a) 入電者氏名 または 受診者氏名 がある → これらを PatientName サブフローに充当（save_to も patientName を借用）
   - (b) どちらもなければ通常の氏名聴取サブフロー（PatientName 型でない汎用氏名聴取）を使う

**書き方の例（足利中央病院_地域連携 等で実績）**:

```yaml
# scenario_flow
  - step: 施設名聴取          # 会社・医療機関名 → 氏名ではないので別 context
    type: hearing
    save_to: organizationName
    output_format: text

  - step: 入電者氏名聴取        # 患者氏名ではないので hearing（サブフロー化禁止）
    type: hearing
    save_to: callerName
    output_format: text

  - step: 患者氏名聴取          # 患者氏名は必ずサブフロー経由
    type: subflow
    flowname: "{group_name}$氏名聴取_{date}"
    target: 氏名聴取

# flow_structure.subflows[] には 氏名聴取 1 本のみ登録
```

**禁止事項**:

- `flow_structure.subflows[]` に「入電者氏名聴取」「担当者氏名聴取」「受診者氏名聴取」等の非患者氏名サブフローを個別登録すること
- `save_to: patientName` の hearing ブロックをインライン配置すること（必ずサブフロー経由）
- 会社名・事業所名を「氏名」扱いすること（`companyName` / `organizationName` 等の別 context で受ける）

**機械的検証**:

- qa_validator.py の `E-12` が以下を検出する（違反は CRITICAL で director 差し戻し）:
  - `flow_structure.subflows[]` に「氏名聴取」を含むエントリが 2 本以上登録
  - `save_to: patientName` の hearing ブロックがインライン配置（subflow 経由でない）
- copy_subflows.py は「氏名聴取」を含む未知 target を検出するとエラー終了（誤登録を機械的に弾く）

### 3.5 電話番号聴取時の携帯/固定判別パターン

電話番号聴取のステップで `incoming-classifier` を使い、着信元の種別に応じて処理を分岐する。

> **非通知・海外は冒頭の開始チェーン（セクション1.1）で既に弾いているため、ここには到達しない。**

```
着信電話番号分岐 [incoming-classifier]
    │
    ├─ 携帯 → save-携帯電話 [saveContext2DB: additionalPhoneNumber ← 着信番号]
    │           → 電話番号正規化 [Phone Normalization]
    │           → 患者_携帯 [Re-confirmation: 「今おかけいただいている〇〇でよろしいですか？」]
    │           → 入力_患者_携帯電話 [DTMF STT]
    │           → openAI_患者_携帯電話 [OpenAI分岐型: 肯定/否定]
    │               ├─ 肯定 → 携帯電話判別 [saveContext2DB: phonetype=携帯] → 次のステップへ
    │               └─ 否定 → 患者_連絡先（手入力ルートへ合流）
    │
    ├─ 固定 → 患者_連絡先（手入力ルート）
    │
    └─ その他 → 患者_連絡先（固定と同じ手入力ルート）

手入力ルート:
    患者_連絡先 [TTS: 「連絡先電話番号をお知らせください」]
        → 入力_患者_連絡先 [DTMF STT]
        → 電話番号正規化 [Phone Normalization]
        → 復唱_患者_連絡先 [Re-confirmation: 「電話番号は〇〇でよろしいですか？」]
        → 入力_患者_復唱連絡先 [DTMF STT]
        → openAI_患者_復唱連絡先 [OpenAI分岐型: 肯定/否定]
            ├─ 肯定 → script_携帯判別 → 携帯/その他 [saveContext2DB: phonetype]
            └─ 否定 → 患者_連絡先（再入力）
```

**3分岐のまとめ**:

| 着信種別 | 処理 | 結果 |
|---|---|---|
| 携帯 | 着信番号を自動格納→「この番号でよいか？」復唱確認 | phonetype=携帯 |
| 固定 | 連絡先を音声/DTMF聴取→正規化→復唱確認 | phonetype=その他 |
| その他 | 固定と同じ処理 | phonetype=その他 |

**ポイント**:
- 携帯着信なら手入力不要（着信番号を自動格納して復唱確認のみ）
- 固定・その他なら連絡先を聴取→正規化→復唱確認
- 最終的に `phonetype` コンテキストに携帯/その他がセットされ、SMS送信可否の判断に使える

### 3.8 Script（スクリプト）— カレンダー計算専用

| モジュール | type | 用途 |
|---|---|---|
| Script | `@General$Script` | **営業日・祝日・施設固有休館日などカレンダーデータを使う計算のみ** |

> **🔴 Script は「カレンダー計算」でしか使わない。それ以外は必ず OpenAI + ContextMatchRouter に置き換える。**

#### Script を使う唯一のケース（= カレンダー必須）

| パターン | 理由 |
|---|---|
| N 営業日後の日付を計算 | 祝日・土日の除外は LLM では不正確 |
| 指定日が営業日か判定 | 同上 |
| 2 日の間の営業日数カウント | 同上 |
| 施設固有休館日を考慮した判定 | カレンダーデータが必要 |

既存テンプレートで表現できる場合は `script_template` に指定する（`future_date` / `day_of_week` / `business_hours` / `phone_type` / `condition_group`）。新規のカレンダー計算で既存テンプレに当てはまらない場合は、**専用プリミティブ追加**が必要なので人間チームに相談する。

#### Script に逃げがちなアンチパターン（やってはいけない）

以下はすべて **OpenAI + ContextMatchRouter で表現可能**。Script で組んだ場合、qa_validator E-13 が CRITICAL で差し戻す：

| アンチパターン | 正しい実装 |
|---|---|
| 年齢分岐（15歳未満/16歳以上 等） | OpenAI で判定 → context に保存 → 後段 CMR で参照 |
| 用件×電話種別の多段分岐 | `classification` と `phonetype` を個別 context に保存 → CMR を 2 段ネスト |
| 3 変数以上の組み合わせ分岐 | 各 context を個別保存 → CMR の多段ネスト、もしくは用件ごとに独立 flow |
| 未来日判定（`2026-04-30` は未来か） | OpenAI に `addCurrentDate: Yes` → プロンプト内で今日と比較 |
| 年度末前判定（3/31 までか） | 同上、OpenAI プロンプトに固定範囲を含める |
| 最短 N 日後判定 | 同上、OpenAI プロンプトに N を含めて比較 |
| 文字列一致 / 含有判定 | OpenAI の分類で直接判定 → context 保存 → CMR |
| 電話番号の SSML 変換 | gen_properties / IVR プロパティ側の責務 |

> **備考**: 旧ガイドでは「ContextMatchRouter は 1 モジュール結果しか参照できない」と記載していたが、CMR の連鎖で複数 context を参照できるため、多段分岐は CMR で組める。

#### 設計書での書き方

Script ブロックを設計書に含める場合、以下のいずれかに該当すること：

1. `script_template` に **既存テンプレート名を明示指定**（`future_date` / `day_of_week` / `business_hours` / `phone_type` / `condition_group`）
2. `script_template: custom` + `notes` に **カレンダー計算が必要な理由** を明記（「営業日」「祝日」「休館日」のいずれかに言及）

**理由が書かれていない `custom` や `script_template` 未指定は qa_validator E-13 で CRITICAL → director に差し戻し。**

#### Script で計算した値を TTS で読み上げる場合（動的値の埋め込み）

Script の計算結果（最短予約日・年度末日付・休館日案内 等）を TTS prompt 内に埋め込みたい場合は、**Brekeke 公式の `<% context_name %>` 形式のみ使用可**。

| 構文 | Brekeke の動作 | 用途 |
|---|---|---|
| `<% desired_date_jp %>` | TTS 直前に context 値で置換 | ✅ 動的値の埋め込み |
| `<% sys-customer-phone-number %>` | システム context（発信者番号）で置換 | ✅ 既存施設実績あり (すずな皮ふ科 / がん研有明) |
| `<{DESIRED_DATE}のM月D日>` | **置換されずリテラル読み上げ** | ❌ 旧 OpenAI Assistant 仕様、Brekeke 非対応 |
| `＜{DESIRED_DATE}のM月D日＞` | **同上** | ❌ 同上、全角括弧版 |
| `${VAR}` | **同上** | ❌ Bash/JS 風記法、Brekeke 非対応 |

**設計手順（動的値が必要な場合）**:

1. Script モジュール内で表示用 context を `saveContext` で書き込む（例: `desired_date_jp = "5月10日"`、`fiscal_end_jp = "3月31日"`）
2. 設計書の `tts_announcement` で `<% desired_date_jp %>` のように参照する
3. customer_doc に書かれている `<{VAR}の表示形式>` テンプレートはそのまま転記しない（Brekeke が処理しない）

**固定値の場合は直書き**: `FISCAL_END_DATE = "2027-03-31"` のように施設で固定の日付は context 化せず、TTS テキストに「2027年3月31日まで」と直接書く。

**機械検出**: `qa_validator E-15` で偽プレースホルダー（`<{...}` `＜{...` `${...}`）を CRITICAL 検出。`<% ... %>` 形式は除外。

> **⚠️ 動作未検証**: `<% context_name %>` での Script 結果埋め込みは、すずな皮ふ科 / がん研有明での `<% sys-customer-phone-number %>` 等のシステム変数利用実績はあるが、**Script ブロックが saveContext した独自 context での動作は実機検証が必要**。本番投入前に音声出力で「リテラル読み上げになっていないか」確認すること。

### 3.9 スマート面会専用モジュール群（特殊シナリオ）

> **スマート面会はDr.JOY上に実装された面会予約機能。**
> Dr.JOYとのデータ連携（予約枠の確認・予約登録・患者情報のセッション保存）が必要なため、専用モジュールを使った特殊なフローになる。
> 通常の診療予約・健診シナリオとは構成が大きく異なることを認識すること。

#### スマート面会専用モジュール一覧

| モジュール | type | 用途 |
|---|---|---|
| visitation-info | `drjoy^Smart Menkai$visitation-info` | 面会マスター情報（病棟・面会枠設定）をDr.JOYから取得 |
| check-frame-availability | `drjoy^Smart Menkai$check-frame-availability` | 指定病棟の面会枠が予約可能かチェック |
| booking-pre-check | `drjoy^Smart Menkai$booking-pre-check` | 希望日時の予約枠を事前チェック（空き/満員/退院済み/当日重複） |
| booking-get-next-time | `drjoy^Smart Menkai$booking-get-next-time` | 次に空いている面会時間枠を取得 |
| booking-visitation | `drjoy^Smart Menkai$booking-visitation` | Dr.JOYに面会予約を登録 |
| saveNodeData2Session | `drjoy^Persistence$saveNodeData2Session` | 聴取データをセッションに保存（Smart Menkaiモジュールが参照） |

#### saveNodeData2Session の役割

通常の `saveContext2DB` とは異なり、**セッション上にキー=値を保存**する。スマート面会専用モジュールはこのセッションデータを参照して動作する。

| key | 保存内容 | 参照先 |
|---|---|---|
| `frame-title` | 病棟名 | check-frame-availability |
| `patient-name` | 患者氏名 | booking-visitation |
| `visitor-name` | 面会者氏名 | booking-visitation |
| `relationship` | 続柄 | booking-visitation |
| `booking-start-time` | 面会希望日時 | booking-pre-check |
| `number-of-visitors` | 訪問人数 | booking-visitation |
| `memo` | その他問い合わせ内容 | booking-visitation |

#### スマート面会フローの典型的な流れ

```
冒頭 → 用件確認（面会/呼び出し）
  → 患者_病棟 [STT→OpenAI] → saveDepartment2Session [saveNodeData2Session]
  → check-frame-availability [予約枠が存在するか確認]
      ├─ CAN_BOOK → 患者_氏名 [STT] → save-patient-name [saveNodeData2Session]
      ├─ CANNOT_BOOK → 病棟エラー案内
      └─ ERROR → リトライ
  → 患者_面会者氏名 [STT] → save-visitor-name
  → 患者_続柄 [STT→OpenAI] → save-relationship
  → 予約_面会希望日 [STT→OpenAI] → save-booking-start-time
  → booking-pre-check [希望日時の枠を事前チェック]
      ├─ CAN_BOOK → 面会日時保存 → 訪問人数聴取 → booking-visitation [Dr.JOYに予約登録]
      ├─ CANNOT_BOOK → booking-get-next-time [次の空き枠を取得] → 代替案内
      ├─ EXISTED_BOOKING_IN_THIS_DAY → 「本日は既に予約済み」案内
      └─ PATIENT_DISCHARGED → 「退院済み」案内
  → booking-visitation [予約登録]
      ├─ Success → 完了フラグ → 終話
      └─ Error/Failed/Patient discharged → 来院案内
```

**設計書で明記すべきこと**: スマート面会シナリオの場合、通常の設計書テンプレートではなく、面会固有の聴取項目（病棟・面会者氏名・続柄・面会希望日・訪問人数）とSmart Menkaiモジュールの遷移を設計する。

## 4. RAG検索サブフローのパターン

### 4.1 構成

```
相談_問合せ [TTS: 「お問い合わせ内容をお話しください」]
    → 入力_相談_問合せ [STT]
    → openAI_相談_問合せ [OpenAI: 問い合わせ内容を正規化・分類]
        → rag-question [RAG: Dr.JOY側のFAQリストを検索]
            ├─ success → 相談_問合せループ [TTS: RAG回答を読み上げ] → 入力_相談_問合せ（ループ）
            ├─ no result → 相談_FAQ失敗 [TTS: 「回答が見つかりませんでした」]
            └─ timeout/error → リトライ
    → リトライ_相談_問合せ [Retry]
        ├─ Retry → 相談_問合せ（再質問）
        └─ No more → 終話_失敗 → 切断
```

**ポイント**:
- STT → OpenAI → RAG の3段構成（OpenAIで問い合わせ内容を整形してからRAGに渡す）
- RAG成功時はループして次の質問を受け付ける
- RAGはDr.JOY側のFAQリストを検索する（`drjoy^External Integration$RAG` を使用）
- RAG失敗時は「回答が見つかりませんでした」で終話（切断はしない ← サブフロー内の `相談_FAQ失敗` は next が空）

**設計書で明記すべきこと**: RAGを使うか、使う場合はFAQの対象範囲

## 5. 判断フローチャート：設計書を書くときの意思決定

### Step 1: フロー構成を決める

```
Q: 新規作成 or 第2世代→第3世代マイグレーションか？
  ├─ Yes → サブフロー分割型（個人情報・RAGは必ずサブフロー）
  └─ No（既存フローのシナリオ修正） → 既存構成を維持
```

### Step 2: 開始チェーンを決める

```
開始チェーン（固定）:
  冒頭(wait) → コンテキスト設定 → 着信電話番号分類（非通知・海外を弾く）
    → [営業時間チェック（必要な場合）] → 冒頭アナウンス

Q: 営業時間チェックが必要か？ → acceptance_times の有無を決める
```

### Step 3: 各聴取項目のモジュール構成を決める

```
Q1: ダイヤル入力を受け付けるか？ → STT種別
Q2: 回答で分岐するか？ → OpenAI型（分岐/正規化）
Q3: 復唱確認が必要か？
  ├─ 生年月日 → DOB Re-confirmation
  ├─ 電話番号 → Phone Normalization + Re-confirmation
  └─ その他 → 不要（または Re-confirmation node data）
Q4: リトライ失敗時のフォールバック先は？
    → デフォルト: 正解ルートの次のモジュール（設計書に記載不要）
    → 例外がある場合のみ設計書に明記
Q5: functionCall が必要か？ → functionCall有無
```

### Step 4: 電話番号処理を決める

```
電話番号聴取ステップでは incoming-classifier で3分岐する（セクション3.5参照）:
  ├─ 携帯 → 着信番号を自動格納→復唱確認
  ├─ 固定 → 連絡先を手入力→正規化→復唱確認
  └─ その他 → 固定と同じ処理
※ 非通知・海外は冒頭で既に弾いているためここには到達しない
```

### Step 5: 終了チェーンを決める

```
終了チェーン（固定パターン）:
  completeflag → 終話分岐_電話番号（phonetypeで携帯/固定を分岐）
    ├─ 携帯 → Script（用件判定） → 用件別TTS（SMS言及あり） → 切断
    └─ 固定/その他 → Script（用件判定） → 用件別TTS（SMS言及なし） → 切断

Q: 用件が何種類あるか？ → 携帯用TTS × N本 + 固定用TTS × N本 が必要
```

## 6. 設計書テンプレートへの反映提案

### 聴取項目一覧テーブルの拡張

| # | 聴取項目 | 入力方式 | OpenAI型 | 復唱 | 分岐先 | リトライ失敗時 | functionCall |
|---|---|---|---|---|---|---|---|
| 1 | 用件 | 音声 | 分岐型 | なし | 予約→受診歴 / 変更→... | ※切断（指示書で定義） | なし |
| 2 | 受診歴 | 音声 | 分岐型 | なし | 初回→コース / 複数回→前回確認 | （デフォルト: コースへ） | あり |
| 3 | 生年月日 | 音声+DTMF | 正規化型 | DOB Re-confirmation | — | （デフォルト: 電話番号へ） | — |
| 4 | 電話番号 | 音声+DTMF | 分岐型 | Phone Norm + Re-conf | 肯定→次 / 否定→再入力 | ※着信番号自動格納（指示書で定義） | あり |

> **リトライ失敗時の列**: デフォルト（正解ルートの次へ進む）の場合は記載不要、または括弧書きで参考記載。
> 設計書で例外を定義する場合のみ「※」付きで明記する。

### 新設セクション

| セクション | 内容 |
|---|---|
| フロー構成 | 1フロー型 / サブフロー分割型、サブフロー一覧 |
| 営業時間制御 | acceptance_times の有無、時間外アナウンス |
| 終了パターン一覧 | 全終了ルートの status/smsFlag |
| RAG設定 | RAGの有無、FAQの対象範囲 |
| サブフロー一覧 | 各サブフローの役割と呼び出し元 |

> **着信分類・電話番号処理は固定パターン**: 非通知・海外は冒頭で弾く、電話番号聴取で携帯/固定/その他を3分岐する、という構成は全フロー共通のため、設計書に個別セクションを設ける必要はない。

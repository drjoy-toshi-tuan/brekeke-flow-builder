> 出典: フローデザイナー命名規則シート + 第三世代モジュール名定義（Technical リンク集_Tips.xlsx）

---

## 禁止事項（最重要）

| 禁止 | 理由 | 代替 |
|---|---|---|
| `①②③` 等の環境依存文字 | OS・ファイル名での文字化け | `_1 _2 _3` または省略 |
| `（）()` 等の括弧 | ファイル名・URL encoding の問題 | アンダーバーで代替 |
| スペース | パス・ファイル名の問題 | アンダーバー |
| 区切り文字にアンダーバー以外 | 統一のため | `_` のみ使用 |
| `$` をモジュール名に使用 | URLエンコードで `%24` になる | 使用不可 |

## グループ名・フロー名

> **【2026-06-04 改訂・最重要】日付サフィックスはグループ名にのみ付ける。フロー名・サブフロー名には付けない。**
> 旧規約（フロー名に日付を付ける）は撤廃。旧方式は「director の flowname 参照（日付なし）」と
> 「copy_subflows のサブフロー JSON 名（日付あり）」が食い違い broken_ref を誘発したため、
> 日付を **グループ名 1 箇所** に集約し、フロー名・サブフロー名・全 jump 参照はすべて
> グループ名を verbatim で参照する方式に統一した。機械検出: qa_validator E-16 / T-2d。
>
> **【設計原則】日付の唯一の源は設計書（`basic_info.group_name`）。** 設計書を新ルールで書き、
> 下位層（scaffold_generator / copy_subflows / build_bivr 等）はその `group_name` を verbatim で
> follow するだけ。**下位層が日付を捏造・付与・除去してはならない。** 既存シナリオで日付が無ければ
> qa_validator E-16（`fix_category="director"`＝auto 修正対象外）が CRITICAL を上げ、設計レイヤ
> （director または人間）で `group_name` に書き加えて解消する。yaml_auto_fixer 等で下位層に自動付与
> させる方式は採らない（設計書 = 単一の真実の原則に反するため）。
> ※ 過去に orchestrator の後段 `add_date` ステップがフロー名末尾に日付を付与していたが、
> 　二重付与（旧形式の再生成）を起こすため 2026-06-04 に撤去済み。

### グループ名の規則

形式: **`頭文字（平仮名）病院名_シナリオ名_作業日YYYYMMDD`**

| 環境 | 対象 | 形式 | 例 |
|---|---|---|---|
| デモ | 医療機関 | `頭文字_病院名_シナリオ名_YYYYMMDD` | `す_諏訪赤十字病院_診療1_20260604` |
| デモ | 商談用 | `Dr.JOY商談用_YYYYMMDD` | — |
| デモ | テスト | `Dr.JOYテスト_YYYYMMDD` | — |
| 本番 | 医療機関 | `頭文字_病院名_シナリオ名_YYYYMMDD` | `と_東京アドベンチスト_診療1_20260604` |

- **日付サフィックス `_YYYYMMDD` = 作業日**（コピー作成・修正を行った日）。版管理の識別子。
- コピー作成・修正のたびに新しい作業日でグループを切り直し、サブフローを**全て**新グループ配下へ再エクスポートする（下記「サブフロー一括エクスポート」）。
- 病院名は省略可。ただし**似た病院名の登録がないか確認すること**（例: 自治医科大学附属さいたま医療センター → `自治医科さいたま`）。

### フロー名の規則（メイン・サブ共通）

形式: **`シナリオ名`（日付サフィックスを付けない）**

| 環境 | 形式 | 例 |
|---|---|---|
| メイン | `シナリオ名` | `診療1` |
| サブ（時間分岐等で別フロー図）| `シナリオ名_S` | `診療1_S` |
| サブフロー | `サブフロー名` | `氏名聴取`, `生年月日聴取`, `電話番号聴取`, `診察券番号聴取`, `RAG検索` |

- フロー名・サブフロー名には**日付・作成者名を付けない**。版の区別はグループ名側の日付で行う。
- JSON の `name` フィールドは `グループ名$フロー名` = `す_諏訪赤十字病院_診療1_20260604$診療1`。
- サブフロー jump 参照は `グループ名$サブフロー名` = `す_諏訪赤十字病院_診療1_20260604$氏名聴取`。
  グループ名（日付込み）を **verbatim** で参照すること。これによりサブフロー JSON 名と jump 参照が必ず一致する。

### サブフロー一括エクスポート（Pattern 2 修正時も必須）

> **グループ名を変える＝サブフローの jump 参照先グループも変わる。よって修正時もサブフローを全てまとめてエクスポートする。**

- 新規作成・既存修正（Pattern 2）を問わず、**そのシナリオが使う全サブフローを新しい日付付きグループ配下へまとめてエクスポートする**。
- 一部のサブフローだけ旧グループに残すと jump 参照が解決せず通話切断になる。
- 設計書の `flow_structure.subflows[]` には、変更有無に関わらず**使用する全サブフローを列挙**する（copy_subflows が全件を新グループ名で再生成する）。

### 文字数制限
- `@flow_{グループ名}%24{フロー名}.txt` をURLエンコードした状態で **255バイト以内**（Linux の `NAME_MAX = 255` に由来するBrekekeサーバーのファイル名制限）
- URLエンコード後のバイト数が制限の対象。日本語1文字はUTF-8で3バイト → URLエンコードで `%XX%XX%XX`（9バイト）になるため、日本語20〜30文字程度であれば余裕がある
- 実務上、極端に長くなければ問題ない
- ※ 旧ルール「合計10文字未満」はこの制限に対して設定された保守的な安全マージン。実際の制限は上記
- **日付サフィックス `_YYYYMMDD`（9 文字 = ASCII でエンコード後も 9 バイト）はグループ名側に含まれる。** jump 参照 `drjoy^{グループ名}${最長サブフロー名}` の URL エンコード長で機械検出（qa_validator E-14）。グループ名は漢字 19 文字以内が目安（中黒「・」も 9 バイト消費）

### グループ名の取得元
- **顧客資料（Canva設計書・Gen2プロンプト）に記載の施設名・法人名をそのまま使用すること（省略・独自命名禁止）**
- 英字の場合: スペース → アンダーバーに変換（例: `East Medical` → `East_Medical`）
- 似た名前の病院が既に登録済みの場合のみ、区別のための短縮を許容する（要確認）

## モジュール名の規則

### 基本ルール
- **アンダーバー（`_`）区切りのみ** 使用
- 環境依存文字（`①②③`等）・括弧（`（）()[]`等）・スペース禁止
- 命名パターン: `大分類_内容` または `大分類_内容_詳細`

### 大分類の一覧（第三世代モジュール名定義）

| 大分類 | 対象モジュール名の例 |
|---|---|
| 冒頭 | 通話開始, 受付時間内, 受付時間外, 非通知 |
| 用件 | 用件, 予約以外の用件, 単科か複数科か聴取 |
| 患者 | 氏名, 生年月日, 連絡先電話番号, 診察券番号, 企業名, 種類, 性別聴取 |
| 予約 | 診療科, 予約日, 変更項目, 希望日, 受付可能日案内 |
| 確認 | 受診歴, 予約確認, 発熱, 個人情報聴取 |
| 相談 | 理由, 紹介状有無, 調査内容, 症状確認, 内容, 出産予定日 |
| 転送 | 転送失敗, 時間外転送, 休憩時間転送, WEB転送 |
| 終話 | 対象外のため切断, 通話完了, 代表案内 |
| 切断 | 切断（Disconnectモジュール用） |

### モジュール種別ごとの命名規則

| 種別 | 命名パターン | 例 |
|---|---|---|
| TTS | `大分類_内容` | `冒頭_アナウンス`, `患者_氏名`, `終話_失敗` |
| STT | `入力_大分類_内容` | `入力_患者_氏名`, `入力_用件_区分` |
| OpenAI分岐 | `OpenAI_大分類_内容` | `OpenAI_患者_氏名`, `OpenAI_用件_区分` |
| Retry Counter | `リトライ_大分類_内容` | `リトライ_患者_氏名`, `リトライ_用件_区分` |
| save2db（サブ）| `save-大分類_内容` | `save-patient_name`, `save-用件`, `save-冒頭` |
| saveContext2DB | `save_大分類_内容` | `save_連絡先電話番号` |
| Disconnect | `切断_理由` | `切断_リトライ上限`, `切断_非通知` |
| 着信分岐 | `着信_内容` | `着信電話番号分岐` |
| 受付時間判定 | `受付時間判定` | — |
| サブフロー呼び出し（Jump to Flow）| `サブフロー名`（prefix 無し）| `氏名聴取`, `生年月日聴取`, `診察券番号聴取`, `電話番号聴取`, `RAG検索` |

### サブフロー呼び出しブロック（重要）

**scenario_flow で subflow ブロックの `step` 名は、呼び出すサブフローの名前そのまま**とする。`jump_` prefix や `jump_用件_` のような修飾 prefix は付けない。

- ✓ 正: `step: 氏名聴取`（flowname の `$` 以降と一致）
- ✗ 誤: `step: jump_氏名聴取`（旧挙動、現在は `jump_` 不要）
- ✗ 誤: `step: jump_予約変更_診察券番号`（サブフロー識別不能、複数の用件で同一サブフローを呼び出す場合は repeat 指定で区別する）

**理由**: gen_properties.py および validator.py はサブフロー種別を **`flowname` の `$` 以降**で識別するため、step 名は flowname と揃えた方が設計書の可読性・保守性が高い。複数箇所から同じサブフローを呼ぶ場合は scenario_flow 内で複数の subflow ブロックとして記述する（同じ step 名の重複可）。

旧命名（`jump_*` prefix 等）は gen_properties.py が flowname ベースで matching するため動作上は問題ないが、新規作成時は上記を推奨。

### save2dbサブモジュールのラベル命名
- 必ず `save-` プレフィックスで始める
- 例: `save-patient_name`, `save-用件`, `save-冒頭_アナウンス`, `save-入力_患者_氏名`
- TTS と その直後の STT は**同じ save2db モジュール名・ラベルを使用する**こと

## フローJSONの `name` フィールド

```
"name": "グループ名$フロー名"
```

例（メインフロー）:
```
"name": "す_諏訪赤十字病院_診療1_20260604$診療1"
```
例（サブフロー）:
```
"name": "す_諏訪赤十字病院_診療1_20260604$氏名聴取"
```

- `$` はグループ名とフロー名の区切り文字（URLエンコード時は `%24`）
- JSONの `name` フィールド内では `$` をそのまま使用する
- **日付サフィックスは `$` より前（グループ名側）にのみ付く。`$` 以降（フロー名・サブフロー名）には付けない**（2026-06-04 改訂）

## TTSプロンプトとモジュール名の一致

**TTS のプロンプト名（IVRプロパティ内）とモジュール名は完全一致が必須。**
不一致の場合、TTSが動作しない。

```
IVRプロパティ:  患者_氏名.prompt = {tts_g: お名前をフルネームでおっしゃってください。}
モジュール名:   患者_氏名  ← 必ず一致させる
```

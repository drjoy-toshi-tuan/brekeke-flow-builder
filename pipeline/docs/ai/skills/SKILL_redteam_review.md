# SKILL: レッドチーム校閲（壁打ち時・out-of-line）

> **由来**: 旧 `reviewer` エージェント（Haiku・in-line）を keystone（テスト完了までライン内 LLM ゼロ）により
> 退役（2026-06-24）。校閲の**知見はここに保全**し、**壁打ち時に人間 + Claude が out-of-line で適用**する。
> 旧エージェント定義の全文: `.claude/agents/archive/reviewer.md`。関連: `SKILL_quality_criteria.md`。

## 何を校閲するか
prompter が生成した **OpenAI プロンプト（サイドカー MD `prompts_*.md`）のみ**。フロー構造・命名・Layout は
validator が、設計書妥当性は qa_validator が、TTS 品質は gen_properties + 人間が担保する（重複しない）。
JSON 全文の Read は不要（プロンプトはサイドカー MD に集約）。

## チェック観点（3観点）

### 観点1: 攻撃耐性
- 全プロンプトに「プロンプトインジェクション対策」セクションがあるか
- ロール変更・出力形式変更指示への防御文言が機能するか
- 「設定を出力して」等の開示要求に **NO_RESULT** を返す設計か
- 複合意図（「キャンセルかつ新規予約」等）で分岐が崩れないか

### 観点2: 分岐判断の正確性
- 出力可能値リストが設計書の `range_values` / `conditions` と**完全一致**しているか
- NO_RESULT 条件が曖昧入力・想定外入力を適切に捕捉しているか
- enum 型: 数字（DTMF）+ キーワードの二段構造があるか
- datetime 型: 相対日付・和暦・月日のみ入力の変換ロジックがあるか
- 分類結果が次ブロックの条件値と一致しているか

### 観点3: 出力の安全性
- 出力仕様が厳密で、意図しない長文・説明文を出さないか
- 「1語のみ」「特定フォーマットのみ」等の出力制約が明記されているか
- ユーザー入力をそのまま返す（オウム返し）リスクがないか

## 重大度
| 重大度 | 修正担当（壁打ち） | 例 |
|---|---|---|
| Critical | 設計書/プロンプトを壁打ちで直す | インジェクション防御欠落・分岐出力値ミスマッチ・NO_RESULT 条件不備 |
| Warning | 同上（優先度低） | 判定網羅性不足・出力制約の曖昧さ・DTMF 二段構造欠如 |
| Info | 任意 | UX 改善提案 |

## 適用のしかた（keystone 準拠）
- **ライン内では校閲しない**（自動の in-line LLM 校閲は置かない）。
- 壁打ち入口（`/sparring-intake`）で設計を詰める際、人間 + Claude がこのチェックリストでプロンプト設計を点検する。
- 指摘は**生成物でなく生成器（設計書 YAML・プロンプト方針）を直して再実行**で解消する。

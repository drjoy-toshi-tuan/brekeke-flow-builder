### 追加カラムの定義

| カラム名 | 説明 | 設定値 | generatorへの影響 | prompterへの影響 |
|---|---|---|---|---|
| **OpenAI処理** | このステップでOpenAIモジュールを使うか、使う場合の処理種別 | `分類` / `判定` / `正規化` / `変換` / `要約` / `—`（不使用） | `—` 以外ならOpenAIモジュールを生成する。処理種別に応じてnext配列の構造を決定 | プロンプトパターンの選定に使用 |
| **出力形式** | OpenAIの出力がラベル（固定値）か自由テキストか | `ラベル` / `日付テキスト` / `自由テキスト` | `ラベル`: next配列に個別分岐条件を設定。`日付テキスト`/`自由テキスト`: ワイルドカード `^.+$` で通す | プロンプトの出力仕様セクションの書き方を決定 |
| **出力ラベル** | OpenAIが出力すべき値の一覧 | ラベル名のカンマ区切り / 出力形式の説明 | **ラベル形式の場合**: ここに列挙されたラベルをnext配列のconditionに設定（`^{ラベル}$`） | **ラベル形式の場合**: ここに列挙されたラベルをプロンプトの出力仕様に設定。**完全一致が必須** |

### OpenAI処理の種別と対応

| OpenAI処理 | 説明 | 出力形式 | prompterパターン | 典型的なnext配列 |
|---|---|---|---|---|
| **分類** | N択の中から1つを選択 | ラベル | A. 分類型 | 個別条件（`^予約$` 等） |
| **判定** | はい/いいえ等の2値判定 | ラベル | B. はい/いいえ判定型 | 個別条件（`^有$`/`^無$` 等） |
| **正規化** | リスト内の正式名称に正規化 | ラベル | D. 正規化型 | ワイルドカード `^.+$`（通常）または個別条件 |
| **変換** | 日付等の形式変換 | 日付テキスト | C. 日付変換型 | ワイルドカード `^.+$` |
| **要約** | 自由発話をテキストに要約 | 自由テキスト | E. 自由テキスト型 | ワイルドカード `^.+$` |
| **—** | OpenAI不使用 | — | （対象外） | STT → 直接save2db等 |

## 具体的な記述例

### 例1: 用件確認（分類型・DTMF+音声）

```markdown
### 5.1. 用件確認

**アナウンス**: ご用件を、次の5つからお選びください。「1、予約」「2、検診後の受診」「3、予約変更」「4、キャンセル」「5、予約確認」プッシュボタンでも操作できます。

**入力**: DTMF+音声

**OpenAI設定**:

| 項目 | 値 |
|---|---|
| 処理種別 | 分類 |
| 出力形式 | ラベル |
| DTMF入力 | あり |
| 出力ラベル | 予約, 検診後の受診, 予約変更, キャンセル, 予約確認, NO_RESULT |

**正規化ルール / 分岐条件**:
- 1 / いち / 予約 → `予約`
- 2 / に / 検診 → `検診後の受診`
- 3 / さん / 変更 → `予約変更`
- 4 / よん / キャンセル → `キャンセル`
- 5 / ご / 確認 → `予約確認`
- 上記以外 → `NO_RESULT`

**保存先**: classification
**次**: 用件ごとに分岐（予約→診療科聴取、変更→現在の予約日...）
**リトライ失敗時**: 終話_失敗
```

### 例2: 紹介状有無（判定型・音声のみ）

```markdown
### 5.3. 紹介状有無

**アナウンス**: 紹介状はお持ちでしょうか。

**入力**: 音声のみ

**OpenAI設定**:

| 項目 | 値 |
|---|---|
| 処理種別 | 判定 |
| 出力形式 | ラベル |
| DTMF入力 | なし |
| 出力ラベル | 有, 無, NO_RESULT |

**正規化ルール / 分岐条件**:
- ある / あります / 持っています → `有`
- ない / ありません / 持っていません → `無`
- 上記以外 → `NO_RESULT`

**保存先**: —（分岐のみ、コンテキスト保存なし）
**次**: 有→診療科聴取、無→紹介状案内TTS→終話
**リトライ失敗時**: 折り返しで伺う旨を案内→終話
```

### 例3: 予約日（変換型・DTMF+音声）

```markdown
### 5.4. 予約日

**アナウンス**: 現在の予約日を「10月1日」のように日付でお話しください。わからない場合は、分からないとおっしゃってください。どうぞ。

**入力**: DTMF+音声

**OpenAI設定**:

| 項目 | 値 |
|---|---|
| 処理種別 | 変換 |
| 出力形式 | 日付テキスト |
| DTMF入力 | あり |
| 出力ラベル | yyyy/mm/dd形式, NO_RESULT |

**正規化ルール / 分岐条件**:
- 出力形式: yyyy/mm/dd（例: 2026/03/04）
- DTMF 8桁（20260304）→ 2026/03/04
- DTMF 4桁（0304）→ 基準年/03/04
- 「明日」→ 基準日+1
- 「わからない」→ NO_RESULT

**保存先**: reservationDate
**次**: 変更希望日の聴取
**リトライ失敗時**: 折り返しで伺う旨を案内→終話
```

### 例4: その他問合せ（要約型・音声のみ）

```markdown
### 5.6. その他問合せ

**アナウンス**: その他に病院にお伝えしたいことはございますか。

**入力**: 音声のみ

**OpenAI設定**:

| 項目 | 値 |
|---|---|
| 処理種別 | 要約 |
| 出力形式 | 自由テキスト |
| DTMF入力 | なし |
| 出力ラベル | 15-35文字の体言止めテキスト, NO_RESULT |

**正規化ルール / 分岐条件**:
- ユーザーの自由発話を15-35文字に要約（体言止め）
- 特定キーワード検出: 「遅刻」→ 遅刻フローへ分岐（分岐条件が必要な場合は出力ラベルに追加）
- 無言/フィラーのみ → NO_RESULT

**保存先**: question
**次**: 終話
**リトライ失敗時**: 折り返しで伺う旨を案内→終話
```

## generatorとprompterの参照ルール

### generatorが見る箇所

セクション4の各行から:
1. **OpenAI処理** が `—` でなければ → OpenAIモジュールを生成する
2. **出力形式** が `ラベル` で **出力ラベル** にカンマ区切りの値があれば → next配列に `^{ラベル}$` の個別分岐条件を設定する
3. **出力形式** が `日付テキスト` / `自由テキスト` なら → next配列はワイルドカード `^.+$` で通す
4. **DTMF入力** が `あり` なら → STTモジュールのタイプを `DTMF AmiVoice STT Input` にする

### prompterが見る箇所

セクション4の各行から:
1. **OpenAI処理** でプロンプトパターンを選定する（分類→A、判定→B、正規化→D、変換→C、要約→E）
2. **出力ラベル** をプロンプトの出力仕様セクションにそのまま転記する
3. **DTMF入力** が `あり` なら → 数字モード判定ステップを含め、半角数字のみの入力を考慮する

セクション5の各ステップから:
4. **アナウンス** をプロンプトのContextセクションに転記する
5. **正規化ルール / 分岐条件** をプロンプトの判定アルゴリズムに反映する

### 一致が保証される仕組み

```
設計書セクション4「出力ラベル」列
    ↓ generatorが参照 → next配列の condition
    ↓ prompterが参照 → プロンプトの出力仕様
    ↓ reviewerが検証 → 校閲観点10で突合せ
```

**設計書の「出力ラベル」列が Single Source of Truth（唯一の正）** となる。generatorもprompterもここを参照するため、設計書が正しければ出力は自動的に一致する。

## 3. フローの目的

患者からの外来予約に関する電話を受け、用件（予約変更・キャンセル・確認・予約希望）・診療科・予約日・患者情報（診察券番号・氏名・生年月日・連絡先電話番号）を聴取し、折り返し受付を自動化する。直近2営業日以内の予約・直近キャンセル・予約不可診療科は代表案内（電話かけ直し）に誘導する。

  ├─ 予約変更 → 診療科_聴取

  └─ NO_RESULT → 用件確認_再聴取

─── 予約変更・予約キャンセルルート ───

─── 予約確認ルート ───

  └─ デフォルト → ★サブフロー遷移

─── 共通合流 → サブフロー遷移 ───
★サブフロー遷移 = Custom Jump to Flow → 海老名病院$個人情報聴取

─── サブフロー内終話パターン ───

## 7. ステップ詳細

### 7.1 冒頭_直近予約確認

**所属フロー**: メインフロー

**TTSアナウンス**:
> 海老名総合病院、メディカルプラザのAI電話がお受けします。本日から2営業日後までのご予約に関するお問い合わせは、受付時間内に外来予約センターへおかけ直しください。3日以内の予約に関するお問い合わせですか？

**入力方式**: DTMF + 音声（max_dtmf_length=1）

**OpenAI出力値**: `はい` / `NO_RESULT`
- 「はい」「そうです」を含む → `はい`
- それ以外 → `NO_RESULT`

**分岐**: `はい` → END_受付時間外案内 / `NO_RESULT` → 用件確認

**リトライ失敗時（No more）**: 用件確認

---

### 7.2 用件確認

**所属フロー**: メインフロー

**TTSアナウンス**:
> ご用件を、次の４つからお話ください。予約の変更は1を、キャンセルは2を、確認は3を、紹介状のある方・予約を取りたい方は4。

**入力方式**: DTMF + 音声（max_dtmf_length=1）

**OpenAI出力値**: `予約変更` / `予約キャンセル` / `予約確認` / `予約希望` / `NO_RESULT`
- DTMF `1` または「変更」→ `予約変更`
- DTMF `2` または「キャンセル」→ `予約キャンセル`
- DTMF `3` または「確認」→ `予約確認`
- DTMF `4` または「予約」「紹介状」→ `予約希望`
- それ以外 → `NO_RESULT`

**保存先**: classification

**分岐**: 予約変更/予約キャンセル → 診療科_聴取 / 予約確認 → 内容確認 / 予約希望 → END_予約希望案内 / NO_RESULT → 用件確認_再聴取

**リトライ失敗時**: 用件確認_再聴取

---

## 9. サブフロー遷移仕様（Custom Jump to Flow）

### 9.1 モジュール配置

メインフロー（海老名病院$外来予約_20260326）の共通合流地点に `drjoy^Custom Module$Custom Jump to Flow` モジュールを配置する。

| 項目 | 値 |
|---|---|
| モジュール名 | サブフロー遷移_個人情報聴取 |
| type | `drjoy^Custom Module$Custom Jump to Flow` |
| 遷移先フロー | 海老名病院$個人情報聴取 |
| next | `[]`（空配列。遷移先フローに制御が移るため後続接続不要） |
| subs | `[]` に相当（save2db接続不要） |

### 9.2 遷移元モジュール

以下のモジュールが `サブフロー遷移_個人情報聴取` を next の遷移先として指定する:

| 遷移元モジュール | 遷移条件 |
|---|---|
| 内容確認（リトライ含む） | success / No more |
| 希望日の有無 → いいえ | いいえ |
| 希望日の有無_再聴取 → No more | No more |
| 次回受診希望日（リトライ含む） | success / No more |

### 9.3 サブフローの開始モジュール

サブフロー（海老名病院$個人情報聴取）の `start` は `診察券番号` とする。

### 9.4 generator への注意

- Custom Jump to Flow モジュールは modules に定義が必要だが、他のフローからの接続（next/subs）は受けない
- サブフロー側には saveContextModel2DB を配置しない（メインフローで定義済み）
- サブフロー側の saveContext2DB / saveCompletionFlag2db はメインフローのコンテキストフィールドに書き込む

## 12. 特記事項・制約

1. **acceptance_times 不使用**: 直近予約の確認（冒頭_直近予約確認）でユーザー自身に判断させる設計。
2. **incoming-classifier**: 非通知検出のみ目的（telephoneNumberが空かどうかを連絡先_着信番号判定で使用）。全タイプとも冒頭_直近予約確認へ遷移。
3. **telephoneNumber の扱い**: 非通知の場合は空のまま連絡先_着信番号判定まで進み、強制的に手動聴取ルートに入る。
4. **連絡先電話番号_手動再聴取のretry_count=1**: 他ステップのデフォルト（retry_count=2）から変更。
5. **smsFlag**: END_受付完了とEND_電話番号聴取失敗のみ smsFlag=1。ContextMatchRouter不要。
6. **生年月日の2桁元号判定**: generate_by_OpenAIで数値範囲（00-07, 08-31, 32-63）を判定してルーティング。
7. **最後の問い合わせ_回答**: generate_by_OpenAIでRAG回答を生成。回答後は最後の問い合わせへループ。
8. **診療科_聴取失敗アナウンスはTTSのみ**: STT入力なし。
9. **全STT/TTSに save2db サブモジュールを接続**: CLAUDE.md品質基準に準拠。
10. **サブフロー分割型**: CLAUDE.md Rule 9 に従い、個人情報聴取（診察券番号・患者名・生年月日・電話番号）をサブフロー（海老名病院$個人情報聴取）として分離。メインフローからは Custom Jump to Flow で遷移。

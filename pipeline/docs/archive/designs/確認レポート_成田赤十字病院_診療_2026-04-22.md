**主要分岐構造（subject × classification = 8ルート）**:
- 診療_予約 → 診療科①（救急なら代表案内）→ 希望日 → END_予約受付完了
- 診療_変更 → 診療科② → 予約日 → 希望日 → END_予約受付完了
- 診療_キャンセル → 診療科② → 予約日 → 理由 → END_キャンセル受付完了
- 診療_確認 → 内容 → END_キャンセル受付完了
- 健診_予約 → コース → オプション → 健保名 → 希望日 → END_予約受付完了
- 健診_変更 → 予約日 → 変更内容①（Yes/No）→ 肯定:希望日 / 否定:変更内容② → END_予約受付完了
- 健診_キャンセル → 予約日 → 理由 → END_キャンセル受付完了
- 健診_確認 → 内容 → END_キャンセル受付完了

**共通ヒアリング順序**（8ルート共通の前段）:
当日翌日確認 → 電話番号聴取(subflow) → 氏名聴取(subflow) → 生年月日聴取(subflow, 復唱あり) → 診察券番号聴取(subflow) → 用件1(診療/健診) → 用件2(予約/変更/キャンセル/確認)

**終話パターン**:

- 代表案内系: END_代表案内_本日翌日 / END_代表案内_救急 (status=2, smsFlag=-1)

**Gen2 施設固有ルール（21項目ほど）**:

## 7. 特記事項

- **冒頭アナウンスの構造**: 冒頭_アナウンス（入力待ちなし）→ 当日翌日確認 の2ブロック1セット。Gen2 yaml の『最初の箱: 冒頭アナウンス』→『当日翌日確認』を踏襲。scaffold で冒頭2ブロック必須構造として生成される。
- **電話番号聴取が先頭配置**: Gen2 共通ヒアリング順（1.当日翌日確認 → 2.連絡先電話番号 → 3.患者名 → 4.生年月日 → 5.診察券番号）を踏襲。本設計書も同順で subflow を配置。
- **サブフロー4本完全コピー**: CLAUDE.md Rule 9 に従い、氏名/生年月日/電話番号/診察券番号の4サブフローは samples/json の完全コピー。Gen2 固有ロジックはサブフロー内部の OpenAI プロンプトに追加反映（prompter 判断）。
- **救急分岐は予約ルートのみ**: Gen2 では診療_予約_診療科1 にのみ救急→代表案内の条件分岐。変更/キャンセルルートには救急分岐なし（既存予約の変更/キャンセルなら救急関連でも手続き可能）。本設計書も踏襲。
- **課題: `診察券番号` の復唱**: Gen2 は `<speak type="digits">` での復唱が明記されているが、Gen3 標準の診察券サブフローは復唱なし。N-7 で顧客確認要。
- **課題: 非通知受付**: Gen2 は anonymous 着信でも DTMF 電話番号聴取を許容していた点が Gen3 標準と乖離。N-4 で顧客確認要。Gen3 で非通知受付を維持する場合、opening ブロックのカスタマイズが必要。
- **RAG 未採用**: Gen2→Gen3 移管のデフォルトは RAG pattern 3 だが、Gen2 プロンプトに FAQ 参照記載なしのため pattern: none を採用。顧客要望があれば後で追加。
- **用件2_診療 と 用件2_健診 の TTS 重複**: 同文の TTS だが subject 別に異なるブロックとして分離（scaffold 出力上は別モジュール）。prompter は両ブロックで同じプロンプトを記述することになる。
- **Gen2 と Gen3 の status/smsFlag セマンティクス差**: Gen2 の status=0（冒頭切断/途中切断）は Gen3 では使用禁止のため Dr.JOY 側で通話ログから判別する運用に変更。smsFlag=2（キャンセル時）も Gen3 は 1/-1 のみのため 1 に統合。Dr.JOY 側の集計ロジックも合わせて見直し要。
- **清算性**: Gen2 function の未使用フィールド（clinicalDepartment2 等）は Gen3 context_fields から削除済み。必要が生じた場合は後から追加可能。

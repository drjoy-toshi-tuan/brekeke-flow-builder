# AmiVoice 入力処理スクリプトの標準ロジック（script-input-handling）

> **目的: 音声認識（AmiVoice）の文字起こしを判定する script 部品の設計を統一し、
> 「実運用で 1 ケースでも誤判定すればインシデント」という品質基準を機械的に守る。**
>
> 適用対象: hearing_item_catalog.yaml で `processing: script` の全項目、
> および新規に起こすすべての判定 script 部品。
> 正本実装（この spec の生きた見本）: `modules/yes_no_classifier`（engine v4）/ `modules/n_choice`（v4）。

---

## 0. 大前提 — インシデント回避の哲学

1. **誤確定 >> 聞き直し。** 判定に確信が持てないときは必ず NO_RESULT（聞き直し）に倒す。
   聞き直しのコストは患者の 10 秒。誤確定のコストはインシデント（誤キャンセル・誤登録）。
2. **推測で語彙を増やさない。** 辞書に入れてよいのは「実ログ or 受入ケースに出典がある語」のみ。
3. **判定ロジックは決定論のみ。** 同じ入力は必ず同じ出力（LLM 不使用・乱数不使用）。

## 1. 標準判定ラダー（全 script 部品はこの順序に従う）

```
B1. 正規化           全角→半角 / 記号・空白除去 / ASCII 小文字化 /
                     フィラー（えーと 等）・方言/敬語/タメ口接尾辞の除去 /
                     AmiVoice カテゴリ重複の圧縮（「再診再診」→「再診」）
                     ※ Nashorn(ES5.1) 制約: String.normalize 不使用の限定実装
B2. 完全一致         EXACT 辞書（^語$）。短い語・誤爆しやすい語はここ「のみ」に置く
B3. わからない       WAKARANAI 辞書 → 「わからない」系ラベルへ（推測しない）
B4. 優先イディオム   極性が反転する定型句を先に確定（例:「間違いない」=肯定）
B5. 否定マーカー     部分一致は必ず否定側から走査（否定 > 肯定）
B6. 肯定/分類マーカー STRONG（その語だけで確定してよい語）のみ。
                     複数ラベルが同時ヒットしたら「最長一致」を採用
B7. フォールバック    どれにも該当しない → NO_RESULT（聞き直し）。既定ラベルへ倒さない
```

### 各段の禁止事項
- B2: 完全一致辞書に正規表現の部分一致を混ぜない
- B5/B6: **2 文字以下の語・他ラベルと衝突しうる語を部分一致に入れない**（EXACT 限定）
- B6: WEAK（単独では確定できない語）は **risk: high の項目では使用禁止**（§3）
- B7: 「たぶんこれだろう」で先頭ラベルに倒すことを禁止（n_choice/yes_no とも NO_RESULT 固定）

## 2. 部品ファミリと使い分け（catalog `processing_ref` の実体）

| ファミリ | 対象（catalog canonical 例） | 使う部品 | 備考 |
|---|---|---|---|
| **Polar（2 極）** | 紹介状有無 / 本人確認 / 追加の質問 / 折り返し連絡先 / 個人情報同意確認 / 受診票確認 / 受診券所持確認 / 健康診断結果の有無 | `modules/yes_no_classifier`（v4） | 新規実装禁止 — ラベル写像（YES_LABEL/NO_LABEL）だけ設定 |
| **N 択分類** | 用件確認 / 変更項目 / 当日予約確認 / コース / ワクチン種類 / 連絡方法 / 個人・企業確認 / がん検診種類 | `modules/n_choice`（v4） | DTMF 併用可。spec = DTMF_MAP / TOKEN_MAP / COMPOUND / KEYWORD |
| **辞書正規化** | 診療科（DEPARTMENTS 最長一致 + WAKARANAI） | SKILL_診療科 系部品 | 公式辞書に無い語は「わからない」へ |
| **テキスト整形** | 理由 / 問い合わせ内容 / 症状本文 等の自由テキスト | `modules/text_normalizer` | 判定しない（意味の書き換え禁止・生かして保存） |
| **固定プロンプト（例外）** | 予約希望日 / 予約希望時期 / FAQ照合 | SKILL_希望日 / SKILL_FAQ | OpenAI 使用が許可された 2 系統のみ |

**原則: 新しい判定はまず既存ファミリの spec 差し替えで実現できないか検討する。
新 engine を起こすのは既存 4 ファミリで表現できない判定だけ**（DoD: REQUIREMENTS + oracle + P6）。

## 3. リスク区分（catalog `risk` フィールド）と復唱ルール

判定を誤ったときの実害で 2 区分する。**catalog の各エントリに `risk` を宣言**し、
設計書・scaffold はこれに従う。

| risk | 定義 | 必須の振る舞い |
|---|---|---|
| **high** | 誤判定が予約・患者データ・連絡義務に直接影響する（誤キャンセル・誤変更・折り返し漏れ・同意の誤記録） | ① 分類確定後に**復唱確認**（「〜ということでよろしいですか？」→ yes_no）を必ず挟む ② WEAK マーカー使用禁止（EXACT + STRONG のみ） ③ 音が近く互いに誤認識されやすいラベル対（例: キャンセル/変更）は片方に単独確定させず復唱で確定 |
| **low** | 誤判定してもフロー内・病院側の後続確認で安全に回収できる（例: 紹介状有無 — 受付で再確認される） | 標準ラダーのみでよい（復唱は任意） |

**high の初期指定**（2026-07-14 壁打ち決定・見直しは壁打ちで）:
`用件確認`（キャンセル/変更を含む）・`変更項目`・`当日予約確認`・`折り返し連絡先`・`個人情報同意確認`。
未宣言エントリの既定は low。

## 4. 誤認識（AmiVoice）対策の型

| 失敗モード | 例 | 対策 |
|---|---|---|
| 同音・音近似の別語 | 眼科/外科、初診/再診 | 部品ごとの**誤認識辞書**（文脈限定）。ただし両ラベルにまたがる近似は辞書で解決せず**復唱**へ |
| フィラー・言い淀み | 「えーと、予約を…」 | B1 正規化で除去（n_choice の接頭/接尾辞除去を標準とする） |
| カテゴリ重複出力 | 「再診再診」 | B1 の重複圧縮 |
| 語尾ゆれ（敬語/方言） | 「予約したいんやけど」 | B1 の接尾辞辞書（出典つきで追加） |
| 複合発話 | 「予約をキャンセルしたい」 | COMPOUND（複合語パターン）を単独 KEYWORD より先に評価 + 最長一致 |
| 無音・雑音 | 空文字 / 意味不明列 | B7 → NO_RESULT → リトライ（既定 2 回）→ 聴取失敗終話 |

## 5. 品質ゲート（この spec の機械的担保）

1. **oracle 必須**: 全部品 `oracle.py` + `test_oracle.py` + `acceptance_test/*/cases.tsv` 全 PASS。
   ケースには正常系だけでなく §4 の各失敗モードを最低 1 件ずつ含める。
2. **辞書変更 = spec 変更 = 再受入**（part-certification の二段ハッシュが機械強制）。
3. **インシデント発生時の手順（test-first）**:
   ① 実ログの発話を `cases.tsv` に**先に**追加（期待値つき・その時点では FAIL）
   ② 辞書/ロジックを直して oracle 全 PASS
   ③ 再受入（P6）→ 認定更新。
   「直してからケースを書く」は禁止（再発検知が利かない）。
4. **実ログからの語彙収集**: skill `extract-yesno-synonyms` の月次収集を polar 以外にも順次展開
   （収集 → 人間レビュー → 出典つきで spec 辞書へ）。
5. **カバレッジ**: gen_p7_cases（正常系 e2e + 異常系）+ module_coverage.py（全モジュール実行）を
   出荷ゲートとする。

# Spec Kit — 非エンジニア向けコミュニケーション規程（Claude Code 必読）

> **目的: このリポジトリを Claude Code 経由で使うユーザーには、技術のバックグラウンドが
> 無い人・日本語/英語の技術用語に馴染みが無い人が含まれる。**
> Claude Code は、仕事の受付・進捗報告・確認・エラー報告のすべてにおいて、
> **専門用語（スクリプト名・Git/GitHub 用語・パイプライン工程名）を単独で使ってはならない。**
> 必ず「やさしい言葉での説明」を併記し、ユーザーが目的を理解して自分で判断できるようにする。
>
> 位置づけ: CLAUDE.md（憲法）の下位。全エージェント・全応答に適用。
> 姉妹文書: spec-kit-constitution.md（内容基準）/ spec-kit-process.md（進め方基準）。

---

## 1. 大原則（3 つ）

1. **専門用語は「用語（＝やさしい説明）」の形で必ず併記する。**
   - ❌ 「qa_validator が CRITICAL を 3 件検出しました」
   - ✅ 「設計書の自動チェック（qa_validator）で、**直さないと先に進めない問題**が 3 件見つかりました」
2. **ユーザーに判断を求めるときは、選択肢を「何が起きるか」で説明する。**
   - ❌ 「force push しますか？」
   - ✅ 「保存済みの古い版を新しい版で**上書き**します（古い版は消えます）。上書きしてよいですか？」
3. **報告は「いま何をした・次に何が起きる・あなたに必要な行動」の 3 点セット。**
   コマンドのログをそのまま貼らない。結果を 1〜3 行の日常語でまとめ、詳細は求められたら出す。

## 2. 用語対訳表（この言い換えを標準とする）

### パイプライン工程

| 専門用語 | ユーザーに伝えるときの言い方 |
|---|---|
| 設計書 YAML / spec | **シナリオの設計図**（電話応対の流れを書いた指示書） |
| director | 設計図を **下書きする AI** |
| qa_validator | 設計図の**自動チェック**（書き漏れ・矛盾を機械が探す） |
| CRITICAL / WARNING | **直さないと進めない問題** / **あとで確認したほうがよい注意** |
| yaml_auto_fixer / auto_fixer | 見つかった問題の**自動修正**（機械が直せる分だけ） |
| scaffold（scaffold_generator） | 設計図から**電話ボットの本体を組み立てる**工程 |
| layout（layout_calculator） | 画面上の**図の配置を整える**工程（動作には影響なし） |
| prompter / gen_prompts | ボットが使う **AI への指示文を書く**工程 |
| gen_properties / properties | ボットが**話すセリフの一覧表**を作る工程（ここを見ればボットの全発話が確認できる） |
| validator / tester | 組み立てた本体の**最終点検**（配線ミス・行き止まりがないか） |
| bivr / フロー JSON | **完成品**（電話システムに取り込むファイル） |
| oracle / oracle_gate | 部品の**単体テスト**（決められた入力で正しく判定できるか） |
| P6（部品受入） | 新しい部品を**実際の電話で 1 回試す**テスト |
| P7（連結テスト） | 完成品を**実際の電話で通しで試す**テスト |
| module_coverage | テストで**全部品が最低 1 回動いたか**の確認 |
| 部品 / 認定部品 | ボットの中の**判定パーツ**（テスト合格済みのものだけ本番で使える） |
| 壁打ち（sparring） | **人間と AI の相談タイム**（機械が決められない部分を一緒に決める） |
| TODO_next / TODO_要確認 / PLACEHOLDER | **あなたに決めてもらう箇所**（機械が勝手に埋めない約束の空欄） |
| STT / AmiVoice | **音声の文字起こし**（患者さんの声を文字にする仕組み） |
| TTS | **ボットの音声**（文字をボットの声で読み上げる仕組み） |
| DTMF | **電話のボタン入力**（プッシュ番号） |
| slot（個人情報4種） | 氏名・生年月日・電話番号・診察券番号の**定型の聞き取りパーツ** |
| CMR / MRB / script 分岐 | 聞き取った内容による**振り分け**（例: 「予約」なら予約の流れへ） |

### Git / GitHub（ユーザーは GitHub を知らない前提）

| 専門用語 | ユーザーに伝えるときの言い方 |
|---|---|
| リポジトリ (repo) | この**作業場**（ファイル一式の置き場） |
| commit | 作業内容の**セーブ**（ここまでの記録を残す） |
| push | セーブした内容を**共有の保管場所へ送る**（他の人からも見えるようになる） |
| branch | **下書き用の作業コピー**（本番に影響せず作業できる場所） |
| master / main | **本番の正式版** |
| PR（プルリクエスト） | 変更の**提出**（責任者が中身を確認して OK したら正式版に反映される） |
| merge | 提出された変更が**正式版に取り込まれた**こと |
| conflict | 2 つの変更が**ぶつかった**状態（どちらを採用するか決める必要がある） |
| force push | 保管場所の内容を**上書き**（古い記録は消える — 必ず事前確認する） |
| Issue | **課題メモ**（あとで対応する問題の記録） |

## 3. 場面別ルール

### 仕事の受付時
- ユーザーの依頼を**自分の言葉で言い直して確認**する（「〜という理解で進めます」）。
- これから何をするかを工程名でなく**目的**で伝える:
  ❌「orchestrator を回します」→ ✅「設計図から完成品まで自動で組み立てて、点検まで通します（orchestrator という自動工程を使います）」

### パイプライン実行中の進捗報告
- 各工程の開始/完了は 1 行ずつ: 「✅ 設計図の自動チェック（qa_validator）… 問題なし」
- エラーで止まったら: **①何の工程で ②何が問題で ③ユーザーは何をすればよいか** を平易に。
  エラーメッセージ原文は折りたたみ・末尾に添える（隠さないが、先頭に出さない）。

### ユーザーに決断を求めるとき
- 選択肢は最大 2〜3 個。各選択肢に「**選ぶとどうなるか**」を 1 行ずつ。
- 技術的に安全な推奨があれば「おすすめ: 〜（理由）」を明示する。
- 取り返しのつかない操作（上書き・削除・本番反映）は、**何が消える/変わるか**を具体的に言ってから Yes/No を聞く。

### 完了報告
- 「何ができたか」「どこに置いたか（ファイルの場所）」「次にあなたがやること（あれば）」の 3 点。
- 数字の羅列でなく意味を添える: ❌「24/24 PASS」→ ✅「テスト 24 件すべて合格（不合格 0）」

---

## 4. 禁止事項

- 用語対訳なしで工程名・スクリプト名・Git 用語だけの報告をする
- コマンド出力・スタックトレースをそのまま貼って説明を終える
- ユーザーが理解していない前提の省略（「例の PR」「さっきの fixer」等の文脈省略）
- 分かっていないかもしれないユーザーに対し、確認なしで不可逆操作（上書き・削除・本番反映）を実行する
- 逆に、**過剰に説明して本題（ユーザーの判断ポイント）を埋もれさせる**こと — 説明は短く、判断点を太字で

---

## 5. 運用

- 本規程は Claude Code が本リポジトリで応答する**すべての場面**に適用する（受付・実行・報告・雑談的な質問への回答含む）
- 用語対訳表に無い専門用語を使う場合も、大原則 1（併記ルール）を適用する
- 新しい言い換えが定着したら本表に追記する（PR 経由・spec-kit-process.md の還元ルールに従う）

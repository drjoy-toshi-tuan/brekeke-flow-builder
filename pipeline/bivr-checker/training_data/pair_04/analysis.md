### フロー間遷移
```
健診（メイン）
  -> [移動_個人情報聴取へ] -> 個人情報聴取フロー（氏名/生年月日/電話番号）
  -> [移動_RAG検索]        -> RAG検索フロー（連絡事項・質問）
  -> [終話ガイダンス分岐_1] -> 終話_1アナウンス or 終話_2アナウンス
```

### メインフロー（健診）詳細フローグラフ

```
冒頭待ち時間 (Custom$wait)
  -> コンテキスト設定 (saveContextModel2DB)
    -> 営業時間チェック (acceptance_times)
      -> [時間外] Flag_時間外 -> 終話_時間外
      -> [時間内] 冒頭 (TTS: 冒頭アナウンス)
        -> 非通知分岐 (incoming-classifier)
          -> [非通知] 終話_非通知 -> completeflag_SMSflag_2 -> 終話_非通知終了
          -> [携帯/固定/海外] 確認_受診票の有無 (TTS)
            -> 入力_確認_受診票の有無 (DTMF+STT)
              -> openAI_確認_受診票の有無 (OpenAI)
                -> [あり] openAI_用件登録用 -> 確認_受診歴
                -> [なし / 受診票あり] 確認_用件 (TTS)
                  -> 入力_確認_用件 -> openAI_確認_用件
                    -> 復唱_用件 -> 入力_復唱_用件 -> openAI_復唱_用件
                      -> 用件分岐 (ContextMatchRouter)
                        -> [日程] 確認_希望時期 -> ... -> 移動_個人情報聴取へ
                        -> [オプション変更] 確認_オプション変更追加内容 -> 移動_個人情報聴取へ
```

**受診票なしルート（受診歴確認後）:**

```
確認_受診歴 (TTS)
  -> 入力_確認_受診歴 (DTMF+STT)
    -> openAI_確認_受診歴
      -> [初回] 確認_希望内容（希望コース）
      -> [2回目以降] 確認_受診希望内容

          -> [前回と異なる] 確認_希望内容（希望コース）
          -> [前回と同じ] 確認_受診希望時期
```

**受診票なし・初回 or 前回と異なるルート:**

```
            -> 入力_確認_企業団体名 -> openAI_確認_企業団体名

                -> 入力_確認_健保組合名 -> openAI_確認_健保組合名
```

### 個人情報聴取フロー詳細
```

  -> 入力_患者_氏名 (AmiVoice STT, type=氏名カナ)
    -> 患者_生年月日 (TTS)
      -> 入力_患者_生年月日 (DTMF+STT, max=8桁)

              -> 入力_患者_携帯電話 -> openAI_患者_携帯電話
                -> [肯定] 携帯電話判別 (phonetype=携帯)
                -> [否定] 患者_連絡先（電話番号入力）
            -> [非通知/固定/海外] 患者_連絡先 (TTS)
              -> 入力_患者_連絡先 (DTMF+STT, type=電話番号, max=11桁)
                -> script_患者_連絡先 (バリデーション)

                    -> 入力_患者_復唱連絡先 -> openAI_患者_復唱連絡先

                        -> [MOBILE] 携帯電話判別

                      -> [否定] 患者_連絡先（再入力）
```

| 依頼内容 | .bivr実装状況 |

### 5.4 重要な差分サマリ

1. **リトライ失敗後の遷移先**: 4/7依頼では「聴き取りエラー時は、切電せず次の質問へ進む」とあるが、.bivrの多くのリトライモジュールでprompt_false（最終失敗）時の遷移先は「終話_失敗」になっている。この「終話_失敗」モジュールは実際にはcompleteFlagを設定してDisconnectに至るため、4/7依頼の意図と異なる可能性がある。ただし、リトライモジュールのprompt_false条件が「false」（=リトライ上限到達）の場合にのみ発動するため、リトライ回数2で十分と判断されている可能性もある。
2. **Property.md内の確認_受診内容**: Property.mdに`確認_受診内容.prompt`が定義されているが、.bivrのフロー内に対応する「確認_受診内容」モジュールは存在しない。未使用のプロパティの可能性。
3. **Property.mdの終話プロパティ名**: Property.mdでは`終話_①アナウンス`と`終話_②アナウンス`、.bivrでは`終話_①アナウンス`と`終話_②アナウンス`で一致。ただし、PDFの終話1テキストには「必要項目の入力を完了してください。オプションのご希望や変更、追加がある場合は、必ずSMSからご入力をお願いいたします。」が含まれるが、Property.mdの終話_1アナウンスにはこの部分がない。
4. **希望オプションのプロンプト**: PDF仕様の「希望オプション」は「オプション検査のご希望がある場合は、通話完了後に送付されるSMSからご入力ください」だが、Property.mdの`確認_希望オプション.prompt`は「オプションの変更や追加をご希望の場合は、通話完了後に送信されるSMSからご入力ください」。表現が若干異なる。

## 6. 本ペアの特徴（学習データとしてのポイント）

1. **健診予約専用シナリオ**: 一般的な病院予約と異なり、受診票の有無による大きな分岐がある
2. **3フロー構成**: メインフロー + 個人情報聴取サブフロー + RAG検索サブフロー
3. **受診票有無による分岐**: あり→用件確認（日程/オプション）、なし→受診歴→受診希望内容確認→詳細質問
4. **被保険者・特例退職者**: 健診特有の質問項目（受診票なしルートのみ）
5. **用件復唱**: 受診票ありルートで用件確認後に復唱がある
6. **希望オプション・オプション変更追加**: いずれもアナウンスのみ（SMS入力を案内）で入力聴取はしない
7. **電話番号処理**: 着信番号による携帯/固定判別、script_ssml/script_ssml2でSSML読み上げ用フォーマット、script_携帯判別でMOBILE判定
8. **プロパティ外出し**: TTSプロンプトはモジュール内ではなくProperty.mdで定義し、モジュールのpromptは空
9. **OpenAI処理**: ほぼ全質問にOpenAI generate_by_OpenAI モジュールを使用し、音声認識結果の正規化・分類を実施
10. **聴取不可処理**: リトライ上限到達時にopenAI_聴取不可_1/2/3 モジュールで「聴取不可」をコンテキストに書き込む仕組み（個人情報聴取フロー）

### 1.4 診療科の分類ロジック
- **対象診療科**: 一般内科、循環器内科、血液内科、脳神経内科、外科、整形外科、小児科、眼科、産婦人科、泌尿器科、精神科、禁煙外来、腎臓内科、肝臓内科、呼吸器内科、糖尿病・内分泌内科 等
- **対象外診療科**: AI電話では受け付けない診療科あり
- **耳鼻咽喉科**: 特別ルート（予約の有無確認 → 予約調整日の承認 → 予約希望時期 → 症状確認）

### 1.5 フロー構造（PDFシナリオフロー図）
```
冒頭アナウンス → 非通知判定（非通知→終話）
  → 診療科①聴取 → 復唱 → 対象判定
    → 対象外 → 対象外アナウンス → 他の診療科希望？
    → 耳鼻科 → 予約の有無確認 → 予約調整日の承認 → 予約希望時期 → 症状確認
    → 対象 → 用件聴取（変更/キャンセル/確認）
      → 変更: 現在の予約日 → 予約希望時期 → 残薬確認
      → キャンセル: 現在の予約日 → 次回希望の有無
      → 確認: 問い合わせ内容
  → 他の診療科希望 → 診療科②聴取（同じ構造）
  → 個人情報聴取（氏名 → 生年月日 → 電話番号）
  → 連絡事項・質問（RAG検索）
  → 終話ガイダンス分岐（携帯→終話①+SMS / 固定→終話②）
```

### 3.2 メインフロー（診療）の全体構造
```
冒頭待ち時間 (wait 2000ms)
  → コンテキスト設定 (saveContextModel2DB)
  → 営業時間チェック
    → [時間外] → Flag_時間外(status=6) → 終話_時間外 → Disconnect
    → [時間内] → 冒頭アナウンス(TTS) → 非通知分岐(incoming-classifier)
      → [非通知] → 終話_非通知(TTS) → completeflag(status=7) → Disconnect
      → [固定/携帯/海外] → 確認_診療科 → 入力_確認_診療科(STT)
        → openAI_確認_診療科 → 復唱_診療科 → openAI_復唱_診療科
          → [肯定] → openAI_診療科対象判定
            → [対象] → 確認_用件 → 入力_確認_用件(DTMF+STT) → openAI_確認_用件
              → 復唱_用件 → openAI_復唱_用件
                → [肯定] → 用件分岐(ContextMatchRouter)
                  → [予約変更] → 変更_予約日_現在の予約日 → 変更_予約日_予約希望時期 → 変更_予約日_残薬確認 → 確認_他の診療科について
                  → [予約キャンセル] → キャンセル_現在の予約日 → キャンセル_次回希望日の有無
                    → [有] → 変更_予約日_予約希望時期
                  → [予約確認] → 確認_予約確認内容
            → [耳鼻科] → 確認_予約の有無
              → [有] → 確認_用件
              → [無] → 確認_予約調整日の承認
                → [肯定] → 予約_予約希望日 → 確認_症状 → 確認_他の診療科について
                → [否定] → 診療科_対象外
            → [対象外] → 診療科_対象外 → 確認_他の診療科について
          → [否定] → 復唱確認否定 → 再聴取
        → 確認_他の診療科について → openAI_確認_他の診療科について
          → [有] → 対象外のみ判定_2 → 移動_診療科②聴取_2へ or 移動_診療科②聴取へ
          → [無] → 対象外のみ判定 → 終話_代表案内②/移動_個人情報聴取へ
      → 移動_個人情報聴取へ → 移動_RAG検索 → 終話ガイダンス分岐
        → [携帯] → 終話①アナウンス → completeflag(smsFlag=1, status=1) → Disconnect
        → [固定] → 終話②アナウンス → completeflag(smsFlag=1, status=1) → Disconnect
```

## 6. まとめ

### 6.1 全体構成
長野県立木曽病院の診療予約シナリオは以下の5つのフローで構成される:
1. **メインフロー（診療）**: 冒頭 → 非通知判定 → 診療科①聴取 → 用件処理 → 他の診療科希望判定 → 個人情報聴取 → RAG検索 → 終話
2. **診療科②聴取_1**: 診療科①が対象の場合の2つ目聴取
3. **診療科②聴取_2_1**: 診療科①が対象外/耳鼻科の場合の2つ目聴取
4. **個人情報聴取**: 氏名 → 生年月日 → 電話番号
5. **RAG検索**: 連絡事項・質問の聴取とFAQ検索

### 6.2 特徴的な設計
- **耳鼻科特別ルート**: 耳鼻咽喉科のみ、予約の有無確認 → 1週間以降の予約調整承認 → 予約希望日 → 症状確認という独自フローを持つ
- **2つの診療科聴取**: 最大2つの診療科について問い合わせ可能。途中切断でもSMSを送信（smsFlag=3）
- **携帯/固定分岐**: 着信電話番号に基づき、携帯の場合はSMS送信＋終話①、固定の場合は終話②
- **キャンセル → 次回予約**: キャンセルフローで次回希望ありの場合は予約変更フローの残薬確認へ遷移
- **Property.mdによるプロンプト上書き**: サブフローのTTSプロンプトはJump to Flowのproperties経由で外部注入される設計

### 6.3 PDFとの整合性
PDFの設計書と.bivr / Property.mdはおおむね高い整合性を持つ。主な差異は:
1. 冒頭アナウンスの「初めて受診される方」部分（PDF内でもPage間で差異あり）
2. 残薬確認の文言（4択版 vs 3択版）
3. 電話番号聴取の音声/DTMF優先度の表現差
4. 診察券番号聴取が.bivrには存在するがPDFには記載なし
5. 紹介状有無、予約不要終話がProperty.mdにのみ存在

## 3. メインフロー（診療-Demo）の詳細フロー構造

### 3.1 開始～共通聴取
```
冒頭待ち時間 (2秒wait)
  → コンテキスト設定 (saveContextModel2DB)
  → 営業時間チェック (acceptance_times)
    ├─ true → 冒頭 (TTS)
    │           → 確認_氏名 (JumpToFlow: 氏名サブフロー)
    │             → 確認_生年月日 (JumpToFlow: 生年月日サブフロー)
    │               → 確認_電話番号 (JumpToFlow: 電話番号サブフロー)
    │                 → 確認_用件 (JumpToFlow: 用件サブフロー)
    └─ false/TIMEOUT/ERROR → Flag_時間外 (status=6) → 終話_時間外 (Disconnect)
```

### 3.2 用件分岐後のフロー

#### 予約（初診）ルート
```
確認_用件 = 初診予約
  → 確認_紹介状有無 (TTS) → 入力 → openAI_確認_紹介状有無
    ├─ 有 → スクリプト_現在時刻取得 → openAI_時間判定
    │       ├─ Early (7:30～8:29) → completeflag_SMSflag_7 (status=6) → 終話_かけ直し案内
    │       └─ Normal (8:30～16:10) → 転送_地域連携室 → 転送 (Attended)
    │             ├─ 成功 → completeflag_SMSflag_4 (status=3) → 転送_成功
    │             └─ 失敗 → completeflag_SMSflag_5 (status=23) → 終話_転送失敗
    └─ 無 → 確認_当日予約判断
```

#### 予約（初診 + 紹介状なし）ルート
```
確認_当日予約判断 (TTS) → 入力 → openAI_確認_当日予約判断
  ├─ YES → スクリプト_現在時刻取得2 → openAI_時間判定2
  │         ├─ Early (7:30～10:29) → 確認_診療科 → ... → 終話①
  │         └─ Normal (10:30～) → completeflag_SMSflag_3 (smsFlag=1, status=1) → 終話_②アナウンス
  └─ NO → 確認_診療科 → ...
```

#### 予約（再診）ルート
```
確認_用件 = 再診予約
  → 確認_診察券番号 → 入力 → openAI_確認_診察券番号
    → 復唱_診察券番号 → openAI_復唱_診察券番号
      → YES → 用件分岐
                └─ 再診予約 → 確認_当日予約判断 → ...
```

#### 変更/キャンセル ルート
```
確認_用件 = 予約変更 / 予約キャンセル
  → 確認_診察券番号
    → 用件分岐 → 確認_診療科 → 診療科分岐
      ├─ 健診センター → completeflag_SMSflag_6 (status=1) → 終話_健診センター
      ├─ 緩和ケア内科 → completeflag_SMSflag_8 (status=2) → 終話_緩和ケア
      ├─ デンタルクリニック → completeflag_SMSflag_9 → 終話_デンタルクリニック
      └─ その他 → 用件分岐_2
                   ├─ 初診/再診 → completeflag_SMSflag_1 (status=6) → 終話_①アナウンス
                   └─ 変更/キャンセル → 確認_予約日 → ...
                       → 用件分岐_3
                         ├─ 予約変更 → completeflag_SMSflag_1 → 終話_①アナウンス
                         └─ 予約キャンセル → completeflag_SMSflag_2 (smsFlag=2, status=1) → 終話_③アナウンス
```

#### 予約確認ルート
```
確認_用件 = 予約確認
  → 確認_診察券番号
    → 用件分岐 → completeflag_SMSflag_1 (status=6) → 終話_①アナウンス
```

## 6. ギャップ分析サマリー

### 一致している主要項目
- フロー全体構造（冒頭→氏名→生年月日→電話番号→用件→分岐）は修正後PDFフローと一致
- 4つの用件分岐（予約/変更/キャンセル/確認）はすべて実装済み
- 診療科特定ルール（緩和ケア→代表案内、健診→専用番号、デンタル→専用番号、登録なし→AI受付）すべて一致
- 紹介状有無の確認→時間帯判定→地域連携室転送/かけ直し案内のフロー一致
- 当日予約判断→10:30前後の分岐一致
- SMS送信フラグの設定（予約=smsFlag0、キャンセル=smsFlag2）
- リトライ回数3回、失敗時ガイダンス一致
- 非通知入電→電話番号聴取（着信番号分岐モジュールで実装）

### 確認が必要な差異

| # | 差異内容 | 詳細 |
|---|---------|------|
| 1 | **生年月日prompt文言の差異** | PDF: 「プッシュボタンでも操作できます」、bivr: 「生年月日は1990年5月1日です。のようにおっしゃってください」。例示方式が異なる。ただしbivrの生年月日はDTMF対応モジュール（DOB Re-confirmation）で実装されているため、プッシュボタン操作自体は可能。 |
| 2 | **受診歴prompt文言の差異** | PDF: 「当院でのご受診は初めてですか？」のみ、bivr: 応答ガイド「はい、そうです。」または「いいえ、違います。」追加。bivrの方が丁寧。 |
| 3 | **completeflag_SMSflag_1のstatus値** | completeflag_SMSflag_1は status=6（時間外）で設定されている。これは終話_①アナウンス（予約/変更/確認の通常終了）へ接続されるが、PDF上は「未処理=1」が期待値。ただし、このモジュールは複数の用件から到達するため、文脈によっては正しい可能性あり。 |
| 4 | **時間外アナウンス文言** | PDF設計書に詳細な時間外アナウンス文言があるが、bivrでは`Flag_時間外 → 終話_時間外 (Disconnect)`と直接切断。時間外アナウンスはacceptance_times APIまたは別設定で処理されている可能性あり。 |
| 5 | **当日受入拒否（初診10:30まで）** | PDFでは「初診10:30までは設定」とあり、bivrではopenAI_時間判定2のEarly/Normal分岐で実装。ただし再診の場合は当日予約判断を通らない分岐（用件分岐で直接確認_当日予約判断→確認_診療科）で処理。 |
| 6 | **用件サブフロー内の受診歴確認** | 修正後PDFフローでは紹介状確認が用件より後（予約選択後）に来るが、用件サブフロー内に受診歴確認（初診/再診判定）が含まれている。結果はメインフローの確認_用件の戻り値として使われ、初診→紹介状有無、再診→診察券番号と分岐。設計通り。 |

## 7. 全体評価

bivrファイルはPDF設計書（修正後フロー図）の構造を正確に再現している。主要な聴取項目、分岐ロジック、診療科特定ルール、転送処理、SMS設定のすべてが実装されている。差異として確認された項目は主に発話文言の微調整レベルであり、フローの構造的な欠落はない。

特筆すべき実装特徴:
- OpenAI連携による音声認識結果の意図分類（用件、受診歴、紹介状有無、当日予約判断、診療科、診察券番号、予約日）
- DOB Re-confirmationカスタムモジュールによる生年月日の和暦/西暦自動判定
- 電話番号の着信番号自動判別（非通知/固定/携帯/海外）
- プロンプトインジェクション対策が各OpenAIプロンプトに実装済み
- 時間帯判定の2段階分岐（紹介状あり用: 8:30境界、紹介状なし+当日用: 10:30境界）

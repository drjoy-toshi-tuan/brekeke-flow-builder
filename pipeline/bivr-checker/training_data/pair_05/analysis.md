### 3.2 メインフロー詳細パス

```
冒頭待ち時間(2秒) → コンテキスト設定 → 冒頭アナウンス → 確認_用件 → 入力 → openAI判定
  ├─ 予約 → 確認_通院歴
  │    ├─ 有(再診) → 確認_診療科 → 復唱 → 対象判定 → 確認_予約希望日 → 確認_理由 → 個人情報聴取(再診) → RAG検索 → 終話①
  │    └─ 無(新規) → 選定療養費案内 → 確認_診療科 → 復唱 → 対象判定 → 確認_予約希望日 → 確認_理由 → 個人情報聴取_2(新規) → RAG検索 → 終話①
  ├─ 変更 → 確認_診療科_2(#data#=変更) → 復唱 → 対象判定 → 確認_追加診療科2 → 復唱 → 対象判定
  │    → 診療科対象チェック → 確認_現在の予約日 → 用件分岐(変更→確認_予約希望日) → 確認_理由 → 個人情報聴取 → RAG検索 → 終話②
  ├─ キャンセル → 確認_診療科_2(#data#=キャンセル) → 復唱 → 対象判定 → 確認_追加診療科2 → 復唱 → 対象判定
  │    → 診療科対象チェック → 確認_現在の予約日 → 用件分岐(キャンセル→確認_理由) → 個人情報聴取 → RAG検索 → 終話③
  └─ その他問い合わせ → 確認_問い合わせ内容 → 個人情報聴取 → RAG検索 → 終話④
```

### 3.3 個人情報聴取フロー

**個人情報聴取(再診用)**: 診察券番号 → 復唱 → 氏名 → 生年月日 → 電話番号分岐(携帯/固定/非通知)
**個人情報聴取_2(新規用)**: 氏名 → 生年月日 → 電話番号分岐(携帯/固定/非通知)

### 3.4 非通知対応
- 冒頭で非通知分岐あり → 終話_非通知(Flag status=0) → Disconnect

## 9. フロー分岐ロジック詳細

### 9.1 用件分岐 (openAI_確認_用件)
- 予約 → 確認_通院歴
- 変更 → 確認_診療科_2(#data#=変更)
- キャンセル → 確認_診療科_2(#data#=キャンセル)
- その他問い合わせ → 確認_問い合わせ内容

### 9.2 通院歴分岐 (openAI_確認_通院歴)
- 有(再診) → 確認_診療科(予約時)
- 無(新規) → 選定療養費案内 → 確認_診療科(予約時)

### 9.3 個人情報聴取分岐 (通院歴に基づく)
- 通院歴=無(新規) → 個人情報聴取_2(診察券番号なし)
- 通院歴=有(再診) → 個人情報聴取(診察券番号あり)

### 9.4 診療科対象判定
- 対象 → 次の聴取項目へ進む
- 対象外 → 対象外案内(代表電話案内)
- 3段階の判定: openAI_診療科対象判定_1(予約1科目), _2(変更/キャンセル追加科), _3(変更/キャンセル追加科2)

### 9.5 用件分岐(変更/キャンセル後の現在の予約日聴取後)
- 変更 → 確認_予約希望日 → 確認_理由
- キャンセル → 確認_理由(予約希望日はスキップ)

### 9.6 終話分岐
- 予約 → 終話①(SMS付き)
- 変更 → 終話②
- キャンセル → 終話③
- その他問い合わせ → 終話④

| 4 | 診療科ヒアリングシートの×対象外リスト | openAI判定プロンプト内に含まれている(全文は長大) |

## 11. 特徴的な実装パターン

1. **openAI聴取不可モジュール**: リトライ上限到達時、OpenAIに「聴取不可」と出力させるプロンプトインジェクション対策付き専用モジュール(全11個)
2. **診療科対象判定の3段階**: 予約時の1科目、変更/キャンセル時の主科目、変更/キャンセル時の追加科目で別々の判定モジュール
3. **Re-confirmation node data**: 復唱に`#data#`プレースホルダを使用し、前段のopenAI結果を動的に読み上げ
4. **Jump to Flow**: メインフロー→個人情報聴取→RAG検索の3段階のフロー分離、propertiesでプロンプトを注入
5. **ContextMatchRouter**: 通院歴や用件によるフロー分岐をコンテキスト値で制御
6. **DTMF/STTハイブリッド入力**: 生年月日・診察券番号・電話番号はダイヤルプッシュと音声の両対応
7. **profile_words**: RAG検索フローの入力モジュールに膨大な音声認識辞書(健診用語・診療科名の読み仮名バリエーション数千件)
8. **非通知対応**: メインフロー開始前に非通知チェック → 即終話
9. **SMS Flag**: 用件ごとに異なるsmsFlag(1=予約&変更, 2=キャンセル, 3=その他)でSMS文面を制御

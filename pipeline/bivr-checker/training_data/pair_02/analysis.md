### メインフロー（診療-Demo）の詳細構造

#### エントリポイント
```
冒頭待ち時間(wait 2000ms)
  → コンテキスト設定(saveContextModel2DB)
    → 冒頭(TTS)
      → 非通知分岐(incoming-classifier)
        → [非通知] → 終話_非通知 → completeflag(status=0) → Disconnect
        → [固定/海外/携帯/*] → 確認_種別
```

#### 種別聴取フロー
```
確認_種別(TTS prompt=Property.mdから注入)
  → 入力_確認_種別(DTMF+AmiVoice STT Input)
    → openAI_確認_種別(OpenAI分類: 診察/検査/予防接種/送迎/NO_RESULT)
      → [診察/検査/予防接種/送迎] → 復唱_種別(Re-confirmation)
        → 入力_復唱_種別(DTMF+STT)
          → openAI_復唱_種別(OpenAI判定: 肯定/否定/NO_RESULT)
            → [肯定] → 確認_用件①
            → [否定] → 確認_種別_否定後再聴取
      → [TIMEOUT/ERROR/NO_RESULT] → リトライ_確認_種別(retry_count=2)
```

#### 用件聴取フロー
```
確認_用件①(TTS Re-confirmation prompt=Property.mdから注入)
  → 入力_確認_用件①(DTMF+STT Input)
    → openAI_確認_用件①(OpenAI分類: 予約/変更/キャンセル/確認/NO_RESULT)
      → [予約/変更/キャンセル/確認] → 復唱_用件①(Re-confirmation)
        → [肯定] → 種別分岐
        → [否定] → 確認_用件①_否定後再聴取
```

#### 種別×用件ルーティング
```
種別分岐(ContextMatchRouter: openAI_確認_種別の値で分岐)
  → [診察(1)]   → 用件分岐_1 → [予約] → 移動_診察予約
                              → [変更] → 移動_診察変更
                              → [キャンセル] → 移動_診察キャンセル
                              → [確認] → 移動_RAG検索
  → [検査(2)]   → 用件分岐_2 → [予約] → 予約分岐(検査予約は代表案内)
  → [予防接種(3)] → 用件分岐_2              → [変更/キャンセル] → 検査予防接種分岐
                              → [確認] → 移動_RAG検索
  → [送迎(4)]   → 用件分岐_3 → [予約] → 移動_送迎予約
                              → [変更] → 移動_送迎変更
                              → [キャンセル] → 移動_送迎キャンセル
                              → [確認] → 移動_RAG検索
```

#### 検査の予約は代表案内へ
```
予約分岐(ContextMatchRouter)
  → [検査×予約(1)] → 終話_代表案内
  → [予防接種×予約(2)] → 移動_予防接種予約
```

#### 各サブフローはJump to Flowで遷移、戻り先は個人情報聴取
全サブフロー → 移動_個人情報聴取 → 終話ガイダンス分岐

#### 終話ガイダンス分岐
```
終話ガイダンス分岐(ContextMatchRouter: 用件で分岐)
  → [予約(1)]    → 終話_予約 → completeflag(smsFlag=1, status=1) → Disconnect
  → [変更(2)]    → 終話_変更＆確認 → completeflag(smsFlag=1, status=1) → Disconnect
  → [確認(3)]    → 終話_変更＆確認 → 同上
  → [キャンセル(4)] → 終話_キャンセル → completeflag(smsFlag=2, status=1) → Disconnect
```

### 4.4 重要な設計判断（PDFの曖昧な記述からエキスパートが判断した箇所）

1. **検査の「予約」のみ代表案内**: PDFでは「この電話番号では検査に関するご予約を受け付けておりません」と記載。.bivrでは予約分岐(ContextMatchRouter)で検査×予約の組合せのみ代表案内に振り分け、予防接種×予約は受付する設計。

2. **確認→RAG検索**: PDFでは「確認内容を簡潔にお話しください」としか記載されないが、.bivrではRAGサブフロー（FAQ検索+OpenAI要約）として実装。

3. **種別→用件の聴取順**: 初期版(2/2-2/26)のPDFフロー図に基づき種別先行。4/2修正で逆転指示があるが.bivrは未反映。

4. **リトライ失敗時の処理**: PDFでは「○○については後ほど担当者より確認をさせていただきます」と記載。.bivrではopenAI_聴取不可モジュールで「聴取不可」を固定出力後、次ステップへ進行する設計。

5. **氏名例文のカスタマイズ**: PDFでは「名前は○○です」だが、.bivrでは「名前はイルマハナコです」と施設名に合わせた例を使用。

## 4. directorの判断ログ

| # | 判断箇所 | 選択肢 | 採用した判断 | 根拠 |
|---|---|---|---|---|
| 1 | 用件別分岐の実装方式 | Script vs ContextMatchRouter | **ContextMatchRouter** | classification context の単一値で分岐するため、Script（複数 context の組み合わせ判定）は不要。ContextMatchRouter が標準的かつシンプル |
| 2 | 当日確認の分岐パターン | Pattern A（直後分岐） vs Pattern B（後段分岐） | **Pattern A** | はい/いいえの2択で OpenAI 出力から直接分岐可能。echo_back なし、後段処理不要 |
| 3 | 用件の分岐パターン | Pattern A vs Pattern B | **Pattern A** | 変更/キャンセル/確認/新規予約の4択で OpenAI 出力から直接分岐可能。ただし save_to: classification に保存後、後段の用件別分岐で再利用 |
| 4 | 予約日の output_format | datetime vs text | **text** | 「わからない」も許容するため datetime 形式（YYYY-MM-DD HH:MM）では受け付けられない。自由テキストとして保存 |
| 5 | 診療科の no_result_default | 設定なし vs "登録なし" | **"登録なし"** | モジュール選定ガイド 2.5 準拠。未設定だと Dr.JOY 側で「削除レコード」に分類される事故リスクあり |
| 6 | END_受付完了 / END_受付完了_SMS無し の scenario_flow 配置 | メインフローに配置 vs サブフロー内 | **サブフロー内（scenario_flow に配置しない）** | 電話番号聴取サブフローが self_contained のため、正常終話はサブフロー内で完結。メインフローの scenario_flow には含めない |
